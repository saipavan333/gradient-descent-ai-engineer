Gradient descent: how models actually learn

⏱ 28 min Lesson 1.8 ⬇ Runnable notebook
Why this existsThis is where everything becomes 'learning.' Gradient descent is the algorithm that tunes a model's parameters to reduce error — the loop inside every trained model from linear regression to GPT.
By the end, you can
  • State the update rule and the role of the learning rate
  • Explain why stepping opposite the gradient reduces loss
  • Compare batch, stochastic, and mini-batch gradient descent
  • Diagnose a learning rate that is too large or too small

Concept Learning, at last: make the error as small as possible

This is the lesson where everything so far becomes "learning." Training a model means choosing its numbers (its parameters) so its predictions are as close to the truth as possible. We measure "how wrong it is right now" with a loss function $L$ — a single number that is large when the model is bad and small when it's good. (A common loss is the average squared difference between predictions and the true answers.) So learning is just making $L$ small — what mathematicians call minimizing it.

Picture $L$ as a landscape: every possible setting of the parameters is a spot on the ground, and the height there is the loss. We want the lowest valley. We can't see the whole landscape (it has millions of dimensions), but wherever we're standing we can compute the gradient $\nabla L$ (Lesson 1.6), which tells us which way is uphill. So we do the obvious thing: take a small step downhill, then look again, then step again.

Concept The update rule

This is the whole algorithm — gradient descent — in one line. Over and over, replace each parameter with itself minus a small step in the downhill ($-$gradient) direction:

The number $\eta$ (the Greek letter "eta," rhymes with "beta") is the learning rate — how big a step to take. Notice the division of labor: the gradient says which direction to go; the learning rate says how far. Run this update repeatedly and the parameters slide down the landscape until the gradient is nearly zero, which (from Lesson 1.6) means you've reached a flat spot — ideally the bottom.

Concept The learning rate is the dial that makes or breaks training

That innocent $\eta$ is the most important setting you'll tune. Too small and training crawls, taking forever. Too large and the steps overshoot the valley, bouncing back and forth or even flying off to infinity. Just right and the loss glides smoothly down. (A setting like the learning rate, which you choose rather than learn, is called a hyperparameter.)

Concept How much data per step? Batch, stochastic, mini-batch

Each step needs a gradient of the loss — but computed over which data? Three choices:

  • Batch gradient descent uses all the data for every single step: very accurate gradients, but slow and memory-hungry on big datasets.
  • Stochastic gradient descent (SGD) uses just one random example per step: fast but noisy — the path jiggles around. ("Stochastic" simply means "involving randomness.")
  • Mini-batch uses a small handful (say 32–256 examples) per step: the practical sweet spot, and what essentially everyone uses.
!
What can go wrong

The loss landscape isn't always a single clean bowl. Gradient descent can get stuck in a shallow local minimum (a small dip that isn't the deepest valley), stall on a flat plateau, or — with too high a learning rate — diverge (fly off). In practice the noise of mini-batch SGD actually helps jostle the model out of bad spots, and the upgrades in the deep dive (momentum, Adam) smooth the ride. The landscapes of big neural nets turn out to be surprisingly friendly to this simple method.

Worked example Gradient descent from scratch, then the PyTorch way

First, minimize the simple bowl $f(w) = (w-3)^2 + 1$ by hand-coded gradient descent. Its derivative (chain rule on the square) is $f'(w) = 2(w-3)$, and the true lowest point is at $w = 3$.

w = 0.0          # start far from the answer
lr = 0.1         # the learning rate (eta)
for step in range(25):
    grad = 2 * (w - 3)       # the slope f'(w) at the current w
    w = w - lr * grad        # the update rule: step downhill
print("learned w:", round(w, 3))   # close to 3
Outputlearned w: 2.996

Starting at $0$, the parameter slid to $\approx 3$ — gradient descent found the bottom without being told where it was. Now the real thing: fitting a line $y \approx w\,x$ to data where the true slope is $2$, using PyTorch's optimizer loop — the exact pattern you'll write for every model in this course:

import torch

x = torch.tensor([1., 2., 3., 4.])
y = torch.tensor([2., 4., 6., 8.])       # the true relationship is y = 2x
w = torch.tensor(0.0, requires_grad=True)
optimizer = torch.optim.SGD([w], lr=0.01)

for epoch in range(200):                 # one "epoch" = one pass over the data
    pred = w * x
    loss = ((pred - y) ** 2).mean()      # mean squared error
    optimizer.zero_grad()                # clear old gradients (see note below)
    loss.backward()                      # backprop: compute w.grad
    optimizer.step()                     # apply the update rule to w

print("learned w:", round(w.item(), 3), " final loss:", round(loss.item(), 6))
Outputlearned w: 2.0 final loss: 0.0

The model discovered $w = 2$ on its own by repeatedly stepping downhill. Memorize that four-line core — zero_grad(), compute loss, backward(), step() — because it is the heartbeat of training every neural network in Track 4 and beyond. (Why zero_grad()? PyTorch adds up gradients each time you call backward(), so you must reset them to zero every step or last step's gradient would contaminate this one.)

★ Key points to remember
  • Learning = minimizing a loss function $L$ over the model's parameters.
  • Gradient descent repeats: $w \leftarrow w - \eta\,\nabla L$ — step opposite the gradient.
  • The learning rate $\eta$ sets step size: too small crawls, too big overshoots/diverges.
  • Mini-batch SGD (a small batch per step) is the practical default — fast and stable enough.
  • The PyTorch loop — zero_grad() → loss.backward() → optimizer.step() — is the heartbeat of training.

Practice Build the reasoning yourself

1Minimizing $f(w) = (w-3)^2$ with $f'(w) = 2(w-3)$, start at $w = 0$ with learning rate $\eta = 0.1$. Compute the first two updates by hand.
Show solution & reasoning

Step 1: gradient $= 2(0 - 3) = -6$. Update: $w \leftarrow 0 - 0.1(-6) = 0 + 0.6 = 0.6$.

Step 2: gradient $= 2(0.6 - 3) = -4.8$. Update: $w \leftarrow 0.6 - 0.1(-4.8) = 0.6 + 0.48 = 1.08$.

Notice the parameter is climbing toward $3$ and the gradient is shrinking ($-6 \to -4.8$) as we approach the bottom — so the steps get smaller automatically. That self-slowing is why gradient descent settles into a minimum instead of overshooting it (as long as $\eta$ isn't too large).

2Two engineers train the same model. One uses $\eta = 0.001$, the other $\eta = 10$. Describe what each likely sees in the loss curve, and what each should do.
Show solution & reasoning

$\eta = 0.001$ (too small): the loss decreases, but agonizingly slowly — a nearly flat, gently sloping curve. After many epochs it's still far from done. Fix: raise the learning rate (try 10×) or use an adaptive method like Adam.

$\eta = 10$ (too large): the loss jumps around or shoots up to NaN (not-a-number) within a few steps. The updates overshoot the valley repeatedly. Fix: lower the learning rate sharply (try $0.01$ or less). The reasoning: the learning rate must match the landscape's steepness — and the loss curve is your diagnostic. Slow-and-smooth means raise it; jagged-and-exploding means lower it.

3Gradient descent reaches a point where the gradient is (nearly) zero, but it's a shallow dip, not the deepest valley. What is this called, and name one reason mini-batch SGD often escapes it where full-batch might not.
Show solution & reasoning

This is a local minimum — a dip where every small move increases the loss, so the gradient is zero, but a deeper (global) minimum exists elsewhere.

Mini-batch SGD often escapes because its gradients are noisy: each step uses a different random subset of data, so the estimated gradient jitters. That noise can bump the parameters out of a shallow basin and over a small ridge into a better one — whereas full-batch descent, with its exact gradient, would sit still at the bottom of the shallow dip. It's a rare case where less precise is more useful: the randomness acts like a gentle shake. (Modern view: in huge networks, truly bad local minima are rare; the bigger obstacles are saddle points and plateaus, which noise also helps with.)

★ Deep dive: momentum, Adam, and learning-rate schedules▸

Plain gradient descent works, but a few upgrades make it dramatically faster and more robust — and you'll see all of them as one-word options in real training code.

Momentum. Instead of stepping by the current gradient alone, keep a running average of recent gradients — a "velocity" — and step by that. Like a ball rolling downhill, it builds speed in consistent directions and damps out the zig-zag across narrow valleys. One setting (often $0.9$) controls how much history to keep.

Adam. The default optimizer for most deep learning. It keeps per-parameter running averages of both the gradient (like momentum) and the squared gradient, using the latter to give each parameter its own effective learning rate — bigger steps for parameters with small, steady gradients; smaller steps for noisy ones. This adaptivity is why Adam "just works" across many problems with little tuning. You'll write torch.optim.Adam(model.parameters(), lr=1e-3) constantly.

Learning-rate schedules. The best learning rate isn't constant: you often want larger steps early (to make fast progress) and smaller steps late (to settle precisely into the minimum). Schedules shrink $\eta$ over time — step decay, cosine annealing, or a "warmup" that ramps $\eta$ up for the first few hundred steps before decaying it (essential for training transformers).

Finally, the landscape itself: if the loss is convex (a single bowl, like linear and logistic regression), gradient descent is guaranteed to find the global minimum. Neural networks are non-convex (many hills and valleys), so there's no guarantee — yet in practice, with these tricks and enough scale, gradient descent reliably finds solutions that work well. Understanding exactly why that works is still a research frontier, but the simple algorithm you just learned is the one doing it.
