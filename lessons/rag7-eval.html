Evaluating RAG

⏱ 26 min Lesson 8.7 ▶ Open in Colab
Why this existsRAG can fail at retrieval or generation — and you cannot fix what you cannot measure. Recall@k, MRR, and the faithfulness triad let you evaluate each stage and improve the system with eval-driven development.
By the end, you can
  • Evaluate retrieval and generation separately
  • Compute recall@k, precision@k, and MRR
  • Explain the RAG triad (faithfulness, relevance)
  • Build an eval set and use LLM-as-judge

Concept RAG can fail in two places — so measure both

A RAG system has two stages, and it can fail at either: retrieval might not find the relevant chunks, or generation might produce a bad answer even from good chunks. If you only look at the final answer, you can't tell which broke — so the cardinal rule of RAG evaluation is to measure retrieval and generation separately. Most RAG failures are actually retrieval failures (Lesson 8.1), and you'll never know unless you check retrieval on its own.

Evaluating retrieval is a ranking problem (you have ground-truth relevant chunks for each test query), so it uses the ranking metrics from information retrieval:

The three core metrics: Recall@k — of all the relevant chunks that exist, how many did we get into the top $k$? (Did we find them?) Precision@k — of the top $k$ we returned, how many are relevant? (Are our results clean?) MRR (Mean Reciprocal Rank) — how high did the first relevant chunk rank? (1/rank, so rank 1 = 1.0, rank 2 = 0.5.) For RAG, recall@k matters most — if the answer-bearing chunk isn't retrieved, generation is doomed. (nDCG refines this further by weighting higher ranks more, but recall@k and MRR cover the essentials.)

Concept Evaluating the generated answer: the RAG triad

Good retrieval doesn't guarantee a good answer — the model might ignore the context, misread it, or wander off. To evaluate generation, three questions form the RAG triad:

Faithfulness (or groundedness): is every claim in the answer supported by the retrieved context, or did the model make something up? Answer relevance: does the answer actually address the question (not a grounded-but-off-topic response)? Context relevance: was the retrieved context actually relevant (a check on retrieval quality from the generation side)? These are harder to measure than retrieval metrics because there's no single right answer — so they're often scored by an LLM-as-judge: prompt a strong LLM to rate, say, "is this answer fully supported by this context? yes/no" (a technique covered in Track 12). The triad catches the distinct failure modes: unfaithful (hallucinated), irrelevant (off-topic), or poorly-retrieved (bad context).

Worked example Computing retrieval metrics

Given a ranked retrieval result and the known-relevant chunks for a test query, let's compute the three core metrics directly from their definitions.

Read each number. Recall@3 = 1.00: both relevant chunks (d1 and d7) are in the top 3, so we found all the relevant content — excellent, since generation now has everything it needs. Precision@3 = 0.67: 2 of the top 3 are relevant (d3 is not), so one slot is "wasted" on an irrelevant chunk — minor, but it costs prompt space. MRR = 0.50: the first relevant chunk (d7) appeared at rank 2, giving 1/2; ideally it would rank 1st (MRR 1.0). Together these tell a clear story: retrieval found everything relevant (high recall) but ranked a junk chunk first (MRR could be better). That's exactly the kind of diagnosis reranking (Lesson 8.5) targets — push the relevant chunks to the top. Computing these on a test set of queries (each with labeled relevant chunks) turns "the RAG feels off" into specific, fixable numbers — and crucially isolates whether the problem is retrieval (low recall) or generation (good chunks, bad answer).

★ Key points to remember
  • RAG can fail at retrieval or generation — evaluate them separately to know which.
  • Retrieval (ranking) metrics: Recall@k (found the relevant chunks?), Precision@k (clean results?), MRR (first relevant ranked high?).
  • For RAG, recall@k matters most — a missed answer-bearing chunk dooms generation.
  • Generation: the RAG triad — faithfulness (grounded?), answer relevance (on-topic?), context relevance (good retrieval?).
  • Triad metrics are often scored by LLM-as-judge; build a test set of queries with known relevant chunks.

Practice Build the reasoning yourself

1For a query, the relevant chunks are {A, C} and the system returns [B, A, D, C, E]. Compute Recall@2, Precision@2, and MRR.
Show solution & reasoning

Top-2 = [B, A]. Relevant in top-2: just A → 1 of 2 relevant found. Recall@2 = 1/2 = 0.50. Precision@2 = 1/2 = 0.50 (1 of the 2 returned is relevant). First relevant (A) is at rank 2, so MRR = 1/2 = 0.50.

Interpretation: at k=2 we found only half the relevant chunks (C is at rank 4, below the cutoff) — so if we only feed the model the top 2, it misses C's information. Raising k to 4 would give Recall@4 = 1.0 (both A and C now included), at the cost of a longer prompt. This illustrates the recall-vs-prompt-size tradeoff in choosing k: bigger k catches more relevant chunks (higher recall) but uses more tokens and may add noise. You tune k by measuring recall@k on your eval set against your token budget.

2Your RAG system has high retrieval recall (the right chunks are always retrieved) but users report wrong answers. Which RAG-triad metric would you check, and what does a failure there mean?
Show solution & reasoning

Since retrieval is good (high recall), the problem is in generation, so check faithfulness (and answer relevance). A faithfulness failure means the model is not grounding its answer in the retrieved context — it's ignoring the correct chunks and answering from its own (possibly wrong) parametric memory, or misreading the context, i.e., hallucinating despite having the right information available.

Fixes target generation, not retrieval: strengthen the prompt instruction ("answer ONLY using the context; if it's not there, say you don't know"), make sure the relevant chunk is prominent (reranking, Lesson 8.5, so it's at the top where the model attends most), reduce distracting irrelevant chunks (lower k or higher precision), or use a more capable model. The diagnostic value of separating the metrics is exactly this: high recall + wrong answers points the finger squarely at generation/faithfulness, saving you from fruitlessly tuning retrieval that's already working.

3You want to evaluate your RAG system but have no labeled data. Outline how you'd build an evaluation set, and how an LLM can help.
Show solution & reasoning

Build a set of representative test queries with ground truth. For retrieval, label which chunks are relevant to each query (so you can compute recall@k/MRR); for generation, write or approve reference answers. To bootstrap this without manual labor, use an LLM to generate question–answer pairs from your documents: feed it a chunk and ask "write a question this chunk answers," giving you (query, relevant-chunk) pairs automatically — a synthetic eval set you then spot-check for quality.

For ongoing scoring, use LLM-as-judge: prompt a strong model to rate faithfulness ("is this answer supported by this context? yes/no"), answer relevance, and context relevance — automating the triad without human raters on every example (Track 12 covers LLM-as-judge's pitfalls, like bias and the need to validate it against human judgments). The result is an eval-driven workflow: a test set plus automated metrics that let you measure changes (new chunking, a reranker, a different model) objectively rather than by vibes. This discipline — you ship your eval, not just your model — is the single biggest factor in improving a RAG system reliably, and it's the bridge to Track 12 on evaluation and safety.

★ Deep dive: eval-driven development, and why RAG evaluation is hard▸

The most important meta-lesson is eval-driven development: build the evaluation harness first, then improve the system against it. Without metrics, optimizing RAG is guesswork — you change the chunk size, it "feels better," but you can't tell if you helped or hurt, or whether a fix for one query broke ten others. With a test set and the metrics above, every change becomes measurable: does this reranker raise MRR? does this chunking raise recall@5? does this prompt raise faithfulness? You can run ablations, catch regressions, and make principled tradeoffs. Teams that evaluate systematically improve RAG steadily; teams that don't plateau and ship brittle systems. This is the same discipline as model selection in Track 3 (cross-validation, held-out test sets) applied to a generative pipeline, and it's the central theme of Track 12. "Ship your eval, not your model" is the slogan — your evaluation set is the durable asset that lets you keep improving.

RAG evaluation is genuinely hard in ways worth respecting. There's no single right answer for most questions, so generation metrics are inherently fuzzy — two faithful, relevant answers can be worded completely differently, which is why exact-match scoring fails and LLM-as-judge (despite its own biases — Track 12) has become the practical tool. Relevance labels are subjective: whether a chunk is "relevant" to a query is sometimes a judgment call, so retrieval ground truth has noise. The metrics interact: optimizing recall by raising k can hurt precision and add distracting context that lowers faithfulness, so you can't maximize everything independently. And real usage drifts from your test set — new question types appear, documents change — so evaluation is continuous, not one-time (you mine production logs for new test cases, a connection to monitoring in Track 10). Specialized frameworks (RAGAS, TruLens, and others) implement the triad and retrieval metrics with LLM-judges to make this practical. The honest synthesis: perfect RAG evaluation is impossible, but good-enough evaluation — separating retrieval from generation, tracking recall@k/MRR and the faithfulness/relevance triad on a maintained test set — is entirely achievable and transforms RAG from a craft into an engineering discipline. It's the capstone skill of this track, because a RAG system you can't measure is a RAG system you can't reliably improve — and improvement, guided by evaluation, is what turns a demo into a dependable product.
