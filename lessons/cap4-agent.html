Capstone 4: a tool-using agent

⏱ 120 min Lesson 13.5 ⬇ Runnable notebook
Why this existsBuild a tool-using agent — an LLM that reasons, calls tools, and loops to accomplish multi-step tasks, wrapped in guardrails. Demonstrates both the power of agents and the defense-in-depth discipline they require.
By the end, you can
  • Build the think-act-observe loop with tools
  • Add permission and budget guardrails
  • Reason about long-horizon reliability
  • Apply least privilege

Concept The project: an LLM that acts, not just answers

This capstone builds an agent — an LLM that doesn't just respond but acts: it reasons about a goal, calls tools (a calculator, a search, an API, code execution), observes the results, and loops until the task is done. It's a compelling portfolio project because agents are the frontier of applied AI, and building one demonstrates you understand the act–observe loop, tool use, and — critically — the guardrails that make an acting system safe. Where the RAG capstone gave a model knowledge, this one gives it the ability to do things to accomplish multi-step tasks.

The core is the agent loop: the LLM reasons about the goal (Think), chooses and calls a tool (Act), reads the result (Observe), and repeats — the ReAct pattern from Track 8. Tools are how it acts: you give it a registry of functions it can call (with descriptions), and it emits structured requests to invoke them. But because an agent takes real actions, the essential addition is guardrails: a permission allowlist (it can only call approved tools), a budget (a cap on steps so it can't loop forever), and for high-stakes actions, human approval. These constrain what the agent can do from the outside, so even if it misbehaves or is manipulated, it can't cause serious harm — the defense-in-depth principle of Track 8.

Worked example An agent loop with guardrails

Here's the skeleton: an agent works through a plan of tool calls, but every action passes guardrails first — a permission allowlist and a budget. Watch how a disallowed action is blocked while permitted ones run:

def calculator(expr): return eval(expr)        # a tool
tools = {"calculator": calculator}
ALLOWED = {"calculator"}                        # guardrail: permission allowlist
budget, used = 2, 0

def agent(goal):
    global used
    print(f"goal: {goal}")
    plan = [("calculator", "12*8"), ("delete_files", "/"), ("calculator", "100-58")]
    for tool, arg in plan:                       # think -> act -> observe, with guardrails
        if tool not in ALLOWED:
            print(f"  BLOCKED: '{tool}' not permitted"); continue
        if used >= budget:
            print(f"  BLOCKED: budget {budget} exceeded"); continue
        used += 1
        print(f"  Thought: use {tool}  ->  Action: {tool}({arg})  ->  Observation: {tools[tool](arg)}")
    return "done"

agent("compute some values, safely")
Outputgoal: compute some values, safely Thought: use calculator -> Action: calculator(12*8) -> Observation: 96 BLOCKED: 'delete_files' not permitted Thought: use calculator -> Action: calculator(100-58) -> Observation: 42

The agent works through its plan, and each action passes the guardrails before executing. The two calculator calls are on the allowlist and within budget, so they run — you see the Think→Act→Observe trace producing 96 and 42. But delete_files is blocked by the permission gate — it's not on the allowlist, so no matter what the LLM "decided," the action simply can't execute. That's the essence of safe agent design: the guardrails are independent of the model, so the agent's safety doesn't depend on the agent behaving. In the real project, the "plan" isn't hardcoded — the LLM generates each Thought and Action dynamically based on the goal and the observations so far (real ReAct), you'd give it genuinely useful tools (web search, code execution, an API), and the guardrails would be more elaborate (human approval for anything irreversible, sandboxing, logging every action for audit). But the architecture is exactly this: an LLM looping through think–act–observe, with every action gated by permission and budget checks it cannot bypass. Building this — a real agent with real tools that accomplishes a multi-step task, wrapped in guardrails — demonstrates you understand both the power of agents (the loop and tools) and the discipline they require (defense-in-depth safety), which is precisely the balanced, current understanding that makes an agent project impressive rather than a naive demo.

★ Key points to remember
  • The project: an agent — an LLM that reasons, calls tools, observes results, and loops to accomplish multi-step tasks.
  • The core is the think → act → observe loop (ReAct); tools are functions the model can call.
  • Because agents act, they need guardrails: permission allowlists, budgets (step caps), and human approval for high-stakes actions.
  • Guardrails are independent of the model — safety doesn't depend on the agent behaving (defense in depth).
  • A real build uses genuine tools and dynamic LLM-generated reasoning; the guardrails are what make it safe to demo and deploy.

Practice Build and reason about it

1Plan a useful agent project. What task, what tools, and what guardrails?
Show a strong plan

"I'd pick a task that genuinely needs multi-step tool use — say a research assistant that answers a question by searching the web, reading results, and synthesizing an answer, or a data assistant that answers questions about a dataset by writing and running code. Tools: for the research assistant, a web search tool and a page-reading tool (plus maybe a calculator); for the data assistant, a code-execution tool over the data. Each tool gets a clear description so the LLM knows when to use it. Loop: a ReAct loop — the LLM reasons about what it needs, calls a tool, reads the result, and continues until it can answer, with a step budget so it terminates. Guardrails — essential: a permission allowlist (only the intended tools), a step/budget cap (no runaway loops), sandboxing for the code tool (so it can't touch anything dangerous), treating retrieved/search content as untrusted (prompt-injection awareness), and logging every action for audit. For anything irreversible or sensitive, human approval. Evaluation: test on real multi-step questions, checking it completes tasks correctly and stays within its guardrails."

The plan shows you understand what makes agents useful (multi-step tool use for tasks a single LLM call can't do) and, crucially, what makes them safe (the specific guardrails matched to the tools' risks — sandboxing the code tool, allowlisting tools, capping steps, treating external content as untrusted). The reasoning skill is choosing a task that genuinely benefits from an agent (not one a single call could handle), giving it the right tools, and — the mark of maturity — designing the guardrails deliberately based on what could go wrong with those tools. A naive agent demo shows the loop; an impressive one shows the loop plus the thoughtful safety design, which is exactly the balanced understanding (power and discipline) that Track 8 taught and that interviewers look for. Being able to explain the task choice, the tools, and especially the guardrails and why each is there makes this a rich, credible project.

2Your agent works on simple tasks but fails or gets lost on longer ones. Why, and how would you improve it?
Show reasoning

"This is the compounding-error problem: a long task is a chain of steps, and even a modest per-step failure rate multiplies over many steps into a high chance of overall failure — plus the agent can lose track of the goal as its context fills with intermediate results. Fixes, roughly in order: explicit planning — have the agent draft a plan first and follow it, so it keeps the overall goal in view instead of drifting step to step; decomposition — break the task into smaller, checkable subtasks so each is more reliable; reflection/self-correction — after steps, have the agent check its work against the goal and retry on errors, raising per-step reliability; memory management — summarize or externalize earlier steps so the context doesn't overflow and bury the goal; and sometimes reducing the autonomy needed — a more constrained workflow with fewer free-form steps. I'd also make sure the agent handles tool failures gracefully (retry, or report what it got) rather than derailing."

The reasoning demonstrates you understand why agents are hard — long-horizon reliability is the central challenge, driven by compounding errors and context limits — and that the levers to address it (planning, decomposition, reflection, memory) each attack that problem from a different angle. This is exactly the depth from Track 8, and being able to diagnose 'it fails on long tasks' as a compounding-error/goal-drift issue (not a random bug) and prescribe structured fixes shows real understanding of agent engineering. It's also an honest, mature framing: agents are powerful but long-horizon reliability is genuinely difficult and partly unsolved, so you improve it with these techniques rather than expecting perfection. Being able to discuss this trade-off and the mitigations makes your agent project a credible demonstration of understanding, not just a demo that works on easy cases — and it's precisely the kind of nuanced answer that distinguishes someone who has actually built and struggled with agents from someone who has only read about them."

3Someone says "let's give our agent full access to all our systems so it can do anything the user asks." How do you respond?
Show reasoning

"I'd push back firmly, because that violates the most important principle of safe agent design: least privilege. Giving an agent full access to everything means that any failure — a hallucination, a bug, a prompt-injection attack, a misunderstanding — could cause catastrophic harm, because the agent can do anything. And agents will sometimes fail: they hallucinate, they can be manipulated via prompt injection (malicious instructions in content they read), and errors compound over steps. So 'full access' means the worst case is unbounded. Instead, the agent should get the minimum permissions its task actually requires, with the most dangerous or irreversible actions gated behind human approval, everything logged, and sensitive systems simply out of reach. That way, even a fully hijacked or badly-confused agent can't cause serious damage — the worst case is bounded by what it's able to do, not by whether it behaves.

The deeper point — and why this is such an important thing to get right — is that you cannot make an agent safe by trusting it to behave; you make it safe by constraining what it's able to do. Prompt injection has no clean fix (a determined attacker can often find phrasings that slip past the model's training), so the security model must assume the agent can be compromised and limit the blast radius accordingly. 'Give it full access so it's maximally capable' is exactly the reasoning that produces dangerous systems; 'give it least privilege and gate the dangerous actions' is what makes an acting AI safe to deploy. This connects directly to the guardrails and defense-in-depth of Track 8 and the governance concerns of Track 11 — the responsible-AI insight that powerful, acting systems must be constrained from the outside. Being able to articulate this — why least privilege matters, why you can't rely on the model behaving, and how to bound the worst case — demonstrates exactly the safety-conscious judgment that separates someone who can responsibly build agents from someone who would build a dangerous one. It's the single most important thing to get right when giving an AI the ability to act, and pushing back on 'give it full access' is the correct, mature response."

★ Deep dive: building agents responsibly at the frontier▸

Agents are the most exciting and the most hyped area in AI right now, which makes building one thoughtfully a particularly valuable thing to demonstrate — because it shows you can separate the genuine capability from the hype. The genuine capability is real: an LLM in a loop with tools can accomplish multi-step tasks that a single call cannot — research across sources, operating software, automating workflows — and this composes powerfully with everything else (RAG becomes a tool the agent can call, code execution becomes a tool, other models become tools). The frontier of applied AI is largely about making agents reliable and useful for real work. But the honest reality, which a mature agent project reflects, is that agents are not a solved, plug-and-play technology: they struggle with long-horizon reliability (compounding errors), they're vulnerable to prompt injection with no clean fix, and giving them real actions makes their failures consequential. So building an agent well means holding both truths — the power and the immaturity — and engineering for the latter with guardrails, scoping, and defense in depth. A project that shows a working agent and thoughtful safety design demonstrates exactly the balanced judgment the field most needs.

This capstone also integrates and caps the curriculum's arc. An agent uses LLMs and prompting (Track 6), often RAG as a tool (Track 7, the previous capstone), the agent patterns and guardrails of Track 8, the reliability and monitoring mindset of Track 9, and the safety and defense-in-depth thinking of Track 11 — all in one system that reasons, acts, and stays within safe bounds. It's the clearest demonstration of the theme running through the whole track: that modern AI systems are built by composing capabilities (knowledge, reasoning, tools, action) while engineering carefully for reliability and safety. The disposition to carry forward — and to demonstrate in this project — is the one this curriculum has emphasized throughout: enthusiasm for what these systems can do, paired with sober, disciplined engineering for how they fail. That combination — building capable agents and constraining them responsibly — is exactly what distinguishes an engineer who can be trusted to deploy acting AI in the real world, where the stakes are real. Having built one, with real tools and real guardrails, and being able to reason about both its power and its risks, is compelling evidence that you can work at the frontier responsibly.
