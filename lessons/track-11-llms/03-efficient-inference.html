Efficient inference: KV-cache & quantization

Training gets the headlines, but serving a model to users is where most of the compute is spent. Two techniques make it affordable: the KV-cache (never recompute the past) and quantization (store weights in fewer bits). Both are simple, and you’ll build them.

Track 11 · Lesson 3 Concept + code ~26 min Builds on: Track 5, Lesson 2

A deployed LLM answers millions of requests; shaving its inference cost matters more than almost anything else in production. Generation is autoregressive (Track 5) — one token at a time — which is both the source of its cost and the key to optimizing it. We’ll implement the KV-cache, which turns quadratic work into linear, and quantization, which shrinks the model in memory.

💡 The KV-cache: don’t recompute the past

To generate token \(t\), self-attention needs the keys and values of all previous tokens (Track 5). The naive approach recomputes every past key and value at every single step — to produce \(T\) tokens it does work proportional to \(1+2+\dots+T = O(T^2)\). But the past doesn’t change: token 3’s key is the same whether you’re generating token 4 or token 400. So cache the keys and values as you compute them, and at each new step compute only the one new token’s key and value:

\[ \text{naive: } O(T^2) \text{ key/value computations} \quad\longrightarrow\quad \text{cached: } O(T). \] (3.1) 
K_cache, V_cache = [], []
for t in range(T):
    x = embeddings[t]
    K_cache.append(x @ Wk);  V_cache.append(x @ Wv)   # ONE new key/value per step
    q = x @ Wq
    out.append( attend(q, K_cache, V_cache) )         # reuse all cached past

Verified identical to the naive output — the cache changes the work, not the result. The catch is memory: the cache stores keys and values for every token and layer, so its size grows with context length and becomes the main memory bottleneck for long conversations — which motivates the second technique.

📐 Quantization: fewer bits per weight

Models are trained in 32- or 16-bit floating point, but for inference that precision is overkill. Quantization stores each weight in fewer bits — commonly 8-bit integers (int8), sometimes 4-bit. The mapping is a scale factor: divide by the largest magnitude, round to an integer in \([-127,127]\), and multiply back to recover an approximate float:

\[ q = \operatorname{round}\!\big(W/s\big), \quad s = \frac{\max|W|}{127}, \qquad \hat W = q\cdot s \approx W. \] (3.2)

Our kv_cache.py demonstrates it on a 512×512 matrix: memory drops from 1024 KB (float32) to 256 KB (int8) — 4× smaller — while the average error introduced is ~0.009, tiny relative to the weights. A 4× smaller model fits on 4× less GPU memory (or a much cheaper GPU), and moving fewer bytes also speeds up the memory-bound parts of inference.

Inference is memory-bound. Generating one token multiplies the input by every weight once — very little math reuse — so the bottleneck is usually reading the weights from memory, not the arithmetic. That reframes both techniques: quantization means fewer bytes to read per weight; the KV-cache means not re-reading the past. Combined (e.g. QLoRA fine-tunes a 4-bit base with LoRA adapters from Lesson 2), they’re what let large models run on modest hardware.

🔑 Key results to remember

  • KV-cache: store past keys/values, compute only the new token’s each step — \(O(T^2)\to O(T)\), identical output.
  • Cache memory grows with context length and layers — the main memory cost of long-context generation.
  • Quantization: \(q=\operatorname{round}(W/s)\) with \(s=\max|W|/127\); int8 is 4× smaller than float32 with tiny error.
  • Inference is memory-bound: the win is fewer bytes read (quantization) and not re-reading the past (cache).
  • They compose (e.g. QLoRA) to run and adapt large models on modest hardware.

Answer from memory, then read why.

Q1. The KV-cache speeds up generation by…

  • using lower-precision weights.
    That’s quantization. The cache is about avoiding recomputation, not precision.
  • skipping some tokens entirely.
    No tokens are skipped; every token is still generated. The cache reuses past computation.
  • storing past keys/values and computing only the new token’s each step.
    Correct. Past keys/values don’t change, so caching them turns the per-step cost from “recompute all \(t\)” into “compute 1,” giving \(O(T)\) total instead of \(O(T^2)\).
  • reducing the number of layers.
    The architecture is unchanged; the cache only removes redundant recomputation.

Q2. Does the KV-cache change the model’s output?

  • No — it produces identical results, only faster.
    Correct. The cached keys/values are exactly what the naive method recomputes, so the attention outputs are identical; the demo verifies this. It’s a pure efficiency optimization.
  • Yes — it approximates attention.
    There’s no approximation; the stored values are exact copies of what would be recomputed.
  • Yes — it makes outputs more random.
    Sampling randomness is unrelated to caching; the deterministic computation is unchanged.
  • Only for the first token.
    It’s identical for all tokens; nothing special happens at the first.

Q3. Int8 quantization of a float32 weight matrix reduces its memory by about…

  • no change.
    It changes a lot — 8 bits vs 32 bits per weight.
  • 2×.
    That would be float16; int8 goes further.
  • 8×.
    Too much — that would be 4-bit vs 32-bit. Int8 vs float32 is 8 bits vs 32.
  • 4×.
    Correct. Float32 uses 32 bits/weight, int8 uses 8 — a 4× reduction, as the demo shows (1024 KB → 256 KB), with only tiny rounding error.

Q4. Why is “inference is memory-bound” the key framing for both techniques?

  • Because the GPU has infinite compute.
    Compute isn’t infinite, but for single-token generation the arithmetic is cheap relative to memory movement — that’s the real point.
  • Generating a token reads every weight with little math reuse, so the bottleneck is moving bytes — which quantization (fewer bytes) and the cache (no re-reads) both reduce.
    Correct. With batch-1 generation there’s almost no arithmetic reuse per weight, so time is dominated by reading weights (and the KV-cache) from memory. Both techniques attack exactly that: fewer bytes per weight, and no recomputing/rereading the past.
  • Because models don’t use matrices.
    Models are built on matrix multiplies; the framing is about where their time goes, not whether they use matrices.
  • Because attention is free.
    Attention isn’t free — and the cache exists precisely to cut its repeated cost.

✍️ Practice

1. Run kv_cache.py and change \(T\) to 32, 128, 512. Confirm the naive/cached work ratio grows roughly linearly with \(T\). Why linearly?

Show solution

Naive work is \(\approx T^2/2\) and cached work is \(T\), so their ratio is \(\approx T/2\) — linear in \(T\). Doubling the sequence length doubles the relative advantage of caching. That’s why the cache is not a nice-to-have but essential for long contexts: at \(T=2000\) the naive method would do ~1000× more key/value work. The absolute savings compound across every layer and attention head in a real transformer, turning “impossible” into “interactive.”

2. In the quantization demo, quantize to 4-bit (range \([-7,7]\), scale \(=\max|W|/7\)) instead of int8. What happens to memory and error?

Show solution

Memory drops further — 4 bits is 8× smaller than float32 (twice as small as int8) — but the rounding error roughly doubles, because you have far fewer representable levels (15 vs 255). This is the fundamental quantization trade-off: fewer bits save memory but add noise, and past a point the noise degrades quality. Real 4-bit methods (like the NF4 format in QLoRA) fight this with smarter, non-uniform level placement and per-block scales, recovering most of the quality at 4 bits. Your simple per-tensor scheme is the honest baseline that those methods improve on.

3. Conceptual: why is per-tensor quantization (one scale for the whole matrix) worse than per-channel (one scale per row)?

Show solution

A single scale is set by the largest-magnitude weight in the whole matrix. If a few rows contain big outliers, that one scale becomes coarse for all the other (smaller) weights, wasting precision on them and adding error. Per-channel quantization gives each row its own scale, so rows without outliers keep fine resolution — much less error for the same bit-width. This matters because transformer weights (and especially activations) are known to have outlier channels; handling them well is a central theme in LLM quantization research (e.g. per-channel scales, outlier-aware methods). The general lesson: match the granularity of your scale to the structure of the values you’re compressing.
