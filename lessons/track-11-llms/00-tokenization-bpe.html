Tokenization at scale: Byte-Pair Encoding

Your Track 5 GPT used single characters. Real language models use subword tokens learned by Byte-Pair Encoding — the unglamorous first step that decides how efficiently a model reads, and whether it ever hits an unknown word.

Track 11 · Lesson 0 Concept + code ~24 min Builds on: Track 5

This track bridges the gap between the toy GPT you built and the models that power real systems. It starts where every LLM starts: turning text into tokens. Characters are too fine (long sequences, no word-level meaning); whole words are too coarse (huge vocabularies, and any unseen word becomes “unknown”). Byte-Pair Encoding (BPE) finds the sweet spot — and you’ll build it from scratch.

💡 Concept: learn the vocabulary from data

BPE builds its vocabulary greedily from the training text. Start with every word split into individual characters. Then repeat a simple move: find the most frequent adjacent pair of symbols across the whole corpus and merge it into a single new symbol. Do this a fixed number of times. Frequent chunks — “th”, then “the”, then “theme” — become single tokens, while rare words still decompose into known pieces. The number of merges sets the vocabulary size.

🛠️ From scratch: BPE in ~20 lines

The whole algorithm is two steps in a loop — count pairs, merge the best — from bpe.py: 

def train_bpe(corpus, num_merges):
    words = {" ".join(list(w) + ["_"]): freq for w, freq in count(corpus)}
    merges = []
    for _ in range(num_merges):
        pairs = get_stats(words)              # count every adjacent symbol pair
        best  = max(pairs, key=pairs.get)     # the most frequent pair
        words = merge(words, best)            # fuse it into one symbol everywhere
        merges.append(best)
    return merges                             # the ordered list of merges IS the tokenizer

To tokenize new text you replay the merges in the order they were learned. The learned merge list is the entire tokenizer. Two properties make BPE the industry default:

  • Compression. On our corpus, BPE with 20 merges represents the text in 78% fewer tokens than character-level — shorter sequences mean cheaper attention (which is quadratic in length, Track 5).
  • No unknown words. A never-before-seen word like “themelessness” splits into known subwords (theme, le, s, s, n, e, s, s, _) instead of a useless [UNK] token — so the model can always read something meaningful.
Tokens, not characters, are what the model actually sees. Every length limit (“context window”), every price (“per 1K tokens”), and every quirk (why models miscount letters in a word) traces back to this step. A model literally cannot see inside a token — “strawberry” may be three tokens, so counting its r’s is genuinely hard for it. The tokenizer shapes everything downstream.

🔑 Key results to remember

  • BPE learns a subword vocabulary by greedily merging the most frequent adjacent pair, a fixed number of times.
  • The ordered merge list IS the tokenizer; encoding replays the merges, decoding concatenates.
  • Subwords beat both extremes: shorter than characters, and no unknown-word problem unlike whole words.
  • Vocabulary size = number of merges (plus the base characters) — a key hyperparameter (real LLMs use ~30K–100K+).
  • The model sees tokens, not letters — the source of context limits, token pricing, and letter-counting errors.

Answer from memory, then read why.

Q1. At each step, BPE merges…

  • a random pair of symbols.
    It’s deterministic and greedy, not random — it always picks the most frequent pair.
  • the most frequent adjacent pair of symbols in the corpus.
    Correct. Counting adjacent pairs and merging the top one is the entire learning rule; frequent chunks become tokens first.
  • the longest word in the corpus.
    BPE works on adjacent symbol pairs, not whole words, and frequency (not length) decides.
  • every pair that appears more than once.
    Only one pair — the single most frequent — is merged per step; it repeats for a fixed number of steps.

Q2. Why is subword tokenization preferred over a whole-word vocabulary?

  • Words are always shorter than subwords.
    Backwards — a word token is longer/coarser; subwords are pieces of words.
  • Whole words train faster.
    Training speed isn’t the issue; the problems with whole words are vocabulary size and unknown words.
  • A word vocabulary is huge and still fails on any unseen word (mapping it to [UNK]), while subwords stay compact and always decompose.
    Correct. Subwords bound the vocabulary size and guarantee every input — even novel words — splits into known pieces, so there’s no information-destroying [UNK].
  • Subwords eliminate the need for embeddings.
    Every token, subword or not, still gets an embedding. Subwording changes the token set, not the embedding step.

Q3. Representing text in fewer tokens (via BPE) directly helps a transformer because…

  • self-attention cost grows quadratically with sequence length, so shorter sequences are much cheaper.
    Correct. Attention compares every token to every token (\(O(T^2)\), Track 5), so halving the token count roughly quarters the attention cost — a big efficiency win.
  • fewer tokens mean fewer parameters.
    Parameter count is set by the architecture, not by how many tokens a given text uses.
  • shorter sequences are always more accurate.
    Accuracy isn’t guaranteed by length; the direct benefit is computational cost.
  • it removes the need for positional encodings.
    Positional information is still needed regardless of tokenization.

Q4. A model struggles to count the letters in “strawberry.” The tokenization-based explanation is…

  • the model is too small.
    Even large models show this; size isn’t the core reason.
  • counting is impossible for neural networks.
    Models can count in many settings; this specific failure is about token opacity, not counting in general.
  • letters are stored in a separate memory.
    There’s no separate letter memory; the issue is what the token represents.
  • the word may be a few tokens, and the model can’t “see inside” a token to its individual letters.
    Correct. If “strawberry” is, say, three subword tokens, the letter-level composition is hidden — the model operates on token vectors, not characters, so per-letter reasoning is genuinely hard.

✍️ Practice

1. Run bpe.py with num_merges set to 5, 20, and 100. How do the compression and the vocabulary change?

Show solution

More merges → more (and longer) tokens in the vocabulary → better compression (fewer tokens per text), with diminishing returns. With very few merges you’re close to characters (little compression); with many, common whole words become single tokens. But there’s a trade-off: a bigger vocabulary means a larger embedding table and output layer (more parameters), and extremely long tokens generalize worse to novel text. Real LLMs pick a vocabulary size (often 30K–100K) that balances sequence length against embedding-table size — exactly the knob you’re turning here.

2. Feed your trained tokenizer a word made of common pieces (e.g. “unhappiness”) and a random string (e.g. “xqzptr”). Compare how they tokenize.

Show solution

“unhappiness” tends to split into a few meaningful subwords (like un, happi, ness) if those pieces were frequent in training — efficient and semantically sensible. “xqzptr” has no learned merges, so it falls back to individual characters — longer, but still representable (no [UNK]). This asymmetry is the whole point: BPE spends its vocabulary on frequent patterns and gracefully degrades to characters for anything unusual. It’s also why text in languages or domains underrepresented in training tokenizes into more tokens — and therefore costs more and fills the context faster.

3. Conceptual: GPT models use byte-level BPE (merging bytes, not characters). Why start from bytes?

Show solution

Starting from raw bytes (256 possible values) guarantees that any possible input — any Unicode character, emoji, code, or binary-ish text — is representable with zero unknown tokens, because every byte is in the base vocabulary. Character-level BPE would need every character it might ever see in its base set, which is impractical across all of Unicode. Byte-level BPE gets universal coverage from a tiny 256-symbol base, then learns merges on top. The cost is that rare characters (some emoji, non-Latin scripts) may take several byte-tokens each — again the reason such text consumes more tokens.
