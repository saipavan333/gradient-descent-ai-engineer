Retrieval-augmented generation & in-context learning

A model’s weights are a frozen snapshot of its training data — it can’t know your documents or today’s news. RAG fixes this by giving the model an external memory it searches at question time. It’s the architecture behind most real-world LLM applications.

Track 11 · Lesson 4 Concept + code ~26 min Builds on: Tracks 1, 5, Lesson 0

Everything so far lives in the model’s weights — fixed at training time, expensive to update, and prone to confidently stating wrong “facts.” This final lesson adds the missing piece for real applications: retrieval-augmented generation (RAG), which lets a model pull in relevant text on demand. It rests on two ideas you already own — the dot product (Track 1) and in-context learning — and you’ll build the retrieval core from scratch.

💡 Foundation: in-context learning

A trained transformer exhibits a remarkable ability: in-context learning. Without any weight updates, it can use information placed in its prompt to answer — follow instructions, imitate examples (“few-shot”), or use facts you paste in. The context window acts as a fast, temporary memory that steers the frozen model. RAG is the systematic exploitation of this: if the model will use whatever relevant text is in its prompt, then the job becomes finding the right text to put there.

📐 Retrieval by similarity in embedding space

Represent every document as a vector — an embedding — that captures its meaning, so that similar texts have nearby vectors. Do the same for the user’s question. Then the most relevant documents are simply the ones whose vectors are closest to the question’s, measured by cosine similarity — the normalized dot product from Track 1:

\[ \operatorname{sim}(q, d) = \frac{q \cdot d}{\lVert q\rVert\,\lVert d\rVert}, \qquad \text{retrieve } \arg\max_d \operatorname{sim}(q,d). \] (4.1)

Our rag.py builds this with simple bag-of-words embeddings (unit vectors over the vocabulary, so their dot product is cosine similarity). It retrieves the correct fact for every test query — “capital of France?” → the Paris document, “who created Python?” → the Guido van Rossum document — purely by vector similarity: 

def embed(text):
    v = bag_of_words(text); return v / norm(v)     # unit vector -> dot = cosine

doc_vecs = [embed(d) for d in documents]           # index the knowledge base once
def retrieve(query):
    q = embed(query)
    sims = doc_vecs @ q                             # cosine similarity to every doc
    return documents[argmax(sims)]                  # the closest document
# then: answer = LLM(prompt = query + retrieved_document)
Why RAG matters. It attacks the three biggest weaknesses of a bare LLM at once. Knowledge cutoff: swap in fresh or private documents without retraining. Hallucination: grounding answers in retrieved text (and citing it) makes them checkable. Cost: updating a database is far cheaper than fine-tuning. Real systems replace bag-of-words with learned neural embeddings and a fast approximate-nearest-neighbor index over millions of documents — but the core you just built, cosine similarity in embedding space, is exactly the mechanism.

🔑 Key results to remember

  • In-context learning: a frozen model uses information in its prompt (instructions, examples, facts) with no weight updates.
  • RAG: embed documents and the query, retrieve the closest by cosine similarity (Track 1 dot product), and condition the answer on the retrieved text.
  • Embeddings put meaning in geometry — similar texts get nearby vectors, so relevance becomes nearest-neighbor search.
  • RAG fixes knowledge cutoff, hallucination, and update cost without retraining the model.
  • Production RAG uses learned embeddings + approximate nearest-neighbor indexes over huge corpora — same core mechanism.

Answer from memory, then read why.

Q1. In RAG, how is the “most relevant document” for a query found?

  • By retraining the model on the query.
    No training happens at query time — that’s the point. Retrieval is a fast vector search.
  • By highest cosine similarity between the query’s embedding and each document’s embedding.
    Correct. Both query and documents are vectors; the closest by cosine similarity (normalized dot product) is the most semantically relevant, retrieved instantly without touching model weights.
  • By choosing the longest document.
    Length is irrelevant to relevance; similarity in embedding space is what matters.
  • By asking the model to guess.
    Retrieval is a deterministic vector search, not a model guess — that’s what grounds the answer.

Q2. What is “in-context learning”?

  • Fine-tuning the model on new data.
    That updates weights; in-context learning involves no weight updates at all.
  • Training a fresh model for each query.
    No training occurs; the model is frozen.
  • Storing facts in the model’s parameters.
    That’s parametric knowledge from pretraining; in-context learning uses the prompt, not the weights.
  • A frozen model using information placed in its prompt to answer, with no weight updates.
    Correct. The context window acts as temporary memory — instructions, examples, or retrieved facts in the prompt steer the model’s output without changing a single weight. RAG systematically exploits this.

Q3. Which problem does RAG not directly solve?

  • The model’s core reasoning ability on the retrieved text.
    Correct. RAG supplies relevant facts, but the model still has to reason over them correctly — if it reasons poorly, good retrieval won’t save the answer. RAG improves what the model knows, not how well it thinks.
  • Knowledge cutoff (outdated training data).
    RAG does address this — swap in current documents without retraining.
  • Hallucinated facts.
    RAG helps here by grounding answers in retrieved, citable text.
  • Using private or proprietary documents.
    RAG handles this — put your private docs in the retrievable store.

Q4. Why embed documents as vectors instead of keyword-matching the query text?

  • Vectors are always shorter than text.
    Size isn’t the reason; it’s about matching meaning.
  • Keyword matching is impossible in Python.
    Keyword matching is easy to code; it’s just less effective at capturing meaning.
  • Embeddings capture meaning, so a query matches relevant docs even when they share few exact words (synonyms, paraphrases).
    Correct. Good embeddings place semantically similar texts nearby, so “tallest mountain” can retrieve a document about “highest peak” even with different wording — something literal keyword matching misses.
  • Vectors avoid the need for a question.
    The question is still essential — it’s embedded and used as the search key.

✍️ Practice

1. Run rag.py. Then ask a question whose words don’t appear in any document (e.g. “Where would I climb the world’s loftiest peak?”). Does bag-of-words retrieval still work? Why or why not?

Show solution

It often fails or retrieves the wrong document, because bag-of-words embeddings match on exact shared words — “loftiest peak” shares no tokens with “highest mountain,” so their vectors are nearly orthogonal. This exposes the limitation of simple lexical embeddings and motivates learned neural embeddings, which map synonyms and paraphrases to nearby vectors so meaning-based retrieval works even without word overlap. Swapping the embedding function for a trained sentence encoder (while keeping the exact same cosine-similarity retrieval you built) is precisely how production RAG improves recall — the pipeline is unchanged, only embed() gets smarter.

2. Modify retrieve to return the top-\(k\) documents (say \(k=2\)) instead of the single best. Why do real systems retrieve several?

Show solution

Retrieving top-\(k\) (sort by similarity, take the first \(k\)) hedges against imperfect retrieval: the single best match might miss part of the answer, contain an error, or be one of several relevant passages. Passing a few candidates gives the model more to ground on and lets it synthesize or cross-check — at the cost of using more context-window tokens (Lesson 0’s budget). Choosing \(k\) trades recall against prompt length and noise; too large a \(k\) buries the useful passage among distractors. Tuning \(k\), and re-ranking the retrieved set, are standard RAG engineering knobs.

3. Conceptual: RAG puts retrieved text in the prompt. How does this connect to the KV-cache and context-length limits from earlier lessons?

Show solution

Retrieved documents consume context-window tokens, and every token added grows the KV-cache and the attention cost (Lessons 0 and 3). So RAG lives inside a budget: you can only prepend as much retrieved text as the context window allows, and more retrieved text means more memory and latency per generated token. This ties the whole track together — tokenization sets how much text fits, the KV-cache/quantization determine what long contexts cost, and RAG decides what to spend that context on. Good RAG retrieves little but highly relevant text, because context is a finite, priced resource, not free space.
