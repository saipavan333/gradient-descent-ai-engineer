Ensembles: Random Forests & Gradient Boosting

⏱ 29 min Lesson 3.8 ⬇ Runnable notebook
Why this existsCombining many models is the most reliable accuracy boost in classical ML. Random forests and gradient boosting routinely win on tabular data — the first thing to try on a real spreadsheet.
By the end, you can
  • Explain bagging and the random forest
  • Explain boosting (XGBoost/LightGBM)
  • Contrast variance vs bias reduction
  • Know why ensembles beat single trees

Concept The wisdom of crowds, for models

A single decision tree is flexible but unstable (high variance). The fix is one of the most powerful and reliable ideas in all of applied machine learning: don't trust one model — train many and combine them. This is an ensemble. If each model is decent and they make different mistakes, averaging their predictions cancels the individual errors while keeping the shared signal, like polling a crowd instead of asking one person. Ensembles of trees, in particular, are the models that still win most competitions on tabular data — the everyday spreadsheets of business.

There are two great strategies for building an ensemble, and they attack the two different errors from the bias–variance decomposition. Bagging reduces variance by averaging many independent models. Boosting reduces bias by building models in sequence, each fixing the last one's mistakes.

Concept Bagging and the Random Forest

Bagging (short for "bootstrap aggregating") trains each tree on a different random resample of the data — drawn with replacement, so each tree sees a slightly different world — then averages their votes. Because each tree overfits its own sample in its own way, their errors are largely independent and cancel when averaged, collapsing the variance that plagued the single tree.

The Random Forest adds one more twist: at each split, each tree may only consider a random subset of the features. This stops every tree from leaning on the same one dominant feature, making the trees more different from each other — and more different trees average to an even lower-variance ensemble. Random forests are a fantastic default: strong, robust, and almost impossible to make catastrophically overfit.

Concept Boosting: a team that learns from its mistakes

Boosting takes the opposite approach. Instead of many independent trees, it builds them one after another, where each new tree focuses on the examples the current ensemble gets wrong. The first tree makes rough predictions; the second tree is trained to correct the first's errors (its residuals); the third corrects what's left; and so on. The sum of all these small corrective trees is a single, highly accurate model.

Because each tree explicitly chips away at the remaining error, boosting drives bias down and can reach very high accuracy. The trade-off: it's sequential (can't be trivially parallelized like a forest) and more prone to overfitting if you add too many trees or let them grow too deep, so it needs more careful tuning. The famous libraries XGBoost and LightGBM are highly optimized gradient boosting, and they're the first thing most practitioners reach for on a new tabular problem.

Worked example One tree vs. a forest vs. boosting

We compare a single decision tree against a random forest and a gradient-boosting model on the breast-cancer data. To get a fair, stable read (not a lucky single split), we use 5-fold cross-validation — Lesson 3.12 — and report the mean accuracy and its standard deviation.

from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import cross_val_score
from sklearn.tree import DecisionTreeClassifier
from sklearn.ensemble import RandomForestClassifier, GradientBoostingClassifier

X, y = load_breast_cancer(return_X_y=True)

models = [
    ("single tree",   DecisionTreeClassifier(random_state=0)),
    ("random forest", RandomForestClassifier(n_estimators=300, random_state=0)),
    ("grad boosting", GradientBoostingClassifier(random_state=0)),
]
for name, model in models:
    s = cross_val_score(model, X, y, cv=5)
    print(f"{name:14s} {s.mean():.3f} +/- {s.std():.3f}")
Outputsingle tree 0.917 +/- 0.016 random forest 0.963 +/- 0.021 grad boosting 0.963 +/- 0.021

The single tree averages 91.7%. Both ensembles jump to 96.3% — a 4.6-point gain, which is huge at this end of the scale (it nearly halves the error rate, from ~8.3% to ~3.7%). And we changed nothing about the features or the problem; we just stopped trusting one tree and pooled many. This is why "try a random forest or gradient boosting" is the standard strong baseline on any tabular dataset — it's a large, almost-free accuracy gain over a single model. The cross-validated standard deviations (~0.02) also tell you the gap is real, not noise.

★ Key points to remember
  • An ensemble combines many models; if they err differently, combining cancels error and beats any one.
  • Bagging / Random Forest: many trees on random resamples (+ random feature subsets), averaged → big variance reduction.
  • Boosting (XGBoost, LightGBM): trees built in sequence, each fixing the last's errors → big bias reduction.
  • Tree ensembles are the usual winners on tabular data and a strong default baseline.
  • Forests are robust and parallel; boosting is often more accurate but needs more tuning and can overfit.

Practice Build the reasoning yourself

1You have 100 classifiers that each get 60% accuracy on a balanced two-class problem, and their errors are independent. Intuitively, why can a majority vote of them do far better than 60%?
Show solution & reasoning

Each classifier is a coin biased slightly toward correct (60%). With independent errors, the majority vote is wrong only if most of the 100 happen to err at once — and that's very unlikely when each is more often right than wrong. The law of large numbers pushes the majority toward the truth, so the vote's accuracy climbs well above 60% (toward ~97%+ for 100 independent 60% voters).

The crucial assumption is independence: if all 100 made the same mistakes, the vote would also be 60% — no gain. That's exactly why random forests work so hard (resampling + random features) to make their trees different. Diversity is the fuel of ensembling.

2Your gradient-boosting model gets 99% on training but only 84% on test, while a random forest gets 95% train and 94% test. What's happening, and what would you adjust on the booster?
Show solution & reasoning

The booster is overfitting (big 15-point train–test gap), while the forest is well-balanced. Boosting drives bias down aggressively, so left unchecked it can fit noise — the classic boosting failure mode.

To fix the booster: use fewer trees (or enable early stopping on a validation set), lower the learning rate (each tree contributes less, so the ensemble corrects more gently), reduce tree depth (shallower "weak learners" — depth 3–6 is typical for boosting), and add regularization / subsampling. These all rein in its tendency to memorize. The forest needs little of this because bagging is variance-reducing by nature and hard to overfit — a reminder that the two ensemble families have different default risks.

3A teammate says "ensembles are always better, so never use a single model." Give a balanced response with at least two real costs of ensembles.
Show solution & reasoning

Ensembles usually win on accuracy, but they're not free. (1) Interpretability: a single decision tree can be read like a flowchart and explained to a stakeholder; a forest of 300 trees or a boosted ensemble is a black box (you can get feature importances, but not a simple story). (2) Cost and latency: running hundreds of trees per prediction uses more memory and time, which matters for real-time or edge deployment.

Other considerations: more hyperparameters to tune (especially boosting), and larger model artifacts to store and serve. So the honest rule is: reach for an ensemble when accuracy is the priority, but a single regularized linear model or one shallow tree can be the right call when you need transparency, speed, or simplicity — which is a real engineering trade-off, not a defeat.

★ Deep dive: why "gradient" boosting, OOB error, and the tabular reign▸

Why is it called gradient boosting? Because each new tree is fit to the negative gradient of the loss with respect to the current predictions — which, for squared error, is exactly the residual (the leftover error). So boosting is performing gradient descent (Lesson 1.8), but in "function space": each tree is one downhill step, nudging the ensemble's predictions in the direction that most reduces the loss. The same optimization idea that trains a single model is here orchestrating a whole sequence of them. This view also lets boosting optimize any differentiable loss — squared error, log loss, ranking losses — which is part of why it's so versatile.

Random forests come with a neat free diagnostic: out-of-bag (OOB) error. Since each tree is trained on a bootstrap resample, about a third of the data is left out of each tree's sample. You can evaluate each example using only the trees that didn't see it, giving a validation-quality estimate without a separate held-out set — cross-validation almost for free.

Finally, the open question: why do tree ensembles still beat deep learning on tabular data, when neural nets dominate images and text? Current understanding points to a few things: tabular features are often heterogeneous and non-smooth (a small change in one column can flip the meaning), and trees' axis-aligned splits handle that naturally, whereas neural nets have a bias toward smooth functions. Trees are also far more robust to uninformative features and need almost no preprocessing. Deep learning for tabular data is an active research area, but as of now, "start with gradient boosting" remains excellent advice — and a slightly humbling one for the deep-learning enthusiast heading into Track 4.
