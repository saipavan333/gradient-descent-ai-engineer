Model registry & versioning

⏱ 22 min Lesson 9.4 ⬇ Runnable notebook
Why this existsOnce you train models regularly, you must track which version is live. A model registry versions models and stages them (staging → production → archived), making promotion explicit and rollback instant — like git for models.
By the end, you can
  • Explain a model registry
  • Define versions and stages
  • Describe promotion and rollback
  • Know why old versions are archived, not deleted

Concept Which model is live right now?

Once you're training models regularly (Lesson 9.2), you accumulate many of them — v1, v2, v3, last Tuesday's experiment, the one with the new features. Now urgent questions appear: which model is currently serving production traffic? Which was live last month when that incident happened? If the new model misbehaves, how do we instantly go back to the old one? Answering these by hand — emailing model files around, naming them final_model_v2_REAL.pkl — is how teams lose track of what's actually running. A model registry is the system that solves this: a central, versioned catalog of trained models that tracks every version and which stage each one is in.

A version is simply an immutable, numbered snapshot of a trained model — v1, v2, v3 — so you can refer to an exact model unambiguously. A stage is where a version sits in its journey to production. Typical stages are staging (a candidate being tested), production (the model currently serving live traffic), and archived (a retired former model, kept for history and rollback). The registry's job is to always know, with certainty, which version is in which stage — so "what's live?" has a definite, auditable answer.

Concept Promotion and rollback

The registry turns deployment into two controlled operations. Promotion is the deliberate act of moving a version into production — you don't "accidentally" deploy; you explicitly promote v3 from staging to production after it passes your checks, and the registry records exactly when and by whom. Rollback is the reverse safety net: if the freshly-promoted model turns out to be bad in production (errors spike, accuracy drops), you promote the previous known-good version back to production — instantly returning to a working state. Rollback is only possible because the registry kept the old version (archived, not deleted); this is exactly why you never throw old models away.

This matters because deploying a new model is risky — it might be worse in ways your offline tests didn't catch (recall train/serve skew, Lesson 9.3). The registry makes that risk manageable: every deployment is reversible. The mindset is the same as version control for code (git), where you can always revert to a previous commit — here you can always revert to a previous model. Without it, a bad deployment means scrambling to find and redeploy the old model under pressure; with it, rollback is one promotion away.

Worked example Promotion and instant rollback

Here's a registry in miniature. register adds a new version in staging; promote moves a version to production — and crucially, it first archives whatever was previously in production, enforcing the rule that one model is live at a time. Watch the rollback at the end.

registry = {}
def register(version): registry[version] = "staging"
def promote(version):                          # one model is 'production' at a time
    for v in registry:
        if registry[v] == "production": registry[v] = "archived"
    registry[version] = "production"

register("v1"); promote("v1")
register("v2"); promote("v2")
print("after v2 promote:", registry)
promote("v1")                                  # rollback to the known-good v1
print("after rollback  :", registry)
Outputafter v2 promote: {'v1': 'archived', 'v2': 'production'} after rollback : {'v1': 'production', 'v2': 'archived'}

Trace the stages. We register and promote v1 — it's now production. Then v2 is registered and promoted; promoting v2 first archives v1 (the old production model), so we get {v1: archived, v2: production} — v2 is live, v1 is retired but kept. Now suppose v2 is misbehaving in production. Rollback is a single call: promote("v1") archives v2 and restores v1 to production, giving {v1: production, v2: archived}. We're back to the known-good model instantly, with no retraining and no hunting for files. Two things make this work: the registry never deletes versions (so v1 was still there to roll back to), and promotion always archives the incumbent (so there's exactly one production model and the transition is clean). That's the entire value of a registry — every deployment is an explicit, recorded, reversible stage change, which turns the scary act of shipping a new model into a safe, one-command operation you can undo.

★ Key points to remember
  • A model registry is a central, versioned catalog of trained models — it always knows which version is live.
  • A version is an immutable numbered snapshot; a stage is its status: staging, production, or archived.
  • Promotion is the deliberate, recorded move into production; deployment becomes explicit, not accidental.
  • Rollback = promote the previous known-good version back — instant recovery from a bad deployment.
  • Rollback works only because old versions are archived, not deleted — like git for models.

Practice Build the reasoning yourself

1Your team promotes a new recommendation model on Friday afternoon. By Saturday, click-through has dropped 15%. Walk through how a registry lets you respond, and what you'd do next.
Show solution & reasoning

The immediate response is rollback: promote the previous known-good version back to production. Because the registry archived rather than deleted it, the old model is sitting there ready, and one promotion restores it — click-through should recover within minutes, stopping the bleeding without anyone retraining or hunting for a file. This is the registry's headline value: a bad deployment is a reversible event, not a crisis. (This is also why you don't deploy risky changes Friday afternoon if you can help it — but if you did, rollback saves you.)

Then, with the fire out, you investigate offline: the bad version (v-new) is still archived in the registry, so you can pull it and diagnose why it underperformed — compare its served features against training features (train/serve skew?), check whether its offline metrics actually predicted online success, look at what changed. The registry's history also tells you exactly which version was live during the drop, which is essential for a clean investigation. The general pattern this illustrates is the operational discipline of safe deployment: make every change reversible, roll back first and investigate second (restore service before diagnosing), and keep enough history to do the diagnosis. The registry is the infrastructure that makes this discipline possible for models — without it, a 15% drop on a Saturday is a panicked scramble; with it, it's a routine rollback followed by a calm post-mortem.

2How does a model registry relate to git (version control for code)? Where does the analogy hold, and where does it break down?
Show solution & reasoning

The analogy is strong and intentional. Like git, a registry gives you immutable versioned snapshots (each model version is fixed and referenceable, like a commit), history (you can see what existed when), and the ability to revert (rollback is "checkout the previous version"). The mindset is identical: never lose old states, make changes reversible, always know what's current. Many registries even integrate with git so a model version links to the exact code commit that produced it (lineage, Lesson 9.2).

Where it breaks down is what is being versioned and how you judge it. Code is deterministic and text-based, so git can show a precise line-by-line diff and you can read exactly what changed. A model is a large binary blob of learned weights; you can't meaningfully "diff" v2 against v3 by looking at the numbers, and the difference that matters is behavioral (v3 predicts differently), which you measure with metrics on data, not by reading the file. So a model registry pairs each version with its metrics and metadata (how it scored, what data it trained on) because that — not a textual diff — is how you compare model versions. Also, a model's quality depends on data that lives outside the registry, so reproducing a version requires the registry plus the versioned data and code. The clean takeaway: a registry is "git for models" in spirit — versioning, history, revert — but adapted to artifacts whose meaning is statistical and whose comparison is empirical rather than syntactic. Recognizing both the parallel and its limits is the mark of understanding why model versioning needs its own tooling rather than just throwing model files into git.

3Beyond rollback, why is it valuable for the registry to record which version was in production at any past point in time?
Show solution & reasoning

Because production incidents are investigated after the fact, and you can't debug what you can't identify. When something goes wrong — a wave of bad predictions, a customer complaint, a regulator's inquiry about a decision made three months ago — the first question is "which model produced that?" If the registry records the full timeline of which version was in production during which dates, you can answer precisely: "on March 12 the live model was v4, here it is, here's its training data and metrics." Without that record, you're guessing which model made a decision, and you may be unable to reproduce or explain it at all.

This temporal audit trail matters for several reasons. For debugging: you can pull the exact model that generated a bad output and inspect it. For accountability and compliance: regulated domains (credit, healthcare, insurance) often require proving which model made a given automated decision and how it was built — a legal, not just technical, need. For analysis: when metrics shifted, knowing whether a model change coincided with the shift helps separate "the model caused it" from "the world changed." It connects to the lineage idea from Lesson 9.2 — the registry's timeline plus each version's training lineage gives end-to-end traceability from a production decision back to the data and code that produced the responsible model. The deeper principle is that production systems must be accountable: able to answer "what was running, when, and why" long after the fact, because the questions that matter most always come later. The registry's history is what makes models accountable in that sense.

★ Deep dive: safe deployment strategies — canary, blue-green, shadow▸

A registry makes rollback possible, but the best teams reduce how often they need it by deploying new models gradually rather than flipping a switch for all traffic at once. Canary deployment routes a small slice of traffic (say 5%) to the new model while the rest stays on the old one; you watch the new model's metrics on that slice, and only if it looks healthy do you ramp it to 100% — if it's bad, you've exposed just 5% of users and you roll the canary back. (The name comes from "canary in a coal mine.") Blue-green deployment keeps two full environments, the current ("blue") and the new ("green"); you switch all traffic from blue to green at once but keep blue running idle, so rollback is an instant switch back — trading the cost of running two environments for a very fast, clean cutover. Shadow deployment (or shadow mode) runs the new model alongside the old on real production traffic but doesn't use its predictions — it just logs them, so you can compare the new model's behavior against the old on genuine live data with zero user risk before ever promoting it. Each strategy is a different point on the trade-off between deployment safety, infrastructure cost, and complexity.

What ties these to the registry is that they're all built on the same foundation it provides: multiple versions coexisting, explicit control over which version serves which traffic, and the ability to revert. Canary needs two versions live and a way to shift the split; blue-green needs the old version held ready; shadow needs the new version deployed but inert — all of which presuppose a registry that tracks and serves specific versions on demand. These strategies belong to the broader discipline of progressive delivery, the idea that you de-risk changes by exposing them incrementally and watching real signals rather than betting everything on offline tests — which, given train/serve skew and the gap between offline metrics and online outcomes, is exactly the humility production ML calls for. The throughline of this lesson is risk management: a new model is a bet, offline tests are an incomplete read on that bet, so you make the bet reversible (registry rollback) and, ideally, partial and observable (canary/shadow). That mindset — every deployment is a reversible, gradually-validated experiment, never an irreversible leap — is what separates teams that ship models confidently from teams that fear every deploy. With versioned models that you can deploy and revert safely, the next question is the mechanics of actually serving them to users, which is the next lesson.
