Track 3 · Classical ML

The ML problem setup: features, labels, train/test

⏱ 22 min Lesson 3.1 ▶ Open in Colab
Why this existsBefore any model, you frame the problem: what are the features, what is the label, and how do you measure success on data you have not seen? Get the train/test discipline wrong and every later number is fiction.
By the end, you can
  • Tell supervised, unsupervised, and reinforcement learning apart
  • Read data as a feature matrix X and target vector y
  • Explain generalization and why you hold out a test set
  • Split data without leaking, keeping class balance

Concept What "learning from data" actually means

Up to now you've built the tools: vectors, matrices, gradients, NumPy, pandas. Starting here, you use them to do the actual job — machine learning: writing programs that improve at a task by looking at examples, instead of by being told the rules. The most common and most useful flavor, and the one this whole track is about, is supervised learning: you show the computer many examples where you already know the right answer, and it learns to produce that answer on new examples it has never seen.

Three words you'll use in every lesson from now on. A feature is one measured input — a number describing an example (a house's size, a patient's blood pressure). The label (or target) is the answer you want to predict (the house's price, "tumor / not tumor"). A model is the rule — with adjustable numbers inside — that maps features to a predicted label. Training is the process of tuning those numbers so the predictions match the known answers (exactly the "make the loss small" idea from Lesson 1.8).

Two quick contrasts so the boundaries are clear. Unsupervised learning has no labels — you just have inputs and want to find structure, like grouping customers into segments (you'll meet it in Lesson 3.10). Reinforcement learning has no fixed answers either; an agent takes actions and learns from rewards (think game-playing). If someone hands you a spreadsheet with a column you want to predict, that's supervised learning — and that situation is everywhere.

Concept Your data is a table: a matrix X and a vector y

Almost all supervised learning starts from a table. Each row is one example; each column is one feature; and one special column is the label. Stacking the feature columns gives the feature matrix $X$ (often called the "design matrix") with one row per example and one column per feature, and the labels form a vector $y$. So the shapes line up exactly with Track 1: if you have $n$ examples and $d$ features, $X$ is $n \times d$ and $y$ has length $n$.

One row of $X$ — all the features for a single example — is a feature vector. The model's job is to learn a function $f$ so that $f(\text{feature vector}) \approx$ the true label. Everything in this track is a different idea for what shape that function $f$ can take: a straight line (Lesson 3.2), an S-curve (3.3), a tree of yes/no questions (3.7), and so on.

Concept The whole game is generalization — so you hide some data

Here is the single most important idea in all of applied ML, and the one beginners most often get wrong. You do not care how well the model does on the examples it trained on. You care how well it does on new, unseen data — because that's what "using the model" means. Doing well on fresh data is called generalization, and it is the only thing that matters.

Why the warning? Because a model can "cheat" by simply memorizing the training examples — getting them all right without learning any real pattern. That's overfitting (the star of Lesson 3.5). A memorizing model looks brilliant on training data and falls apart on anything new. To catch this, you hold out some data: split your table into pieces and never let the model learn from the held-out part, so it can serve as an honest stand-in for "the future."

The three pieces each have a distinct job. The training set is what the model learns from. The validation set is what you use to make decisions — which model, which settings (the "hyperparameters" from Lesson 1.8) — without peeking at test. The test set is locked in a drawer until the very end and used exactly once, to estimate real-world performance. A common split is 60% / 20% / 20%, or 80% / 20% when you use cross-validation (Lesson 3.12) instead of a fixed validation set.

!
The cardinal sin: letting test data influence training

If any information from the test set sneaks into training — even indirectly, like scaling your features using the whole dataset's average before splitting — your test score becomes a lie, and it always lies in the flattering direction. This is data leakage (Lesson 2.5). The discipline is simple and absolute: split first, then fit everything (the model, the scaler, the feature choices) on training data only.

Worked example Splitting a real dataset, the right way

We'll use a classic real dataset: 569 breast-tumor biopsies, each with 30 measured features, labeled benign or malignant. The goal is the disciplined first step of every project — split into train and test, while keeping the class proportions the same in both halves (called a stratified split, so neither half accidentally gets too few of a rare class).

The 569 examples split into 455 for training and 114 held back for the final test (20%). Crucially, the benign fraction is almost identical in both halves (0.626 vs 0.632) — that's stratification doing its job. The random_state=0 just fixes the shuffle so you and a teammate get the same split (reproducibility, from Lesson 2.1). From here on, the model only ever sees X_train; X_test stays in the drawer until Lesson 3.4, when we score it.

★ Key points to remember
  • Supervised learning maps features ($X$) to a known label ($y$) by learning from examples.
  • Data is a table: rows = examples, columns = features, plus one target column. $X$ is $n\times d$, $y$ has length $n$.
  • What matters is generalization — performance on unseen data, not on the training set.
  • Split into train (learn), validation (choose), and test (score once). Never train on test.
  • Split before any preprocessing, or you risk data leakage and a flattering, fake score.

Practice Build the reasoning yourself

1You have 10,000 labeled examples and want a 70/15/15 train/validation/test split. How many examples are in each set, and which one does the model's training algorithm actually learn from?
Show solution & reasoning

70% of 10,000 = 7,000 train; 15% = 1,500 validation; 15% = 1,500 test. The training algorithm learns only from the 7,000 training examples. The validation set is for your choices (model and hyperparameters), and the test set is scored once at the end. Note that the validation and test sets are "spent" differently: you can look at validation many times, but each peek at test risks tuning to it, so you look exactly once.

2A teammate standardizes all 10,000 examples (subtracts the mean, divides by the standard deviation) and then splits into train/test. Why is this a bug, and what's the fix?
Show solution & reasoning

It's data leakage. The mean and standard deviation were computed using the test examples too, so information about the test set leaked into the transformation applied to training — the model indirectly "knows" something about the data it's supposed to be judged on. The test score will look better than reality.

The fix: split first, compute the mean and standard deviation on the training set only, then apply those same numbers to transform the test set (exactly the fit on train, transform test pattern from Lesson 2.5). The rule generalizes: every preprocessing step that "learns" anything from data must learn it from training data alone.

3You're predicting whether a transaction is fraud. Only 0.5% of transactions are fraud. Why might a plain random train/test split be risky, and what would you do instead?
Show solution & reasoning

With such a rare class, a random split can land very few (or, in a small test set, almost no) fraud cases in one of the halves, making the score noisy and unrepresentative. Use a stratified split, which preserves the 0.5% fraud proportion in both train and test — just like the breast-cancer worked example kept the benign fraction equal in both halves.

This rare-class situation (called class imbalance) also foreshadows Lesson 3.4: plain accuracy is a terrible metric here, because predicting "never fraud" scores 99.5% while catching zero fraud. You'll need precision and recall instead.

★ Deep dive: validation strategies, cross-validation, and when splits break▸

A single fixed validation set is the simplest approach, but it has a weakness: your model-selection decisions hinge on one particular slice of data, which might be lucky or unlucky. The standard upgrade is k-fold cross-validation (the whole of Lesson 3.12): split the training data into $k$ equal folds, train on $k-1$ of them and validate on the held-out one, rotate through all $k$ choices, and average the scores. Every example serves as validation exactly once, so the estimate is far more stable. The trade-off is compute — you train $k$ times instead of once.

Stratified k-fold combines both ideas, keeping class proportions balanced in every fold; it's the sensible default for classification. For time series, never split randomly: you must train on the past and validate on the future (a "rolling" or "expanding window" split), because shuffling would let the model peek at the future — a subtle, devastating leak.

Finally, a deeper assumption underlies all of this: that your training and test data are drawn from the same distribution. When the world changes after you deploy — new users, new prices, new behavior — that assumption breaks. This is distribution shift (or "drift"), and it's why models decay in production and need monitoring and retraining (Track 10). A clean train/test split estimates performance under today's distribution; keeping that estimate honest over time is a whole discipline of its own.

You now have the setup every model plugs into: features, labels, and an honest way to measure success. Next, in 3.2 Linear regression from scratch, we build the simplest real model — a straight line — and watch it learn its own slope and intercept by minimizing squared error.
