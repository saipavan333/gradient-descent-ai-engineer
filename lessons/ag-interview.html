Track 9 · AI Agents & Tool Use

Interview check: Agents

⏱ 15 min Lesson 9.8
Why this existsAgent interviews test the whole picture — what an agent is, ReAct, planning, memory, multi-agent, and above all reliability and prompt injection. This checkpoint rehearses those questions with balanced, judgment-showing answers.
By the end, you can
  • Define an agent and its loop crisply
  • Explain ReAct, planning, and memory
  • Reason about multi-agent trade-offs
  • Discuss reliability and prompt injection

Concept What an agents interview tests

Agents are the frontier of applied AI, so interviewers use them to test whether you understand the whole picture: what an agent is (an LLM in an act–observe loop, not just a chatbot), how it uses tools, how it reasons and plans, how it remembers, when to use multiple agents, and — most of all — how to make it reliable and safe. The strongest candidates pair enthusiasm for what agents can do with sober judgment about what goes wrong. The model answers below show the reasoning interviewers want.

Model answer "What makes something an agent, versus just an LLM call?"

"A plain LLM call is one-shot: text in, text out. An agent wraps the LLM in a loop where it can act in the world and react to what happens. Each iteration: the model reasons about the goal, chooses an action (usually calling a tool — search, code, an API), the action executes, the result is fed back into the model's context, and it decides the next step — repeating until the task is done. The two essential ingredients are tool use (it can do things, not just talk) and the feedback loop (it observes results and adapts). That loop is what lets an agent handle multi-step tasks that a single call can't: it can search, read the result, search again, then answer. The flip side — and I'd raise this unprompted — is that acting in the world means mistakes have real consequences, so autonomy and risk rise together, which is why guardrails matter. The mental shorthand I use is: agent = LLM + tools + a loop + guardrails."

Strong close: name the trade-off — more autonomy is more capability and more risk — to show judgment, not just hype.

Model answer "Explain ReAct. Why interleave reasoning and acting?"

"ReAct = Reasoning + Acting. Instead of the model picking a tool blindly, each step it first writes a Thought (reasoning about what it needs and why), then an Action (the tool call), then reads the Observation (the result) — and loops. Interleaving matters because the reasoning makes the tool choice deliberate and inspectable: the model thinks 'I need the population, so I'll search,' which leads to better actions than guessing, and the visible Thought trace is a debugging goldmine — you can see exactly why the agent did something when it goes wrong. It also lets the agent adapt: it reasons over each observation before the next action, so it can recover when a tool returns something unexpected. It's essentially chain-of-thought (Lesson 7.4) extended into a loop with actions — thinking and doing, each informing the other. The practical payoff is more reliable tool use and far easier debugging."

Model answer "How do you make an agent reliable enough to trust with real actions?"

"Guardrails and defense in depth — I don't rely on the model behaving; I constrain it from the outside. Permissions: an allowlist of exactly which tools/actions it can use, so it physically cannot do dangerous things (it can't delete files if that tool isn't exposed). Budgets: caps on steps, tool calls, time, and money so a runaway loop self-terminates. Human-in-the-loop: high-stakes, irreversible actions (sending money, deleting data) require explicit human approval before they execute. Sandboxing: run actions in an isolated environment so mistakes can't touch production. Monitoring: log every action for audit and debugging. The principle is that the agent's safety must not depend on the agent — independent external checks hold no matter what the model outputs. I'd also raise prompt injection proactively: malicious instructions hidden in content the agent reads can hijack it, and you can't fully fix that by telling the model to behave — you must limit what a hijacked agent is able to do. The mindset is least privilege: grant the minimum autonomy the task needs, gate the rest."

Model answer "When would you use multiple agents instead of one?"

"Multi-agent helps when a task genuinely decomposes into distinct specialties — say a researcher, a writer, and a fact-checker, each with a focused role, prompt, and tools — because a focused agent often beats a generalist juggling everything, and you get modularity (improve one without touching the others). Common patterns are supervisor/worker (a coordinator delegates subtasks), pipeline (sequential transforms), and debate (agents critique each other to converge — ensembling applied to reasoning). But I'd push back on over-using it. Multi-agent adds real costs: more LLM calls, coordination overhead, lossy handoffs, and more failure points — and the compounding-error math punishes it (five 90%-reliable agents give ~59% end-to-end). Empirically, many elaborate multi-agent systems underperform a single well-built agent. So my default is a single capable agent with good tools, and I add agents only where a specific specialization clearly earns its complexity. The most reliable systems use the fewest agents that get the job done."

Model answer "What's the hardest unsolved problem with agents?"

"Reliability broadly, and prompt injection specifically. On reliability: because an agent chains many steps and each can fail, errors compound — even a small per-step failure rate becomes a large end-to-end one over a long task — so getting agents to complete long, complex tasks dependably is genuinely hard, and it's why guardrails, scoping, and self-correction matter so much. On security: prompt injection is the defining open problem. Agents read external content — web pages, emails, tool outputs — and that content shares the same context as their instructions, so an attacker who controls any of it can try to hijack the agent. There's no clean 'escaping' fix the way there is for SQL injection, because instructions and data are both natural language in one stream. Defenses are layered and imperfect — constrain tools and permissions, require approval for sensitive actions, separate trusted from untrusted input — but the honest state of the art is that it's not solved. That's exactly why the security model assumes the model can be manipulated and constrains capabilities from the outside. Naming these — and not pretending agents are a solved, plug-and-play technology — signals real understanding."

★ Rapid-fire answers to have ready
  • Agent in one line? An LLM in a loop that uses tools and reacts to results — agent = LLM + tools + loop + guardrails.
  • Why a loop? Multi-step tasks need observe-and-adapt: act, read the result, decide the next step, repeat.
  • ReAct? Interleave Thought → Action → Observation; reasoning makes tool use deliberate and the trace debuggable.
  • Planning vs reacting? Plan first for complex multi-step goals; add reflection/self-correction to catch errors.
  • Agent memory? The context window is short-term memory; external stores (often a vector DB) give long-term recall.
  • Multi-agent default? Prefer one good agent; add agents only for genuine specialization — they add cost and failure points.
  • Reliability? Defense in depth: permissions, budgets, human-in-the-loop, sandboxing, monitoring — don't trust the model.
  • Biggest risk? Prompt injection (unsolved) and compounding errors over long tasks.

Practice Mock interview prompts

1"Design an agent that books travel for a user — flights, hotels, the works. Walk me through it." (3-minute answer.)
Show a strong answer

"Core loop: a ReAct-style agent that reasons about the request, calls tools, and adapts. Tools: flight search, hotel search, a calendar/availability check, and a booking API — each with a clear schema. Planning: for a multi-leg trip I'd have it plan first (outbound, hotel, return) then execute step by step, re-planning if something's unavailable. Memory: hold user preferences (aisle seat, budget, loyalty numbers) in long-term memory so it personalizes without re-asking. Now the part I'd emphasize — guardrails, because this agent spends money: a strict permission allowlist (it can search freely but booking is gated), human-in-the-loop approval before any purchase ('Book this $480 flight? [confirm]'), a spending budget cap, and full logging. I'd also treat search results as untrusted (prompt -injection risk) and never let the agent act on instructions embedded in a webpage. Reliability: a step budget so it can't loop forever, and graceful 'I couldn't complete this, here's how far I got' on failure."

This hits every track concept — loop, tools, planning, memory, multi-step — but the senior signal is leading with guardrails for an agent that takes irreversible, costly actions. That's the judgment interviewers reward.

2"Your agent works on simple tasks but fails on long, complex ones — it gets lost or gives up halfway. Why, and what would you do?"
Show a strong answer

"This is the compounding-error problem: a long task is a chain of steps, and even a modest per-step failure rate multiplies over many steps into a high chance of overall failure — plus the agent can lose track of the goal as the context fills with intermediate results. Fixes, roughly in order: (1) Explicit planning — have it draft a plan and follow it, so it keeps the overall goal in view instead of drifting step to step. (2) Decomposition — break the task into smaller, checkable subtasks (possibly specialized sub-agents) so each piece is more reliable and verifiable. (3) Reflection / self-correction — after steps, have it check its own work against the goal and retry on errors, which raises per-step reliability. (4) Memory management — summarize or externalize earlier steps so the context doesn't overflow and bury the goal. (5) Reduce the autonomy needed — sometimes the right answer is a more constrained workflow with fewer free-form steps. The framing I'd give: long-horizon reliability is the hard problem for agents, and the levers are planning, decomposition, self-correction, and memory — each attacks the compounding-error math from a different angle."

Diagnosing it as compounding errors (not a random bug) and offering structured, layered fixes shows you understand why agents are hard — exactly the depth senior interviews probe.

3"Sell me on agents — but also tell me, honestly, why I should be cautious deploying one." (Tests balanced judgment.)
Show a strong answer

"The case for: agents extend LLMs from talking to doing. By giving a model tools and a loop, it can accomplish real multi-step work — research across sources, operate software, automate workflows that used to need a human in the loop for every step. That's a genuine step-change in what AI can automate, and it composes with everything else: RAG becomes a tool, code execution becomes a tool, and the model orchestrates them toward a goal. The case for caution: agents act, so their mistakes have real consequences — a hallucination becomes a wrong action, not just a wrong sentence. They're not reliable on long tasks yet (compounding errors), and the security story is genuinely unsettled — prompt injection has no clean fix, so an agent reading untrusted content can be hijacked. So I'd deploy them deliberately: start with constrained, lower-stakes tasks, wrap them in strong guardrails (permissions, budgets, human approval for anything irreversible), monitor everything, and expand autonomy only as you earn confidence. The synthesis: agents are powerful and risky, and those are inseparable — the engineering discipline of constraining, monitoring, and verifying them is exactly what makes the power safely usable. I'm bullish on the capability and clear-eyed about the caution."

Holding both the promise and the risk — with concrete guardrails as the bridge — is precisely the mature, hype-resistant judgment that distinguishes a strong agents candidate from someone just repeating buzzwords.

★ Deep dive: the agent mindset, and where this fits in your toolkit▸

The throughline of this whole track is a single shift: from a model that talks to a system that acts. Everything followed from putting an LLM in a loop — tools gave it hands (Lesson 9.2), ReAct made its reasoning and acting inform each other (8.3), planning and reflection let it handle complex goals and fix its own mistakes (8.4), memory let it persist information across steps (8.5), multi-agent patterns let specialists collaborate (8.6), and guardrails made the whole thing safe enough to trust (8.7). The mental model to carry into any agent question is the loop wrapped in constraints: think → act → observe → repeat, bounded by permissions, budgets, and human oversight. When you reason about an agent problem, locate it on that map — is this a tool-design question, a planning question, a memory question, a reliability question? — because each has its own well-understood levers, and naming the right one is what separates a crisp answer from a vague one.

The deeper signal interviewers listen for is balanced judgment. Agents are the most hyped area in AI, which means the field is full of people who can gush about what agents might do and few who can soberly say what they reliably do today, why they fail, and how to make them safe. Being the second kind of person — excited by the capability, fluent in the failure modes (compounding errors, prompt injection, over-engineered multi-agent systems), and specific about the engineering that mitigates them (defense in depth, least privilege, human-in-the-loop, monitoring) — is exactly what makes you credible. Connect agents outward, too: they're built on prompting (Track 7), they use RAG as a tool (Track 8), they demand production discipline to deploy (Track 10: serving, monitoring, cost), and they raise safety questions that are an active research frontier (Track 11: alignment, oversight). Seeing agents not as a magic trick but as the integrative capstone of modern applied AI — where reasoning, retrieval, tools, and safety all come together — is the understanding that lets you both build them well and talk about them convincingly. You've now got that end-to-end picture. Next, Track 10 turns to a question every one of these systems eventually faces: once you've built something that works, how do you actually serve it — reliably, affordably, and at scale — to real users?

That completes Track 9 — you can reason about agents that act, plan, remember, and stay safely within bounds. Next, Track 10 (MLOps & Serving) tackles how to deploy, scale, monitor, and maintain these models and systems in production — turning a working prototype into a reliable service.
