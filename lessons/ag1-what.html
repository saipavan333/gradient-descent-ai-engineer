Track 9 · AI Agents & Tool Use

What is an agent

⏱ 22 min Lesson 9.1 ▶ Open in Colab
Why this existsAn agent is an LLM in a loop that can act in the world — use tools, observe results, and adapt — not just reply once. This is the shift from a model that talks to a system that does, and the foundation of everything in this track.
By the end, you can
  • Define an agent as LLM + tools + loop
  • Contrast a one-shot call with an act–observe loop
  • Explain why autonomy and risk rise together
  • Trace one iteration of the agent loop

Concept From answering to acting

So far an LLM has been a function: text in, text out. A single call answers one question; even RAG (Track 8) is one fixed sequence — retrieve, then generate. An agent is different: it's an LLM that decides what to do, takes an action, observes the result, and decides again — looping until it accomplishes a goal. The model is no longer just producing an answer; it's driving a process, choosing its own steps.

The defining feature is autonomy in a loop: the model itself decides, at each step, which action to take next based on what it has learned so far. This unlocks tasks a single call can't handle — anything requiring multiple steps, external information, or actions in the world: "research this topic and write a report," "debug this code and fix it," "book me a flight." The model can search, run code, call APIs, read files, and react to what it finds, chaining these into a solution.

Concept Single call vs pipeline vs agent

It helps to see agents on a spectrum of autonomy:

A single call is best when one shot suffices (summarize this text). A pipeline — a fixed sequence of steps you wrote — is best when the steps are known and always the same (chunk → retrieve → generate). An agent is for when the steps can't be predetermined: the path depends on what the model discovers along the way. The cost of that flexibility is unpredictability — an agent might take a wrong turn, loop, or fail in ways a fixed pipeline can't — which is why much of this track is about making agents reliable (Lesson 8.7). The engineering judgment is: use the least autonomous approach that solves your problem, because autonomy trades control for capability.

Worked example A minimal agent loop

Here's the agent loop stripped to its essence: think (decide), act, observe, repeat until the goal is met. We use a trivial goal — find an $n$ whose square exceeds 50 — with a simple decision rule standing in for the LLM's reasoning, so you can watch the loop structure clearly.

This is the entire skeleton of every agent: a loop where the model looks at the current state, decides an action, the action changes the state, and it decides again — until a stopping condition. Here the "model" is a one-line rule and the "action" is incrementing a number, but swap in a real LLM that reasons in natural language and "actions" that are tool calls (search, run code, call an API), and you have a genuine agent. Three things to notice that recur throughout the track: the loop must have a stopping condition (the goal is met), it needs a max-steps guard (the while steps < 20 — so a confused agent can't loop forever, Lesson 9.7), and each decision depends on the observed result of the last action (the new n). Everything else in this track — tools, ReAct, planning, memory, guardrails — makes each part of this loop more capable and more reliable.

★ Key points to remember
  • An agent is an LLM that decides actions in a loop to reach a goal — it drives a process, not just answers.
  • The loop is think → act → observe → repeat, until a stopping condition (goal met).
  • Spectrum of autonomy: single call (one shot) → pipeline (fixed steps) → agent (model chooses the path).
  • Agents handle tasks whose steps can't be predetermined — but trade control for flexibility (unpredictability).
  • Use the least autonomous approach that solves the problem; agents need stopping conditions and step guards.

Practice Build the reasoning yourself

1For each task, decide single call, pipeline, or agent: (a) translate a paragraph; (b) "research the top 3 competitors and summarize their pricing"; (c) a fixed flow that always retrieves docs then answers.
Show solution & reasoning

(a) Translate → single call. One input, one output, no steps to decide — a plain LLM call. (b) Research competitors → agent. The steps depend on what's found: which competitors, which sources, how many searches — the model must explore and decide as it goes, so it needs the agent loop with tools (web search). (c) Fixed retrieve-then-answer → pipeline. The steps are predetermined and identical every time (RAG, Track 8), so a fixed pipeline is simpler and more reliable than an agent.

The deciding question is always: are the steps known in advance? If yes, single call or pipeline; if the path must adapt to what the model discovers, an agent. Task (b) is the clear agent case because you genuinely can't script the exact searches ahead of time — and that open-endedness is exactly what agents are for, and exactly why they're harder to make reliable.

2An agent is given a vague goal and runs for 80 steps, repeatedly searching the same thing, before timing out. Name two things that went wrong and how the loop structure relates.
Show solution & reasoning

Two failures. (1) Looping / no progress: the agent repeated the same action without using the observation to change course — its decisions weren't conditioned on what it already learned, so it went in circles. (2) Weak stopping condition: with a vague goal, the agent never recognized "done," so nothing triggered an exit — it just ran until the step cap saved it.

This maps directly to the loop: each decision must depend on the accumulated observations (or the agent can't make progress), and there must be a clear goal/stopping condition (or it never knows to stop). Fixes include clearer goals, prompting the model to track what it's tried, reflection to notice it's stuck (Lesson 9.4), and the max-steps guard as a backstop. Looping without progress is one of the most common agent failure modes, and it's why reliability engineering (Lesson 9.7) is central — the flexibility that makes agents powerful is the same flexibility that lets them go wrong.

3How is a basic RAG system (Track 8) a very simple kind of agent, and what would make it a "real" agent?
Show solution & reasoning

Basic RAG is essentially an agent that always takes exactly one predetermined action — retrieve — before answering. It's a one-step, fixed-path "agent": there's no loop, no decision about whether or what to retrieve, no reacting to results. That's why it's better called a pipeline.

It becomes a real agent when you let the model decide: whether it needs to retrieve at all, what to search for (formulating its own queries), whether to retrieve again based on what it found (e.g., "the first results were insufficient, let me search differently"), and when it has enough to answer. Now retrieval is one tool the model chooses to use, possibly multiple times, in a loop driven by its own reasoning — that's "agentic RAG." This illustrates the whole leap of Track 9: turning fixed pipelines into adaptive loops where the model takes the wheel, deciding its own sequence of actions. RAG was the gentlest step; tools and the ReAct loop (next lessons) make it general.

★ Deep dive: the autonomy spectrum and when agents are (and aren't) worth it▸

"Agent" covers a wide spectrum, and conflating its ends causes confusion. At the low end are workflows with LLM steps: mostly-fixed pipelines where an LLM makes a small decision or two (route this ticket, extract these fields) — predictable and easy to control. In the middle are constrained agents: a loop with a small, fixed toolset and tight bounds, free to choose among a few actions for a few steps (the sweet spot for most production agents today). At the high end are autonomous agents: long-running, many tools, open-ended goals, minimal human oversight — powerful in demos but currently unreliable for high-stakes work, because errors compound over long action chains. A useful framing from practitioners: prefer workflows to agents when you can, because fixed structure is more reliable, and reach for full agency only when the task genuinely demands open-ended decision-making. The most successful real systems are usually closer to "structured workflow with some agentic steps" than to "fully autonomous agent," precisely because reliability matters more than impressiveness in production.

Why are agents harder than single calls in a way that's fundamental, not just engineering polish? Because errors compound. If each step is 95% reliable, a 10-step agent is only $0.95^{10} \approx 60\%$ reliable end-to-end — and real steps are often less than 95%, while real tasks need many steps. A single mistake early (a wrong tool call, a misread observation) can derail everything downstream, and the model may not notice. This is the central challenge of the whole field of agents, and it's why so much effort goes into reliability — better tool design (Lesson 9.2), grounding each step in real feedback (ReAct, Lesson 9.3), planning and reflection to catch errors (Lesson 8.4), and hard guardrails (Lesson 9.7). It's also why you should be skeptical of demos: an agent that works impressively once may fail half the time, because the compounding math is unforgiving. The honest state of the art is that constrained, well-guarded agents on bounded tasks work well and are deployed widely (coding assistants, customer-support automation, research helpers), while reliable fully-autonomous agents for complex open-ended goals remain an active frontier. Knowing where on the spectrum your task sits — and deliberately choosing the least autonomy that solves it — is the most important judgment in building with agents, and it's the lens for everything that follows: each lesson adds a capability (tools, reasoning, planning, memory, coordination) and the discipline to keep it reliable.
