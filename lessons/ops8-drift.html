Data & model drift, retraining

⏱ 25 min Lesson 10.8 ▶ Open in Colab
Why this existsModels decay because the world drifts from the training snapshot. Data drift (inputs shift) and concept drift (the input→output rule changes) both degrade accuracy; the fix is detect-and-retrain, the loop that keeps a model current.
By the end, you can
  • Define drift and why models go stale
  • Distinguish data vs concept drift
  • Detect drift (often without labels)
  • Explain the retraining loop

Concept Why models go stale: the world moves

A model is trained on a snapshot of the world — the data as it was during training. It learns the patterns in that data. But the world doesn't hold still: customer behavior shifts, prices change, new products appear, fashions turn, fraudsters invent new tricks. As reality drifts away from the training snapshot, the model's learned patterns become increasingly outdated, and its accuracy quietly decays. This phenomenon — the gradual degradation of a model's performance because the live data no longer matches the training data — is called drift, and it's the single biggest reason deployed models get worse over time. Crucially, the model's code and weights haven't changed at all; the data changed out from under it.

There are two distinct kinds, and the difference matters. Data drift (also called covariate shift) is when the distribution of the inputs changes — the model starts seeing different kinds of data than it trained on. For example, a model trained mostly on customers aged 30–50 starts getting many 18–25-year-old users; the inputs have shifted. Concept drift is subtler and worse: the relationship between inputs and the correct output changes — the very thing the model is trying to predict now works differently. For example, before a recession "high income" strongly predicted loan repayment; after it, that relationship weakens. The inputs might look the same, but the right answer for them has changed.

Why does the distinction matter? Because they're detected and fixed somewhat differently. Data drift you can often spot without labels, just by comparing the live input distribution to the training distribution (the inputs visibly look different). Concept drift is harder — the inputs may look identical, so you often need actual outcomes (labels) to notice that the same inputs now lead to different results. But the practical response to both is usually the same: the model is stale, so you retrain it on fresh data that reflects the new reality.

Concept Retraining: closing the loop

Retraining is training a new version of the model on recent data, so it relearns the current patterns. This is the action that closes the lifecycle loop (Lesson 10.1): monitoring detects drift, which triggers retraining, which produces a new model version, which gets validated and deployed — and then monitored again. The trigger can be scheduled (retrain every week/month regardless, simple but may retrain needlessly or too late) or triggered by drift (retrain when monitoring detects quality has dropped past a threshold, more responsive but needs good drift detection). Either way, retraining isn't a sign of failure — it's the normal, expected maintenance that keeps a model alive as the world changes.

Worked example Detecting drift and triggering a retrain

A simple drift detector compares a live feature's average to the training baseline; if it has moved too far, retrain. Here the training mean was 50; we score two weeks of live data.

The detector measures how far the live data's average has moved from the training average, as a fraction. In week 1, the live values average 50 — exactly the training baseline — so the drift score is 0.0: the data still looks like what the model trained on, no action needed. By week 8, the live values average 70, a 40% shift from the baseline (drift 0.4), well past our 10% threshold — so the detector fires RETRAIN. This is data drift caught without any labels: we never needed to know whether predictions were right, only that the inputs had moved far from training. That distribution shift is an early warning that the model is now operating on data unlike what it learned, so its accuracy is likely degrading — and the automated response is to retrain on recent data so the model relearns the new normal (here, that the typical value is now ~70, not 50). Real drift detectors use more sophisticated statistics than a mean comparison — they compare whole distributions (with tests like population stability index or Kolmogorov–Smirnov) — but the logic is exactly this: quantify how far live data has drifted from the training reference, and when it crosses a threshold, trigger the retraining loop. That single threshold check is what makes the lifecycle self-correcting instead of silently rotting.

★ Key points to remember
  • Drift = model performance decays because live data no longer matches training data; the world changed, not the model.
  • Data drift: the input distribution shifts (different inputs than training).
  • Concept drift: the input→output relationship changes (the right answer for given inputs is now different) — subtler and harder to detect.
  • Data drift is often detectable without labels (compare input distributions); concept drift usually needs outcomes.
  • Retraining on fresh data is the fix — triggered on a schedule or by detected drift; it's normal maintenance, not failure.

Practice Build the reasoning yourself

1A demand-forecasting model that was accurate for two years suddenly performs terribly in March 2020 (the COVID lockdowns). Which kind of drift is this, and why is it so hard for the model?
Show solution & reasoning

This is severe concept drift (with data drift alongside). The pandemic didn't just change which inputs appeared — it broke the relationship between inputs and demand that the model had learned. Patterns that reliably predicted demand for two years (day of week, season, recent trend, store location) suddenly meant something completely different: toilet paper and home goods spiked, travel and restaurant demand collapsed, and the historical input→demand mapping the model encoded became actively wrong. The model kept confidently applying rules that no longer held, so its accuracy cratered.

It's so hard because the model has no way to know the world's rules changed — it can only apply the patterns it learned from history, and when history stops being a guide, a model trained on history is fundamentally mismatched to reality. This is the deepest danger of concept drift: it's not a bug to fix in the model, it's a change in the underlying truth the model was never trained for. The response is to retrain on the new-regime data as it accumulates — but there's a painful catch during a sudden shift: at first you have very little new-regime data to retrain on, so the model is stuck between a stale old model and an undertrained new one until enough new data arrives. Abrupt concept drift like a pandemic or a market crash is among the hardest things in applied ML precisely because it invalidates the core assumption — that the future resembles the past — that all learning from historical data depends on. The practical lesson: monitor for it, retrain aggressively when it hits, and recognize that during a regime change, model performance will degrade no matter how good your MLOps is, because the problem is epistemic, not engineering.

2Compare scheduled retraining (e.g., every month) with drift-triggered retraining. When would you choose each?
Show solution & reasoning

Scheduled retraining runs on a fixed cadence regardless of measured drift. Pros: dead simple to set up and reason about, predictable resource usage, and guaranteed freshness up to the cadence. Cons: it's blind to actual need — it may over-retrain (wasting compute, and risking introducing a worse model) when nothing has drifted, or under-retrain (the model decays badly between scheduled runs if drift hits right after a retrain). It's a good default when drift is gradual and roughly steady, or when you lack good drift detection. Drift-triggered retraining retrains only when monitoring detects quality has dropped or data has drifted past a threshold. Pros: responsive — it retrains exactly when needed, neither wastefully nor too late; efficient with compute. Cons: it requires reliable drift detection and monitoring infrastructure (which can have false alarms or miss subtle concept drift), making it more complex to build and trust.

The choice hinges on your drift profile and your tooling maturity. If drift is slow and predictable and you're early in your MLOps journey, a simple schedule is sensible. If drift is irregular or bursty (sometimes stable for months, sometimes a sudden shift) and you've invested in good monitoring, drift-triggered is far better because it matches effort to reality. In practice, mature teams often use a hybrid: a baseline schedule as a safety net (retrain at least monthly no matter what) plus drift triggers for responsiveness (retrain immediately if drift is detected, regardless of the schedule). This belt-and-suspenders approach gets predictability and responsiveness together. The deeper point is that retraining cadence is a real decision with trade-offs, not an afterthought — and like most MLOps choices, it's driven by understanding your specific system's behavior (how it drifts) rather than by a universal rule.

3Connect drift back to a fundamental assumption from Track 3. Why is drift not a flaw in your model but an inherent property of deploying ML in a changing world?
Show solution & reasoning

Drift connects directly to the i.i.d. assumption (and the train/test-from-the-same-distribution premise) that underlies essentially all of supervised learning from Track 3. Machine learning works by assuming that the data the model will see in the future is drawn from the same distribution as the training data — that the past is representative of the future. Every generalization guarantee, every reason to expect test performance to predict deployment performance, rests on this assumption. Drift is simply the real-world violation of that assumption over time: the future stops resembling the past, so the foundation the model's validity stood on erodes.

This is why drift is inherent, not a flaw. There's nothing wrong with a model that drifts — it learned the patterns of its training distribution correctly, and it would still be accurate if that distribution held. The "problem" is that deploying into a living world means the distribution won't hold forever, because the world isn't stationary. So drift isn't a bug to be eliminated; it's a permanent condition to be managed, which is exactly why retraining is a continuous maintenance activity rather than a one-time fix. Seeing this clearly reframes the whole MLOps lifecycle: the monitor→retrain loop exists precisely because the i.i.d. assumption decays in production, so the loop is the operational mechanism for continually re-establishing that assumption — re-aligning the model's training distribution with the current world. This connection, from a foundational statistical assumption (Track 3) to a core operational practice (this track), is the kind of deep link that shows you understand ML not as a collection of techniques but as one coherent discipline: the theory tells you why models must drift, and the operations tell you what to do about it.

★ Deep dive: detecting drift in practice, and the feedback-loop trap▸

Detecting drift in practice means comparing a reference distribution (usually the training data, or a recent healthy window) against the current distribution of live data, feature by feature and for the predictions. Practitioners use statistical tools for this: the population stability index (PSI) and Kolmogorov–Smirnov test quantify how much a single feature's distribution has shifted; for predictions you watch whether the output distribution has moved. The toy mean-comparison in the worked example is the same idea stripped to its core — these methods just compare whole distributions rather than only their averages, so they catch shifts in shape and spread, not just center. The operational challenge is tuning sensitivity: too sensitive and normal fluctuation triggers false retraining alarms; too lax and real drift slips through — the same alert-calibration problem as monitoring (Lesson 9.7). Teams also distinguish detecting that something drifted from diagnosing what and why, which requires slicing by feature and segment to localize the shift.

A subtle and important trap unique to deployed ML is the feedback loop, where the model's own predictions change the world in ways that induce drift. Consider a recommendation model: it recommends certain items, users click them (partly because they were recommended), that click data becomes training data, which reinforces recommending those items — and the system can spiral into a narrow, self-confirming loop that drifts away from users' true broad preferences. Or a predictive-policing model that sends patrols to areas it flagged, generating more recorded incidents there, which "confirms" the prediction and amplifies the bias. These loops mean the model isn't just observing a changing world — it's causing change, and the drift is partly self-inflicted. Detecting and breaking such loops (through exploration, holdout groups that don't get the model's intervention, or careful causal analysis) is an advanced but crucial concern, especially as models drive more consequential decisions. It also ties to fairness and safety (Track 12): feedback loops are a common mechanism by which models entrench and amplify bias over time. The overarching lesson of this lesson is that a deployed model lives in a dynamic, sometimes reflexive relationship with its environment — the world changes the model's accuracy (drift), and the model can change the world (feedback) — so operating ML responsibly means continuously watching for both, and never assuming that a model validated once stays valid or stays neutral. That vigilance is the heart of the monitor-and-retrain discipline, and the last piece is automating the whole loop so it runs reliably — which is CI/CD, next.

Detecting drift and retraining by hand doesn't scale. Next, 9.9 CI/CD for ML shows how to automate the whole pipeline — testing, building, and deploying code, data, and models — so the lifecycle loop runs reliably without manual steps.
