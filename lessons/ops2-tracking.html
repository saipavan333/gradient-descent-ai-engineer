Experiment tracking & reproducibility

⏱ 24 min Lesson 9.2 ▶ Open in Colab
Why this existsTraining is dozens of experiments; without logging, the best model is lost. Experiment tracking records every run's params, metrics, and artifacts — and reproducibility pins data, code, environment, and the seed so you can rebuild the winner.
By the end, you can
  • Log params, metrics, and artifacts per run
  • Compare runs to find the best
  • List the five reproducibility determinants
  • Explain the random seed

Concept If you didn't log it, it didn't happen

Training a model is not a single act — it's dozens or hundreds of experiments: you try a learning rate of 0.1, then 0.01; two layers, then four; this feature set, then that one. Each run produces a model with some score. The problem is memory: after thirty runs, which settings gave the best result? What data did the good run use? Can you rebuild it exactly? Without a system, the answer is usually "I think it was the one with lr=0.01... or was it 0.001?" — and the best model is lost.

Experiment tracking solves this by automatically recording, for every training run, three things: the parameters (the settings you chose — learning rate, number of layers, etc., also called hyperparameters), the metrics (the results — accuracy, loss, and so on), and the artifacts (the outputs — the saved model file, plots, logs). With every run logged, you can compare them side by side, see exactly which settings produced the best metric, and reproduce that run. Tools like MLflow, Weights & Biases, and Neptune do this; conceptually it's a logbook for your training.

Concept Reproducibility: getting the same result twice

Closely tied to tracking is reproducibility: the ability to run the same training again and get the same model. This sounds trivial but is surprisingly hard, because a model's result depends on more than the code. It depends on the data (exactly which rows, which version), the parameters, the code, the software environment (library versions — a different NumPy can change results), and randomness (weight initialization and shuffling use random numbers). Pin down all five and the run is reproducible; leave any of them floating and "the model that worked last week" may be unrecoverable.

Reproducibility matters for real reasons: you can't debug a model you can't rebuild, you can't pass a model to a teammate who can't regenerate it, and in regulated industries (finance, healthcare) you may be legally required to prove exactly how a model was produced. The practical recipe is to version everything (code in git, data with a version tag, environment in a lockfile or container) and fix the random seed — the starting number for the random generator, which makes "random" initialization repeat identically. Experiment tracking supports this by logging the parameters and often the code version and data reference alongside each run.

Worked example A minimal experiment tracker

Here's the essence of experiment tracking in a few lines: a log_run function records each run's parameters and metrics, and afterward we find the best run by its metric — the comparison that tracking makes trivial.

Each call to log_run stores one experiment's settings and results — exactly what a real tracker like MLflow does, just persisted to a server and a UI instead of a list. Once the runs are logged, finding the winner is a one-line max over the metric: run2, with learning rate 0.01 and 4 layers, at 0.86 accuracy. Notice what the log gives you that memory wouldn't: not just that run2 won, but the exact parameters that produced it, so you can reproduce it. Notice too what makes run3 instructive — it has the same learning rate as run2 but 8 layers instead of 4, and it scored lower (0.83), telling you that more layers hurt here. That kind of controlled comparison — change one parameter, see its effect — is only possible because every run was logged with its parameters attached. This tiny dict is the conceptual core of every experiment-tracking tool: log the inputs and outputs of each run, and the path to the best model becomes visible instead of a guessing game.

★ Key points to remember
  • Experiment tracking logs every training run's parameters, metrics, and artifacts (the model file).
  • It turns "which settings were best?" from a memory game into a side-by-side comparison.
  • Reproducibility = getting the same model again; it needs data, params, code, environment, and the random seed all pinned down.
  • Practical recipe: version code (git), data (a tag), and environment (lockfile/container); fix the random seed.
  • Tools: MLflow, Weights & Biases, Neptune — but the core idea is just a disciplined logbook for training.

Practice Build the reasoning yourself

1A teammate says "our best model scored 0.91 last month but nobody can reproduce it now." List the likely culprits and how tracking would have prevented this.
Show solution & reasoning

The model is unreproducible because at least one of the five determinants wasn't pinned. Likely culprits: the data changed (rows were added or cleaned differently, and no version of last month's exact dataset was saved); the parameters weren't recorded (nobody remembers the exact lr/layers); the code moved on (the training script was edited since); the environment drifted (a library was upgraded); or the random seed was never fixed (so even identical inputs give a different model). Any one of these breaks reproducibility.

Experiment tracking would have prevented it by logging, at the moment of that 0.91 run, the parameters, the metrics, the code version, and a reference to the exact data — plus saving the model artifact itself. With that record, "reproduce last month's model" becomes "check out commit abc123, pull data version v4, set seed 42, run with these logged params" — or, failing that, just reload the saved artifact. The lesson is that reproducibility isn't something you bolt on after the fact; it has to be captured at training time, which is exactly why tracking runs automatically on every run rather than waiting for you to decide a run was important. The best run is often a surprise, so you must log them all.

2Why log every run automatically, instead of just saving the ones that look good?
Show solution & reasoning

Because you don't know which run is "good" until you can compare it to the others, and you often don't know a run mattered until later. If you only save runs you think look good at the time, you lose the comparisons that give results meaning: knowing run2 hit 0.86 is far more useful when you can see run1 got 0.71 and run3 got 0.83, because the contrast reveals which parameter changes helped. You also lose the "failed" runs that are actually informative — a run that scored terribly because you doubled the learning rate is valuable evidence about the learning rate, and you'd want it logged.

There's a deeper reason too: human judgment about which run is important is unreliable and retrospective. The run you dismissed on Monday might become the baseline you desperately want to reproduce on Friday when a "better" idea turns out worse. Automatic, exhaustive logging removes the need to predict importance — you capture everything cheaply and decide what mattered later, with full information. This mirrors a general principle in operations: instrument first, analyze later, because you can't go back and collect data from a run that already happened. The cost of logging a run is tiny; the cost of having needed a run you didn't log is a redo of the work, or worse, a result you can never recover.

3How does experiment tracking connect to hyperparameter tuning (Track 3) and to the rest of this MLOps track?
Show solution & reasoning

It's the connective tissue. Hyperparameter tuning (Track 3) is the act of systematically trying different parameter settings to find the best model — which is precisely a structured set of experiments, each of which must be tracked for the search to mean anything. A tuning sweep that tries 50 learning-rate/layer combinations is 50 logged runs; the tracker is what lets you see the response surface and pick the winner. So tracking is the substrate tuning runs on top of.

For the rest of this track, tracking is the train stage of the lifecycle loop (Lesson 9.1), and it feeds the next stages directly. The winning run's artifact — the saved model — is what gets handed to the model registry (Lesson 9.4) for versioning and promotion to production. The logged metrics become the baseline you later compare against in monitoring (Lesson 9.7): "production accuracy is 0.79, but the model scored 0.86 at training — something's wrong." And the reproducibility discipline (versioned data, code, environment) is what makes the retraining stage trustworthy, since retraining is just re-running a tracked experiment on fresh data. In short, tracking is where the operational loop begins: it's the first place you impose the rigor — log everything, version everything, reproduce anything — that the entire discipline depends on.

★ Deep dive: what a real tracker stores, and the lineage problem▸

A production experiment-tracking tool stores more than our toy dict. For each run it captures the parameters and metrics (often as time series — loss at every epoch, not just the final number, so you can see the training curve and spot overfitting), the artifacts (the model file, plus plots, confusion matrices, and sample predictions), the code version (the git commit), the environment (library versions, hardware), and a reference to the data version. Tied together, these answer not just "which run was best?" but "how, exactly, was this specific model produced?" — a question called lineage or provenance. Lineage is the ability to trace any deployed model back through its training run, its data, and its code to a complete, reproducible origin. In casual projects this feels like overkill; in serious ones it's essential, because when a model misbehaves in production the first question is always "what produced this model, and can we reproduce and inspect it?" — and without lineage, you're guessing.

Lineage becomes critical in regulated domains. A bank using an ML model for credit decisions may be legally required to explain and reproduce exactly how any deployed model was built — which data, which code, which parameters — both for audits and to defend against claims of unfair discrimination. Healthcare and other high-stakes fields have similar demands. This is why the serious end of MLOps treats data, code, and models as jointly versioned artifacts with tracked relationships between them, not as loose files. The discipline also pays off far short of regulation: it's what lets a team of ten data scientists collaborate without overwriting each other's work, what makes a six-month-old result recoverable, and what turns "I think this model is better" into "this model is measurably better than the logged baseline, here's the comparison." The throughline of this lesson — capture everything at the moment it happens, because you can't reconstruct it later — is the same instinct that underlies monitoring, logging, and every other operational practice in this track. It's the operational mindset applied to the training stage, and it's the foundation the deploy and monitor stages build on.

Tracking assumes your data is clean and consistent — but getting it that way is its own engineering challenge. Next, 9.3 Data & feature pipelines, feature stores covers how features are computed reliably and served identically to training and production.
