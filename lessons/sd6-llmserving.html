Designing an LLM serving platform

⏱ 28 min Lesson 11.6 ▶ Open in Colab
Why this existsLLMs are huge and generate one token at a time, so serving them is a specialized discipline: queue → batch → GPU workers with KV cache. Capacity is tokens/sec = QPS × tokens/response; batching keeps GPUs full.
By the end, you can
  • Explain why LLM serving is hard
  • Describe queue, batching, KV cache
  • Size a GPU fleet in tokens/sec
  • Reason about build vs buy

Concept Serving an LLM is a different beast

Serving a classical model is one quick prediction per request. Serving a large language model is fundamentally harder, and understanding why is the key to designing an LLM serving platform — the infrastructure that makes large-model inference fast and affordable at scale. Two properties make LLMs special. First, they're huge: billions of parameters, so each forward pass is billions of operations and the model needs expensive GPU memory just to load. Second, they generate autoregressively — one token at a time, each new token depending on all previous ones — so producing a 500-token answer means running the model 500 times in sequence, not once. This makes LLM inference slow, costly, and latency-sensitive in a way classical serving never was, and it's why LLM serving has become a specialized discipline.

The platform's job is to keep the expensive GPUs busy and shared efficiently. Requests arrive at a queue; a batch scheduler groups compatible requests so the GPU processes many at once (batching, below); a pool of GPU workers holds the model and does the generation, each maintaining a KV cache (the stored intermediate attention values for the tokens generated so far, so each new token doesn't recompute the whole sequence); and generated tokens stream back to the user token-by-token (which is why chatbots appear to "type"). A model registry supplies model versions; the worker pool autoscales with load. Every piece exists to squeeze maximum throughput from scarce, costly GPUs.

Worked example Sizing the GPU fleet

LLM capacity planning works in tokens per second, because generation is token-by-token. Given a target request rate, the tokens each response needs, and one GPU's generation throughput, you can estimate the fleet:

At 100 requests per second, each generating a 500-token answer, the platform must produce 50,000 tokens every second. If one GPU (with batching) generates ~2,500 tokens per second, you need 20 GPUs to keep up. That number is the heart of an LLM serving design, and it exposes the levers. GPUs are expensive, so 20 of them is a real cost — and everything in the platform aims to raise that per-GPU throughput (the 2,500) so you need fewer of them. Batching is the biggest lever: without it, a GPU generating for one request at a time might do a fraction of 2,500, so batching many requests into each pass can multiply throughput several-fold — directly cutting the GPU count and cost. KV caching avoids recomputing past tokens, and quantization (a smaller-precision model) lets each GPU run faster. Flip it around and the design pressures are clear: longer responses (more tokens) or higher QPS raise the demand linearly, so you'd push back on unnecessarily long outputs and cache common ones. This tokens-per-second accounting — demand = QPS × tokens/response, GPUs = demand ÷ per-GPU throughput — is the back-of-the- envelope that anchors any LLM serving design, and it makes concrete why batching and the other optimizations aren't niceties but the difference between 20 GPUs and 100.

★ Key points to remember
  • LLMs are huge and generate autoregressively (one token at a time), making inference slow, costly, and latency-sensitive.
  • An LLM serving platform keeps scarce GPUs busy: a queue → batch scheduler → GPU worker pool (with KV cache) → streamed tokens.
  • Batching packs many requests into one GPU pass — the biggest throughput lever, since GPUs are massively parallel.
  • KV caching avoids recomputing earlier tokens; quantization speeds each GPU up.
  • Capacity math: tokens/sec demand = QPS × tokens/response; GPUs = demand ÷ per-GPU throughput.

Practice Build the reasoning yourself

1Chat requests need low latency (user waiting), but batching adds a small delay to gather a batch. How do you serve both interactive chat and bulk offline jobs on the same platform?
Show solution & reasoning

This is the classic latency-vs-throughput tension, and the answer is to treat the two workloads differently rather than force one policy on both. For interactive chat (latency-sensitive), use small batches and a very short batch-collection window (wait only a few milliseconds to gather a batch, then run) — or continuous batching that adds requests to an in-flight batch — so users aren't kept waiting to form a large batch. You accept somewhat lower GPU utilization as the price of low latency. For bulk offline jobs (throughput-sensitive, nobody waiting — e.g., summarizing a million documents overnight), use large batches and long collection windows to maximize GPU utilization and minimize cost per token, since latency doesn't matter. Route the two workloads to different queues (or different worker pools) with different batching and priority policies.

The design principle is prioritization and workload separation: a serving platform isn't one-size-fits- all; it classifies requests by their latency requirement and applies the batching/priority policy that fits. Interactive requests get priority and small-batch low-latency treatment; batch jobs get whatever capacity is left, packed into large efficient batches, possibly preempted by interactive spikes. Many real platforms implement exactly this — priority queues, separate pools, or continuous batching that interleaves both. The reasoning skill is recognizing that "latency vs throughput" isn't a single global choice but a per-workload one, and that a good platform lets each workload sit at the right point on the trade-off. This connects to requirements (Lesson 11.2): different requests have different SLAs, so the platform must honor each rather than averaging them — a sophistication that shows you understand serving as a multi-tenant resource-allocation problem, not just "run the model."

2Your LLM platform's GPU cost is unsustainable. Beyond batching, list ways to cut it, and note what each sacrifices.
Show solution & reasoning

Several levers, each with a sacrifice. Quantization (run the model at 8-bit or 4-bit precision): each GPU runs faster and the model uses less memory, cutting GPU count — at a small, usually acceptable accuracy cost. Smaller / distilled models (or model routing: a small model handles easy queries, escalating to a big one only when needed): dramatically cheaper for the majority of traffic — sacrificing some quality on harder queries unless routing is accurate. Shorter outputs (cap max tokens, discourage verbose generations): fewer tokens generated means less compute — sacrificing completeness if capped too aggressively. Caching (Lesson 11.7): serve repeated/similar queries from cache, skipping the GPU entirely — sacrificing freshness for changing answers. KV-cache and inference optimizations (continuous batching, speculative decoding, paged attention): raise throughput with little to no quality cost, mostly at the price of engineering complexity. Cheaper/spot hardware or autoscaling down off-peak: pay for capacity only when needed — sacrificing some reliability/complexity.

The reasoning framework is to attack the cost driver — GPU-seconds — from every angle: make each GPU do more (batching, quantization, inference optimizations), send the GPUs less work (shorter outputs, caching, routing to smaller models), and pay less per GPU-second (autoscaling, hardware choices). Most sacrifices are on the accuracy or freshness or complexity axis, and the right combination depends on the requirements — a coding assistant might refuse to shrink the model but happily cap output length and cache; a high-volume classifier might quantize aggressively. The senior move is to quantify where the cost goes (the tokens-per-second math), then pick the levers with the best cost-savings-per- quality-lost for that specific workload, rather than reaching for a single trick. This is the accuracy/cost/latency trade-off (Lesson 11.2) at LLM scale, where the stakes are highest because GPU costs dominate — which is precisely why LLM inference optimization has become such an active and valuable specialty.

3When should you build your own LLM serving platform versus just calling a hosted LLM API? What's the trade-off?
Show solution & reasoning

This is the build-vs-buy decision, one of the most important in modern AI design. Calling a hosted API (a foundation-model provider) means no infrastructure to build or GPUs to manage — you get a state-of-the-art model instantly, pay per token, and scale is someone else's problem. It's the right choice when you value speed-to-market, have moderate or spiky volume, want the best available model, and lack ML-infra expertise — which describes most teams starting out. The sacrifice: per-token cost that can become very expensive at high volume, less control (you can't fully customize the model or guarantee latency), data-privacy considerations (sending data to a third party), and dependence on the provider. Building your own platform (self-hosting an open model) means owning the GPUs, batching, scaling, and ops from this lesson. It's the right choice when volume is high and steady enough that per-token API costs exceed the cost of running your own fleet, when you need full control (custom/fine-tuned models, strict latency, on-prem for privacy/compliance), and when you have the engineering to operate it.

The trade-off crystallizes as cost structure and control vs simplicity and capability. APIs are operational simplicity and top-tier models at a per-use price that scales with volume; self-hosting is a large fixed investment (engineering + GPUs) that pays off at scale and buys control and privacy. The crossover is largely about volume: below some request rate, the API is cheaper and vastly simpler; above it, self-hosting's fixed costs amortize and win — which is exactly a back-of-the-envelope calculation (compare projected API token spend against the cost of the GPU fleet from the worked example). Many teams follow a natural path: start on an API to move fast and validate the product, then, if volume grows enough to justify it, migrate high-volume workloads to a self-hosted platform for cost and control, possibly keeping the API for spikes or the hardest queries (a hybrid). The reasoning skill interviewers want is recognizing this as a real, quantifiable decision driven by volume, cost, control, and privacy requirements — not a religious preference — and being able to reason about when the crossover happens. It's the design-level version of the accuracy/cost trade-off that runs through this whole track.

★ Deep dive: the inference optimizations behind modern LLM serving▸

Modern LLM serving is a rapidly-advancing engineering field, and a strong design touches its key techniques. Continuous (in-flight) batching is the big one: because different requests finish generating at different times, a naive fixed batch wastes GPU capacity waiting for the slowest request; continuous batching dynamically adds new requests to the batch as others finish, keeping the GPU maximally full — a large throughput win that's now standard. PagedAttention (from the vLLM system) manages the KV cache like virtual memory, in pages, so memory isn't wasted reserving worst-case space per request — letting you fit more concurrent requests on a GPU. Speculative decoding uses a small, fast "draft" model to propose several tokens ahead, which the large model then verifies in a single pass; when the draft is right (often), you get multiple tokens for one big-model step, speeding generation with no quality loss. Quantization (running weights at 8-bit, 4-bit, or lower) shrinks memory and speeds compute for a small accuracy cost, and tensor/model parallelism splits a model too large for one GPU across several. Each of these raises the per-GPU throughput that the worked example showed drives the fleet size and cost, which is why they're the object of intense engineering effort.

Zooming out, LLM serving crystallizes the theme that in the LLM era, inference economics are a first-class product and design constraint, not a backend afterthought. When every response costs real GPU-seconds and adds real latency, the cost/latency/quality trade-off decides what products are viable: whether you can call the model on every keystroke or only on submit, whether you use the frontier model or a distilled one, how much you cache, how long you let answers be. A design that ignores this ships a demo that dies on contact with a budget; a design that reasons about tokens-per-second, GPU utilization, and the optimization levers ships something that can actually scale. This also closes a loop with the whole curriculum: the transformer architecture (Track 6) and LLM behavior (Track 7) determine what the model does; MLOps (Track 10) keeps it healthy; and this serving layer determines whether it can be run affordably for millions of users — the piece that turns a capable model into a viable service. The most sought-after skill in applied AI right now sits exactly here: making large-model inference fast and cheap enough to build real products on, which is why this deserves to be its own design lesson and why the final design concern — controlling cost across the whole system through caching, sharding, and more — comes next.

Serving is where the money goes, so controlling it is a design discipline of its own. Next, 10.7 Caching, sharding & cost control covers the cross-cutting techniques that keep a large AI system fast and affordable.
