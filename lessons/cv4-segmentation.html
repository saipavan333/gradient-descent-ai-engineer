Track 14 · Computer Vision

Segmentation: a label for every pixel

⏱ 40 min Lesson 14.4 ⬇ Runnable notebook
Why this existsWhen you need exact shapes — the precise extent of a tumor or the drivable road — a box is too coarse. Segmentation labels every pixel, and the encoder-decoder (U-Net) design that makes it possible reappears in diffusion models later.
By the end, you can
  • Distinguish semantic vs instance segmentation
  • Explain the encoder-decoder (U-Net) and why skip connections matter
  • Evaluate a mask and say why per-class IoU/Dice beats raw pixel accuracy
  • Pick classify vs detect vs segment for a given task

Concept A label for every single pixel

A bounding box (detection) is coarse — it's a rectangle, and rectangles don't capture an object's exact shape. When you need the precise outline, you use segmentation: assigning a class label to every pixel in the image. Instead of "there's a car in this rectangle," segmentation says "these exact pixels are car, those are road, those are sky." It's the finest-grained vision task — a per-pixel classification of the whole image. Two flavors: semantic segmentation labels each pixel with its class (all car-pixels get the label "car," regardless of how many cars); instance segmentation goes further and separates individual objects (car #1's pixels vs car #2's pixels), combining detection's "which object" with segmentation's exact pixel masks.

Segmentation matters wherever exact boundaries are needed. In medical imaging, outlining the precise region of a tumor or organ (not just a box) is what lets doctors measure it. In self-driving, knowing the exact drivable road surface and the precise extent of pedestrians is safety-critical. In photo editing, "select the person" or replacing a background requires a per-pixel mask. Because it produces a full-resolution output (a label per pixel), segmentation needs an architecture that can both understand the image and output at pixel resolution.

Concept The encoder-decoder (U-Net) architecture

How does a network output a label for every pixel? Not with a classifier that gives one answer, but with an encoder-decoder architecture — the classic example being U-Net. The encoder is like a CNN classifier: it progressively downsamples the image (via convolution and pooling), building up rich, abstract features while shrinking spatial size — learning what is in the image. The decoder then does the reverse: it progressively upsamples back to full resolution, turning those features into a per-pixel prediction — recovering where everything is. The key trick is skip connections that pass fine-grained detail from encoder layers directly across to the matching decoder layers, so the precise boundaries (lost during downsampling) are recovered. The result — shaped like a "U" — understands the image globally and localizes precisely.

Worked example Evaluating a segmentation: per-pixel accuracy

Since segmentation predicts a class per pixel, you can evaluate it pixel-by-pixel. Here a tiny predicted mask is compared to the true mask (each cell is a pixel's class), computing what fraction of pixels the model got right:

import numpy as np
true = np.array([[0,0,1],[0,1,1],[2,2,1]])      # each cell = the class of that pixel
pred = np.array([[0,0,1],[0,1,1],[2,2,2]])      # the model's per-pixel prediction
print("per-pixel accuracy:", round((true == pred).mean(), 3))
print("pixels correct:", int((true == pred).sum()), "/", true.size)
Outputper-pixel accuracy: 0.889 pixels correct: 8 / 9

The masks are 3×3 grids where each number is a pixel's predicted class (0 = say background, 1 = object, 2 = something else). Comparing cell by cell, the model matches the truth on 8 of 9 pixels — it got everything right except the bottom-right pixel, which it labeled class 2 instead of the true class 1 — giving 88.9% per-pixel accuracy. This is segmentation evaluation in miniature: because the output is a label per pixel, you compare the predicted and true masks pixel-for-pixel. It also hints at a subtlety, though: per-pixel accuracy can be misleading when classes are imbalanced. If 95% of an image is "background," a model that labels everything background scores 95% accuracy while completely missing the small object that matters — the accuracy paradox from Track 11, in visual form. That's why real segmentation is usually evaluated with IoU per class (the same intersection-over-union idea from detection, but computed on pixel masks: overlap pixels ÷ union pixels for each class) or the closely-related Dice score, which focus on how well each class's region is captured rather than raw pixel counts. So the worked example shows the basic idea (compare masks pixel-by-pixel) while pointing at the real practice (use IoU/Dice per class to avoid being fooled by dominant background). Understanding that segmentation output is a per-pixel mask, evaluated by comparing masks — carefully, with class-aware metrics — is the core of working with the finest-grained vision task.

★ Key points to remember
  • Segmentation assigns a class to every pixel — the finest-grained vision task, giving exact shapes (not boxes).
  • Semantic segmentation labels pixels by class; instance segmentation also separates individual objects.
  • It uses an encoder-decoder (U-Net): encoder downsamples to understand, decoder upsamples to full resolution, skip connections preserve fine detail.
  • Evaluated by comparing masks — but use IoU/Dice per class, not raw pixel accuracy, which imbalanced background can inflate.
  • Used where exact boundaries matter: medical imaging, self-driving, photo editing.

Practice Build the reasoning yourself

1For a task, would you choose classification, detection, or segmentation: (a) "is there a dog in this photo?", (b) "count the cars in this parking lot", (c) "measure the exact area of a skin lesion"?
Show solution & reasoning

Match the task to the granularity you actually need. (a) "Is there a dog?" → classification. You need one yes/no label for the whole image; you don't need location or shape, so the simplest task suffices. Using detection or segmentation would be overkill — more data to label, more compute, more complexity for no benefit. (b) "Count the cars" → detection. You need to find and count individual objects, so you need a box per car (then count the boxes). Classification can't count (it labels the whole image); segmentation could work via instance segmentation but is more than you need — a box per car is enough to count them. (c) "Measure the exact area of a skin lesion" → segmentation. "Exact area" requires the precise pixel-level outline of the lesion (a box would over-count area by including surrounding skin), so only a per-pixel mask gives an accurate area measurement.

The reasoning principle is to use the least-complex task that meets the need: classification for whole-image labels, detection for locating/counting objects, segmentation for exact shapes/areas. Each step up (classify → detect → segment) gives finer information but costs more — more expensive labeling (drawing pixel masks is far more work than a class label), more compute, and more complexity — so you choose the coarsest task that answers your actual question. This is the same "simplest solution that meets the requirements" judgment from system design (Track 10), applied to vision: don't reach for segmentation when a class label will do. Being able to map a real question to the right vision task — and justify not over-engineering — is exactly the practical judgment that shows you understand what each task is for, which is more valuable than knowing how any single one works. The task hierarchy (classify/detect/segment) is a toolkit; picking the right tool for the requirement is the skill.

2Why does segmentation need an encoder-decoder (U-Net) rather than a standard CNN classifier, and what does each half do?
Show solution & reasoning

A standard CNN classifier is built to output one answer for the whole image: it downsamples the image through conv/pool layers into a small, abstract representation and then produces a single label. That's perfect for classification but wrong for segmentation, which needs a full-resolution output — a label for every pixel. You can't get a per-pixel mask from a network that has collapsed the image to a tiny feature vector. So segmentation uses an encoder-decoder. The encoder is exactly a CNN classifier's downsampling path: it progressively shrinks spatial size while building rich, abstract features — answering what is in the image (this region is car-like, that is road-like), but at low spatial resolution. The decoder then reverses the process: it progressively upsamples back toward the original resolution, translating the abstract features into a dense per-pixel prediction — recovering where everything is at full detail.

The crucial addition is skip connections: the encoder's downsampling loses fine spatial detail (exact edges get blurred as resolution drops), so U-Net passes the high-resolution feature maps from each encoder level directly across to the matching decoder level, letting the decoder combine "abstract understanding" (from deep, low-res features) with "precise location" (from shallow, high-res features). This is why the boundaries in a U-Net mask are sharp rather than blobby. The reasoning skill is understanding that segmentation is a dense prediction problem — output at input resolution — which fundamentally requires an architecture that first compresses to understand and then expands to localize, with skips to keep detail. It's a beautiful, reusable design (the "U"), and it appears far beyond segmentation wherever you need pixel-level or high-resolution output. Being able to explain why a classifier won't do and how the encoder-decoder's two halves (plus skips) solve the dense-prediction problem demonstrates real architectural understanding — the same depth this track built for CNNs, now for the segmentation setting.

3Segmentation labels are expensive (someone must outline every object pixel-by-pixel). How does this shape a real segmentation project?
Show solution & reasoning

The cost of pixel-level labels is a defining practical constraint. Drawing a precise mask around every object in an image is vastly more labor than assigning a class label or even a bounding box — it can take many minutes per image — so labeled segmentation data is scarce and expensive, which shapes the whole project. Consequences and responses: Lean heavily on transfer learning — start from a segmentation model pretrained on a large dataset and fine-tune on your smaller labeled set, since you can't afford to label enough to train from scratch. Use aggressive data augmentation to stretch limited labeled images. Consider weaker supervision: sometimes you can train (partly) from cheaper labels — image-level or box-level labels, or a few clicks — rather than full masks (weakly- or semi-supervised segmentation), trading some accuracy for far less labeling. Prioritize labeling effort: label the most informative or hardest examples first (active learning), and reuse model predictions as a starting point for human annotators to correct (model-assisted labeling) rather than drawing every mask from scratch.

The reasoning is that in segmentation, the bottleneck is usually the labels, not the model — proven architectures (U-Net and successors) are readily available, but getting enough high-quality pixel masks is the hard, expensive part. So a real project's effort and cost concentrate on the data: how to get good masks affordably, how to make limited data go far (transfer learning, augmentation), and how to reduce labeling burden (weak supervision, model-assisted annotation). This mirrors a theme across the whole curriculum — that data, not the model, is often the true constraint and the true cost (Tracks 2, 9) — here made especially sharp because segmentation's labels are the most expensive in vision. Recognizing that a segmentation project is largely a data project — planning around the cost of masks, and using transfer learning and weak supervision to cope — is exactly the practical, cost-aware judgment that distinguishes someone who could actually deliver a segmentation system from someone who only knows the architecture. The model is the easy part; the pixel labels are the challenge.

★ Deep dive: the vision task hierarchy, and where segmentation fits▸

It's worth stepping back to see the three core vision tasks as a hierarchy of increasing spatial granularity, all built on the same foundation. Classification asks "what is in the image?" — one label, no location. Detection asks "what objects are where?" — a box and class per object, coarse location. Segmentation asks "what is each pixel?" — exact per-pixel labels, the finest location. Each level gives more spatial information at more cost (in labeling, compute, and complexity), and each builds on the one before: detectors and segmentation models use classification backbones (a CNN or transformer that learned to see) as their feature extractor, then add machinery for localization (boxes) or dense prediction (masks). So the CNN and its learned feature hierarchy — the subject of this track's early lessons — underpins all three tasks; classification is the base capability, and detection and segmentation extend it with location. Instance segmentation sits at the top, essentially combining detection (which objects, where) with segmentation (exact pixels) — models like Mask R-CNN literally add a mask-prediction branch to a detector. Seeing this hierarchy clarifies the whole field: it's not three unrelated problems but a ladder of granularity over the same learned visual features.

The unifying practical wisdom carries across all three: choose the coarsest task that meets your need (granularity costs money), lean on pretrained backbones and transfer learning (the features transfer), watch your data (labels get more expensive as granularity increases, dramatically so for masks), and evaluate with metrics that fit the task and its imbalance (accuracy for balanced classification, mAP with IoU for detection, per-class IoU/Dice for segmentation — the Track 11 discipline throughout). And the same architectural currents run through all of them: CNNs powered the first generation, and transformers are now reshaping each task (DETR for detection, transformer-based segmentation models, Vision Transformers for classification). This connects segmentation forward to the next lessons — the Vision Transformer, which is changing the backbone underneath all these tasks, and generative models, which flip the problem from analyzing images to creating them. Having now covered the three analytical vision tasks (classify, detect, segment) and seen how they form a granularity hierarchy over a shared CNN foundation, you understand the core of "teaching machines to see." The track next turns to the architecture that's modernizing this foundation (ViT) and then to the striking capability of generating images from scratch — but the analytical tasks you've just learned, and the task-selection and data judgment around them, are the bread and butter of applied computer vision.
