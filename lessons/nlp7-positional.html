Positional encoding

⏱ 20 min Lesson 5.7 ▶ Open in Colab
Why this existsAttention is order-blind, so a transformer needs to be told where each word sits. Positional encoding — a vector added per position — is the elegant fix, and it gates the long-context frontier of modern LLMs.
By the end, you can
  • Explain why attention needs positional information
  • Describe sinusoidal, learned, and rotary encodings
  • Compute sinusoidal positional encodings
  • Connect positions to long-context extension

Concept Attention is order-blind, so we must add position

Self-attention has a surprising blind spot: it treats its input as a set, not a sequence. If you shuffle the token order, attention computes the same relationships — just reordered. So on its own, a transformer can't tell "dog bites man" from "man bites dog," even though word order flips the meaning. We have to explicitly tell the model where each token sits, and we do it with positional encoding.

The idea is simple: create a unique vector for each position and add it to the token's embedding before the first block. Now the input to the model encodes both "what" (the word) and "where" (its position), so the attention scores depend on order. The same word in position 1 and position 5 gets slightly different vectors, and the model learns to use that difference.

Concept How to build the position vectors

The original transformer used sinusoidal positional encodings: each dimension of the position vector is a sine or cosine wave of a different frequency, so every position gets a unique pattern and nearby positions get similar ones. A neat property is that the model can, in principle, attend to relative positions ("three tokens back") because shifts in position correspond to predictable rotations of these waves. And because the formula works for any position, it can handle sequences longer than those seen in training.

Modern models often use learned positional embeddings instead (just another lookup table, one vector per position, trained like any weight) or, increasingly, rotary positional embeddings (RoPE), which rotate the query and key vectors by an angle proportional to position — elegantly encoding relative position directly inside the attention computation. The choice affects how well a model generalizes to longer contexts. Whatever the method, the goal is identical: give attention a sense of order.

Worked example Sinusoidal positional encodings

Let's compute the classic sinusoidal encodings for 4 positions in a 6-dimensional space. Even dimensions use sine, odd dimensions cosine, each at a different frequency — producing a unique vector per position.

Each row is the encoding for one position, and they're all different — position 0 is [0,1,0,1,0,1], position 1 is [0.84, 0.54, …], and so on. The left columns (high-frequency sine/cosine) change quickly from row to row, capturing fine position differences; the right columns (low-frequency) barely change, capturing coarse position — together giving each position a distinct "fingerprint." This vector is simply added to the token's embedding, so "cat" at position 0 and "cat" at position 2 enter the transformer as different vectors. That one addition is what rescues the order-blind attention mechanism — a small, elegant fix for a fundamental limitation.

★ Key points to remember
  • Self-attention is order-blind (permutation-equivariant) — it sees a set, not a sequence.
  • Positional encoding adds a unique, position-dependent vector to each token embedding before the blocks.
  • Sinusoidal encodings use sine/cosine waves of different frequencies — unique per position, and extendable to unseen lengths.
  • Alternatives: learned position embeddings and rotary (RoPE), which encodes relative position inside attention.
  • Without positional information, "dog bites man" and "man bites dog" would be indistinguishable to the model.

Practice Build the reasoning yourself

1Concretely, how does adding positional encodings let the model tell "dog bites man" apart from "man bites dog"?
Show solution & reasoning

Without positions, the model sees the same set of token vectors {dog, bites, man} for both sentences, so its attention computes identical relationships — it literally can't distinguish them. With positional encoding, "dog" at position 0 gets embed(dog) + pos(0) while "dog" at position 2 gets embed(dog) + pos(2) — different vectors. So in "dog bites man," the subject slot (position 0) holds dog, and in "man bites dog" it holds man, and the attention scores differ accordingly.

The model can then learn that, say, the token in the subject position is the agent of the verb. Position turns an unordered bag into a sequence the model can reason about grammatically — which is exactly the information word order carries in language.

2In the worked example, the left dimensions of the encoding changed quickly across positions while the right dimensions barely changed. Why is having both fast- and slow-varying dimensions useful?
Show solution & reasoning

It's a multi-scale "clock." The high-frequency (fast-changing) dimensions distinguish nearby positions precisely — positions 3 and 4 differ clearly there. The low-frequency (slow-changing) dimensions distinguish coarse position — they tell position 5 from position 500, where the fast dimensions may have cycled and become ambiguous. Together, the full vector pins down a position uniquely across many scales, like how hour, minute, and second hands together specify a time more precisely than any one alone.

This multi-scale design also helps the model represent relative distances at different ranges, and is part of why sinusoidal encodings can extrapolate to longer sequences — the slow dimensions keep providing a coarse position signal even beyond the training length.

3A model was trained with learned positional embeddings for positions 0–2047. You feed it a 3,000-token document. What likely goes wrong, and why might sinusoidal or rotary encodings fare better?
Show solution & reasoning

Learned positional embeddings are a lookup table with a fixed number of rows (here 2048). Positions 2048–2999 have no embedding — the model never learned vectors for them — so it either errors or behaves unpredictably on the tail of the document. Learned positions don't extrapolate beyond their trained range.

Sinusoidal encodings are computed by a formula valid for any position, so they at least produce a vector for position 2500 (though quality can still degrade past the training length). RoPE tends to extrapolate better still and has variants explicitly designed for length extension. This is why long-context models care a lot about the positional scheme — extending context length is partly a positional-encoding problem, and it's an active research area in modern LLMs (Track 6).

★ Deep dive: why sinusoids, and the long-context frontier▸

Why sine and cosine specifically? The original authors wanted encodings where the model could easily attend to relative positions, and trigonometric functions have a magical property: a shift in position corresponds to a fixed linear transformation (a rotation) of the encoding. Concretely, $\text{pos}(k + \Delta)$ can be written as a rotation of $\text{pos}(k)$ by an angle depending only on $\Delta$ — so the model can learn to detect "the token $\Delta$ positions away" with a single linear operation, regardless of absolute position. This is exactly the relative relationship that matters for language ("the previous word," "the start of this clause"), and it's why the sinusoidal choice wasn't arbitrary. RoPE takes this insight further by applying the rotation directly to the query and key vectors, so relative position falls out of the dot product itself — which is part of why it has become popular in recent LLMs.

Positional encoding sits at the center of one of the most active frontiers in LLMs: extending context length. Going from 2K to 1M tokens of context isn't just about the quadratic attention cost (Lesson 5.5) — it's also about whether the positional scheme still works far beyond the training length. Models often degrade sharply past their trained context because positions they never saw confuse them. A whole toolkit has emerged: position interpolation (squeezing learned/RoPE positions to fit a longer range), NTK-aware scaling, ALiBi (which biases attention scores by distance instead of adding position vectors), and training on progressively longer sequences. The payoff — feeding a model an entire book, codebase, or conversation history — is enormous, which is why "context length" is a headline spec for every new model. It's a satisfying example of how a seemingly minor design detail (how to tell the model where each token sits) turns out to gate a major capability. With tokenization, embeddings, attention, the block, and now position in hand, you understand every ingredient of a transformer — the last lesson assembles them into the different model shapes you'll actually encounter.
