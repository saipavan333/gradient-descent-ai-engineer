Backpropagation, step by step

⏱ 30 min Lesson 4.3 ▶ Open in Colab
Why this existsBackpropagation computes how to adjust every weight — the chain rule, systematized, and the engine that makes learning possible. Understand it and autograd stops being magic.
By the end, you can
  • Explain backprop as forward values then backward gradients
  • Apply the chain rule along a computational graph
  • Verify a gradient numerically
  • Use loss.backward() and autograd

Concept The problem: which way should each weight move?

A fresh network predicts nonsense because its weights are random. To improve, we run gradient descent (Lesson 1.8): nudge each weight a little in the direction that reduces the loss. But that requires knowing, for every single weight, how the loss would change if we wiggled it — its partial derivative $\partial L/\partial w$. A real network has millions of weights. Computing each derivative separately would be impossibly slow. Backpropagation is the clever algorithm that computes all of these gradients in a single backward sweep, about as cheaply as one forward pass. It's the engine that makes training deep networks possible.

The whole thing is the chain rule from Lesson 1.7, applied systematically. Recall: if a small change flows through a chain of functions, you multiply the local slopes to get the overall slope. A network is exactly a long chain of functions (layer after layer), so its gradients are products of local slopes — and backprop is just the bookkeeping that computes those products efficiently.

Concept Forward values, then backward gradients

Picture the network as a computational graph: nodes are operations (multiply, add, activate, loss), edges carry values. Training is two passes over this graph. The forward pass flows left to right, computing each value and finally the loss. The backward pass flows right to left, computing how sensitive the loss is to each intermediate value — the gradients.

The backward pass starts with a trivial fact ($\partial L/\partial L = 1$) and works backward: at each node it takes the gradient flowing in from the right and multiplies by that node's local derivative to get the gradient flowing out to the left. Because every node only needs to know its own simple local derivative and the gradient handed to it, the whole sweep is local and mechanical — perfect for a computer.

The key efficiency: gradients for early layers reuse the gradients already computed for later layers (the $\partial L/\partial a$ term gets passed along), so nothing is recomputed. That reuse is why backprop costs about the same as one forward pass, instead of one pass per weight.

Worked example Backprop by hand, checked against the definition

Take a tiny chain: input $x$, weight $w$, a ReLU, a second weight $v$, and a squared-error loss against target $t$. We'll compute the gradient $\partial L/\partial w$ by the chain rule, then verify it against a brute-force numerical derivative — the gold-standard sanity check.

The chain rule gives $\partial L/\partial w = (-3.0)(1.5)(1.0)(2.0) = -9.0$, and the numerical derivative — literally "change $w$ a hair and measure the loss change" — agrees exactly. The negative sign tells gradient descent to increase $w$ to reduce the loss. That's backprop in miniature: multiply local slopes along the path. A real network does this for millions of weights at once, and PyTorch automates it entirely:

import torch
x = torch.tensor(2.0); t = torch.tensor(3.0)
w = torch.tensor(0.5, requires_grad=True)    # track gradients for w
v = torch.tensor(1.5, requires_grad=True)

loss = (torch.relu(w * x) * v - t) ** 2
loss.backward()                              # backprop: fills w.grad, v.grad
print(w.grad)                                # tensor(-9.)

One call to loss.backward() runs the entire backward pass and stores each parameter's gradient in its .grad — and it reports $-9.0$ for w.grad, matching our hand calculation. This automatic differentiation (autograd) is why you never hand-derive gradients in practice; you just build the forward computation and call backward().

★ Key points to remember
  • Backpropagation computes $\partial L/\partial w$ for every weight in one backward sweep — the chain rule, systematized.
  • Training is two passes: forward computes values and the loss; backward computes gradients right-to-left.
  • At each node, gradient out = (incoming gradient) × (local derivative); early layers reuse later layers' gradients.
  • That reuse makes backprop cost ≈ one forward pass, not one pass per weight.
  • In PyTorch, loss.backward() does it all (autograd); you never hand-derive gradients.

Practice Build the reasoning yourself

1For the chain $z = wx$, $a = z^2$, $L = a$, with $x = 3$ and $w = 2$, compute $\partial L/\partial w$ by the chain rule. (Use $\frac{d}{dz}z^2 = 2z$.)
Show solution & reasoning

Forward: $z = wx = 6$, $a = z^2 = 36$, $L = 36$.

Backward: $\partial L/\partial a = 1$; $\partial a/\partial z = 2z = 12$; $\partial z/\partial w = x = 3$. Multiply: $\partial L/\partial w = 1 \times 12 \times 3 = \mathbf{36}$.

Check directly: $L = (wx)^2 = w^2 x^2 = 9w^2$, so $dL/dw = 18w = 18(2) = 36$. ✓ The chain rule gave the same answer by multiplying local slopes — exactly what backprop automates.

2During the backward pass through a ReLU, the gradient flowing in is 5, but the neuron's forward input $z$ was $-2$. What gradient flows out the other side, and why?
Show solution & reasoning

It flows out as 0. ReLU's local derivative is 1 when $z>0$ and 0 when $z<0$. Here $z=-2<0$, so the local slope is 0, and gradient out = (incoming 5) × (local 0) = 0.

Intuition: the neuron was in its flat "off" region, so nudging its input wouldn't have changed its output at all — therefore it can't have affected the loss through this path, and no gradient should propagate. This is also the seed of the "dying ReLU" issue (Lesson 4.2): a neuron stuck at $z<0$ for all inputs gets zero gradient forever and never recovers.

3You hand-derive a gradient for a custom layer and aren't sure it's right. Describe the "gradient check" you'd run, using the worked example as a template.
Show solution & reasoning

Compare your analytic gradient to a numerical one. For each parameter $w$, compute $\frac{L(w+\epsilon) - L(w-\epsilon)}{2\epsilon}$ with a tiny $\epsilon$ (say $10^{-6}$) — this is the derivative's definition, approximated. If your analytic gradient matches this number to several decimals (as $-9.0$ matched $-9.0$ in the worked example), your derivation is almost certainly correct.

This "finite-difference gradient check" is the standard way to validate hand-written backprop, and it's why the worked example computed both — the numerical version needs no calculus, just the forward pass, so it's a trustworthy referee. (It's too slow for actual training — one forward pass per parameter — but perfect for verifying a small custom piece.)

★ Deep dive: autograd, reverse mode, and why frameworks won▸

What PyTorch actually does is reverse-mode automatic differentiation. As you run the forward pass, it records every operation into a graph (the "tape"), remembering each op's local derivative rule. When you call backward(), it walks that graph in reverse, applying the chain rule mechanically — exactly the by-hand process from the worked example, but for arbitrary computations you wrote in plain Python. You get exact gradients for free, which is why nobody hand-derives backprop anymore and why frameworks (PyTorch, JAX, TensorFlow) transformed the field: researchers can dream up any architecture and the gradients come automatically.

Why reverse mode specifically? Differentiation can sweep forward or backward through the graph. Forward mode is efficient when there are few inputs and many outputs; reverse mode wins when there are many inputs and few outputs. A neural network has millions of inputs (the weights) and exactly one output (the scalar loss) — the ideal case for reverse mode, which gets all million gradients in a single backward sweep. If the loss were vector-valued you'd pay once per output, but it's a single number, so reverse mode is essentially optimal. This match between the shape of the learning problem (many params, one loss) and the strength of reverse-mode autodiff is a quiet reason deep learning scales as well as it does.

One practical gotcha that follows directly: because PyTorch accumulates gradients into .grad every time you call backward() (handy for some advanced tricks), you must zero them each step with optimizer.zero_grad() — the detail from Lesson 1.8 that bites every beginner. It makes sense now: the tape adds the new backward pass's gradients to whatever was already in .grad, so you clear it first. That's the bridge to the next lesson, where this backward pass becomes one beat in the training loop.
