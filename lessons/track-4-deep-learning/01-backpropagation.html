Backpropagation by hand

Backpropagation has a fearsome reputation it does not deserve. It is nothing more than the chain rule from Track 2, applied backward through the computational graph — and it computes the gradient of the loss with respect to every parameter in a single sweep.

Track 4 · Lesson 1 Concept + derivation + code ~28 min Builds on: Lesson 4.0, Track 2 chain rule

💡 Concept: the chain rule, walked backward

To train a network with gradient descent (Track 2), we need \(\partial L/\partial w\) for every weight \(w\) — how the loss changes if we nudge that weight. Backpropagation computes all of these at once. The engine is the chain rule: if a value influences the loss only through a chain \(a \to b \to c \to L\), then its gradient is the product of the local derivatives along that chain:

$$ \frac{\partial L}{\partial a} = \frac{\partial L}{\partial c}\cdot\frac{\partial c}{\partial b}\cdot\frac{\partial b}{\partial a}. $$
(★)

The genius of doing this backward (from \(L\) toward the inputs) rather than forward is efficiency. Each node only needs to know two things: its own local derivative (how its output depends on its immediate inputs) and the gradient flowing in from above (\(\partial L/\partial \text{output}\)). It multiplies them and passes the result to its inputs. One sweep from output to inputs, and every gradient is known.

🧮 Worked example: backprop on \(d = a\times b + c\)

Let’s do it fully by hand on last lesson’s graph, treating \(d\) as the loss \(L\). The forward pass gave \(e = a\times b = -6\) and \(d = e + c = 4\). Now the backward pass. We seed the output’s gradient with \(\partial L/\partial d = 1\) (the loss’s gradient with respect to itself is 1), then move backward, applying each operation’s local rule:

🧮 The backward pass, step by step

The \(+\) node (\(d = e + c\)): addition passes gradients through unchanged, because \(\partial d/\partial e = 1\) and \(\partial d/\partial c = 1\). So

$$ \frac{\partial L}{\partial e} = \frac{\partial L}{\partial d}\cdot 1 = 1, \qquad \frac{\partial L}{\partial c} = \frac{\partial L}{\partial d}\cdot 1 = 1. $$

The \(\times\) node (\(e = a\times b\)): the local derivatives are \(\partial e/\partial a = b\) and \(\partial e/\partial b = a\) — multiplication swaps the values. So

$$ \frac{\partial L}{\partial a} = \frac{\partial L}{\partial e}\cdot b = 1\cdot(-3) = -3, \qquad \frac{\partial L}{\partial b} = \frac{\partial L}{\partial e}\cdot a = 1\cdot 2 = 2. $$

The gradients are \(\partial L/\partial a = -3\), \(\partial L/\partial b = 2\), \(\partial L/\partial c = 1\). Each tells us how the loss responds to a tiny nudge of that input — exactly the information gradient descent needs. Two patterns are worth memorizing because they’ll recur in every network: add distributes the gradient unchanged to all its inputs, and multiply routes the gradient to each input scaled by the other input’s value.

🔑 Why backprop, and not finite differences?

We could estimate each gradient by nudging one input and re-running the forward pass (finite differences, Track 2). But that costs one full forward pass per parameter — hopeless for a model with billions of weights. Backpropagation gets all the gradients in a single backward pass, at roughly the same cost as one forward pass. That efficiency — gradients of millions of parameters for the price of one extra sweep — is the algorithmic breakthrough that makes deep learning feasible at all.

⚠️ Gradients accumulate at shared nodes

If a value feeds into more than one place (like \(b\) in the practice problem \( (a+b)(b+c)\)), its gradient is the sum of the gradients arriving from each path (multivariable chain rule). That’s why real autograd engines write self.grad += ... (add), not = (assign) — a subtle but critical detail you’ll implement next lesson.

🛠️ From scratch: backprop by hand, then verified

code/track-4/backprop_manual.py
# forward
a, b, c = 2.0, -3.0, 10.0
e = a * b          # e = -6
d = e + c          # d = 4  (our "loss" L)

# backward, by hand, starting from dL/dd = 1
dd = 1.0
de = dd * 1.0      # d = e + c  ->  dd/de = 1   (add passes gradient through)
dc = dd * 1.0      # d = e + c  ->  dd/dc = 1
da = de * b        # e = a * b  ->  de/da = b   (multiply swaps the values)
db = de * a        # e = a * b  ->  de/db = a
print("grads:  da =", da, " db =", db, " dc =", dc)   # -3, 2, 1

# verify with finite differences
def L(a, b, c): return a*b + c
h = 1e-6
print("check da:", round((L(a+h,b,c) - L(a-h,b,c)) / (2*h), 4))   # -3.0  ✓

The hand-computed gradients match the finite-difference check exactly — proof our chain-rule reasoning is right. In the next lesson we’ll stop doing this by hand and build an engine that records the graph and runs this backward pass for any expression, automatically.

🔑 Key results to remember

Lock these in

  1. Backprop = chain rule backward through the graph: \(\partial L/\partial a\) is the product of local derivatives along the path.
  2. Seed \(\partial L/\partial L = 1\) at the output, then sweep backward; each node multiplies its incoming gradient by its local derivative.
  3. Add passes gradient through unchanged; multiply routes it to each input scaled by the other input.
  4. One backward pass = all gradients, at ~the cost of one forward pass. Finite differences cost one pass per parameter — infeasible.
  5. Gradients at a value used in multiple places accumulate (sum) — hence grad +=.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. For \(e = a\times b\) with \(a=2, b=-3\), and gradient \(\partial L/\partial e = 1\) flowing in, what is \(\partial L/\partial a\)?

  • \(2\)
    That’s \(\partial L/\partial b = a = 2\). For \(\partial L/\partial a\) you multiply the incoming gradient by the other input, \(b\).
  • \(1\)
    That would be the local derivative of an add node. For a multiply, the local derivative \(\partial e/\partial a = b\), not 1.
  • \(-3\)
    Correct. For a multiply, \(\partial e/\partial a = b = -3\), so \(\partial L/\partial a = 1\cdot(-3) = -3\). Multiply routes the gradient scaled by the other input.
  • \(-6\)
    \(-6\) is the forward value of \(e\), not a gradient. The gradient to \(a\) is the incoming grad times \(b\).

Q2. How does an addition node \(d = e + c\) route gradients during backprop?

  • It passes the incoming gradient unchanged to both \(e\) and \(c\).
    Correct. Since \(\partial d/\partial e = \partial d/\partial c = 1\), addition copies the gradient through to each input — the “gradient distributor.”
  • It splits the gradient in half.
    No halving — each input gets the full incoming gradient (local derivative 1), not a divided share.
  • It multiplies the gradients of \(e\) and \(c\).
    Gradients aren’t multiplied together at a node. Addition simply passes the same gradient to each input.
  • It zeros the gradient.
    Addition is fully differentiable with derivative 1; it preserves the gradient, never zeros it.

Q3. Why is backpropagation vastly preferred over estimating each gradient with finite differences?

  • Finite differences are inaccurate.
    They’re actually quite accurate (and great for checking). The decisive issue is cost, not accuracy.
  • Backprop doesn’t need the chain rule.
    Backprop is the chain rule applied systematically — it relies on it entirely.
  • Finite differences can’t handle nonlinear functions.
    They handle nonlinear functions fine. The problem is scaling to many parameters.
  • Backprop gets all gradients in one backward pass; finite differences need one forward pass per parameter.
    Correct. For billions of parameters, per-parameter forward passes are impossible. Backprop computes every gradient in a single sweep — the key to scalable training.

Q4. A value \(b\) feeds into two different parts of the graph. During backprop, its total gradient is…

  • the larger of the two incoming gradients.
    We don’t take a max. The multivariable chain rule sums contributions from all paths.
  • the sum of the gradients arriving from both paths.
    Correct. When a value influences the loss through multiple paths, its gradient accumulates (adds) across them — which is why engines use grad +=.
  • only the gradient from the first path.
    Ignoring the second path gives a wrong (incomplete) gradient. Both paths contribute.
  • zero, because the paths cancel.
    Paths don’t generally cancel; their gradients add. Assuming cancellation would break training.

✍️ Practice

Problem 1 · by hand

Backprop through \(L = (a + b)\times c\) at \(a=1, b=2, c=4\). Find \(\partial L/\partial a\), \(\partial L/\partial b\), \(\partial L/\partial c\).

Show solution

Forward: \(u = a+b = 3\), \(L = u\times c = 12\). Backward from \(\partial L/\partial L = 1\). The multiply: \(\partial L/\partial u = c = 4\) and \(\partial L/\partial c = u = 3\). The add (\(u=a+b\)) passes its gradient through: \(\partial L/\partial a = \partial L/\partial u\cdot 1 = 4\) and \(\partial L/\partial b = 4\). So \(\partial L/\partial a = 4\), \(\partial L/\partial b = 4\), \(\partial L/\partial c = 3\). (Sanity check: nudging \(a\) up by 1 raises \(u\) by 1, which raises \(L\) by \(c=4\). ✓)

Problem 2 · accumulation

Backprop through \(L = (a+b)\times(b+c)\) at \(a=1, b=2, c=3\), paying attention to \(b\) appearing twice. Find \(\partial L/\partial b\).

Show solution

Forward: \(u = a+b = 3\), \(v = b+c = 5\), \(L = u\times v = 15\). Backward: \(\partial L/\partial u = v = 5\), \(\partial L/\partial v = u = 3\). Now \(b\) flows into both \(u\) and \(v\): from the \(u\) path it gets \(\partial L/\partial u\cdot 1 = 5\); from the \(v\) path it gets \(\partial L/\partial v\cdot 1 = 3\). Accumulate them: \(\partial L/\partial b = 5 + 3 = 8\). (Check: nudging \(b\) raises both factors, so its effect is the sum — exactly why we add gradients at shared nodes.)

Problem 3 · code

Extend backprop_manual.py with a step \(f = d\times 2\), treat \(f\) as the loss, and recompute \(\partial f/\partial a\) by hand. Verify with finite differences.

Show solution 
a, b, c = 2.0, -3.0, 10.0
e = a*b; d = e + c; f = d*2          # forward: e=-6, d=4, f=8
df = 1.0
dd = df * 2.0                         # f = d*2  -> df/dd = 2
de = dd * 1.0                         # add passes through
da = de * b                          # multiply: de/da = b = -3
print("df/da =", da)                 # -6
# check:
F = lambda a: ((a*b)+c)*2; h=1e-6
print("check:", round((F(a+h)-F(a-h))/(2*h), 4))   # -6.0 ✓

The extra \(\times 2\) doubles every downstream gradient: \(\partial f/\partial a = -6\) (was \(-3\) for \(d\)). The chain rule simply carries the factor of 2 all the way back — gradients compound multiplicatively through the graph.

Problem 4 · reasoning

In a deep network, the gradient at an early layer is a long product of many local derivatives. Explain how this leads to the “vanishing” and “exploding” gradient problems.

Show solution

By the chain rule (★), the gradient reaching an early layer is the product of the local derivatives of all the layers above it. If each of those local derivatives is consistently smaller than 1 (as happens when sigmoid/tanh saturate — Lesson 3), the product of many small numbers shrinks toward zero: the vanishing gradient problem, where early layers barely learn. If the local derivatives are consistently larger than 1, the product blows up: the exploding gradient problem, causing unstable, NaN-prone training. Both are direct consequences of backprop being a long multiplicative chain — and they motivate the fixes you’ll meet next: better activations (ReLU, whose positive-side derivative is exactly 1), careful initialization, normalization, and (in Track 5) residual connections. Understanding backprop as a product is the key to understanding why deep networks are hard to train.
