Optimizers & the training loop

Every piece is now in your hands: a model (Lesson 3), an autograd engine for gradients (Lesson 2), and optimizers to follow them downhill (Track 2). This lesson assembles them into the loop that trains every neural network on Earth — and uses it to make a network of your own learn.

💡 Concept: four steps, repeated

Training a neural network is a loop of four steps, and you’ve built every one. (1) Forward pass: run the input through the model to get a prediction (Lesson 0). (2) Compute the loss: measure how wrong the prediction is (a negative log-likelihood — Track 1). (3) Backward pass: call .backward() to get the gradient of the loss with respect to every weight (Lesson 2). (4) Optimizer step: nudge each weight downhill along its gradient (Track 2). Then repeat.

⚠️ Zero the gradients every step

Because our engine accumulates gradients with += (Lesson 2), you must reset every parameter’s .grad to 0 before each backward pass — otherwise this step’s gradients pile on top of last step’s and the update is garbage. In our loop that’s for p in model.parameters(): p.grad = 0.0; in PyTorch it’s optimizer.zero_grad(). Same idea, same reason. Forgetting it is one of the most common training bugs in existence.

🛠️ The capstone: train a network with your own autograd

Here is the whole thing — the loop that trains the MLP from Lesson 3, on the two-moons dataset, using gradients from the engine you built in Lesson 2. No PyTorch. No NumPy autograd. Every gradient is computed by your code.

code/track-4/train_moons.py — the training loop
model = MLP(2, [8, 8, 1])                 # 2 inputs -> 8 -> 8 -> 1 output

for epoch in range(80):
    # 1) FORWARD: predict for every point (targets are -1 / +1)
    preds = [model([Value(x[0]), Value(x[1])]) for x in X]
    # 2) LOSS: mean squared error (a Gaussian negative log-likelihood, Track 1)
    loss = sum((p - y)**2 for p, y in zip(preds, Y)) * (1.0/len(X))
    # 3) BACKWARD: zero grads, then backprop through the whole graph
    for p in model.parameters(): p.grad = 0.0
    loss.backward()
    # 4) OPTIMIZER STEP: gradient descent on every parameter (Track 2)
    lr = 0.5 - 0.4*epoch/80               # a simple learning-rate decay
    for p in model.parameters(): p.data -= lr * p.grad

    if epoch % 20 == 0:
        acc = sum((p.data > 0) == (y > 0) for p, y in zip(preds, Y)) / len(X)
        print(f"epoch {epoch:3d}  loss {loss.data:.4f}  acc {acc:.0%}")
output
epoch   0  loss 1.3690  acc 43%
epoch  20  loss 0.2429  acc 90%
epoch  40  loss 0.1730  acc 94%
epoch  60  loss 0.1118  acc 97%
epoch  79  loss 0.0814  acc 99%

From random weights and 43% accuracy (worse than a coin flip on this balanced data) to 99% — by nothing but forward, backward, step, repeat. And what it learned is not a line but a curve, wrapping around the interlocking moons exactly where a linear model would be helpless:

🔑 This loop scales to everything

The exact four-step loop you just ran is, conceptually, how every modern model is trained — ResNets, GPT, AlphaFold, all of them. The differences are scale (billions of parameters, tensors on GPUs), the loss (cross-entropy for classification/language), the optimizer (Adam instead of plain SGD, Track 2), and tricks to stabilize very deep training (normalization, residuals, schedules). But the heartbeat — forward, loss, backward, step — is identical. You now understand training all the way down.

🔑 Key results to remember

Lock these in

  1. The training loop: forward (predict) → loss → backward (backprop) → optimizer step → repeat.
  2. Zero the gradients before each backward pass (our engine accumulates with +=).
  3. One pass over the data is an epoch; each step typically uses a random mini-batch (SGD, Track 2).
  4. An MLP trained this way learns nonlinear decision boundaries — solving problems linear models can’t.
  5. This loop is how all deep models train; scale, loss, and optimizer change, the four steps don’t.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. What is the correct order of the four steps in a training iteration?

  • Backward → forward → loss → step.
    You can’t backprop before computing a forward pass and a loss — there’d be nothing to differentiate.
  • Loss → step → forward → backward.
    The loss needs predictions (a forward pass) first, and the step needs gradients (a backward pass) first. This order is impossible.
  • Forward → loss → backward → optimizer step.
    Correct. Predict, measure error, get gradients, update weights — the only order that makes sense, repeated each iteration.
  • Forward → step → loss → backward.
    You can’t take an optimizer step before computing gradients (the backward pass). The step must come last.

Q2. Why must you zero the gradients before each backward pass?

  • Because gradients accumulate with +=; otherwise this step adds to last step’s.
    Correct. The engine sums gradients (needed for shared nodes). Without zeroing, each step’s gradient piles on the previous, corrupting the update direction.
  • To reset the weights.
    Zeroing .grad clears the gradients, not the weights. The weights (.data) are exactly what we’re trying to keep and improve.
  • To make training deterministic.
    Determinism is a separate concern (random seeds, batch order). Zeroing grads is about correctness of the accumulation.
  • It isn’t necessary.
    It is essential with an accumulating engine — skipping it is a classic bug that silently wrecks training.

Q3. The MLP reached a curved decision boundary on two-moons. What made the curve possible?

  • The learning-rate decay.
    Decay helps convergence but a linear model with any schedule still draws only a line. Curvature comes from elsewhere.
  • The nonlinear activations between layers.
    Correct. Without nonlinearities the network collapses to a single linear map (a line); the tanh activations are what let stacked layers bend the boundary into a curve.
  • Using more training points.
    More data sharpens the boundary but doesn’t grant curvature — a linear model stays linear no matter how much data.
  • The mean-squared-error loss.
    The loss measures error; it doesn’t supply the model’s flexibility. Nonlinearity does.

Q4. How does this scalar training loop relate to training GPT-scale models?

  • It’s unrelated; large models train differently.
    They train with the very same loop conceptually — only the scale and engineering differ.
  • Large models don’t use backpropagation.
    They depend entirely on backpropagation (autograd over enormous graphs). It’s the only feasible way to get their gradients.
  • Large models skip the optimizer step.
    The optimizer step (usually Adam) is central to training them; nothing skips it.
  • Same four steps; the differences are scale, tensors/GPUs, the loss, and the optimizer.
    Correct. Forward → loss → backward → step is identical; GPT just runs it on billions of parameters as tensors on GPUs, with cross-entropy loss and Adam, plus stabilization tricks.

✍️ Practice

Problem 1 · run it

Run train_moons.py. Confirm the loss falls and accuracy rises toward ~99%. Then remove the p.grad = 0.0 line and rerun — what happens, and why?

Show solution

As written, the loss drops from ~1.37 to ~0.08 and accuracy climbs to ~99% over 80 epochs. Remove the gradient-zeroing line and training breaks: gradients accumulate across all epochs, so each step uses an ever-growing, stale gradient sum — the effective step size explodes and the loss diverges (or thrashes) instead of decreasing. This is the concrete, visible consequence of the accumulation rule from Lesson 2, and exactly why every framework forces a zero-grad call each iteration.

Problem 2 · experiment

Change the network from MLP(2, [8, 8, 1]) to MLP(2, [4, 1]) (a single small hidden layer). Does it still separate the moons? What about MLP(2, [1]) (no hidden layer)?

Show solution

MLP(2, [4, 1]) usually still works — even a small hidden layer provides enough nonlinearity to curve the boundary, though accuracy may be a bit lower or training a touch less stable. MLP(2, [1]) is a single neuron — a linear classifier (one tanh of a linear combination), which can only draw a straight boundary and therefore cannot separate the interleaving moons; it plateaus around 80–88% accuracy, misclassifying the points where the moons interlock. This is a hands-on confirmation of the lesson’s core claim: hidden layers + nonlinearity are what buy you curved boundaries.

Problem 3 · reasoning

Our loop uses full-batch gradient descent (all points each step). For a dataset of 10 million examples, why would we switch to mini-batches?

Show solution

Full-batch means computing the loss and gradient over all 10M examples before taking a single step — enormously expensive per update and often impossible to fit in memory. Mini-batch SGD (Track 2) instead estimates the gradient from a small random sample (say 32–512 examples) each step, giving many cheap, frequent updates that make rapid progress. The gradient estimate is noisier, but that noise is often helpful (it escapes shallow traps, Track 2/3), and over an epoch the model sees all the data anyway. Practically, mini-batching is what makes training on large datasets feasible at all — it trades a little gradient accuracy for a massive gain in updates-per-second.

Problem 4 · synthesis

Trace one training step through every track of this curriculum. Which idea from Tracks 1–4 does each part of the loop use?

Show solution

One step touches the entire curriculum. The forward pass is dot products and matrix multiplies (Track 1) through neurons and activations (Track 4, Lessons 0/3). The loss is a negative log-likelihood — squared error under a Gaussian assumption (Track 1, Lesson 12). The backward pass is the chain rule (Track 2) run over the computational graph by your autograd engine (Track 4, Lessons 1/2). The optimizer step is gradient descent, ideally with momentum/Adam (Track 2). And whether the resulting model generalizes — rather than overfitting these exact points — is the domain of bias–variance and regularization (Track 3). Four tracks, one loop. You didn’t just learn the parts; you can now see them working together in a single line of training — which is exactly what it means to understand deep learning from first principles.
