Build an autograd engine

This is the lesson the whole curriculum has been building toward. In about sixty lines of Python you’ll write a real automatic-differentiation engine — the same machinery, in miniature, that powers PyTorch and every model trained today. Once you’ve built it, deep learning stops being magic.

Track 4 · Lesson 2 From-scratch implementation ~32 min Builds on: Lessons 4.0–4.1

Last lesson we ran backprop by hand. That doesn’t scale — a real network has millions of operations. So we’ll automate it: build a Value object that wraps a number, records how it was computed as we go, and knows how to run the backward pass over the whole graph by itself. This is automatic differentiation (autograd).

💡 Concept: a value that remembers

The key idea: instead of computing with bare numbers (which forget their history), we compute with objects that remember. Each Value stores four things: its data (the number), its grad (the gradient of the final loss with respect to it, initially 0), the set of children that produced it, and a little function _backward that knows how to push gradient to those children. As you do arithmetic, each operation builds a new Value and wires up its _backward — so the computational graph assembles itself automatically.

🛠️ Building it, piece by piece

Start with the object and one operation. The pattern for every operation is identical: compute the output value, then define a _backward closure encoding that operation’s local derivative (Lesson 1).

code/track-4/autograd.py — the core
import math

class Value:
    def __init__(self, data, _children=()):
        self.data = data
        self.grad = 0.0                 # gradient of the loss w.r.t. this value
        self._backward = lambda: None   # how to push grad to children
        self._prev = set(_children)     # the Values that produced this one

    def __add__(self, other):
        other = other if isinstance(other, Value) else Value(other)
        out = Value(self.data + other.data, (self, other))
        def _backward():                # add distributes the gradient
            self.grad  += out.grad
            other.grad += out.grad
        out._backward = _backward
        return out

    def __mul__(self, other):
        other = other if isinstance(other, Value) else Value(other)
        out = Value(self.data * other.data, (self, other))
        def _backward():                # multiply scales by the OTHER input
            self.grad  += other.data * out.grad
            other.grad += self.data  * out.grad
        out._backward = _backward
        return out

    def tanh(self):
        t = math.tanh(self.data)
        out = Value(t, (self,))
        def _backward():                # d/dx tanh = 1 - tanh^2
            self.grad += (1 - t*t) * out.grad
        out._backward = _backward
        return out

Notice three things that are exactly Lesson 1 in code. Each operation knows its local derivative (add → 1; multiply → the other value; tanh → \(1-\tanh^2\)). Each uses += so gradients accumulate at shared nodes. And each closure captures the incoming gradient via out.grad, multiplying it by the local derivative — the chain rule, automated.

The backward pass: topological order

Now the engine’s heart. To backprop, we must process nodes in the right order: a node’s gradient can only be pushed to its children after its own gradient is fully accumulated from everything above it. That ordering is a topological sort of the graph (children before parents); we reverse it to go output → inputs.

the backward sweep
    def backward(self):
        topo, visited = [], set()
        def build(v):                   # depth-first topological sort
            if v not in visited:
                visited.add(v)
                for child in v._prev:
                    build(child)
                topo.append(v)          # children get added before parents
        build(self)
        self.grad = 1.0                 # seed: d(output)/d(output) = 1
        for v in reversed(topo):        # outputs -> inputs
            v._backward()               # apply each local rule in order

That’s the entire algorithm. Build the graph by computing, then call .backward() on the final loss to fill in every .grad. With a few more one-line operations (**, relu, exp, and the reverse/negate helpers that let Values mix with plain numbers), you have a complete engine — see the full code/track-4/autograd.py.

🎯 Proving it works

The acid test: do the engine’s gradients match finite differences (Track 2)? And does it handle a real neuron?

running autograd.py
a = Value(2.0); b = Value(-3.0); c = Value(10.0)
d = a*b + c
d.backward()
print(a.grad, b.grad, c.grad)     # -3.0  2.0  1.0   (matches Lesson 1 by hand!)

# finite-difference check on a:  both give -3.0  ✓

# a full tanh neuron, differentiated automatically:
x1, x2 = Value(2.0), Value(0.0)
w1, w2 = Value(-3.0), Value(1.0)
n = x1*w1 + x2*w2 + Value(6.881373587)
o = n.tanh(); o.backward()
print(round(o.data,4), round(x1.grad,4))   # 0.7071  -1.5
🔑 You just built PyTorch (the important part)

This Value class is autograd. PyTorch’s real engine differs only in engineering, not in concept: it operates on whole tensors (arrays) at once instead of single scalars, runs on GPUs, and has hundreds of operations — but every one still records a local backward rule, and loss.backward() still does a reverse topological sweep accumulating gradients. When you later write loss.backward() in PyTorch, you will know exactly what those two words do, because you wrote them yourself.

🔑 Key results to remember

Lock these in

  1. A Value stores data, grad, its children, and a _backward closure — the graph builds itself as you compute.
  2. Every operation defines its output and a local backward rule (add → pass through; multiply → scale by the other; tanh → \(1-\tanh^2\)), using grad += to accumulate.
  3. backward() = topological sort (children before parents), seed output grad = 1, then call each _backward in reverse.
  4. The gradients match finite differences — the engine is provably correct.
  5. This is PyTorch’s autograd in miniature; the real one adds tensors, GPUs, and scale, not new ideas.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. Why does each operation use self.grad += ... rather than self.grad = ...?

  • To make the code shorter.
    It’s not about brevity — using = would be just as short but wrong.
  • To avoid floating-point errors.
    Accumulation isn’t a numerical-stability trick; it’s about correctness when a value has multiple consumers.
  • Because gradients are always positive.
    Gradients can be negative (we saw \(-3\)). The reason for += is unrelated to sign.
  • A value used in several places receives gradient from each path; they must be summed.
    Correct. The multivariable chain rule sums contributions from every path a value feeds into. Overwriting with = would keep only the last path and corrupt the gradient.

Q2. Why must backward() process nodes in (reverse) topological order?

  • To save memory.
    Ordering is about correctness, not memory. Any order uses similar memory.
  • A node’s gradient must be fully accumulated before it’s pushed to its children.
    Correct. If you propagated a node’s gradient before all its parents had contributed, the children would get an incomplete (wrong) gradient. Topological order guarantees parents finish first.
  • So the forward values are computed correctly.
    Forward values are already computed before backward() runs; ordering here is purely about the gradient sweep.
  • Python requires it.
    Python imposes no such requirement; it’s a mathematical necessity of the chain rule over a graph.

Q3. What value do we seed at the output before the backward sweep?

  • 1, since \(\partial L/\partial L = 1\).
    Correct. The gradient of the loss with respect to itself is 1; this seed is what the rest of the gradients chain off of.
  • 0, so nothing has gradient yet.
    Seeding 0 would make every gradient stay 0 — the chain rule would multiply everything by zero. The output must start at 1.
  • The value of the loss.
    We seed the gradient, not the data. \(\partial L/\partial L = 1\) regardless of the loss’s numeric value.
  • The learning rate.
    The learning rate belongs to the optimizer step, not the backward pass. The seed is the output’s self-gradient, 1.

Q4. How does this scalar engine relate to PyTorch’s autograd?

  • They’re completely different algorithms.
    They’re the same algorithm — reverse-mode autodiff with per-operation local rules and a topological backward sweep.
  • PyTorch doesn’t use backpropagation.
    PyTorch is built entirely on backpropagation; loss.backward() is exactly this sweep.
  • Same core idea; PyTorch adds tensors, GPUs, and many operations.
    Correct. The concept is identical — record local backward rules, sweep in reverse topological order — PyTorch just operates on whole arrays, on GPUs, at scale.
  • PyTorch computes gradients with finite differences.
    No — finite differences are too slow at scale. PyTorch uses autograd (this very mechanism), which is why it’s fast.

✍️ Practice

Problem 1 · extend the engine

Add a .sigmoid() method to the Value class. (Recall \(\sigma(x)=1/(1+e^{-x})\) and \(\sigma'(x)=\sigma(x)(1-\sigma(x))\).) Gradient-check it against finite differences.

Show solution 
def sigmoid(self):
    s = 1 / (1 + math.exp(-self.data))
    out = Value(s, (self,))
    def _backward():
        self.grad += s * (1 - s) * out.grad   # local derivative
    out._backward = _backward
    return out

x = Value(0.5); y = x.sigmoid(); y.backward()
print(x.grad)                                  # sigma'(0.5) ~ 0.2350
# finite diff: ((sig(0.5+h)-sig(0.5-h))/(2h)) ~ 0.2350  ✓

The pattern is mechanical: compute the output, then write self.grad += (local derivative) * out.grad. Once you internalize this template, you can add any differentiable operation to the engine in three lines — which is exactly how PyTorch’s op library grew.

Problem 2 · use the engine

Use the Value engine to compute the gradient of \(L = a^2 + 3b\) at \(a=4, b=2\). What should \(\partial L/\partial a\) and \(\partial L/\partial b\) be?

Show solution 
a = Value(4.0); b = Value(2.0)
L = a**2 + 3*b
L.backward()
print(a.grad, b.grad)        # 8.0  3.0

Analytically \(\partial L/\partial a = 2a = 8\) and \(\partial L/\partial b = 3\), matching the engine. (This needs the __pow__ and __rmul__ helpers from the full file, so a**2 and 3*b work.) You’re now differentiating expressions you never wrote a derivative for by hand — the engine does it.

Problem 3 · reasoning

If you call .backward() twice on the same graph without resetting gradients, what happens, and why?

Show solution

The gradients double (more precisely, accumulate again). Because every _backward uses +=, a second backward pass adds a second copy of every gradient on top of the first. This is the same reason PyTorch makes you call optimizer.zero_grad() (or model.zero_grad()) before each backward pass — to clear the accumulators. Forgetting to zero gradients is one of the most common real-world training bugs, and now you understand precisely why it happens: the += that makes shared-node accumulation correct also means gradients persist until you reset them.

Problem 4 · synthesis

Explain how this lesson unites Track 1, Track 2, and Lesson 4.1 into one working system.

Show solution

The engine is the confluence of everything so far. The forward computations are dot products and sums (Track 1 linear algebra). The local backward rules are derivatives — the power rule, the chain rule, \(\frac{d}{dx}\tanh\) (Track 2 calculus). The backward sweep is the chain rule applied over a graph (Lesson 4.1). And the purpose is to feed gradients to gradient descent so the model can minimize its loss, which is maximum likelihood (Track 1 again). In ~60 lines, the entire mathematical arc of the curriculum becomes a runnable machine that can train any differentiable model. That is why this lesson is the pivot of the whole journey: from here on, building bigger models (MLPs next lesson, transformers in Track 5) is mostly about what graph to build — the engine differentiates it for free.
