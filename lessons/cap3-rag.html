Capstone 3: a RAG assistant over your docs

⏱ 120 min Lesson 14.4 ▶ Open in Colab
Why this existsBuild a RAG assistant that answers from your own documents — the most-deployed LLM pattern. Index offline (chunk, embed, store); serve online (retrieve, generate grounded and cited). Integrates most of the curriculum.
By the end, you can
  • Build the offline index and online retrieve-generate loop
  • Focus on retrieval quality
  • Debug retrieval vs generation failures
  • See RAG as an integration project

Concept The project: an assistant that answers from your knowledge

This capstone builds the most-deployed LLM pattern in production: a RAG assistant that answers questions grounded in your documents — a company knowledge base, a set of manuals, your own notes, any corpus. It's an exceptional portfolio project because RAG is exactly what a huge fraction of real LLM applications are, and building one end to end demonstrates you can turn a pile of documents into a working, grounded question-answering system. As the previous capstone established, when the need is knowledge (answering from specific, current documents), RAG is the right tool, not fine-tuning.

The architecture has the two phases you know from Track 8 and Track 11. Offline indexing (built once, updated when documents change): load the documents, chunk them into passages, embed each chunk into a vector, and store the vectors in a vector database. Online serving (per query): embed the user's question, retrieve the most similar chunks from the vector store, assemble a prompt with those chunks as context plus an instruction to answer only from them (and cite sources), and have the LLM generate the grounded answer. The result is an assistant that answers from your actual documents — with citations, current knowledge, and far less hallucination than an ungrounded model.

Worked example A minimal RAG pipeline

The heart of RAG — retrieve relevant context, then ground the answer in it — is buildable in a few lines. Here we index three documents, retrieve the most relevant one for a query (using TF-IDF similarity as a stand-in for neural embeddings), and assemble the grounded prompt:

The query "how many days do I have for a refund?" correctly retrieves document [1] — the refund policy — even though the wording differs, because similarity matching finds the relevant chunk by content. In the full assistant, you'd then assemble a prompt like "Answer using only this context. Context: [1] Our refund policy allows returns within 30 days. Question: how many days do I have for a refund? Answer:" and send it to an LLM, which would respond "30 days," grounded in and citing the retrieved document. This tiny example is the whole RAG pattern: retrieve the relevant context, then generate grounded in it. The real project scales each piece up — you'd use neural embeddings (which match by meaning far better than TF-IDF, handling synonyms and paraphrase), a proper vector database (for fast search over many documents), chunking that splits your docs sensibly, likely reranking and hybrid search for retrieval quality (Track 8), and a real LLM for generation with a prompt that instructs it to answer only from context, cite sources, and say "I don't know" when the context lacks the answer. But the core loop is exactly this: embed and index offline, then per query retrieve-then-generate. Building it end to end — with real embeddings, a vector store, and an LLM — and evaluating its retrieval and faithfulness (Track 12) is a complete, impressive, deployable project that mirrors what countless production systems do.

★ Key points to remember
  • The project: a RAG assistant that answers questions grounded in your own documents — the most-deployed LLM pattern.
  • Offline: chunk → embed → store in a vector database (built once, updated as docs change).
  • Online: embed the query → retrieve relevant chunks → generate an answer grounded in them, with citations.
  • Grounding in retrieved context gives current, private, citable knowledge and far less hallucination.
  • Real builds add neural embeddings, a vector DB, chunking, reranking/hybrid search, and retrieval + faithfulness evaluation.

Practice Build and reason about it

1Plan this RAG assistant end to end. What are the components, and where would you focus effort for quality?
Show a strong plan

"Setup (Capstone 0): the reproducible foundation. Ingest & chunk: load my documents, handle their formats, and chunk them sensibly (by paragraph/section, with some overlap, keeping metadata for citations). Embed & index: embed each chunk with a retrieval embedding model and store the vectors in a vector database. Retrieve: embed the query, do nearest-neighbor search — ideally hybrid (semantic + keyword) so exact terms aren't missed — and rerank the top candidates for precision. Generate: assemble a prompt with the top chunks, citation tags, and an instruction to answer only from the context and say 'I don't know' if it's absent; call the LLM. Evaluate — where I'd focus: build a small test set of question–answer pairs, and measure retrieval (recall@k — are the right chunks fetched?) separately from generation (faithfulness — is the answer grounded?), because most RAG failures are retrieval failures. Serve & polish: a simple interface, and ideally cache common queries."

Where I'd focus effort: retrieval quality, because it's the ceiling on the whole system — the LLM can only answer from what's retrieved, so if the right chunk isn't fetched, no amount of prompting fixes it. That means investing in chunking (getting passages that contain answers), embeddings (matching meaning), hybrid search (catching exact terms), and reranking (surfacing the best chunks) — and measuring recall@k to know it's working. The reasoning skill is recognizing that RAG is a pipeline whose quality is bounded by its weakest link, usually retrieval, so you build and evaluate each stage (with the retrieval-vs-generation split from Track 12) rather than just wiring an LLM to a vector store and hoping. Being able to explain this — the components, the offline/online split, and especially why retrieval quality is where you focus and how you'd measure it — is exactly the depth that makes this a standout project and a rich interview deep-dive.

2Your RAG assistant gives a confidently wrong answer. How do you debug it?
Show reasoning

"I'd separate retrieval from generation, because in RAG the failure is usually retrieval, and inspect the retrieved chunks for that query. Did the relevant chunk get retrieved? If not, it's a retrieval failure — I'd check chunking (is the answer split across chunks or diluted?), the embedding model (is it matching meaning?), and add hybrid search (for exact terms it's missing) or reranking; I'd quantify with recall@k on a test set. If the relevant chunk was retrieved but the answer is still wrong, it's a generation/faithfulness problem — the model ignored or misread the context. I'd strengthen the prompt ('answer ONLY from the context; say you don't know if it's absent'), rerank so the best chunk is prominent, remove distracting chunks, or use a stronger model, and measure faithfulness. The discipline is to never guess: look at exactly what was retrieved, and let recall@k versus faithfulness tell me which half to fix."

This is the essential RAG debugging methodology, and it demonstrates real hands-on understanding. The key insight is that 'the assistant gave a wrong answer' has two very different causes — the right information wasn't retrieved, or it was retrieved but the model didn't use it — and they have completely different fixes, so the first move is always to inspect the retrieved chunks and localize the failure. Most RAG problems turn out to be retrieval problems, which is why 'look at what was retrieved first' is the highest-value diagnostic. Being able to walk through this — the retrieval-vs-generation split, the specific checks for each, and the metrics (recall@k, faithfulness) that pinpoint the cause — is exactly what distinguishes someone who has actually built and debugged a RAG system from someone who has only wired one up, and it's a frequent, revealing interview question."

★ Deep dive: from a demo to a production-grade RAG assistant▸

The gap between a RAG demo and a production-grade RAG assistant is where the real engineering — and the most impressive version of this project — lives. A demo wires an LLM to a vector store and works on easy questions. A production-grade system adds the layers that make it reliable: retrieval quality (hybrid search combining semantic and keyword so exact terms aren't missed, reranking with a cross-encoder to surface the best chunks, and careful chunking that keeps answers intact), rigorous evaluation (a maintained test set, retrieval metrics like recall@k separate from generation metrics like faithfulness, so you can measure and improve each stage rather than guess), grounding discipline (prompts that force answering only from context and abstaining when the answer is absent, plus citations so answers are verifiable), freshness (incremental indexing so new or changed documents become searchable promptly), and cost/latency management (controlling how many chunks you retrieve since they're LLM input tokens, caching common queries, and choosing model sizes — Track 11). Building even a few of these beyond the basic pipeline is what turns your capstone from a tutorial clone into a demonstration of production judgment, and it's exactly the material that makes for a standout interview deep-dive.

This capstone also sits at the center of what modern applied AI work actually is. RAG has become the dominant pattern for building LLM applications over private or current knowledge — which is most enterprise LLM use — so being able to design, build, evaluate, and reason about a RAG system is one of the most directly employable skills you can demonstrate. And it integrates the whole curriculum: it's the clearest example of how the separate topics you learned — embeddings, retrieval, LLMs, prompting, evaluation, serving, safety — come together into a coherent, valuable system, which is the entire point of the capstones. As you build it, hold onto the two habits that make RAG (and most ML systems) succeed: retrieval quality is the ceiling, so invest and measure there first; and evaluate each stage separately, so you improve deliberately rather than guessing. Those habits, plus the ability to explain how the system draws on everything you've learned and what you'd do to make it production-grade, turn this project into powerful evidence that you can build the kind of real, grounded, evaluated AI application that teams urgently need — which is precisely what makes it, for many candidates, the single most valuable project in the portfolio.
