Track 9 · MLOps & Serving

Data & feature pipelines, feature stores

⏱ 25 min Lesson 9.3 ▶ Open in Colab
Why this existsModels eat features, not raw data, and the pipeline that computes them is fragile and critical. Train/serve skew — features computed differently at train vs serve time — silently wrecks accuracy; a feature store computes them once for both.
By the end, you can
  • Explain feature pipelines
  • Define train/serve skew and its danger
  • Fix skew with one shared definition
  • Describe a feature store (offline/online)

Concept Models are only as good as the data plumbing

A model doesn't eat raw data — it eats features: the processed numerical inputs computed from raw data (Lesson 2.x). "Raw data" might be a database row like {amount: 100, day: "Sat"}; a feature is something the model can use, like amount_log = 4.6 or is_weekend = True. The code that turns raw data into features, repeatedly and reliably, is a data pipeline (or feature pipeline): an automated sequence of steps that ingests raw data, cleans and transforms it, and produces features ready for training or prediction. In production this pipeline runs constantly, and if it breaks or drifts, the model silently gets garbage — which is why "models are only as good as their data plumbing" is an MLOps proverb.

Concept Train/serve skew: the silent killer

Here's a subtle, devastating bug. Suppose at training time you compute a feature one way (say, average purchase over 30 days using a pandas script), and at serving time — live, in production — a different engineer computes "the same" feature a slightly different way (using a SQL query that rounds differently or handles missing values differently). Now the model was trained on features that don't quite match the features it sees in production. This mismatch is called train/serve skew, and it quietly wrecks accuracy: the model performs great in testing and mysteriously worse in production, because it's effectively seeing inputs it was never trained on. It's one of the most common and hardest-to-diagnose production ML failures, precisely because nothing crashes — the numbers are just subtly wrong.

The fix is to compute each feature in exactly one place, with one definition, used by both training and serving. That's the central job of a feature store: a system that computes, stores, and serves features consistently across training and production. A feature store has an offline side (large historical feature tables for training) and an online side (fast lookups of current feature values for live prediction), both fed by the same feature definitions — so the average-purchase feature is computed once and the model gets the identical value whether it's learning from history or scoring a live request. Feature stores also let teams reuse features across models and avoid recomputing them, but preventing skew is the headline benefit.

Worked example One feature definition, no skew

The antidote to train/serve skew is a single feature function called in both places. Here the same features function is applied at training time and serving time; because it's literally the same code, the features are guaranteed identical.

Because both training and serving call the same features function, the same raw input produces byte-for-byte identical features — amount_log = 4.615, is_weekend = True — and the final check prints identical: True. This looks almost too obvious to state, and that's the point: the discipline of a single shared definition is the whole solution. Train/serve skew creeps in precisely when this obvious rule is broken — when the training pipeline and the serving pipeline are written separately by different people in different languages (pandas for training, SQL or Java for serving) and slowly diverge. A feature store enforces the rule at the infrastructure level: there is one canonical definition of amount_log, computed once, and both training and serving read from it, so they cannot disagree. The toy function here is doing in code what a feature store does as a system — guaranteeing that the model learns and predicts on the same features.

★ Key points to remember
  • A data/feature pipeline turns raw data into the features a model consumes — automatically and repeatedly.
  • Train/serve skew = features computed differently at training vs serving time; it silently wrecks production accuracy and nothing crashes.
  • The fix: compute each feature in one place, with one definition, used by both training and serving.
  • A feature store does this — an offline side for training tables and an online side for fast live lookups, fed by the same definitions.
  • Bonus: feature stores let teams reuse features across models and avoid recomputation.

Practice Build the reasoning yourself

1A fraud model scores 0.94 AUC in offline testing but only catches a fraction of the fraud it should in production. The team swears the model is identical. What should you suspect first, and how would you check?
Show solution & reasoning

Suspect train/serve skew first. The tell is exactly this pattern — excellent offline, poor in production, model unchanged — which points not at the model but at the features it's being fed in production differing from those it trained on. The likely cause: the training features were computed by one pipeline (say a pandas notebook over historical data) and the serving features by a different one (a real-time service in another language), and they diverge on something subtle — a different default for missing values, a different time window, a rounding or time-zone difference, or a feature that's computed on stale data at serving time.

To check, log the actual feature values the model receives in production and compare them, for the same entities, against the features computed by the training pipeline. If avg_purchase_30d is 220.5 offline but 198.0 online for the same customer, you've found the skew. This logging-and-comparison is the standard diagnostic, and it's why mature teams emit the served features to a log precisely so this comparison is possible. The fix is to unify the computation — ideally route both training and serving through a single feature store or shared feature library so the divergence can't recur. The broader lesson: when a model is "great in test, bad in prod" and the model truly hasn't changed, the data plumbing is the prime suspect, not the model.

2Beyond preventing skew, why might a company with many models invest in a shared feature store?
Show solution & reasoning

The second big payoff is reuse and efficiency. In a company with dozens of models, the same features recur constantly — "customer's average spend," "days since last login," "item popularity" are useful to a recommendation model, a churn model, and a fraud model alike. Without a feature store, each team recomputes these from scratch, duplicating effort, wasting compute, and — worse — computing them slightly differently, so the same conceptual feature has five inconsistent implementations across the company. A shared feature store lets one team define a feature once, and every other team consumes the identical, already-computed feature.

This compounds into several wins: less duplicated work (define once, reuse everywhere), consistency across models (everyone's "average spend" means the same thing), lower compute cost (compute the feature once, not per-team), and faster model development (a new model can pull from a catalog of ready features instead of building a pipeline from zero). It essentially turns features into a shared, governed asset — like a well-organized library of reusable components — rather than something each project rebuilds. The skew-prevention and the reuse benefits reinforce each other: the single source of truth that prevents skew is the same single source that enables reuse. For a one-model startup a feature store is overkill; for an organization running many models on overlapping data, it becomes core infrastructure.

3Why is the data pipeline often considered the most important — and most fragile — part of a production ML system?
Show solution & reasoning

Most important because everything downstream depends on it: the model can only be as good as the features it's fed, so a flawless model on broken features produces broken predictions. The pipeline is the foundation the entire system stands on, which is why practitioners say "garbage in, garbage out" — and in ML the "garbage in" is usually a data-pipeline problem, not a modeling one. Most real-world ML quality problems trace back to data issues (missing values, schema changes, skew, stale data) rather than to the model architecture.

Most fragile because the pipeline depends on the outside world, which changes without warning. An upstream team renames a database column, a third-party data source changes its format, a sensor starts reporting in different units, a logging bug drops 5% of events — any of these can silently corrupt the features while the model code and the pipeline code are both unchanged. Unlike a logic bug you can catch with a unit test, these are data failures that only show up when bad data flows through. This is why production ML invests so heavily in data validation (checking incoming data against expected schemas, ranges, and distributions before it reaches the model) and in monitoring data quality as a first-class signal (Lesson 9.7). The fragility is inherent: you're building on data you don't fully control, so the engineering response is constant vigilance — validate aggressively, monitor continuously, and treat any data anomaly as a potential model failure. The pipeline is where the messy real world meets your clean model, and that boundary is always the most dangerous part of the system.

★ Deep dive: batch vs streaming features, and point-in-time correctness▸

Feature pipelines come in two flavors by freshness. Batch pipelines recompute features on a schedule (say, nightly) — fine for slow-changing features like "customer's lifetime spend," which doesn't need to be up-to-the-second. Streaming pipelines compute features in real time as events arrive — necessary for fast-changing features like "number of transactions in the last 5 minutes," which a fraud model needs now, not from last night's batch. Many systems run both: a batch layer for historical and slow features, a streaming layer for fresh ones, with the feature store unifying them so the model just asks for "the current value" and gets the right freshness behind the scenes. Choosing batch vs streaming per feature is a real design decision, trading infrastructure complexity (streaming is much harder to build and operate) against how fresh the feature truly needs to be.

The subtlest issue in feature engineering for ML is point-in-time correctness, also called avoiding data leakage through time. When you build a training set from history, each training example's features must reflect only what was known at that moment in the past — not information from after the event you're predicting. For example, if you're training a model to predict whether a customer churns in month N, the feature "customer's total complaints" must be computed as of the start of month N, using only complaints up to that point; if you accidentally include complaints from month N+1 (which you'd have in your historical table today), the model "cheats" by peeking at the future, scores beautifully offline, and fails in production where the future isn't available. This is one of the nastiest bugs in applied ML because, like skew, it inflates offline metrics and only reveals itself in production — and it's easy to introduce when naively joining historical tables. A good feature store handles point-in-time joins correctly, reconstructing each feature's value as of the relevant timestamp. Understanding leakage-through-time, along with train/serve skew, marks the difference between someone who can train a model on a clean dataset and someone who can build a feature pipeline that actually works in production — the latter being far rarer and exactly what this track is teaching. Both bugs share a moral: in production ML, the data plumbing deserves at least as much rigor as the model, because that's where the failures that matter most actually hide.
