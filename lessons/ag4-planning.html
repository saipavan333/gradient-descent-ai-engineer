Track 8 · AI Agents & Tool Use

Planning & task decomposition

⏱ 25 min Lesson 8.4 ⬇ Runnable notebook
Why this existsComplex goals need more than reacting step to step. Planning has the agent draft a strategy first; reflection has it check and correct its own work — the levers that let agents handle long, multi-step tasks.
By the end, you can
  • Contrast planning with pure reaction
  • Explain plan-then-execute
  • Describe reflection and self-correction
  • Know when planning is worth the cost

Concept Giving an agent direction: planning

A purely reactive agent (ReAct, Lesson 8.3) decides one step at a time, which is flexible but can wander, lose the thread on a complex goal, or get stuck in loops. Planning adds direction: before diving in, the agent breaks the goal into a sequence of smaller, ordered subtasks — a plan — and then executes them. Decomposition turns an overwhelming open-ended goal ("write a research report on electric vehicles") into a manageable checklist (research → outline → draft each section → review), which the agent can work through systematically.

The two styles trade off. Reactive (ReAct) adapts freely to what it finds — great for tasks where the path is genuinely unpredictable, but prone to losing direction. Plan-and-execute commits to a plan first, giving structure and a clear sense of progress, but is less flexible if reality diverges from the plan. Many real systems combine them: make a high-level plan, then execute each step reactively (with ReAct), and re-plan if a step reveals the original plan was wrong. The plan is a scaffold, not a straitjacket.

Concept Reflection: catching your own mistakes

The other major addition is reflection (self-correction): after attempting a task or step, the agent critiques its own work — "did that actually work? is this correct?" — and revises if not. Because an agent's first attempt is often imperfect, having it review and retry catches errors that would otherwise propagate.

Reflection is powerful because models are often better at recognizing a mistake than avoiding it the first time — asking "is this code correct? does it handle edge cases?" can surface bugs the initial generation missed. A coding agent that writes code, runs the tests, sees a failure, and fixes it is using reflection grounded in real feedback (the test results). Self-critique plus retry is one of the highest-leverage reliability techniques for agents — and like ReAct, it's most effective when the critique is grounded in real signals (test results, validation errors) rather than the model just second-guessing itself in the abstract.

Worked example Decompose a goal, then execute

Here's the plan-and-execute skeleton: the planner breaks a goal into ordered steps, then the executor works through them. (The planner is a one-liner here; a real agent would prompt the LLM to generate the plan.)

goal = "Write a short blog post about coffee"

# PLAN: the model decomposes the goal into ordered subtasks
plan = ["research coffee history", "outline the post", "write the draft", "proofread & polish"]

print("GOAL:", goal)
for i, step in enumerate(plan, 1):
    print(f"  plan step {i}: {step}")

# EXECUTE: work through the plan in order (each step would itself use tools / ReAct)
print("executing:")
for step in plan:
    print(f"  [done] {step}")
print("post complete.")
OutputGOAL: Write a short blog post about coffee plan step 1: research coffee history plan step 2: outline the post plan step 3: write the draft plan step 4: proofread & polish post complete.

The agent turned one vague goal into four concrete, ordered steps, then executed them in sequence. Notice the structure this provides: each step is a smaller, well-defined task the agent can tackle (and each could itself be a ReAct loop using tools — research uses web search, proofread uses the model's own review). The plan gives direction (the agent always knows what's next and how much remains) and makes progress visible, which a purely reactive agent lacks. In a fuller system, you'd add reflection: after "write the draft," the agent critiques it ("is it clear? on-topic? the right length?") and revises before moving on; and if a step fails or reveals a problem, the agent re-plans the remaining steps. So the complete pattern is: plan → execute each step (reactively, with tools) → reflect and revise → re-plan if needed. Even this trivial example shows why decomposition helps — "write a blog post" is daunting as one action, but "research, then outline, then draft, then polish" is a clear path the agent can follow and check off.

★ Key points to remember
  • Planning decomposes a goal into ordered subtasks, giving a reactive agent direction and visible progress.
  • Reactive (ReAct) adapts step-by-step; plan-and-execute commits to a plan first — many systems combine them and re-plan as needed.
  • Reflection (self-critique + retry) catches mistakes the first attempt missed — a major reliability boost.
  • Models are often better at recognizing errors than avoiding them, which is why reflection helps.
  • Reflection works best grounded in real signals (test results, validation), not abstract self-doubt.

Practice Build the reasoning yourself

1Decompose this goal into a plan an agent could execute: "Find the three highest-rated Italian restaurants near me that are open now and make a reservation at one."
Show solution & reasoning

A reasonable plan: (1) get the user's location; (2) search for Italian restaurants nearby; (3) filter to those open now and sort by rating; (4) take the top three and present them; (5) ask the user which one (or apply a preference); (6) make the reservation via a booking tool; (7) confirm back to the user.

Notice several things: the steps are ordered with dependencies (you can't filter before searching, can't book before choosing), some steps use tools (search, booking API), one step involves the user (choosing — a human-in-the-loop checkpoint, Lesson 8.7, especially wise before an irreversible action like booking), and the plan makes the daunting goal tractable. A reactive agent could stumble through this, but the plan gives direction and ensures it doesn't, say, try to book before checking hours. And if step 2 finds no open restaurants, the agent would re-plan (widen the radius, suggest alternatives). This is exactly the kind of multi-step, tool-using, partly-irreversible task where planning plus reflection plus guardrails earn their keep.

2A coding agent writes a function, and the tests fail. Describe how reflection grounded in the test results helps, versus the agent just "trying again."
Show solution & reasoning

Grounded reflection: the agent reads the actual test failure ("AssertionError: expected 6, got 5 for input [1,2,3]") as feedback, reasons about why ("off-by-one — I'm not including the last element"), and makes a targeted fix. The real signal (the failing test) tells it precisely what's wrong, so the revision is directed, not random. This is reflection at its best — critique grounded in concrete evidence.

Just "trying again" without reading the error would have the agent regenerate code blindly, possibly reintroducing the same bug or a new one, with no information about what was wrong — essentially guessing. The difference is the feedback signal: tests, error messages, and validation results turn vague "is this right?" into specific "here's exactly what failed and why," which the model is very good at acting on. This is why coding agents are among the most successful agent applications — code provides immediate, precise, automatic feedback (does it compile? do the tests pass?), making the reflect-and-fix loop tight and reliable. Whenever you can give an agent a real verification signal, reflection becomes dramatically more powerful.

3When would you choose a reactive (ReAct) agent over a plan-and-execute one, and vice versa?
Show solution & reasoning

Reactive (ReAct): when the path is genuinely unpredictable and depends heavily on what's discovered — exploratory research, debugging where each finding changes the next step, open-ended investigation. A rigid upfront plan would be useless because you can't know the steps in advance. Plan-and-execute: when the task has a recognizable structure that benefits from upfront organization — a multi-part report, a known multi-step procedure, a goal with clear sub-goals. The plan keeps the agent on track and prevents wandering.

In practice, the best systems often combine them: make a high-level plan for direction, execute each step reactively (with ReAct, since each step's details are unpredictable), and re-plan when a step reveals the plan was wrong. So it's less "either/or" and more "plan for structure, react for adaptability, re-plan for robustness." The judgment is about how much the path can be known in advance: more predictable structure favors planning; more uncertainty favors reactivity; most real tasks have both, so you blend. This mirrors the autonomy-spectrum thinking from Lesson 8.1 — match the approach to the task's actual nature.

★ Deep dive: planning approaches, reflection patterns, and the reliability frontier▸

Planning for agents spans a range of sophistication. The simplest is decompose-then-execute: prompt the model to list the steps, then run them — easy and surprisingly effective for well-structured tasks. More advanced is plan-and-solve with dynamic re-planning, where the agent revises the remaining plan after each step based on results. At the high end are search-based methods like Tree of Thoughts, which explore multiple possible plans/reasoning paths, evaluate them, and pursue the most promising — trading much more compute for better solutions on hard problems. There's also hierarchical planning: a high-level plan whose steps are themselves decomposed into sub-plans, useful for very complex goals. Which to use depends on task difficulty and your compute budget; for most applications, decompose-then-execute with re-planning hits the sweet spot, and the elaborate search methods are reserved for genuinely hard reasoning where their cost is justified.

Reflection, too, comes in flavors. Self-refinement has the model critique and improve its own output in a loop (helpful but limited — the model's blind spots can persist, since it's grading its own work). Tool-grounded reflection (run the tests, validate the output, check the result) is far more reliable because the critique rests on objective signals rather than the model's self-assessment. Reflexion maintains a memory of past failures and lessons across attempts, so the agent learns within a task. The key principle — echoing ReAct (Lesson 8.3) — is that feedback grounded in reality beats introspection: an agent that can actually check its work (compile the code, query the database to verify, re-read the source) corrects itself reliably, while one that only ponders "am I sure?" is on shakier ground. This is why the most successful agentic systems are in domains with cheap, automatic verification: coding (tests, compilers), math (checkers), and tasks where actions have observable, checkable outcomes.

Stepping back, planning and reflection are the field's primary answers to the compounding-error problem from Lesson 8.1. Planning reduces wandering (fewer wasted, error-prone steps); reflection catches errors before they propagate (so a mistake at step 3 doesn't doom steps 4–10); and re-planning recovers when things go off track. Even so, reliable long-horizon autonomy remains hard — the more steps, the more chances to fail, and no amount of planning or reflection fully eliminates the compounding. This is why the practical state of the art favors bounded agents: constrained scope, limited steps, human checkpoints for high-stakes actions, and verification wherever possible (the next lessons on memory, coordination, and guardrails build out this reliability toolkit). Planning gives agents foresight and reflection gives them hindsight — together they're much of what turns a wandering, error-prone loop into a directed, self-correcting one. But the honest frontier is that we're still learning how to make agents reliable over long, complex, high-stakes tasks, which is exactly why the remaining lessons focus relentlessly on reliability — the difference between an impressive demo and a system you can trust.

Plans and reflections live in the agent's context — but that context is limited and forgotten between sessions. Next, in 8.5 Memory (short & long-term), we give agents the ability to remember across steps and across conversations.
