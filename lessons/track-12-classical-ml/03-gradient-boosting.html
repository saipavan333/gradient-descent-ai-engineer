Gradient boosting

Random forests build trees in parallel and average them. Boosting builds them one at a time, each fixing the last one’s mistakes. Done right, it’s gradient descent in function space — and the reigning champion of tabular machine learning.

Track 12 · Lesson 3 Concept + code ~26 min Builds on: Lessons 1–2, Track 2

Bagging (Lesson 2) reduces variance by averaging independent trees. Boosting takes the opposite tack: build trees sequentially, each one trained to correct the errors the ensemble has made so far. The remarkable insight is that “correct the errors” means “follow the negative gradient of the loss” — so boosting is literally gradient descent, but stepping in the space of functions. XGBoost, LightGBM, and CatBoost — the tools that win most tabular competitions — are all gradient boosting.

💡 Concept: fit the residual, again and again

Start with a trivial prediction (the mean). Look at the residuals — how far off each prediction is. Train a small “weak” tree to predict those residuals, and add a shrunken version of it to the ensemble. Now the predictions are a little better; compute the new residuals and repeat. Each stage nudges the ensemble toward the data:

\[ F_{m}(x) = F_{m-1}(x) + \eta\, h_m(x), \qquad h_m \approx \text{residual } r = y - F_{m-1}(x), \] (3.1)

where \(\eta\) is the learning rate (shrinkage) and \(h_m\) is the weak learner (a shallow tree, here a one-split “stump”). The key connection: for squared-error loss \(L=\tfrac12(y-F)^2\), the negative gradient with respect to the prediction \(F\) is exactly the residual, \(-\partial L/\partial F = y-F\). So fitting the residual is fitting the negative gradient — gradient descent (Track 2), performed in function space one weak learner at a time. That’s why it’s called gradient boosting.

🛠️ From scratch: gradient boosting in a loop

The whole algorithm is Equation (3.1) in a loop, from gradient_boosting.py: 

pred = mean(y)                          # F_0: the simplest possible model
for stage in range(n_stages):
    resid = y - pred                    # = negative gradient of ½(y - pred)²
    h = fit_stump(x, resid)             # weak learner fits the residual
    pred = pred + lr * h(x)             # add a shrunken step (learning rate lr)
Boosting can overfit — forests can’t (much). This is the crucial difference from Lesson 2. Adding trees to a forest only refines an average and never hurts. Adding stages to a booster keeps driving down training error and will eventually fit noise — too many stages overfit. The defenses: a small learning rate \(\eta\) (each step is gentle), shallow weak learners, and stopping early when validation loss bottoms out (Track 9). Boosting trades forest-style safety for higher accuracy when tuned carefully.

🔑 Key results to remember

  • Boosting is sequential: each weak learner fits the current residual; \(F_m = F_{m-1} + \eta\,h_m\).
  • Residual = negative gradient (for squared error), so boosting is gradient descent in function space.
  • Weak learners (shallow trees/stumps) + a small learning rate \(\eta\) = gentle, accurate steps.
  • Too many stages overfit — unlike bagging; use early stopping and shrinkage.
  • Gradient boosting (XGBoost/LightGBM) is the state of the art for tabular data.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. How does boosting differ from bagging (random forests)?

  • Boosting trains all trees independently in parallel.
    That’s bagging. Boosting is sequential — each tree depends on the previous ones’ errors.
  • Boosting builds trees sequentially, each correcting the current ensemble’s residuals.
    Correct. Where bagging averages independent trees, boosting adds them one at a time, each fitting what the ensemble still gets wrong.
  • Boosting uses distance, not trees.
    Boosting uses weak learners (usually trees); distance is k-NN.
  • Boosting never uses a learning rate.
    Shrinkage (a learning rate) is central to boosting, controlling step size.

Q2. Why is it called gradient boosting?

  • Because the trees have gradients.
    Trees aren’t differentiable in the usual sense; the gradient is of the loss w.r.t. predictions.
  • Because it uses gradient descent to train each tree’s splits.
    Splits are chosen by impurity/error search, not gradient descent. The “gradient” is elsewhere.
  • Because it needs a GPU.
    Hardware is unrelated to the name.
  • Because fitting the residual equals fitting the negative gradient of the loss — gradient descent in function space.
    Correct. For squared error, \(-\partial L/\partial F = y - F = \) the residual, so each weak learner takes a gradient-descent step in the space of functions.

Q3. Adding more stages to a booster, unlike adding trees to a forest, can…

  • eventually overfit, driving training error down while test error rises.
    Correct. Boosting keeps reducing training loss stage by stage, so past a point it fits noise — hence early stopping and shrinkage. A forest, by contrast, just refines an average and plateaus.
  • only ever help.
    That’s true of bagging, not boosting. Extra boosting stages can hurt generalization.
  • reduce the number of features.
    Stages don’t change the feature set.
  • make it non-sequential.
    Boosting stays sequential regardless of stage count.

Q4. What is the role of a small learning rate \(\eta\) (shrinkage) in boosting?

  • It removes the need for weak learners.
    Weak learners are still essential; \(\eta\) scales their contribution.
  • It makes each tree deeper.
    Depth is separate; \(\eta\) shrinks each stage’s step, and boosters usually keep trees shallow.
  • It makes each stage a gentle step, so many small corrections generalize better than a few big ones.
    Correct. Small \(\eta\) means no single stump dominates; the ensemble improves slowly and smoothly, which — with enough stages and early stopping — generalizes better than large, greedy steps.
  • It guarantees zero test error.
    Nothing guarantees zero test error; \(\eta\) is a regularizer, not a magic bullet.

✍️ Practice

1. In gradient_boosting.py, raise the learning rate to 1.0 and lower it to 0.05. How do the fit and the RMSE curve change?

Show solution

With \(\eta=1.0\), the ensemble drops error fast in the first few stages but can become jumpy and start fitting noise sooner (higher variance). With \(\eta=0.05\), each stage barely moves the prediction, so error falls slowly and you need many more stages to reach a good fit — but the final fit is typically smoother and generalizes better. This is the classic boosting trade-off: a small learning rate plus many stages usually beats a large rate plus few, at the cost of compute. Tuning \(\eta\) against the number of stages (with early stopping on validation) is the central knob of gradient boosting.

2. Track training vs a held-out test RMSE across stages. Where should you stop, and why does training RMSE keep falling past that point?

Show solution

You should stop where test (validation) RMSE bottoms out — beyond that, additional stages keep fitting training-set noise, so training RMSE continues down while test RMSE turns up (overfitting). This “early stopping” is the standard way to pick the number of boosting rounds and directly applies the honest-evaluation discipline of Track 9: the training curve alone is misleading because a booster can always reduce it further. Real libraries automate this with a validation set and a patience parameter.

3. Conceptual: gradient boosting reduces bias by adding corrections, random forests reduce variance by averaging. When would you reach for each?

Show solution

Reach for a random forest as a strong, low-effort baseline: it’s robust, needs almost no tuning, resists overfitting, and gives OOB estimates and feature importance — ideal when you want a reliable answer fast. Reach for gradient boosting when you want the last few points of accuracy and are willing to tune (learning rate, depth, stages, regularization) with careful validation — it usually wins tabular benchmarks but is easier to overfit and slower to tune. In practice many workflows start with a forest to gauge signal, then switch to a tuned booster (XGBoost/LightGBM) for the final model — using exactly the model-selection rigor from Track 9.
