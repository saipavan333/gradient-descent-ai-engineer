Decision trees

A model that thinks in questions: “is this feature above a threshold?”, again and again, splitting the data into ever-purer groups. Trees are interpretable on their own — and the building block of the two most powerful classical models.

Track 12 · Lesson 1 Concept + code ~24 min Builds on: Lesson 0, Track 3

A decision tree is the flowchart you’d draw by hand: a cascade of yes/no questions that narrows down to an answer. Each question splits the data to make the resulting groups purer — more dominated by one class. Trees are readable, need no feature scaling, and handle mixed data — but their real importance is as the atom of random forests and gradient boosting, the next two lessons.

💡 Concept: split to reduce impurity

At each node the tree asks “is feature \(j\) below threshold \(t\)?”, sending points left or right. To choose the best split, we need to measure how mixed a group of labels is. The Gini impurity of a node with class proportions \(p_c\) is

\[ G = 1 - \sum_c p_c^2, \] (1.1)

which is \(0\) for a pure node (all one class) and largest when classes are evenly mixed. The tree tries every feature and threshold and picks the split that most reduces the weighted impurity of the two children, then recurses on each side — a greedy, top-down partition. It stops at a maximum depth (or when a node is pure), and each leaf predicts its majority class.

🛠️ From scratch: CART, and the overfitting knob

Our decision_tree.py implements CART: at each node, search features × thresholds for the split with the lowest weighted Gini, recurse to a maximum depth. The core of the search: 

for j in features:                                  # each feature
    for t in candidate_thresholds(X[:, j]):         # each threshold
        left, right = y[X[:, j] < t], y[X[:, j] >= t]
        impurity = (len(left)*gini(left) + len(right)*gini(right)) / len(y)
        keep the (feature, threshold) with the lowest impurity
The tree’s blessing and curse. A single tree is wonderfully interpretable — you can read the exact rules it used — and needs no scaling. But it is a high-variance learner: change a few training points and the greedy splits can cascade into a very different tree. That instability is precisely what makes trees perfect raw material for ensembling: average many unstable trees and the variance cancels, which is where their real power lives.

🔑 Key results to remember

  • A tree splits on “feature \(j\) < threshold \(t\)?”, greedily choosing the split that most reduces impurity, and recurses.
  • Gini impurity \(G = 1-\sum_c p_c^2\): 0 = pure, max = evenly mixed. (Entropy is an alternative, from Track 13.)
  • Axis-aligned splits ⇒ rectangular decision regions.
  • Depth is the capacity knob: deep trees overfit (train↑, test↓); prune or limit depth.
  • Interpretable but high-variance — which makes trees the ideal atom for ensembles.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. What does Gini impurity measure at a node?

  • The depth of the node.
    Depth is structural; Gini is about the label mix within the node.
  • How mixed the class labels are — 0 if pure, larger if evenly split.
    Correct. \(G=1-\sum p_c^2\) is zero when all points share a class and maximal when classes are balanced; the tree splits to reduce it.
  • The distance between two points.
    That’s k-NN’s metric; trees use impurity, not distance.
  • The number of features.
    Gini depends on class proportions, not feature count.

Q2. Why are a decision tree’s decision regions rectangular?

  • Because it uses Euclidean distance.
    Trees don’t use distance at all; the shape comes from how they split.
  • Because of the Gini formula.
    Gini chooses where to split, not the geometric shape of the regions.
  • Because it standardizes features.
    Trees don’t need standardization, and scaling wouldn’t create rectangles.
  • Because each split thresholds a single feature (axis-aligned), so regions are bounded by axis-parallel cuts.
    Correct. A split “\(x_j < t\)” is a cut perpendicular to axis \(j\); stacking such cuts partitions space into axis-aligned boxes.

Q3. As you increase a single tree’s max depth, train accuracy rises toward 100% but test accuracy peaks then drops. This is…

  • overfitting — the tree memorizes training noise with ever-finer rectangles.
    Correct. Deep trees carve tiny regions around individual points, fitting noise; that lifts train accuracy but hurts generalization — the bias–variance story from Track 3.
  • underfitting.
    Underfitting is the shallow-tree end (both accuracies low). Here train is high and test falls — that’s overfitting.
  • a bug in the impurity calculation.
    It’s expected behavior of unconstrained trees, not a bug.
  • the curse of dimensionality.
    That concerns many features/distances; this is about capacity (depth), not dimensionality.

Q4. Why is a single tree’s instability (high variance) actually useful?

  • It makes the tree faster.
    Instability isn’t about speed; it’s about how much the tree changes with the data.
  • It improves interpretability.
    Instability doesn’t aid interpretability; a single stable tree is easiest to read.
  • Averaging many different unstable trees cancels their variance, which is how forests and boosting gain power.
    Correct. If individual trees vary a lot but are roughly unbiased, averaging their predictions cancels the noise (variance reduction) — the principle behind random forests (next lesson).
  • It guarantees zero training error.
    A deep tree can hit zero train error, but that’s overfitting, not the reason instability helps ensembles.

✍️ Practice

1. Run decision_tree.py. At which depth does test accuracy peak, and what does training accuracy do beyond that point?

Show solution

Test accuracy climbs then peaks around depth 7–8 and declines slightly as depth grows further, while training accuracy keeps rising toward ~0.97–1.0. The gap between them widening is the visual signature of overfitting: the extra depth buys fit on the training set that doesn’t transfer. In practice you’d cap the depth (or set a minimum samples-per-leaf, or prune) at the point that maximizes validation accuracy — the same model-selection discipline from Track 9, now controlling a tree instead of a neural net.

2. Swap Gini for entropy \(H = -\sum_c p_c\log p_c\) as the split criterion. Does the tree change much?

Show solution

Usually very little. Gini and entropy are both maximized at even mixtures and minimized at purity, and they agree on most splits; entropy (from Track 13) penalizes impurity slightly differently but rarely changes the resulting tree meaningfully. The choice is mostly convention and a tiny bit of speed (Gini avoids a log). The deeper point: what matters is having a sensible purity measure to greedily optimize, not the exact one — a reminder not to over-tune details that don’t move the result (Track 9).

3. Conceptual: a tree can perfectly separate the training data yet a linear model can’t. Does that make trees “better”?

Show solution

No — fitting training data perfectly is easy and usually bad (overfitting). A deep tree’s power to shatter the training set is exactly why an unconstrained tree generalizes poorly, while a simpler linear model, though it can’t fit everything, may generalize better on truly linear problems (lower variance). “Better” always means better on unseen data for your problem, judged by validation — not raw flexibility. The right model depends on the data’s structure; the value of a broad toolbox is being able to match model to problem rather than forcing one everywhere.
