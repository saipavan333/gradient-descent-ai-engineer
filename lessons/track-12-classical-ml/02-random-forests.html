Random forests

One tree overfits. So grow a whole forest of them — each on a different random slice of the data — and let them vote. Averaging many unstable trees cancels their errors, giving one of the most reliable off-the-shelf models in all of ML.

Track 12 · Lesson 2 Concept + code ~24 min Builds on: Lesson 1, Track 9

A single deep tree is a high-variance learner — it fits noise. The fix is beautifully simple: build many trees, each on a randomized version of the problem, and average their predictions. Because each tree’s errors are somewhat independent, averaging cancels them — the same variance-reduction principle you used to average seeds in Track 9, now applied to whole models. The result, the random forest, is accurate, robust, and famously hard to misuse.

💡 Concept: bagging + random features

Two sources of randomness make the trees differ. First, bagging (bootstrap aggregating): each tree is trained on a bootstrap sample — a dataset of the same size drawn with replacement from the original, so each tree sees a slightly different set of points. Second, feature randomness: at each split, a tree may only choose from a random subset of features. Together these decorrelate the trees, which is what makes averaging them powerful.

Why does averaging help? If each tree has roughly the same (low) bias but high, largely independent variance, averaging \(B\) of them cuts the variance by up to a factor of \(B\) while leaving the bias unchanged. The trees individually overfit in different directions; their consensus does not.

🛠️ From scratch: a forest beats a deep tree

Our random_forest.py reuses the Lesson 1 tree, adding bootstrap sampling and per-split feature subsampling: 

for b in range(n_trees):
    idx = rng.integers(0, n, n)                 # bootstrap: sample rows WITH replacement
    tree = fit_tree(X[idx], y[idx], max_depth=8, max_features=1)   # random feature per split
    trees.append(tree)

def forest_predict(trees, X):
    votes = [predict(t, X) for t in trees]
    return majority_vote(votes)                 # the ensemble's consensus
A free validation set: out-of-bag. Because each bootstrap sample leaves out ~37% of the data, every point is “out-of-bag” for the trees that didn’t train on it. Averaging those trees’ predictions on their out-of-bag points gives an honest accuracy estimate for free, with no separate hold-out — a neat, practical bonus of bagging that ties straight back to the honest-evaluation discipline of Track 9.

🔑 Key results to remember

  • Random forest = many trees, averaged. Two randomness sources: bootstrap samples (bagging) + random feature subsets per split.
  • Averaging decorrelated trees cuts variance while keeping bias low — turning an overfitting learner into a robust one.
  • Smoother boundary, better generalization than one deep tree, with little tuning.
  • Out-of-bag error gives a free, honest performance estimate.
  • A strong default for tabular data — hard to misconfigure, and it reports feature importance.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. A “bootstrap sample” used to train each tree is…

  • a random subset of the features.
    That’s the feature-randomness component. A bootstrap sample resamples rows, not features.
  • the first half of the data.
    It’s a random resample, not a fixed split.
  • a dataset the same size as the original, drawn with replacement (so some rows repeat, others are absent).
    Correct. Sampling \(n\) rows with replacement from \(n\) rows makes each tree see a slightly different dataset — the core of bagging.
  • the data with noise added.
    No noise is injected; the variety comes from resampling existing rows.

Q2. Why does averaging many trees reduce error?

  • Their individual (largely independent) variances cancel when averaged, while bias stays low.
    Correct. Roughly unbiased but high-variance predictors that err independently average toward the truth — variance drops by up to a factor of the number of trees, bias unchanged.
  • Each tree becomes more biased.
    Averaging doesn’t change bias; it targets variance.
  • The trees all make the same prediction.
    If they were identical, averaging would do nothing — the point is that they differ (decorrelation).
  • It increases training accuracy to 100%.
    The benefit is on test error via variance reduction, not maximizing train accuracy.

Q3. Choosing splits from a random subset of features (not all) is meant to…

  • make each tree more accurate individually.
    It usually makes each tree slightly weaker individually — but that’s fine, because the goal is the ensemble.
  • speed up prediction only.
    There’s a minor speed effect, but the purpose is statistical, not computational.
  • reduce the number of trees needed.
    It doesn’t reduce tree count; it changes how the trees relate to each other.
  • decorrelate the trees, so their errors are more independent and average better.
    Correct. If every tree could use the one dominant feature, they’d look alike; forcing random feature choices makes trees diverse, which is what makes averaging effective.

Q4. What is “out-of-bag” error?

  • The error on a purchased external test set.
    No external data is involved; OOB comes from the bootstrap process itself.
  • An accuracy estimate from evaluating each point using only the trees that didn’t train on it.
    Correct. Each point is left out of ~37% of bootstrap samples; predicting it with just those trees gives an unbiased performance estimate for free, no separate hold-out needed.
  • The training error of the forest.
    OOB is specifically a held-out-style estimate, not training error.
  • The error before averaging.
    OOB is about which trees see which points, not about a pre-averaging quantity.

✍️ Practice

1. Run random_forest.py and increase the number of trees: 1, 5, 25, 100, 500. How does test accuracy behave, and does it ever get worse with more trees?

Show solution

Test accuracy rises quickly then plateaus — most of the variance reduction comes in the first few dozen trees, with diminishing returns after. Crucially, adding trees does not cause overfitting: more trees only refine the average, so accuracy flattens rather than declining. This is a hallmark of bagging (unlike boosting, next lesson, where too many stages can overfit). Practically, you pick “enough” trees for the accuracy plateau and stop — extra trees just cost compute, not generalization.

2. Set max_features to use all features at every split (i.e. plain bagging, no feature randomness). Does the forest get better or worse, and why?

Show solution

It usually gets slightly worse (or no better), because without feature randomness every tree tends to pick the same strong feature at the top and the trees become correlated — they make similar mistakes, so averaging cancels less variance. Feature subsampling deliberately weakens and diversifies individual trees to strengthen the ensemble. This is the counterintuitive heart of random forests: making each learner a bit worse and more different makes the whole better — a lesson that recurs in ensembling.

3. Conceptual: random forests and deep neural networks both fight overfitting, but differently. Contrast their strategies.

Show solution

A random forest reduces variance by averaging many diverse high-variance models — it embraces overfitting in each tree and cancels it across the ensemble. A deep network instead trains one large model and controls variance through regularization (weight decay, dropout, early stopping — Track 4) and huge data. Forests need almost no tuning and give feature importance and OOB estimates for free, which is why they dominate small/medium tabular problems; networks scale to raw high-dimensional data (images, text) where hand-built features fail. Different medicines for the same disease — and knowing both lets you match the tool to the data.
