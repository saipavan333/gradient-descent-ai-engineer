Track 9 · MLOps & Serving

Scaling inference

⏱ 26 min Lesson 9.6 ▶ Open in Colab
Why this existsOne server can't serve millions. Horizontal scaling (replicas + load balancer + autoscaling) handles volume; caching, batching, quantization, and better hardware cut per-prediction cost — the economics that decide whether an AI product is viable.
By the end, you can
  • Explain horizontal scaling and autoscaling
  • Use caching, batching, quantization
  • Reason about the accuracy/cost trade-off
  • Know why batching suits GPUs

Concept One server can't serve a million users

A single packaged model behind a single server (Lesson 9.5) works fine in a demo. But real traffic — thousands or millions of requests, with spikes at peak hours — will overwhelm one server: requests queue up, latency balloons, and eventually it falls over. Scaling inference is the practice of serving predictions reliably and affordably under heavy, variable load. There are two broad strategies: run more copies of the model (scaling out), and make each prediction cheaper (optimization). Production serving uses both.

The primary tool for handling volume is horizontal scaling (scaling out): run many identical copies of the model — called replicas — and put a load balancer in front of them. A load balancer is a component that receives all incoming requests and distributes them across the replicas (e.g., round-robin: first request to replica A, next to B, next to C, and around again), so the work is spread out and no single replica is overwhelmed. Need to handle 3× the traffic? Run 3× the replicas. Because the replicas are identical and stateless (each just runs the model), you can add or remove them freely.

The companion idea is autoscaling: automatically adjusting the number of replicas based on current load. When traffic surges, the system spins up more replicas; when it drops (overnight), it removes them. This matters because traffic is uneven — provisioning enough servers for peak traffic and running them 24/7 wastes money during quiet hours, while provisioning for average traffic means crashing at peak. Autoscaling gives you enough capacity exactly when you need it and not a server more, which is the key to serving reliably and affordably.

Concept Making each prediction cheaper

Scaling out handles load by adding hardware, but you also want each prediction to cost less, which lets fewer replicas do more. Four common levers: batching — group several incoming requests and run them through the model together in one pass, which is far more efficient on GPUs (one batched pass beats many separate ones); caching — store the results of recent or common inputs and return the saved answer instead of re-running the model when the same input recurs; quantization and model compression — shrink the model (e.g., store weights at lower numerical precision, or distill it into a smaller one, Lesson 6.x) so it runs faster and cheaper, usually with minimal accuracy loss; and better hardware — GPUs and specialized accelerators run inference much faster than CPUs.

Worked example Caching cuts the work

Caching is one of the simplest, highest-leverage optimizations. Here a cache stores each input's result; when an input repeats, we return the saved answer instead of re-running the (costly) model. Count how many times the model actually runs.

cache = {}
calls = 0
def predict(x):
    global calls
    if x in cache: return cache[x]       # cache hit: skip the model
    calls += 1                            # cache miss: run the (costly) model
    cache[x] = x * x
    return cache[x]

requests = [3, 3, 5, 3, 5, 7]
results = [predict(x) for x in requests]
print("results   :", results)
print("model runs:", calls, "for", len(requests), "requests")
Outputresults : [9, 9, 25, 9, 25, 49] model runs: 3 for 6 requests

Six requests came in, but the model only ran 3 times — once each for the distinct inputs 3, 5, and 7. The repeated requests (the second and fourth 3, the second 5) were cache hits: served instantly from the stored result without touching the model. We did half the work for the same output. The results list is identical to what you'd get with no cache — caching changes cost, not correctness. In production this is enormously valuable whenever inputs repeat, which they often do (popular search queries, the same product viewed by many users, common inputs): a cache can absorb a large fraction of traffic, so your replicas only compute the genuinely novel requests. The trade-off to watch is staleness — if the right answer for an input can change over time, a cached result can go out of date, so caches use expiry rules. But for the many cases where the same input deserves the same answer, caching is close to free throughput. It pairs with the other levers: batch the cache misses, run them on a quantized model, on a GPU, behind a load balancer across autoscaled replicas — and a modest fleet serves enormous traffic.

★ Key points to remember
  • Horizontal scaling = run many identical model replicas behind a load balancer that spreads requests across them.
  • Autoscaling adjusts the replica count with load — enough capacity at peak, no waste off-peak (reliable and affordable).
  • Make each prediction cheaper with batching (group requests), caching (reuse results), quantization/compression (smaller model), and better hardware (GPUs).
  • Caching changes cost, not correctness — but watch staleness when answers can change over time.
  • Serving runs forever, so these optimizations compound into the dominant cost saving.

Practice Build the reasoning yourself

1Your model serves fine on weekday mornings but times out during a flash sale when traffic spikes 10×. Walk through how you'd make serving survive the spike.
Show solution & reasoning

The core fix is horizontal scaling with autoscaling. Put the model behind a load balancer with a fleet of replicas, and configure autoscaling to add replicas automatically when load rises (measured by request rate, latency, or CPU/GPU utilization). When the flash sale hits and traffic jumps 10×, the autoscaler spins up roughly 10× the replicas, the load balancer spreads requests across them, and per-replica load stays manageable — no timeouts. When the sale ends, replicas scale back down so you're not paying for idle capacity. This is exactly the problem autoscaling exists to solve: large, unpredictable spikes.

Layer in optimizations to make the spike cheaper to absorb. Caching helps a lot during a flash sale, because many users hit the same few sale items — those repeated requests become cache hits, so your replicas only compute distinct inputs. Batching lets each replica process grouped requests more efficiently under load. You'd also set sensible limits (a queue with a max, graceful degradation — e.g., fall back to a cheap default recommendation if the model is saturated) so that if the spike exceeds even the scaled capacity, the system degrades gracefully rather than collapsing. The reasoning pattern is: scale out to add capacity, autoscale to match the spike's timing, and optimize (cache, batch) to reduce the per-request cost so each replica goes further — and add a safety valve for the worst case. This combination is how real services survive events like Black Friday.

2Your team's large model is accurate but too slow and expensive to serve at scale. Besides adding more servers, what options reduce the per-prediction cost, and what's the trade-off?
Show solution & reasoning

The options are the model-cheapening levers. Quantization: store and compute the model's weights at lower numerical precision (e.g., 8-bit integers instead of 32-bit floats), which shrinks memory and speeds up inference substantially, usually with only a small accuracy drop. Distillation/compression: train a smaller "student" model to mimic the large one (Lesson 6.x), getting much of the accuracy at a fraction of the size and cost. Caching for repeated inputs and batching for efficiency. And better hardware (GPUs/accelerators) if you're on CPUs. Often you combine several: a distilled, quantized model, batched, behind a cache.

The central trade-off is accuracy vs cost/latency. Quantization and distillation typically cost a little accuracy in exchange for big speed and cost wins — and the right question is whether that small accuracy loss is acceptable for the use case. Frequently it is: a model that's 0.5% less accurate but 4× cheaper and faster is a clear win for most production systems, because the slightly-better model that's too slow or expensive to serve at scale is useless. This is a recurring theme — the best offline model is often not the best model to serve, because production imposes latency and cost budgets that offline evaluation ignores (Lesson 9.5). The mature move is to treat the accuracy/cost trade-off as a deliberate design decision: measure how much accuracy you'd lose from compression, weigh it against the serving savings, and pick the point that meets your latency and budget constraints. Reflexively serving the biggest, most accurate model is a common and expensive mistake.

3Why does batching requests together make inference more efficient, especially on a GPU?
Show solution & reasoning

Because a GPU is a massively parallel processor — it has thousands of cores designed to do many arithmetic operations simultaneously. Running a single input through a model uses only a sliver of that parallel capacity; most of the GPU sits idle while it processes one example. Inference is dominated by matrix multiplications (Track 1), and a GPU can multiply a batch of inputs against the model's weights almost as fast as a single input, because it packs the batch into one big parallel matrix operation that fills the cores. So processing 32 requests as one batch can take nearly the same wall-clock time as processing 1 — yielding roughly 32× the throughput from the same hardware in the same time.

There's also fixed overhead per model call (moving data to the GPU, launching the computation, post-processing) that gets amortized across a batch: pay it once for 32 inputs instead of 32 times. The trade-off is a small latency cost — to form a batch, the server waits a few milliseconds to collect several requests before running them, so an individual request may wait slightly longer than if it ran alone. This is the latency-vs-throughput tension again: batching boosts throughput (and lowers cost per prediction) at the price of a little added latency per request. Systems tune the batch size and the max wait time to balance the two — large batches for throughput-oriented workloads, small batches or short waits when latency is critical. Understanding why the GPU favors batches (its parallelism wants to be filled) is what lets you reason about this knob rather than just turning it blindly, and it's the same hardware fact that made GPUs central to deep learning in the first place (Track 4).

★ Deep dive: the cost of LLM serving and why it reshapes everything▸

Scaling inference has become one of the defining problems of modern AI because large language models are expensive to serve in ways classical models never were. An LLM has billions of parameters, so a single response requires billions of arithmetic operations, and — because it generates text one token at a time, each token depending on all previous ones — it can't produce a long answer in one shot; it runs the model once per generated token. This makes LLM serving costly and latency-sensitive in a distinctive way, and it's why a whole subfield of inference optimization has exploded around it. Techniques you'll hear about include KV caching (storing the model's intermediate attention computations for the tokens generated so far, so each new token doesn't recompute them — a form of caching specific to transformers), continuous batching (dynamically batching token-generation steps across many concurrent requests, since different requests are at different points in their generation), quantization (running the model at 8-bit or 4-bit precision to cut memory and speed up inference), and speculative decoding (using a small fast model to draft several tokens and a large model to verify them in one pass). All of these are the same levers from this lesson — caching, batching, compression — specialized and intensified for the brutal economics of LLM inference.

The strategic point is that inference cost shapes what products are viable. When each LLM call costs real money and adds real latency, the cost/latency/quality trade-off stops being a backend detail and becomes a core product constraint: it dictates whether you can afford to call the model on every keystroke or only on submit, whether you use the big model or a distilled one, how much you cache, and even what features you can offer at what price. This is why serving and scaling are not a sideshow to the "real" ML of training models — for deployed AI, and especially for LLMs, the economics of inference are often the deciding factor in whether a system succeeds. It also closes a loop with the rest of this track: a model is trained once but served forever (Lesson 9.5), so the cumulative inference cost dominates, and the engineering discipline of making that inference cheap, fast, and scalable is where much of the value — and much of the hardest systems work — in applied AI now lives. Mastering these trade-offs is increasingly what separates teams that can ship AI products people can actually afford to use from teams whose impressive demos never survive contact with a budget. With serving scaled, the remaining question is keeping the deployed system healthy over time — which is monitoring, next.

A scaled, fast service can still silently go wrong. Next, 9.7 Monitoring & observability covers watching both the system's health and the model's quality in production — so you catch failures (and silent decay) before your users do.
