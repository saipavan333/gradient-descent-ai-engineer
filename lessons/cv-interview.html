Track 5 · Computer Vision

Interview check: Computer Vision

⏱ 35 min Lesson 5.8
Why this existsConsolidates the whole track for interviews: place any problem on the task ladder (classify/detect/segment/generate), justify CNN vs ViT from your data, and explain diffusion and CLIP in plain terms — the connected understanding interviewers actually probe for.
By the end, you can
  • Choose classification, detection, or segmentation for a task and justify it
  • Reason about CNN vs ViT trade-offs from dataset size and pretraining
  • Explain in plain terms how diffusion generates images and how CLIP enables zero-shot
  • Connect vision to transfer learning, the transformer, and imbalance-aware evaluation

Concept What a computer-vision interview tests

A computer-vision interview is less about memorizing architectures and more about whether you can match a task to the right tool and justify the trade-offs. Interviewers want to see that you can place a problem on the task ladder (classify → detect → segment → generate), reason about CNN-vs-transformer choices from the data you have, explain how modern generative and multimodal systems actually work in plain terms, and — throughout — connect it back to the fundamentals from earlier tracks (transfer learning, attention, evaluation under imbalance). The signal they're after is connected understanding: that vision isn't a bag of separate models but a hierarchy of tasks built on shared features, now increasingly unified with language through the transformer. The model answers below show that mindset — each one names the core idea, reasons about when it applies, and ties it to the wider picture.

Model answer "Classification, detection, segmentation — what's the difference and when do you use each?"

"They're a hierarchy of increasing spatial granularity. Classification answers 'what is in this image?' — one label for the whole image, no location. I'd use it for 'is there a defect on this part?' or 'what breed is this dog?' Detection answers 'what objects are where?' — a bounding box plus a class per object. I'd use it to find and count things: cars in a lot, people in a frame, products on a shelf. Segmentation answers 'what is each pixel?' — a per-pixel class label, giving exact shapes. I'd use it when I need precise boundaries or areas: the exact extent of a tumor, the drivable road surface, or a 'select the person' mask for photo editing. The key judgment is to use the coarsest task that meets the need, because granularity costs money — pixel masks are far more expensive to label than boxes, which are more expensive than class labels, plus more compute and complexity. So I don't reach for segmentation when a class label answers the question. They also build on each other: detectors and segmentation models use a classification backbone as their feature extractor, then add machinery for localization or dense prediction."

Strong close: it's one ladder over shared features — pick the least-granular rung that answers the actual question.

Model answer "CNN or Vision Transformer — how do you choose?"

"It comes down to inductive bias versus data. A CNN builds in strong, correct assumptions about images — locality (nearby pixels relate) and translation invariance (an object is the same wherever it appears) — so it learns efficiently from modest data. A Vision Transformer builds in almost none of that: it splits the image into patches, treats each as a token, and lets self-attention relate every patch to every other, which gives it a global view from the first layer and lets it scale beautifully — but it has to learn image structure from data, so it's data-hungry. So my rule of thumb: on small-to-medium datasets with no pretraining, use a CNN — its priors are a gift when data is scarce. On large datasets or with large-scale pretraining, a ViT can win and keep improving with scale, because it assumed less and can learn a better solution than the hard-coded one. In practice I'd rarely train either from scratch — I'd start from a pretrained backbone (CNN or ViT) and fine-tune, which sidesteps the small-data problem for both. And honestly the modern answer is often a hybrid — designs like Swin reintroduce CNN-like locality into transformers — so it's less a war than a convergence, chosen by your data regime and compute budget."

Model answer "How does a diffusion model generate an image from a text prompt?"

"The core idea is to learn to reverse noise. During training, you take real images and repeatedly add a little noise until they become pure static — that forward process needs no learning. Then you train a network to do the opposite: given a noisy image, predict the noise so it can be subtracted, recovering a slightly cleaner image. That's a stable, well-posed regression, which is a big reason diffusion beat GANs — GANs are fast but unstable and prone to mode collapse, while diffusion trains reliably and gives high, diverse quality. To generate, you start from pure random noise and apply that learned denoising step hundreds of times, and a brand-new image emerges. For text-to -image, you condition each denoising step on the prompt — the encoded text (often from a CLIP text encoder) is fed in so denoising is steered toward images matching the words. And to make it fast, Stable Diffusion uses latent diffusion: an autoencoder compresses the image to a small latent, all the denoising runs in that cheap space, and a decoder expands the result back to pixels. So: encode to a latent, denoise many steps while conditioning on the prompt, decode to an image. The denoiser itself is usually a U-Net — the same encoder-decoder-with-skips as segmentation, because denoising is also a dense, full-resolution image-to-image task."

Model answer "What is CLIP, and why was it important?"

"CLIP aligns images and text in one shared embedding space. It trains an image encoder and a text encoder together with contrastive learning — on hundreds of millions of web image-caption pairs, it pulls each image close to its true caption and pushes it away from the other captions in the batch. The result is that a picture and its description land at nearly the same point, so you can compare vision and language directly by vector similarity, with no manual labels — the caption is the supervision. Why it mattered: it unlocked zero-shot classification. To classify an image into categories you never trained on, you just write each category as text, embed the candidates, and pick the one closest to the image's vector — a working classifier for arbitrary, described-in-words categories with zero labeled examples. That was a paradigm shift from 'train a fixed-class model on labeled data.' Beyond that, CLIP became a building block everywhere: it guides diffusion models' text conditioning, powers multimodal search and retrieval, and its image encoder seeds modern vision-language models. The one caveat I'd raise is that web-scale training is double-edged — the same breadth that gives zero-shot generality also absorbs the web's biases and raises data-consent questions, so CLIP-based systems need bias evaluation before deployment."

Model answer "You have 2,000 labeled images for a new classification task. How do you get the best model?"

"Two thousand images is far too few to train a strong vision model from scratch, so my whole strategy is transfer learning. I'd take a model pretrained on a large dataset like ImageNet — a CNN such as ResNet/EfficientNet, or a pretrained ViT — and fine-tune it on my 2,000 images. The pretrained backbone already learned general visual features (edges, textures, shapes, parts) that transfer, so I only need my small set to adapt the final layers to my specific classes. I'd pair that with aggressive data augmentation — random flips, crops, rotations, color shifts — to stretch the limited data and improve generalization essentially for free. I'd be careful about overfitting given the small set: watch the validation curve, use early stopping and regularization, and possibly freeze early layers and only fine-tune later ones. I'd also check class balance and evaluate with the right metric — if classes are imbalanced I'd look at per-class precision/recall, not just accuracy (the Track 12 discipline). And if labels are the real bottleneck, I'd consider a CLIP zero-shot baseline first to see how far describing the classes in words gets me before investing more labeling. The theme is: with little data, never start from scratch — stand on pretrained features and make the small data go far."

Strong close: small data is a transfer-learning problem — pretrained backbone + augmentation + overfitting care beats any from-scratch model.

★ Key points to remember
  • The task ladder: classify (whole-image label) → detect (box + class per object) → segment (per-pixel mask) → generate (diffusion). Use the coarsest task that meets the need — granularity costs labeling, compute, and complexity.
  • CNN vs ViT = inductive bias vs data: CNNs (strong priors) win on small data; ViTs (weak priors, global attention) win at scale or with pretraining. Prefer pretrained backbones + fine-tuning either way.
  • Diffusion generates by learning to reverse noise: predict-the-noise regression (stable, unlike GANs); generate from random noise; condition on text; latent diffusion makes it cheap. The denoiser is a U-Net.
  • CLIP aligns image and text in a shared space via contrastive learning, enabling zero-shot classification and seeding diffusion conditioning and modern VLMs — but inherits web-scale biases.
  • Everything connects: transfer learning throughout, the transformer as a universal token processor (patches→tokens, image tokens→LLM), and evaluation under imbalance from Track 12.

Practice Rehearse out loud

1"Design a system to detect manufacturing defects on a production line, given only a few hundred labeled defect images." Talk through your approach.
Show model approach & reasoning

"First I'd frame the task. If I only need 'defective vs not,' that's classification. If I need to locate the defect, that's detection; if I need its exact extent/area, segmentation. I'd pick the coarsest that meets the requirement — for a pass/fail line, classification; if operators need to see where, detection. Second, the defining constraint is few hundred labels, so my whole plan is transfer learning: start from a pretrained backbone and fine-tune, never train from scratch. Third, heavy augmentation — flips, rotations, crops, brightness/contrast shifts that mimic real lighting variation — to stretch the tiny dataset. Fourth, defects are usually rare, so I'd expect severe class imbalance and evaluate with precision/recall on the defect class, not accuracy — a model calling everything 'good' would score high while catching nothing (the accuracy paradox). I'd tune the decision threshold to the cost trade-off: missing a defect (false negative) is usually far worse than a false alarm, so I'd favor recall. Fifth, because defects can be things I've never seen, I'd consider framing it partly as anomaly detection — learn what 'normal' looks like and flag deviations — since I can't label every possible defect. I'd deploy with monitoring and a human-in-the-loop for flagged items, and plan to keep collecting labeled examples of new defect types to improve over time."

The reasoning being tested is connecting the whole curriculum to a real constraint: task selection (the ladder), small-data strategy (transfer learning + augmentation), imbalance-aware evaluation (Track 12), threshold tuning to error costs, and the maturity to consider anomaly detection when the label space is open. A strong answer never says 'train a CNN' in a vacuum — it reasons from the data you actually have (few labels, rare positives) to a plan that will work in production. That end-to-end, constraint-driven thinking is exactly the signal a vision interview is looking for.

2"Why is the transformer showing up everywhere — language, vision, and generation?" Give the unifying answer.
Show model approach & reasoning

"Because the transformer is a universal token processor. Its core operation — self-attention over a set of tokens — makes no assumption about what the tokens are, so it applies to any data you can turn into a sequence of tokens. In language, tokens are word-pieces. In vision, the ViT showed an image is 'worth 16×16 words' — split it into patches and each patch is a token. The same move works for audio frames, video, even protein sequences. So bringing a new modality to the transformer is really just two steps: tokenize the data, and tell the model where the tokens belong with positional information (because attention is order-blind). That's why the architecture unifies fields that used to be separate. It also powers generation and multimodality: diffusion models increasingly use transformer denoisers, and vision-language models feed image tokens (from a ViT) straight into a language model's attention alongside text tokens, so one model can reason over both. The deep reason the transformer wins is that it's flexible (any token attends to any token — a global view) and it scales — more data and compute keep making it better — which suits the modern era of huge datasets."

This question tests whether you see modern AI as one coherent architecture rather than a pile of tricks. The strongest answer names the unifying principle (tokenize + attention), traces it across modalities (words, patches, and beyond), and connects it to generation and multimodal systems — showing you understand why the field converged on the transformer, not just that it did. That systems-level, connect-the-dots understanding is what separates a memorizer from an engineer who can reason about where the field is going."

3"A product manager wants to add 'generate a product image from a text description' to your app. What do you tell them about how it works, and what to watch for?"
Show model approach & reasoning

"How it works, in plain terms: a text-to-image diffusion model (like Stable Diffusion) generates images by starting from random noise and repeatedly removing noise, steered by the text prompt, until a coherent image matching the words appears — it's not retrieving or pasting existing images, it's synthesizing a new one guided by what it learned from millions of image-caption pairs. Practically we'd likely call a hosted model or run an open one, and control quality with the prompt, the number of denoising steps (more steps = better quality but slower/costlier), and guidance strength. What to watch for: (1) Cost and latency — generation is many forward passes per image, far more expensive than classification, so it affects unit economics and response time. (2) Reliability — outputs are stochastic and imperfect; text rendering, fine details, and precise instructions can fail, so we'd want human review or regeneration for anything customer-facing. (3) Safety and misuse — we need guardrails against generating harmful, infringing, or misleading images, plus a policy for deepfake-style misuse. (4) Legal/ethical — the model learned from scraped data, so there are open copyright and consent questions, and outputs can reflect training-data biases; I'd flag this to legal and consider provenance/watermarking. (5) Bias — the model may stereotype (e.g., who it depicts for a given profession), which we'd need to test."

The reasoning tested here is translating a technical capability into product reality: explaining diffusion without hand-waving ('generate from noise, steered by text' — no magic), then surfacing the concerns that actually matter in production — cost, reliability, safety, legal, and bias. This mirrors the responsible-AI discipline of Track 12 and the systems cost-awareness of Track 11, applied to generative vision. An interviewer wants to see that you can be the person who both understands the tech and anticipates its practical and ethical pitfalls before they become incidents — which is exactly what makes someone trustworthy to ship these systems."

★ Deep dive: the one-paragraph story of computer vision▸

If you can tell the whole arc of this track in a breath, you've got it. An image is a grid of numbers — a tensor of pixels — and everything follows from teaching a model to find patterns in that grid. CNNs did it first, sliding small learned filters to detect features (edges → textures → parts → objects), with skip connections (ResNet) letting them go very deep; transfer learning from big pretrained backbones made them practical on modest data. On top of classification grew a task ladder of increasing granularity: detection (box + class per object, evaluated with IoU and mAP), then segmentation (a class per pixel, via encoder-decoder U-Nets, evaluated with per-class IoU/Dice to avoid the background-imbalance trap). Then the Vision Transformer re-expressed images as sequences of patch-tokens so the language-model architecture could process them — trading a CNN's built-in priors for global attention that wins at scale, which is really a story about inductive bias versus data. Diffusion models flipped analysis into generation: learn to reverse noise, then denoise from static into new images, conditioned on text and made cheap by latent diffusion — with a U-Net denoiser tying it back to segmentation. Finally CLIP and vision-language models joined vision to language in a shared token space, enabling zero-shot classification and the multimodal assistants that now see, read, and reason.

The connective themes are the real prize, because they generalize beyond vision. Transfer learning and pretraining run through everything — you almost never start from scratch; you reuse features learned on large data. The transformer is a universal token processor — tokenize a modality, add positions, and attention does the rest, which is why vision, language, and generation converged on one architecture. Scale and data drive capability — and also import bias, so capability and harm often share a root in the training data. And evaluation must fit the task and its imbalance — accuracy for balanced classification, precision/recall for rare classes, IoU/mAP for detection, per-class IoU/Dice for segmentation, human and bias evaluation for generative and multimodal systems (the Track 12 discipline throughout). Hold those four themes and the specific models become easy to place and reason about. You began the whole curriculum with numbers in a matrix and a gradient step; you end it able to explain how machines see, create, and reason about the visual world — and, more importantly, able to reason about which approach fits a real problem and why. That judgment, not any single architecture, is what makes someone job-ready — and it's exactly what a computer-vision interview is built to find."

That completes the Computer Vision track — and with it, a full arc from an image as a tensor to multimodal reasoning. You can now place any vision problem on the task ladder and justify the architecture, data, and evaluation choices around it.
