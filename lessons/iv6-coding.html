Concept The coding round, old and new

The coding round tests whether you can actually implement, not just discuss. It spans two flavors: general problem-solving (data structures and algorithms) and ML-specific coding — implementing algorithms or metrics from scratch (k-means, k-nearest-neighbors, gradient descent, precision/recall, cosine similarity), or writing practical data/model code. And there's a new reality: at some companies the coding round now allows an AI assistant, which shifts what's tested from "can you produce code" to "can you judge and integrate code" — prompt well, catch the assistant's mistakes, and build something correct. Prepare for both formats: enough unaided practice to be fluent, and enough AI-assisted building to be fast and discerning.

Worked example Implement gradient descent from scratch

A classic ML coding question: "fit a line to data using gradient descent, from scratch." Here we fit y = w·x (true w = 3) by repeatedly stepping w opposite the gradient of the mean-squared error — the core learning loop, in a few lines:

xs = [1, 2, 3, 4]
ys = [3, 6, 9, 12]                 # true relationship: y = 3x
w, lr = 0.0, 0.01                  # start at 0, learning rate 0.01
for step in range(1000):
    # gradient of mean squared error wrt w:  mean of 2*(w*x - y)*x
    grad = sum(2 * (w * x - y) * x for x, y in zip(xs, ys)) / len(xs)
    w -= lr * grad                 # step downhill
print(f"learned w: {w:.3f}  (true 3.0)")
print(f"predict x=5 -> {w*5:.2f}")
Outputlearned w: 3.000 (true 3.0) predict x=5 -> 15.00

Starting from w = 0, the loop converges to w = 3.000 — it learned the true relationship — and correctly predicts 15 for x = 5. This tiny example is exactly the kind of from-scratch implementation ML coding rounds ask for, and it tests several things at once: do you understand that gradient descent steps opposite the gradient to reduce loss; can you derive the MSE gradient (the 2*(w*x - y)*x term); do you write a clean, correct loop; and do you sanity-check the result (converges to the known answer). In the interview you'd narrate as you go: "I'll compute the gradient of MSE with respect to w, then step w against it, and I expect it to converge to 3 since that's the true slope" — making your reasoning visible, which matters as much as the code. You'd also discuss extensions the interviewer might probe: adding a bias term (y = w·x + b), vectorizing with NumPy for speed, handling the learning rate (too high diverges, too low crawls), and using multiple features. The point of these from-scratch questions is that implementing an algorithm proves you understand it — you can't fake gradient descent if you have to code it — which is why they're a staple even in an era of AI assistants: the skill being tested is comprehension you can implement and verify, not memorized boilerplate. If an assistant were allowed, the test would shift to whether you can prompt it to write this, spot if it got the gradient wrong, and confirm the output converges — judging the code, not just producing it.

★ Key points to remember
  • The coding round spans general algorithms and ML-specific implementation (k-means, kNN, gradient descent, metrics from scratch).
  • Communicate out loud: clarify → state your approach → implement → test/debug — your reasoning is judged, not just the final code.
  • From-scratch ML implementations prove you understand the algorithm (you can't fake coding gradient descent).
  • AI-assisted coding rounds are emerging: the skill shifts to prompting, spotting the assistant's bugs, and verifying correctness — judging code.
  • Prepare both modes: unaided fluency (for no-AI rounds and to judge AI output) and fast, discerning AI-assisted building.

Practice Mock interview prompts

1"Implement k-means clustering from scratch. Walk me through your approach before coding."
Show a strong approach

"I'd state the algorithm first: k-means partitions points into k clusters by alternating two steps until convergence. (1) Assign: assign each point to the nearest centroid (by Euclidean distance). (2) Update: move each centroid to the mean of the points assigned to it. Repeat until assignments stop changing (or a max iteration). Initialize the k centroids somehow — randomly picking k data points is simple; k-means++ is smarter. Before coding I'd note the pieces I need: a distance function, an assignment step (for each point, argmin over centroids), an update step (mean of each cluster's points), and a convergence check. Then I'd implement it cleanly, testing on a tiny dataset where I know the answer. I'd narrate trade-offs: k-means is sensitive to initialization (so real code runs it several times and keeps the best by inertia), you must choose k (the elbow method or silhouette score help), and it assumes roughly spherical, similarly -sized clusters — so it struggles with elongated or varied-density clusters, where other methods fit better. I'd also mention complexity (O(n·k·iterations·dimensions)) and that vectorizing with NumPy matters for real data." This lands because stating the algorithm and its properties before coding shows you understand it, the from-scratch implementation proves it, and volunteering the caveats (initialization sensitivity, choosing k, cluster-shape assumptions) demonstrates depth beyond a rote solution — exactly the reasoning-plus-implementation the round tests. If an assistant were allowed, I'd prompt it for the implementation and focus my attention on verifying the assignment/update logic and the convergence check are correct.

2"Write a function to compute precision, recall, and F1 given lists of predicted and true labels."
Show a strong approach

"I'd start by restating the definitions to make sure we agree, then implement directly. Count the four outcomes by comparing predictions to truth: TP (predicted positive, actually positive), FP (predicted positive, actually negative), FN (predicted negative, actually positive). Then precision = TP/(TP+FP), recall = TP/(TP+FN), F1 = 2·precision·recall/(precision+recall). The clean implementation:

def prf1(pred, true, pos=1):
    tp = sum(p == pos and t == pos for p, t in zip(pred, true))
    fp = sum(p == pos and t != pos for p, t in zip(pred, true))
    fn = sum(p != pos and t == pos for p, t in zip(pred, true))
    precision = tp / (tp + fp) if tp + fp else 0.0
    recall    = tp / (tp + fn) if tp + fn else 0.0
    f1 = 2*precision*recall/(precision+recall) if precision+recall else 0.0
    return precision, recall, f1

The thing I'd emphasize while coding is the edge cases: division by zero when there are no positive predictions (precision) or no actual positives (recall) — I guard each with a check returning 0.0, which is exactly the kind of robustness interviewers watch for. I'd test it on a small hand-computed example to confirm. I'd also mention that this is binary; for multiclass you'd compute per-class and average (macro or weighted), and that in practice you'd use a library, but implementing it proves I understand what the metrics actually measure — which connects to knowing why you'd choose precision vs recall by error cost." This stands out because it pairs a correct, clean implementation with explicit edge-case handling (the zero-division guards) and the conceptual grounding — showing both coding rigor and ML understanding, the combination ML coding rounds reward.

3"If AI assistants are allowed in the coding round, how do you make the most of one without letting it hurt you?"
Show a strong approach

"I'd treat the assistant as a fast pair-programmer whose work I'm responsible for verifying — because the skill being tested is judgment, not the assistant's output. Concretely: I'd still clarify the problem and plan my approach first, so I'm directing the assistant with a clear intent rather than hoping it figures out the problem — a vague prompt gets vague code. I'd prompt for specific, bounded pieces and read every line it produces critically, because assistants confidently generate plausible-but-wrong code (off-by-one errors, wrong edge cases, subtle logic bugs, even a wrong gradient) — catching those is exactly what's being evaluated. I'd test the code (run it, try edge cases, trace a small example) rather than trusting it, and I'd keep narrating my reasoning — what I'm asking for, why, and how I'm verifying it — so the interviewer sees my judgment. The failure mode I'd avoid is over-reliance: pasting the assistant's output without understanding it, so I can't explain or debug it when the interviewer probes ('why does this work? what if the input is empty?'). If I don't understand what it wrote, I can't judge it, and that's the whole test. So the balance is: use the tool to move fast on boilerplate and syntax, but own the design, verify the correctness, and understand every line — because I'm accountable for the result. That's genuinely how good engineers use AI tools day-to-day, which is why the format exists." This lands because it demonstrates exactly the judgment-over-production mindset the new format tests: directing the tool with clear intent, critically verifying its output, testing rigorously, and never ceding understanding — the difference between using AI as a competent engineer versus as a crutch.

★ Deep dive: what the coding round really tests, in an AI-assisted era▸

The coding round has always tested more than whether you can produce working code — it probes problem-solving, communication, and the ability to translate understanding into correct implementation. That's why interviewers care how you clarify the problem, how you reason about approach and trade-offs, how you handle edge cases, and how you debug — the process reveals how you'd actually work. For ML roles specifically, from-scratch implementation questions serve a sharp purpose: they verify that your conceptual knowledge is real and not just vocabulary. You can memorize that gradient descent steps opposite the gradient, but coding it forces you to actually know the gradient, the update rule, and the loop — comprehension you can't fake. This is why ML coding questions favor implementing the fundamentals (gradient descent, k-means, kNN, metrics, a forward pass) over obscure algorithm puzzles: they're checking that the machinery you discuss in the other rounds is machinery you genuinely understand.

The emergence of AI-assisted coding interviews is reshaping this in a way that's worth understanding because it reflects a real shift in the profession. When an assistant can produce the code, the differentiating human skill becomes judgment: understanding the problem well enough to direct the tool, reading its output critically enough to catch its confident mistakes, testing and verifying rather than trusting, and integrating pieces into a correct whole. The often-quoted framing is that the skill being hired for is shifting from "can you produce code" to "can you judge code" — which is exactly how experienced engineers already work with AI tools. This doesn't make understanding less important; it makes it more important, because you can't judge code you don't understand — an assistant that writes a subtly wrong gradient is only caught by someone who knows what the right gradient is. So the deep preparation is the same as it's always been: genuinely understand the algorithms and the fundamentals, so that whether you're implementing from scratch unaided or verifying an assistant's output, you can tell correct from plausible-but-wrong. The practical advice is to build both muscles — enough unaided coding to be fluent and to have the understanding that lets you judge, and enough AI-assisted practice to be fast and discerning with the tool — because you may face either format, and both ultimately rest on real comprehension. The through-line, fitting this whole curriculum, is that in an era where code and answers are increasingly cheap to generate, the scarce and valuable skill is the understanding and judgment to direct, verify, and integrate them — which is precisely what deep learning of the fundamentals gives you, and what every round of the interview, coding included, is ultimately trying to measure.
