Track 7 · RAG & Vector Search

Interview check: RAG

⏱ 15 min Lesson 7.8
Why this existsRAG interviews test the full pipeline and debugging: walk through RAG, fix a hallucinating system, pick a chunking strategy, design an evaluation. This checkpoint rehearses those with model answers.
By the end, you can
  • Walk through a RAG pipeline
  • Debug retrieval vs generation failures
  • Choose chunking and hybrid search
  • Design a RAG evaluation

Concept What a RAG interview tests

RAG questions are staples of modern ML interviews because RAG is the most-deployed LLM technique. Interviewers probe whether you understand the full pipeline, can debug it (most failures are retrieval failures), and know how to evaluate it. The model answers below show the reasoning interviewers want.

Model answer "Walk me through a RAG pipeline."

"Two phases. Offline indexing: load the documents, chunk them (recursive splitting on paragraphs/sentences with some overlap, plus metadata for citations and filtering), embed each chunk with a retrieval embedding model, and store the vectors in a vector database. Online querying: embed the user's question with the same model, run nearest-neighbor search (ANN index like HNSW) — ideally hybrid, combining dense embeddings with BM25 keyword search — optionally rerank the top candidates with a cross-encoder, then assemble a prompt with the top chunks, their citation tags, and an instruction to answer only from the context and say 'I don't know' otherwise. The LLM generates a grounded, cited answer. The key design point is separating the phases: you pay the heavy embedding cost once at index time, so each query just embeds one short question and searches a pre-built index. And I'd have an evaluation harness measuring retrieval (recall@k) and generation (faithfulness) separately."

Strong close: mention that retrieval quality is the ceiling on RAG quality, so most effort goes there.

Model answer "Your RAG system is hallucinating. How do you debug it?"

"First I'd separate retrieval from generation, because the cause is usually retrieval. I'd inspect the retrieved chunks for the failing queries: does the relevant chunk appear at all? If not, it's a retrieval failure — check chunking (is the answer split badly or diluted?), the embedding model (is it matching meaning?), and consider hybrid search (for exact terms it's missing) and query rewriting. I'd measure recall@k on a test set to quantify it. If the relevant chunk is retrieved but the answer is still wrong, it's a generation / faithfulness problem — the model is ignoring the context or misreading it. I'd strengthen the prompt ('answer ONLY from the context; say you don't know if it's absent'), rerank so the best chunk is at the top where the model attends most, reduce distracting irrelevant chunks, or use a stronger model. The discipline is: never guess — look at what was retrieved, and let recall@k vs faithfulness tell you which half to fix."

Model answer "How do you choose a chunking strategy?"

"Chunking is consequential because you can only retrieve what your chunks contain, and the core tradeoff is size: too small loses context, too large dilutes the embedding match and wastes prompt space. My default is recursive splitting — break on paragraphs, then sentences, keeping chunks under a token limit — with some overlap so ideas spanning a boundary aren't lost, plus metadata (source, section) for citation and filtering. I match the strategy to the content: structure-aware splitting for code (by function) and markdown (by section), since blind character cuts break those badly. For the size itself, I tune it empirically — build an eval set and measure whether retrieval actually surfaces answer-bearing chunks at different sizes. If precise facts get diluted, I'd consider the small-to-big pattern: search on small chunks but return their larger parent context to the model. There's no universal best; it depends on the documents and queries, so I measure."

Model answer "Semantic search vs hybrid search — when do you need hybrid?"

"Pure semantic (dense embedding) search is great at meaning — synonyms, paraphrase — but it blurs exact matches: specific names, error codes, SKUs, function names, where the precise token carries the signal and embeddings can't distinguish them well. Keyword search (BM25) nails those exact matches but misses synonyms. Hybrid search runs both and fuses the rankings — commonly Reciprocal Rank Fusion, which combines by rank so you don't need to calibrate incomparable score scales. I reach for hybrid whenever the domain has exact identifiers that matter, which is most enterprise and technical content. It works because dense and sparse make uncorrelated errors — they fail on different queries — so combining them is like ensembling: the union of their strengths beats either alone. For pure natural-language prose with no special identifiers, semantic alone may suffice, but hybrid is a safe production default."

Model answer "How do you evaluate a RAG system?"

"Separately for retrieval and generation, because either can fail. For retrieval, I build a test set of queries with labeled relevant chunks and compute ranking metrics — recall@k (did we fetch the relevant chunks? — most important for RAG, since a missing chunk dooms the answer), precision@k, and MRR (how high the first relevant chunk ranked). For generation, the RAG triad: faithfulness (is the answer grounded in the context, no hallucination?), answer relevance (does it address the question?), and context relevance. These have no single right answer, so I score them with an LLM-as-judge — validated against some human labels. To bootstrap the eval set, I generate question–answer pairs from the documents with an LLM and spot-check them. The whole point is eval-driven development: with these metrics I can measure whether a change — new chunking, a reranker, a different model — actually helped, rather than guessing. You ship your eval, not just your model."

★ Rapid-fire answers to have ready
  • Why RAG over fine-tuning for facts? Fine-tuning instills behavior, not specific facts; RAG injects current/private knowledge and can cite it.
  • Most common RAG failure? Retrieval — the relevant chunk wasn't fetched. Inspect retrieved chunks first.
  • Bi- vs cross-encoder? Bi-encoder embeds separately (fast, scalable); cross-encoder reads query+chunk together (accurate, rerank only).
  • Why ANN over brute force? O(N) is too slow at millions of vectors; ANN is ~100× faster for ~99% recall.
  • Same embedding model for docs and queries? Yes — different models give incompatible vector spaces.
  • Reduce hallucination? Ground in retrieved context, instruct "answer only from context / say I don't know," cite sources.

Practice Mock interview prompts

1"Design a RAG system for a law firm's 100,000 case documents, where citations and access control are mandatory." (3-minute answer.)
Show a strong answer

"Indexing: structure-aware chunking (by section/clause, with overlap and rich metadata — case id, date, client, access level), embed with a strong (ideally legal-domain) model, store in a vector DB with metadata indexing. Querying: hybrid search (semantic + BM25, since legal text has exact citations and statute numbers that must match), filtered by the user's access level so they only retrieve authorized documents — non-negotiable for confidentiality. Then cross-encoder reranking for precision, and prompt assembly with citation tags and an instruction to answer only from context, cite every claim, and say 'I don't have that information' if absent — critical in legal, where an uncited or hallucinated answer is worse than none. Evaluation: a test set of real legal questions with labeled relevant documents, tracking recall@k and faithfulness, plus expert review. Operations: incremental indexing as cases are added, audit logging of what was retrieved, and careful handling of prompt-injection risk in documents. The themes — citations, access control, faithfulness, exact-match retrieval — are exactly what a high-stakes domain demands."

This shows you can translate RAG fundamentals into a domain's real constraints (legal: citations, access, exactness) — exactly the system-design judgment senior roles want.

2"Your RAG works great in testing but degrades in production over a few months. What's happening and how do you address it?"
Show a strong answer

"Several drift factors. Knowledge drift: documents change but the index wasn't updated, so retrieval returns stale chunks — fix with incremental re-indexing and freshness filters. Query drift: users ask new kinds of questions the system wasn't tuned for, hurting retrieval — mine production logs for new query patterns and expand the eval set. Coverage gaps: questions about topics not in the knowledge base, which should trigger 'I don't know' but may hallucinate — monitor abstention rates. I'd add observability: log what's retrieved and a sample of answers, track recall and faithfulness on a continuously-updated eval set drawn from real traffic, and alert on metric drops. This connects to monitoring and drift from Track 9 — a deployed RAG system isn't 'done,' it needs ongoing evaluation and maintenance because both the documents and the questions evolve."

Recognizing that RAG degrades from drift (not a fixed bug) and proposing monitoring + continuous evaluation shows production maturity beyond the happy path.

3"Explain to a skeptical executive why we should invest in RAG instead of just waiting for a bigger model that 'knows everything.'"
Show a strong answer

"A bigger model still won't know our private documents — they were never in its training — and its knowledge is frozen at training time, so it can't know today's data no matter how large. It will also still hallucinate confidently, which is a liability for us. RAG solves all three: it grounds answers in our actual, current documents, lets us cite sources so answers are verifiable and auditable, and we can update knowledge instantly by changing the document store — no waiting for or paying for retraining. It's also far cheaper and more controllable than fine-tuning a model on our data, and it works with whatever base model we choose, including future better ones. In short, scale improves the model's general reasoning, but RAG is what makes it know and faithfully use our business's knowledge — and that's where the value is. The two are complementary: a better model plus RAG beats either alone."

Framing RAG's value in business terms (private data, currency, citations, cost, control) and positioning it as complementary to scale — not competing with it — is exactly the executive-level communication that distinguishes a strong candidate.

★ Deep dive: reasoning about RAG systems end to end▸

The strongest RAG answers treat it as a system with a quality ceiling set by retrieval. The mental model is a chain — chunk → embed → store → retrieve → rerank → assemble → generate → evaluate — where each link can be the weak one, and the final answer is only as good as the chunks that reach the prompt. So when reasoning about any RAG problem, the expert move is to localize the failure: is the relevant chunk in the index at all (chunking)? is it retrieved (embeddings, hybrid, ANN recall)? is it ranked high enough (reranking)? does the model use it (faithfulness, prompt)? Each has distinct fixes, and the only way to know which link broke is to look at the intermediate outputs (especially the retrieved chunks) and measure each stage. Candidates who say "I'd inspect the retrieved chunks and check recall@k separately from faithfulness" immediately signal they've debugged real RAG, because that's the move that resolves the overwhelming majority of issues.

Three habits sharpen RAG reasoning. First, retrieval is the bottleneck — internalize that most "the LLM is dumb" complaints are really "we didn't retrieve the answer," so disproportionate effort belongs in chunking, embeddings, hybrid search, and reranking, and the first diagnostic is always "what did we retrieve?" Second, evaluate to improve — RAG without measurement is guesswork; with recall@k and the faithfulness triad on a maintained eval set, every change is testable, which is the difference between a system that steadily improves and one that plateaus. Third, connect to the bigger picture: RAG is grounded generation, it pairs with prompting (Track 6) and is a building block for agents (Track 8, where retrieval becomes one tool the model can choose), it demands production discipline (Track 9: monitoring, drift, cost), and it raises safety concerns (Track 11: prompt injection via retrieved content, faithfulness as a safety property). Showing that you see RAG not as an isolated trick but as the practical heart of making LLMs useful on real, private, changing knowledge — with retrieval quality as the lever and evaluation as the compass — is exactly the system-level understanding that the best teams hire for. You've now built that understanding end to end, which is precisely what Track 8 extends when the model stops just retrieving and starts acting."

That completes Track 7 — you can ground a model in your own knowledge and prove it works. Next, Track 8 (AI Agents & Tool Use) lets the model stop merely retrieving and start acting — calling tools, planning, and using memory to accomplish multi-step goals.
