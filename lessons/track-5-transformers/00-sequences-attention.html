From sequences to attention

Track 4 built a network that maps a fixed input to an output. But language is a sequence, where meaning depends on context — “it” refers to something said earlier. The transformer’s answer is one of the most consequential ideas in AI: let every token look directly at every other token.

Track 5 · Lesson 0 Concept + code ~24 min Builds on: Tracks 1, 4

This is the track where it all pays off. Using the autograd engine and training loop you built in Track 4, plus one genuinely new idea — attention, built from Track 1’s dot products — you’ll implement and train a small GPT that learns to generate text. We begin with how text becomes numbers, and why attention is the right way to process a sequence.

💡 Concept: text becomes a sequence of vectors

A model only computes with numbers, so the first step is to turn text into them. Tokenization splits text into a sequence of tokens (characters, or more often subword pieces) and maps each to an integer id. Then an embedding table — a learned matrix with one row per vocabulary item — maps each id to a vector in \(\mathbb{R}^d\) (Track 1). After this, a sentence is just a sequence of vectors, and we can do linear algebra on it.

📐 The problem: context lives elsewhere in the sequence

The meaning of a token depends on the others around it. In “the cat sat because it was tired,” the word “it” means the cat — but the cat is four words back. To represent “it” properly, the model must gather information from the right earlier tokens. Earlier architectures (recurrent networks) passed information along the sequence one step at a time, which is slow and tends to forget distant context. The transformer’s radical simplification: let every token look at every other token directly, in parallel, and pull in what’s relevant.

🤖 Why this changed everything

Because attention connects all positions directly (not step-by-step), it processes a whole sequence in parallel — perfect for GPUs — and it can relate tokens any distance apart in a single step. This combination of parallelism and unlimited-range context is why transformers replaced recurrent networks and now power essentially every large language model. The 2017 paper that introduced it was titled, fittingly, “Attention Is All You Need.”

🛠️ From scratch: tokenize and embed

code/track-5/tokenize.py
import numpy as np

text = "the cat sat"
chars = sorted(set(text))                 # the vocabulary
stoi = {c: i for i, c in enumerate(chars)}    # char -> id
itos = {i: c for c, i in stoi.items()}        # id -> char
ids = [stoi[c] for c in text]
print(ids)                                # [6, 4, 3, 0, 2, 1, 6, 0, 5, 1, 6]
print("".join(itos[i] for i in ids))      # 'the cat sat'  (round-trips)

# Embedding table: one learned vector per vocabulary item
vocab_size, d = len(chars), 6
embed = np.random.default_rng(0).normal(0, 1, (vocab_size, d))
seq = embed[ids]                          # (sequence_length, d) -- a matrix of vectors
print(seq.shape)                          # (11, 6)

Real models use subword tokenization (byte-pair encoding) so frequent words are single tokens and rare words split into pieces — but the pipeline is identical: text → ids → vectors. From here on, a sentence is a matrix \(X\in\mathbb{R}^{T\times d}\) of \(T\) token vectors.

🔑 Key results to remember

Lock these in

  1. Tokenization turns text into a sequence of integer ids; an embedding table maps each id to a learned vector in \(\mathbb{R}^d\).
  2. A sentence becomes a matrix \(X\in\mathbb{R}^{T\times d}\) (\(T\) tokens, each a \(d\)-vector).
  3. Attention lets every token gather context directly from every other token, with learned weights.
  4. Unlike recurrent networks, attention is parallel (GPU-friendly) and relates distant tokens in one step — why transformers dominate.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. What does an embedding table do?

  • Splits text into tokens.
    That’s tokenization, the step before embedding. The embedding table acts on the ids that tokenization produces.
  • Maps each token id to a learned vector in \(\mathbb{R}^d\).
    Correct. It’s a trainable matrix with one row per vocabulary item; looking up a token’s id returns its vector — the model learns these during training.
  • Computes attention weights.
    Attention comes later and operates on the embedded vectors. The table just produces those vectors.
  • Predicts the next token.
    Prediction is the model’s final step. Embedding is the first step — turning ids into vectors.

Q2. Why is attention better suited to long-range context than a recurrent network?

  • It uses fewer parameters.
    Parameter count isn’t the point (transformers are often larger). The advantage is how information moves between positions.
  • It processes tokens one at a time.
    That describes recurrent networks, the slow approach. Attention does the opposite — all positions at once.
  • It connects any two positions directly in a single step, in parallel.
    Correct. Attention relates distant tokens immediately (no step-by-step relay) and processes the whole sequence in parallel — solving both the forgetting and the speed problems of recurrence.
  • It doesn’t need training.
    Attention’s weight matrices are very much learned during training. The benefit is architectural, not training-free.

Q3. After tokenization and embedding, a sentence of \(T\) tokens is represented as…

  • a matrix \(X \in \mathbb{R}^{T\times d}\) — \(T\) vectors of dimension \(d\).
    Correct. Each token is a \(d\)-dimensional vector; stacking \(T\) of them gives a \(T\times d\) matrix, ready for the linear algebra of attention.
  • a single scalar.
    Far too little — a scalar can’t hold a sequence of rich token meanings. Each token alone is a whole vector.
  • a \(d\times d\) square matrix.
    The shape is \(T\times d\) (tokens × dimensions), which is square only if \(T=d\) by coincidence.
  • an image.
    Text becomes a sequence of vectors, not an image grid (though both are ultimately tensors of numbers).

Q4. In “the cat sat because it was tired,” resolving what “it” refers to is an example of…

  • tokenization.
    Tokenization just splits the text into pieces; it doesn’t resolve meaning or references.
  • embedding.
    Embedding gives each token a vector but doesn’t, by itself, link “it” to “cat.” That linking is the job of attention.
  • a recurrent bottleneck.
    A bottleneck is a limitation, not the task. The task is relating “it” to its referent — which attention does well.
  • using context from elsewhere in the sequence — exactly what attention provides.
    Correct. “It” means the cat only by looking back at an earlier token; attention’s whole purpose is to let a token gather such context from the rest of the sequence.

Problem 1 · code

Run tokenize.py on a sentence of your own. Confirm encoding then decoding round-trips back to the original text. What is the vocabulary size for “hello world”?

Show solution

For “hello world”, the distinct characters are \(\{\text{space}, d, e, h, l, o, r, w\}\) — 8 unique characters, so vocab size 8 (note ‘l’ and ‘o’ repeat but count once). Encoding maps each character to its id and decoding inverts it, so "".join(itos[i] for i in ids) reproduces the input exactly — a lossless round-trip, which any correct tokenizer must guarantee.

Problem 2 · reasoning

Character-level tokenization gives tiny vocabularies but long sequences; word-level gives short sequences but huge vocabularies (and can’t handle unseen words). Why is subword tokenization a good compromise?

Show solution

Subword tokenization (e.g. byte-pair encoding) keeps a moderate vocabulary by representing frequent words as single tokens while splitting rare or novel words into reusable pieces (“tokenization” → “token”+“ization”). This gives the best of both: sequences stay reasonably short (efficiency, since attention cost grows with sequence length), the vocabulary stays bounded (a manageable embedding table), and there are no unknown words — anything can be built from subword pieces, down to individual bytes. That robustness is why essentially all modern LLMs use subword tokenizers.

Problem 3 · reasoning

The embedding table is a learned matrix. What would you expect to happen to the vectors of two tokens that are used in very similar contexts (e.g. “cat” and “dog”) after training?

Show solution

They should end up close together in embedding space (high cosine similarity, Track 1). Training pushes tokens that play similar roles — appearing in similar contexts and needing similar predictions — toward similar vectors, because that’s what lets the model treat them similarly where appropriate. This is the same “nearby = similar meaning” geometry we previewed with word embeddings in Track 1, now arising naturally as a byproduct of training the whole model end-to-end. The embedding table isn’t told what words mean; it discovers a useful geometry from the task.

Problem 4 · reasoning

Attention lets a token attend to all others. For a sequence of length \(T\), how does the cost of computing all pairwise attention scores grow with \(T\), and why is this a concern for long documents?

Show solution

Computing a score for every (query, key) pair means \(T\times T\) scores, so the cost grows as \(O(T^2)\) — quadratic in sequence length. Doubling the context length quadruples the attention compute and memory. For long documents (tens of thousands of tokens) this becomes the dominant cost and the main bottleneck on context length, which is why a whole research area exists around making attention cheaper (sparse, linear, and flash attention). It’s the price of attention’s superpower — every token seeing every other token — and a concrete example of the kind of efficiency trade-off researchers constantly navigate.
