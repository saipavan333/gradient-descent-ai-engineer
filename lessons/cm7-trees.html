Track 3 · Classical ML

Decision trees

⏱ 25 min Lesson 3.7 ⬇ Runnable notebook
Why this existsDecision trees reason in human-readable yes/no questions and capture nonlinear patterns no line can. They are interpretable alone and the building block of the ensembles that dominate tabular ML.
By the end, you can
  • Trace a decision tree from root to leaf
  • Explain how splits are chosen by impurity
  • Recognize the axis-aligned boundary
  • See why a single deep tree overfits

Concept A model you could explain to your grandparent

Every model so far drew a straight boundary. A decision tree works completely differently and more intuitively: it asks a sequence of yes/no questions about the features, like a flowchart, and the answer you land on is the prediction. "Is the tumor radius > 15? If yes, is the texture > 20? If yes, predict malignant." It's the way a person might reason, which is exactly why trees are so easy to interpret.

The vocabulary: the top question is the root, each yes/no question is a split (or "node"), and each end-point with a prediction is a leaf. A tree handles numbers and categories, needs no feature scaling (it only compares thresholds), and naturally captures interactions — "high size AND old age" — that a linear model would miss.

Concept How a tree chooses its questions: impurity

How does the tree decide which question to ask first, and where to put the threshold? It's greedy: at each node it tries every feature and every split point, and picks the one that best separates the classes — that makes the resulting groups as pure as possible (ideally all-one-class). Purity is measured by Gini impurity or entropy; both are near 0 for a pure group and largest for a 50/50 mix. The tree chooses the split that drops impurity the most, then repeats inside each child.

Because each question is "feature > threshold," the boundary a tree draws is made of horizontal and vertical cuts — an axis-aligned staircase, not a slanted line. Stack enough cuts and a tree can carve out any region, capturing curved and nonlinear patterns that defeat a single logistic line.

Concept Why a single tree overfits

Here's the catch. If you let a tree keep splitting, it will eventually carve the data so finely that every leaf holds a single training example — at which point it has memorized the training set and scores 100% on it. That's textbook overfitting (Lesson 3.5): the tree's depth is its complexity dial, and an unlimited tree has enormous variance, drawing a jagged boundary that changes completely if you reshuffle the data. The cure is to limit complexity — cap the depth, require a minimum number of samples per leaf, or "prune" branches that don't help on validation data.

Worked example Watching a tree memorize as it deepens

Back to the breast-cancer data. We grow trees of increasing maximum depth and compare training accuracy to test accuracy. Depth is the complexity knob, so we expect the now-familiar overfitting signature to appear.

from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import train_test_split
from sklearn.tree import DecisionTreeClassifier

X, y = load_breast_cancer(return_X_y=True)
X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.2, random_state=0, stratify=y)

for depth in [1, 3, None]:                       # None = grow until pure
    tree = DecisionTreeClassifier(max_depth=depth, random_state=0).fit(X_tr, y_tr)
    print(f"depth {str(depth):4s}:  train {tree.score(X_tr, y_tr):.3f}   test {tree.score(X_te, y_te):.3f}")
Outputdepth 1 : train 0.927 test 0.886 depth 3 : train 0.974 test 0.921 depth None: train 1.000 test 0.939

Read the gap. A depth-1 stump (one question!) already gets 88.6% — underfitting a little, with train and test close. At depth 3, a tree a human could read on one page reaches 92.1% test. Let it grow unlimited and training accuracy hits a perfect 1.000 — it has memorized every training example — while test sits at 93.9%, opening a 6-point train–test gap. That perfect training score next to a visible gap is the overfitting fingerprint. The unlimited tree squeezed out a bit more test accuracy here, but it did so by memorizing, which is fragile: on noisier data that gap widens into real test-error loss. The fix that keeps the flexibility and kills the variance is coming next — combine many trees.

★ Key points to remember
  • A decision tree is a flowchart of yes/no splits; you follow answers down to a leaf prediction.
  • It chooses each split greedily to most reduce impurity (Gini or entropy) — making child groups purer.
  • Its boundary is an axis-aligned staircase, so it captures nonlinear patterns a line can't.
  • Trees need no feature scaling and are highly interpretable.
  • A single deep tree overfits (memorizes, train → 100%); limit depth/leaf size, or ensemble it (next lesson).

Practice Build the reasoning yourself

1A node contains 8 examples: 4 class A and 4 class B. A candidate split sends {4 A, 0 B} left and {0 A, 4 B} right. Qualitatively, what happens to impurity, and is this a good split?
Show solution & reasoning

The parent is a perfect 50/50 mix — maximum impurity (Gini = 0.5). The split produces two children that are each 100% one class — zero impurity. Impurity dropped from 0.5 to 0, the largest possible reduction, so this is the best possible split: the tree would choose it eagerly.

This is the engine of tree-building in miniature: search for the question that turns a muddled group into clean ones. A split that left both children at 50/50 would reduce impurity by nothing and would never be chosen.

2Two features perfectly predict the label when combined ("buys" only if income is high AND age is young), but neither alone is very predictive. Would logistic regression or a decision tree more naturally capture this, and why?
Show solution & reasoning

A decision tree captures it naturally. It can split first on income, then — only within the high-income branch — split on age, encoding the "high income AND young" interaction directly as a path through the tree. Each leaf represents a specific combination of conditions.

Plain logistic regression sums independent per-feature contributions ($w_1\cdot\text{income} + w_2\cdot\text{age}$), so it can't represent "only when both hold" unless you manually add an interaction feature (income × age). Capturing interactions automatically is one of the things trees (and tree ensembles) are genuinely good at, and a reason they shine on tabular data.

3You train two unlimited-depth trees on two different 90% samples of the same data and get very different-looking trees. What property is this, and what general technique does it motivate?
Show solution & reasoning

This is high variance (instability): a single deep tree is extremely sensitive to the exact training sample, because a small change near the top can cascade into completely different splits below. Same concept as Lesson 3.5, made vivid.

It motivates ensembling — and specifically averaging many trees. If you train many high-variance trees on different resamples and average their predictions, the individual quirks cancel out while the shared signal reinforces, slashing variance without adding bias. That's precisely the random forest, the subject of the next lesson. Trees' instability is a weakness alone but a strength in a crowd.

★ Deep dive: regression trees, feature importance, and pruning▸

Trees aren't just for classification. A regression tree predicts numbers: same splitting idea, but it minimizes variance (squared error) within each node instead of class impurity, and each leaf predicts the average target of the training examples that land there. The result is a piecewise-constant prediction surface — flat within each box. That's why a lone regression tree can look like a staircase when fit to a smooth trend, and why ensembles (which average many of them) produce the smoother, stronger predictions you'll use in practice.

Trees give you feature importance almost for free: sum up how much each feature reduced impurity across all the splits that used it, and you get a ranking of which features the model leaned on. It's a quick, intuitive read on what matters — though be cautious, as the standard "impurity-based" importance can be biased toward high-cardinality features; permutation importance (shuffle a feature, see how much accuracy drops) is a more reliable alternative.

Finally, controlling overfitting. Beyond capping max_depth, you can set min_samples_leaf (don't make a leaf from too few examples) or use cost-complexity pruning: grow a full tree, then snip back the branches whose complexity isn't paying for itself on validation data — the tree analog of regularization's "penalize complexity." All of these are hyperparameters you'd tune with cross-validation (Lesson 3.12). But honestly, in modern practice you rarely tune a single tree to perfection; you reach for a forest or a boosted ensemble, which is exactly where we go next.

A single tree is interpretable but unstable. The fix is one of the most important ideas in applied ML. Next, in 3.8 Ensembles: Random Forests & Gradient Boosting, we combine many trees into models that routinely win on tabular data.
