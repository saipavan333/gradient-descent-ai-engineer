Decoding strategies

⏱ 24 min Lesson 6.4 ⬇ Runnable notebook
Why this existsThe model gives a distribution; decoding picks the token — the dial between dull and creative. Temperature, top-k, and top-p are among the most practical levers you have when using any LLM.
By the end, you can
  • Contrast greedy, beam, and sampling
  • Explain temperature, top-k, and top-p
  • Match decoding settings to the task
  • Diagnose repetitive or incoherent output

Concept The model gives a distribution; decoding chooses the token

At every step, a language model outputs a probability distribution over the next token (Lesson 5.3). But to actually generate text it must commit to one token. Decoding is the strategy for picking that token from the distribution — and it matters enormously: the same model with different decoding settings can produce dull repetition, incoherent randomness, or fluent creative text. Decoding is the dial between "safe and boring" and "diverse and risky."

The two families are search (deterministically pick high-probability tokens) and sampling (randomly draw from the distribution). The simplest search is greedy decoding: always take the single most likely token (the argmax). It's fast and coherent but tends toward repetitive, generic text, because always choosing the safest word produces bland output and can get stuck in loops ("the the the").

Beam search improves on greedy by keeping the $k$ best partial sequences (the "beams") at each step rather than committing to one, so it can find a higher-probability overall sequence. It's standard for translation and summarization, where there's a roughly correct output. But for open-ended generation (stories, chat), beam search still tends to be repetitive — and that's where sampling shines.

Concept Sampling: temperature, top-k, top-p

Instead of always taking the top token, sampling draws randomly according to the probabilities — so a token the model rates 30% likely is chosen about 30% of the time. This injects the variety that makes text feel natural and creative. Three knobs control it:

  • Temperature $T$ rescales the logits before softmax. $T < 1$ sharpens the distribution (more confident, more repetitive), $T > 1$ flattens it (more diverse, more random). $T \to 0$ approaches greedy.
  • Top-k keeps only the $k$ most likely tokens and samples among them — cutting off the long tail of unlikely (often nonsensical) tokens.
  • Top-p (nucleus sampling) keeps the smallest set of tokens whose cumulative probability reaches $p$ (say 0.9) — an adaptive cutoff that keeps more tokens when the model is uncertain and fewer when it's confident.
Worked example Every decoding strategy on one distribution

Let's apply each strategy to the same set of next-token scores (logits) for six candidate tokens, and watch how the resulting probabilities change.

import numpy as np
def softmax(z, T=1.0):
    z = z / T; e = np.exp(z - z.max()); return e / e.sum()

logits = np.array([3.0, 2.0, 1.0, 0.5, 0.2, -1.0])   # scores for 6 candidate tokens

print("greedy pick:    token", logits.argmax())          # always the top token
print("T = 1.0:", softmax(logits).round(3))
print("T = 0.5 (sharper):", softmax(logits, 0.5).round(3))
print("T = 1.5 (flatter):", softmax(logits, 1.5).round(3))

p = softmax(logits)
# top-k = 3: keep the 3 most likely, renormalize
idx = p.argsort()[::-1][:3]; m = np.zeros_like(p); m[idx] = p[idx]
print("top-k = 3:", (m / m.sum()).round(3))
# top-p = 0.9: keep smallest set with cumulative prob >= 0.9
order = p.argsort()[::-1]; cum = np.cumsum(p[order]); keep = order[:np.searchsorted(cum, 0.9) + 1]
m2 = np.zeros_like(p); m2[keep] = p[keep]
print("top-p = 0.9:", (m2 / m2.sum()).round(3))
Outputgreedy pick: token 0 T = 1.0: [0.601 0.221 0.081 0.049 0.037 0.011] T = 0.5 (sharper): [0.859 0.116 0.016 0.006 0.003 0. ] T = 1.5 (flatter): [0.457 0.234 0.12 0.086 0.071 0.032] top-k = 3: [0.665 0.245 0.09 0. 0. 0. ] top-p = 0.9: [0.665 0.245 0.09 0. 0. 0. ]

Watch each knob. Greedy just takes token 0. At T = 1.0 the top token has 60% probability; T = 0.5 sharpens it to 86% (more deterministic, more repetitive); T = 1.5 flattens it to 46% (more diverse, riskier — even the unlikely token 5 now gets 3%). Top-k = 3 zeros out the bottom three tokens and renormalizes the top three. Top-p = 0.9 happened to keep the same three tokens here (their cumulative probability first exceeds 0.9 at the third), but its cutoff is adaptive — on a flatter distribution it would keep more. In practice you combine them: a typical setting is temperature ≈ 0.7–1.0 with top-p ≈ 0.9, giving fluent text with controlled variety. Lowering temperature (or using greedy) for factual/code tasks where you want determinism; raising it for brainstorming and creative writing. That single choice is one of the most practical levers you have when using any LLM.

★ Key points to remember
  • The model outputs a distribution; decoding picks the token — the dial between safe/repetitive and diverse/creative.
  • Greedy = always the top token (coherent but dull); beam search keeps the best partial sequences (good for translation).
  • Sampling draws randomly by probability, adding natural variety.
  • Temperature sharpens ($T<1$) or flattens ($T>1$); top-k keeps the k best; top-p keeps the smallest set summing to p.
  • Common default: temperature ≈ 0.7–1.0 + top-p ≈ 0.9; lower temperature for factual/code, higher for creative.

Practice Build the reasoning yourself

1In the worked example, $T = 0.5$ gave the top token 86% probability while $T = 1.5$ gave it 46%. Explain mechanically why temperature has this effect.
Show solution & reasoning

Temperature divides the logits before the softmax. Dividing by $T < 1$ (e.g., 0.5 = multiply by 2) magnifies the gaps between logits, so the softmax — which exponentiates differences — produces a much more peaked distribution (the top token dominates). Dividing by $T > 1$ shrinks the gaps, so the logits become more similar and the softmax flattens toward uniform.

Concretely, the gap between the top two logits (3.0 and 2.0) is 1.0. At $T=0.5$ it becomes 2.0 (a bigger gap → sharper split, top token 86%); at $T=1.5$ it becomes 0.67 (smaller gap → flatter, top token 46%). So temperature is literally a "contrast" knob on the distribution: low temperature increases contrast (confident), high temperature decreases it (exploratory). At $T \to 0$ the gaps become infinite and you get greedy; at $T \to \infty$ everything becomes uniform random.

2A user complains their LLM keeps repeating phrases and producing bland, generic stories. They're using greedy decoding. What would you change, and why might pure high-temperature sampling overcorrect?
Show solution & reasoning

Switch from greedy to sampling with a moderate temperature (≈0.8–1.0) and top-p ≈ 0.9. Greedy's "always the safest token" is exactly what produces bland, repetitive text and loops; sampling injects the variety that makes stories feel alive, and top-p keeps the choices sensible by trimming the nonsensical tail.

Pure high-temperature sampling without a top-k/top-p cutoff can overcorrect: flattening the distribution gives real probability to the long tail of unlikely, often incoherent tokens, so the text becomes random and loses coherence ("the elephant quantum sang Tuesday"). That's why temperature is almost always combined with top-p (or top-k): top-p removes the garbage tail, and temperature controls diversity among the sensible options. The sweet spot is "varied but still coherent," which neither greedy nor unconstrained high-temperature achieves alone.

3Top-k always keeps a fixed number of tokens; top-p keeps a variable number. Give a scenario where top-p's adaptiveness is clearly better.
Show solution & reasoning

Consider two different steps. (1) The model is very confident — one token has 0.95 probability. Top-k = 40 would still keep 39 other low-probability tokens, occasionally sampling something wrong when the answer was nearly certain. Top-p = 0.9 keeps just the one token (it alone exceeds 0.9), correctly staying decisive. (2) The model is very uncertain — probability spread over 100 plausible tokens. Top-k = 40 arbitrarily cuts off 60 reasonable options; top-p = 0.9 keeps as many as needed to reach 0.9, preserving the genuine diversity.

So top-p adapts the cutoff to the model's confidence: few tokens when confident, many when uncertain — which matches what you actually want. Top-k's fixed count is either too permissive (when the model is sure) or too restrictive (when it's unsure). This adaptiveness is why top-p (nucleus) sampling became the more popular default, though many systems use both together as a belt-and-suspenders cutoff.

★ Deep dive: why beam search fails at open-ended text, and repetition penalties▸

A counterintuitive finding: for open-ended generation, the highest-probability sequence is often worse than a sampled one. Beam search maximizes total sequence probability, yet its outputs are frequently repetitive and strangely bland — the famous "neural text degeneration" result. Why? Because human language isn't maximally probable; real text constantly takes mildly surprising turns, while the single most-likely continuation tends to be a safe, generic, often looping phrase. Maximizing probability optimizes for the wrong thing in creative settings. This is why chatbots and story generators use sampling (which deliberately includes some lower-probability tokens) rather than beam search, even though beam finds "better" sequences by the probability metric. Beam search remains excellent for translation and summarization, where there genuinely is a high-probability correct answer and you want the most likely output — the task structure, not the model, decides which decoder is right.

Several practical add-ons round out real decoding. Repetition penalties down-weight tokens (or n-grams) that have already appeared, directly attacking the "the the the" loop and phrase repetition — a simple, effective fix layered on top of sampling. Min-p and other newer methods refine the cutoff further. Beam search with sampling hybrids exist. And for structured output (valid JSON, code that parses, a token from a fixed list), constrained decoding masks out any token that would violate the grammar before sampling — guaranteeing well-formed output, which is invaluable for tool use (Track 8) and API responses. The broad lesson is that decoding is a genuine, underappreciated lever: two products built on the same model can feel completely different based on these settings, and matching the decoder to the task (deterministic for facts/code, sampled-and-penalized for creative text, constrained for structured output) is a real skill. When an LLM's output feels off — too repetitive, too random, malformed — the decoding settings are often the first place to look, before you blame the model itself.

Decoding controls how the model speaks; the prompt controls what you ask and how well it responds. Next, in 6.5 Prompting techniques, we learn the patterns — zero/few-shot, chain-of-thought, structure — that reliably get better answers from the same model.
