Track 3 · Classical ML

Interview check: Classical ML

⏱ 15 min Lesson 3.13
Why this existsKnowing classical ML is not the same as explaining it under pressure. This checkpoint rehearses the exact Track-3 questions interviewers ask, with model answers that show the reasoning.
By the end, you can
  • Explain bias-variance, L1 vs L2, precision vs recall crisply
  • Choose bagging vs boosting and justify it
  • Define and prevent data leakage
  • Walk through a tabular problem end to end

Concept What a classical-ML interview is really testing

Knowing the models isn't the same as explaining them under pressure. A classical-ML interview round probes whether you understand why things work, can pick the right tool for a situation, and can reason about tradeoffs out loud. The questions below are the ones that come up again and again — for each, the model answer shows not just the fact but the reasoning, which is what interviewers actually score.

Model answer "Explain the bias–variance tradeoff."

"Total expected error breaks into three parts: bias², variance, and irreducible noise. Bias is error from a model being too simple to capture the pattern — it underfits, with high error even on training data. Variance is error from a model being too sensitive to the particular training sample — it overfits, acing training but failing on new data. As I increase model complexity, bias falls but variance rises, so test error is U-shaped, and I want the bottom of that U. Irreducible noise is the data's inherent randomness — a floor no model beats. In practice I diagnose this from the train–test gap: a small gap with high error means high bias (add complexity); a large gap means high variance (simplify, regularize, or get more data)."

Why it lands: it defines each term, connects to the U-curve, and ends with the actionable diagnostic — exactly the depth interviewers want.

Model answer "Why does L1 regularization produce sparse models but L2 doesn't?"

"Both add a penalty on weight size, but the shape of the penalty differs. Picture the constraint region: L2 is a circle, L1 is a diamond with corners on the axes. The loss contours grow until they touch that region. A circle gets touched at a generic point where all weights are nonzero, so L2 just shrinks weights. A diamond's corners stick out on the axes, and the contours tend to touch a corner — where some weight is exactly zero. So L1 zeros out features, giving automatic feature selection. Algebraically, L1's pull toward zero is constant regardless of weight size, so it can push a weight all the way to zero, while L2's pull shrinks as the weight shrinks and never quite reaches zero."

Bonus point to drop: "L1 is lasso, L2 is ridge, and Elastic Net blends them for correlated features."

Model answer "When would you choose precision over recall, or vice versa?"

"It depends entirely on the cost of each error. Recall = of all real positives, how many I caught; optimize it when a miss (false negative) is expensive — cancer screening, fraud detection. Precision = of everything I flagged, how many were right; optimize it when a false alarm (false positive) is expensive — say, auto-blocking accounts, where wrongly punishing a real user is costly. There's a tradeoff: lowering the decision threshold raises recall but lowers precision. If I need one number I'll use F1, but I'd push back that the right answer is to assign costs to each error and pick the threshold that minimizes expected cost. And I'd never use plain accuracy on imbalanced data — predicting the majority class can score 99% while being useless."

The "assign costs, minimize expected cost" line signals senior thinking.

Model answer "Bagging vs boosting — what's the difference?"

"Both are tree ensembles, but they attack different errors. Bagging (random forests) trains many deep trees independently on random resamples and averages them — the trees' errors are roughly independent and cancel, so it mainly cuts variance. It's parallel, robust, and hard to overfit. Boosting (XGBoost, LightGBM) builds shallow trees sequentially, each one fitting the errors the previous ensemble left behind, so it mainly cuts bias and usually reaches higher accuracy. The cost: boosting is sequential and easier to overfit, so it needs more careful tuning — learning rate, number of trees, depth. On a new tabular problem I'd start with a random forest for a robust baseline, then try gradient boosting if I need more accuracy and can invest in tuning."

Model answer "What is data leakage, and how do you prevent it in cross-validation?"

"Leakage is when information from outside the training data sneaks into the model, making evaluation over-optimistic. The classic case: scaling or imputing using statistics computed over the whole dataset before splitting, so the model indirectly 'sees' the test set. In cross-validation it's subtler — if I scale once up front, every fold's training data was scaled using its own validation fold's statistics. The fix is structural: put every step that learns from data (scaling, imputation, feature selection, PCA) inside a pipeline, so cross-validation refits the whole chain on each fold's training portion only. And I report the final number on a test set the tuning never touched, or use nested CV. Most 'too good to be true' results are leakage."

★ Rapid-fire answers to have ready
  • Why squared error? Penalizes big misses, smooth and convex, = maximum likelihood under Gaussian noise.
  • Why sigmoid + cross-entropy? Maps scores to probabilities; convex pairing; punishes confident wrong answers; = maximum likelihood for Bernoulli.
  • Generative vs discriminative? Discriminative models (logistic regression) learn the boundary $P(y|x)$; generative learn $P(x,y)$.
  • Why standardize? Distance- and variance-based methods (k-means, SVM, PCA, regularized models) are scale-sensitive.
  • Curse of dimensionality? In high dimensions points become near-equidistant and data sparse; motivates dimensionality reduction.
  • How pick k in k-means? Elbow of inertia, or peak silhouette score.

Q1. An interviewer says: "My model has 99% accuracy on a dataset where 1% of cases are fraud." Best response?

  • A"Great, ship it."
  • B"Accuracy is the only metric that matters."
  • C"Accuracy is misleading here — predicting 'no fraud' always scores 99%. What are precision and recall on the fraud class?"
  • D"You need a neural network."
99% could mean it catches zero fraud — never accept accuracy alone on imbalanced data.
Accuracy is exactly the wrong sole metric under imbalance.
Correct. Naming the trivial-majority baseline and asking for precision/recall is exactly the insight being tested.
Model choice isn't the issue; the metric is.

Q2. "You're overfitting badly. Name three distinct fixes." A strong answer includes:

  • AIncrease model complexity, train longer, raise the learning rate
  • BAdd regularization, get more data, reduce model complexity (and/or ensemble via bagging)
  • CRemove the test set, report training accuracy, lower the threshold
  • DSwitch metrics until it looks good
These mostly increase variance — the opposite of what overfitting needs.
Correct. All three reduce variance: penalize complexity, average out noise with more data, or use a simpler/averaged model.
These hide the problem rather than fix it — and removing the test set is malpractice.
Metric-shopping is self-deception, not a fix.

Q3. "Your classes are two interleaving spirals. Which model handles this with least fuss?"

  • APlain logistic regression
  • BA linear SVM
  • CLinear regression
  • DAn RBF-kernel SVM or a tree ensemble — both capture nonlinear boundaries
Logistic regression draws a straight boundary — hopeless on spirals without engineered features.
A linear SVM is still linear; same problem.
Linear regression isn't even a classifier here.
Correct. A nonlinear kernel or a tree ensemble can carve the curved boundary spirals require.

Q4. "Why report a 5-fold CV score instead of a single validation score?"

  • AIt averages over 5 splits, removing the luck of one split and giving a stability estimate (the std)
  • BIt trains faster than one split
  • CIt eliminates the need for a test set
  • DIt guarantees the model won't overfit
Correct. Rotating the validation fold and averaging removes single-split luck and quantifies reliability.
It's slower — k trainings — but more reliable.
You still want an untouched test set (or nested CV) for the final number.
CV estimates performance; it doesn't prevent overfitting by itself.

Practice Mock interview prompts

1"Walk me through how you'd approach a brand-new tabular classification problem, end to end." (2-minute answer.)
Show a strong answer

"First I'd frame it: confirm the target, the features, and the metric that matches the business cost (precision vs recall vs something custom). I'd split off a test set immediately, stratified, and not touch it. Then EDA — check distributions, missingness, leakage, class balance. For modeling I'd start simple: a regularized logistic regression as an interpretable baseline, with all preprocessing inside a pipeline to avoid leakage. Then I'd try a gradient-boosting model, which usually wins on tabular data. I'd tune hyperparameters with cross-validation (random search), compare models on the CV score and its variance, and only at the very end evaluate the chosen model once on the test set. I'd report the metric with a confidence range, inspect the confusion matrix and feature importances, and sanity-check for leakage if a score looks too good."

This hits every box of the loop and signals the discipline (split first, pipeline, CV, test once) that interviewers prize. Saying "start simple, then boost" shows judgment, not just tool-knowledge.

2"Your random forest gets 94% but the business needs to know why each prediction is made. What do you do?"
Show a strong answer

"There's a tension between accuracy and interpretability. First I'd quantify the cost of switching to a transparent model: train a single shallow decision tree or a regularized logistic regression and see how much accuracy I lose. If it's small, the interpretable model may be worth it for the explainability. If the forest's accuracy is essential, I'd keep it but add post-hoc explanation: global feature importances (ideally permutation importance) for overall drivers, and per-prediction explanations with something like SHAP values to show which features pushed a given case toward its prediction. I'd also confirm with stakeholders what 'why' they need — a global sense of drivers, or a defensible per-decision reason (e.g., for regulatory reasons), since those call for different tools."

The strong move is naming the accuracy/interpretability tradeoff explicitly and offering both a model-swap and a post-hoc-explanation path rather than treating it as either/or.

3"A colleague's model scored 0.97 in offline tests but performs terribly in production. List the most likely culprits."
Show a strong answer

"Top suspect: data leakage — a feature that won't be available at prediction time, or preprocessing fit on the full dataset before splitting, which inflates offline scores. Second: distribution shift — production data differs from training data (new users, seasonality, changed behavior), so the i.i.d. assumption broke. Third: train/serving skew — the features are computed differently in the training pipeline than in the live system. Fourth: an over-optimistic evaluation — e.g., reporting the best of many tuned configurations without nested CV, or a non-stratified split on imbalanced data. I'd start by auditing the features for leakage and comparing the training vs production feature distributions."

This shows you understand that a great offline number is necessary but not sufficient — the gap between offline and production is where real ML engineering lives, and it previews Track 10 (MLOps).

★ Deep dive: how to communicate, not just compute▸

The biggest difference between candidates who pass ML interviews and those who don't is rarely knowledge — it's communication of reasoning. A few habits that consistently score well. Define terms before using them: say "recall — of all real positives, the fraction I caught — " so the interviewer knows you understand, not just memorized. Always name the tradeoff: almost every good answer in ML ends with "...but the cost is X," because mature practitioners think in tradeoffs (accuracy vs interpretability, bias vs variance, precision vs recall, compute vs quality). Tie choices to the problem: "I'd use recall here because a missed fraud costs more than a false alarm" beats reciting definitions.

When you don't know something, reason out loud from fundamentals rather than guessing — interviewers are testing how you think, and "I'm not sure, but I'd reason about it like this..." often scores better than a confident wrong fact. And resist the urge to name the fanciest model: proposing a regularized logistic regression baseline before a deep net signals the judgment that you fit the tool to the problem. The throughline of this whole track — frame honestly, choose deliberately, evaluate without fooling yourself — is exactly the mindset these interviews are built to detect.
