RAG Pipeline Lab

Retrieval-Augmented Generation in one screen: your question is embedded, compared to a library of documents by cosine similarity, and the best matches are handed to the model as context. Ask a question and watch every stage run — real vector retrieval, no libraries.

The whole point of RAG is that the model answers from retrieved facts, not just its memory. Type a question below and watch it become a vector, pull the most similar chunks out of the knowledge base, and assemble the exact context an LLM would read before answering. Then ask something the library doesn't cover — and watch retrieval come up empty.

▶ Ask the knowledge base

8 document chunks. Your query is embedded into the same space, then ranked by cosine similarity. The top-k become the model's context.

1 Retrieval — every chunk scored by cosine similarity to your query
2 Context assembled — only the top-k are sent to the model
3 Grounded answer — the model responds using that context

Why not just ask the model directly? The model's weights are frozen at training time — it can't know your private docs or today's news, and it will confidently make things up. RAG fixes both: it fetches real, current, citable text and puts it in front of the model, so the answer is grounded in something you can check. Retrieval quality is everything — if the right chunk isn't fetched, the model can't use it.

Try these

1. Watch a good match win. Ask about the learning rate. The gradient-descent chunk scores near 1.0 and floats to the top; unrelated chunks score near zero. That gap is what makes retrieval work.

2. Shrink k to 1. Now only the single best chunk reaches the model. Great when retrieval is confident, risky when the answer spans several documents — a real tuning decision.

3. Ask something off-topic. Ask about cooking or football. Every score collapses toward zero — the library simply doesn't contain the answer. A good RAG system detects this and says "I don't know" instead of hallucinating.

What this lab proves
  • RAG = retrieve then generate. Embed the query, find the nearest chunks by cosine similarity, feed them in as context.
  • Similarity is a dot product of directions. Cosine ignores length and measures alignment of meaning — that's the whole retrieval engine.
  • You can only answer what you retrieve. If the right chunk isn't fetched, the model can't use it — retrieval quality caps the whole system.
  • Grounding beats memorizing. The answer cites fetched text, so it stays current, private-aware, and checkable.
