Track 3 · Classical ML

Loss functions & evaluation metrics

⏱ 26 min Lesson 3.4 ▶ Open in Colab
Why this existsA model is only as trustworthy as the metric you judge it by. Accuracy hides disasters on imbalanced data; precision, recall, and AUC expose what is really happening — and which mistakes you are making.
By the end, you can
  • Explain why accuracy misleads on imbalanced data
  • Read a confusion matrix (TP/FP/FN/TN)
  • Compute precision, recall, and F1
  • Use thresholds and ROC-AUC; match the metric to cost

Concept Why accuracy can be a dangerous lie

Accuracy — the fraction of predictions you got right — is the obvious metric, and for balanced problems it's fine. But it collapses the moment one class is rare. Suppose 99.5% of credit-card transactions are legitimate. A model that blindly predicts "legitimate" for everything scores 99.5% accuracy while catching exactly zero fraud. It's useless, yet it looks excellent. This is the trap of class imbalance, and escaping it is the whole reason this lesson exists.

To measure a classifier honestly we need to look at the kinds of mistakes it makes, not just the count. A missed fraud and a false alarm are completely different errors with completely different costs, and a good metric keeps them separate.

Concept The confusion matrix: four outcomes

Every prediction on a two-class problem lands in one of four boxes, named by combining whether the prediction was right (true) or wrong (false) with what it guessed (positive = the class of interest, negative = the other one):

  • True Positive (TP): predicted positive, and it really was. ✓
  • True Negative (TN): predicted negative, and it really was. ✓
  • False Positive (FP): predicted positive, but it was negative — a false alarm.
  • False Negative (FN): predicted negative, but it was positive — a miss.

Which error is worse depends entirely on the application — a false alarm on a spam filter is mildly annoying, but a missed tumor is catastrophic. Naming the four counts lets us build metrics that reflect those stakes.

Concept Precision and recall: two questions, two denominators

The two most important metrics both focus on the positive class, but ask different questions. Precision asks: of everything I flagged positive, what fraction was actually positive? — $\text{TP}/(\text{TP}+\text{FP})$. It punishes false alarms. Recall (also called sensitivity) asks: of all the real positives out there, what fraction did I catch? — $\text{TP}/(\text{TP}+\text{FN})$. It punishes misses.

There's almost always a tension between them: catch more positives (higher recall) and you'll inevitably flag some negatives too (lower precision), and vice versa. When you want a single number that balances both, use the F1 score — the harmonic mean $F_1 = 2\cdot\frac{\text{precision}\cdot\text{recall}}{\text{precision}+\text{recall}}$, which is only high when both are high.

Concept The threshold dial, and ROC-AUC

Recall from Lesson 3.3 that a classifier outputs a probability, and you choose a threshold to turn it into a decision. Sliding that threshold trades precision for recall: lower it and you catch more positives but raise false alarms. Rather than commit to one threshold, the ROC curve plots the trade-off across all thresholds — true-positive rate against false-positive rate — and the AUC (area under the curve) summarizes it in one number: 1.0 is a perfect classifier, 0.5 is a coin flip.

One caveat: on very imbalanced data, ROC-AUC can look flatteringly high. There the precision-recall AUC (PR-AUC) is more informative, because it ignores the easy true negatives and focuses on how well you handle the rare positive class.

Worked example Reading a real classifier's mistakes

We score the logistic-regression tumor classifier from Lesson 3.3 on its held-out test set — but instead of one accuracy number, we look at where it errs. Here the class we truly care about catching is malignant (label 0), so we'll treat that as the positive class.

from sklearn.metrics import confusion_matrix, precision_score, recall_score
y_pred = clf.predict(X_test_s)

print(confusion_matrix(y_test, y_pred))   # rows = actual, cols = predicted; order [0, 1]
Output[[40 2] [ 0 72]]

Reading it: of 42 truly malignant tumors, 40 were caught and 2 were called benign (the dangerous misses); of 72 benign tumors, all 72 were labeled correctly. Now the metrics, with malignant as the positive class:

print("recall (malignant caught):", round(recall_score(y_test, y_pred, pos_label=0), 3))
print("precision (malignant):     ", round(precision_score(y_test, y_pred, pos_label=0), 3))
Outputrecall (malignant caught): 0.952 precision (malignant): 1.0

The 98.2% accuracy from Lesson 3.3 hid something important: recall on malignancy is only 95.2% — the model misses about 1 in 20 cancers. In a screening tool that might be unacceptable, and you'd lower the threshold to raise recall, accepting more false alarms (precision would drop from its perfect 1.0). That decision is invisible in the accuracy number and obvious in the confusion matrix — which is exactly why we never judge a classifier by accuracy alone.

★ Key points to remember
  • Accuracy lies on imbalanced data — a "predict the majority" model can score high while being useless.
  • The confusion matrix (TP, TN, FP, FN) separates the two kinds of error so you can weigh their costs.
  • Precision = of flagged, how many right (punishes false alarms); recall = of real positives, how many caught (punishes misses).
  • F1 balances precision and recall; ROC-AUC measures ranking quality across all thresholds.
  • Pick the metric that matches the real-world cost of each mistake — there is no universal "best" metric.

Practice Build the reasoning yourself

1A model's confusion matrix has TP = 80, FP = 20, FN = 40, TN = 860. Compute accuracy, precision, and recall.
Show solution & reasoning

Total = 80+20+40+860 = 1000. Accuracy = (TP+TN)/total = (80+860)/1000 = 0.94.

Precision = TP/(TP+FP) = 80/100 = 0.80. Recall = TP/(TP+FN) = 80/120 ≈ 0.67.

Notice accuracy (0.94) looks great while recall (0.67) reveals the model misses a third of the positives — hidden by the large, easy true-negative count. This is the imbalance trap in miniature: always check precision and recall, not just accuracy.

2Two models for the same task: Model A has precision 0.9, recall 0.5; Model B has precision 0.6, recall 0.85. Which is "better"?
Show solution & reasoning

There's no answer without knowing the cost of each error — that's the point. Model A rarely raises a false alarm but misses half the positives; Model B catches most positives but with more false alarms.

For cancer screening or fraud detection (misses are deadly/expensive), B's high recall wins. For something like flagging emails for a human to delete (false alarms waste a person's time and a miss is harmless), A's high precision wins. If you must compare with one number, the F1 scores are A: $2(0.9)(0.5)/(1.4)\approx 0.64$ and B: $2(0.6)(0.85)/(1.45)\approx 0.70$, slightly favoring B — but F1 assumes the two errors matter equally, which is exactly the assumption you should question first.

3Someone reports "my model has 0.97 ROC-AUC" on a dataset where 0.5% of examples are positive, and claims it's nearly perfect. Why be skeptical, and what would you ask to see?
Show solution & reasoning

On extreme imbalance, ROC-AUC is easy to inflate because it rewards getting the vast pool of negatives ranked below positives — and with 99.5% negatives, a model can score high on AUC while still being impractical at any usable threshold.

Ask for the precision-recall curve and PR-AUC, plus precision and recall at the specific operating threshold they'd actually deploy. PR metrics focus on the rare positive class and ignore the easy true negatives, so they expose whether the model is genuinely useful for catching the needle in the haystack. Also ask how many false alarms per real positive their threshold produces — a concrete, deployable number.

★ Deep dive: averaging across classes, and matching metrics to money▸

With more than two classes, precision and recall are computed per class and then averaged — and how you average matters. Macro-averaging takes the simple mean across classes, treating a rare class as equally important as a common one (good when you care about minority classes). Micro-averaging pools all the TP/FP/FN counts first, which lets common classes dominate (and equals overall accuracy in the single-label case). Weighted averaging splits the difference by class frequency. Reporting the wrong average can make a model look much better or worse than it is on the classes you care about.

The deepest version of "pick the right metric" is to skip the proxy and compute expected cost directly. Assign a dollar (or utility) value to each cell of the confusion matrix — a missed fraud costs $200, a false alarm costs $5 of review time — and choose the threshold that minimizes total expected cost on the validation set. This turns a fuzzy "precision vs recall" debate into an optimization with a clear answer, and it's how mature teams actually set thresholds in production. Metrics like F1 are convenient defaults; the cost matrix is the real objective.

Finally, a reminder that links back to Lesson 3.3: all of these threshold-based metrics assume the model's probabilities mean something. If the model is poorly calibrated, a "0.8" might really be a 0.5, and your carefully chosen threshold won't behave as expected. Good evaluation pairs a discrimination metric (AUC, F1) with a calibration check — a theme we'll develop fully in Track 12 on evaluation and safety.
