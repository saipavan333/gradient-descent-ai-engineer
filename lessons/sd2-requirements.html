Requirements & SLAs: latency, throughput, cost

⏱ 22 min Lesson 10.2 ⬇ Runnable notebook
Why this exists'Fast and cheap' isn't a design. Requirements split into functional (what it does) and non-functional (latency, throughput, cost); SLAs target percentiles (p95/p99), not averages, because the tail is what users feel.
By the end, you can
  • Distinguish functional and non-functional requirements
  • Build a latency budget
  • Explain why SLAs use percentiles
  • Size a fleet from QPS and latency

Concept Turning "fast and cheap" into numbers

"The system should be fast, cheap, and handle lots of users" is not a design — it's a wish. The first real work of system design is turning vague desires into requirements precise enough to design against, split into two kinds. Functional requirements describe what the system does ("recommend 10 videos," "return search results"). Non-functional requirements describe how well it must do it — the qualities of service: how fast, how much load, how reliable, how cheap. In ML systems the non-functional requirements often dominate the architecture, because a model that's accurate but too slow or too expensive to serve is useless. The three that come up constantly are latency, throughput, and cost.

Latency is the time to handle one request — the delay a user feels. Throughput is how many requests the system handles per unit time (closely tied to QPS, Lesson 10.1). Cost is the money to run it — compute, storage, serving. These trade off against each other and against accuracy: a bigger model raises accuracy but worsens latency and cost; more replicas raise throughput but cost more. A design is largely the act of balancing these to meet the requirements.

A powerful way to make latency concrete is a latency budget: the total allowed response time, divided among the stages of handling a request. If the budget is 200ms and the network, feature lookup, and post-processing consume 110ms between them, the model gets only 90ms — which directly limits how big and slow a model you can use. This is why latency isn't an afterthought: it constrains the model itself, sometimes forcing a smaller or compressed model (Lesson 10.6) even at a small accuracy cost, because an accurate model that blows the budget is unusable.

Concept SLAs and the tyranny of the tail

Requirements get formalized as an SLA (Service Level Agreement): a concrete, promised target, like "99% of requests return in under 200ms" or "99.9% uptime." The subtle, crucial detail is that good SLAs are written on percentiles, not averages. The p95 latency is the value that 95% of requests come in under (5% are slower); p99 is the same for 99%. Why not just use the average? Because averages hide the tail — the slowest requests — and the tail is what users actually suffer. An average latency of 50ms sounds great, but if the p99 is 2 seconds, then 1 in 100 requests is painfully slow, and a user making many requests will hit that tail regularly.

Worked example Sizing the fleet from latency and QPS

Requirements turn into capacity. Given the ~7,000 peak QPS from Lesson 10.1 and a per-request latency of 100ms, how many server replicas do you need? A simple capacity estimate:

import math
peak_qps   = 6_944
latency_ms = 100
per_replica_qps = 1000 / latency_ms     # 1 worker, 100ms/request -> 10 req/s
replicas = math.ceil(peak_qps / per_replica_qps)
print(f"per-replica QPS : {per_replica_qps:.0f}")
print(f"replicas needed : {replicas}")
print(f"with 30% headroom: {math.ceil(replicas * 1.3)}")
Outputper-replica QPS : 10 replicas needed : 695 with 30% headroom: 904

The logic chains the requirements together. If one worker takes 100ms per request and handles them one at a time, it serves 10 requests per second. To handle 7,000 QPS at peak, you need 7,000 ÷ 10 = ~695 replicas — and you'd add headroom (say 30%) for safety and traffic above the estimate, giving roughly 900 servers. That's a striking, design-shaping number: it tells you this system needs a serious fleet with a load balancer and autoscaling (Lesson 10.6), and it makes the latency-cost link vivid — if you could cut per-request latency from 100ms to 50ms (via a smaller model or batching), each replica would handle 20 QPS and you'd need half the servers, roughly halving the cost. That's the lever latency gives you over the bill. (The model here is simplified — real servers handle requests concurrently, so the true number is lower — but the back-of-the-envelope reasoning and the conclusion, "this needs hundreds of replicas, and latency directly drives that count," is exactly what an interview wants.) Notice how requirements (QPS, latency) became capacity (replicas) became cost — the chain that turns "fast and cheap" into an actual architecture.

★ Key points to remember
  • Functional requirements = what it does; non-functional = how well (latency, throughput, cost, reliability).
  • Latency = time per request; throughput = requests per second; cost = money to run — they trade off against each other and accuracy.
  • A latency budget divides the total time among stages; the model's slice constrains how big a model you can use.
  • An SLA is a promised target, written on percentiles (p95, p99), not averages — because the tail is what users feel.
  • Requirements → capacity → cost: QPS and latency together size the server fleet.

Practice Build the reasoning yourself

1A product manager says "the recommendation API just needs to be fast." Turn this into a concrete, designable latency requirement, and explain each choice.
Show solution & reasoning

"Fast" needs three specifications: a number, a percentile, and the scope. A concrete version: "p99 latency under 150ms for the recommendation API, measured server-side." Each part matters. The number (150ms) comes from the user experience — recommendations load as part of a page, and research/UX norms say sub-200ms feels instant, so 150ms leaves room for the rest of the page. The percentile (p99, not average) ensures even the slow tail is acceptable, because a user scrolling a feed makes many requests and would hit a bad tail repeatedly — an average target would let 1% of requests be terrible and still "pass." The scope (server-side, this API specifically) clarifies what's measured, since end-to-end latency includes network and client time outside your control.

The reasoning skill is recognizing that "fast" is underspecified along exactly these axes, and that each must be pinned to design against it. Once you have "p99 < 150ms," you can build a latency budget (allocate the 150ms across network, feature lookup, model, post-processing), which then constrains the model size and the serving design — the chain from requirement to architecture. You'd also confirm the requirement is right by asking what the cost of missing it is (does a slow recommendation just feel sluggish, or does it lose the sale?), since that tells you how hard to push. Turning a fuzzy "fast" into "p99 < 150ms server-side" is the small but pivotal act that makes the rest of the design possible — and shows the interviewer you know that requirements must be measurable to be useful.

2Your model's p50 latency is 40ms but its p99 is 800ms. Users complain the app feels "randomly slow." Explain what's happening and how you'd approach it.
Show solution & reasoning

The gap between p50 (40ms) and p99 (800ms) is a classic long-tail latency problem: most requests are fast, but the slowest 1% are 20× slower, and those are what users notice — hence "randomly slow." Because a user making many requests will hit that tail repeatedly, the experience feels unreliable even though the median is excellent. The average would look fine and hide this entirely, which is exactly why you monitor percentiles. The first step is to investigate the tail's causes: common culprits include occasional cache misses (most requests hit a fast cache, the few misses are slow), garbage-collection pauses, cold starts (a replica just spun up), contention/queueing under load bursts, certain expensive inputs (a very long document), or one slow dependency (a feature lookup that occasionally stalls).

Then you target the specific cause. If it's cache misses, warm the cache or speed the miss path. If it's queueing under bursts, add capacity or autoscale faster. If it's expensive inputs, cap or special-case them. If it's a slow dependency, add a timeout and a fallback. A general tactic for tail latency is hedging: send the request to two replicas and take the first response, so one slow replica doesn't sink the request. The key reasoning move is treating tail latency as its own problem with its own causes, distinct from average latency — you don't fix a p99 problem by making the median faster; you find and address what makes the slow requests slow. Recognizing that the p50/p99 gap is the issue, and reasoning about tail-specific causes and mitigations, is exactly the depth that separates someone who's operated real latency-sensitive systems from someone who only optimizes averages.

3Explain the three-way tension between accuracy, latency, and cost in serving an ML model, with an example of trading one for another.
Show solution & reasoning

These three pull against each other, and you usually can't max all three. Accuracy vs latency/cost: a bigger, more accurate model takes longer to run (worse latency) and uses more compute (higher cost). Latency vs cost: you can cut latency by adding more/faster hardware (higher cost), or by batching (better cost/throughput but slightly worse per-request latency). So improving one axis typically costs you on another, and the "best" point depends on the requirements. Example: suppose a large model gives 94% accuracy at 300ms and $0.01/request, while a distilled model gives 92% at 60ms and $0.002/request. If your latency budget is 150ms, the large model is simply infeasible — you must take the distilled one, trading 2 points of accuracy for a 5× latency improvement and 5× cost reduction. The requirement (150ms budget) forced the trade.

The reasoning skill is seeing that there's no universally "right" model — only the model that best fits the requirements' weighting of these axes. For a high-stakes medical diagnosis where accuracy dominates and a few seconds is fine, you'd pick the big accurate model. For a real-time feed where latency and per-request cost dominate at massive scale, you'd pick the fast cheap one and accept slightly lower accuracy. This is why the "best offline model" (Track 9) is often not the right model to serve: offline evaluation optimizes accuracy alone, ignoring the latency and cost constraints that production imposes. A strong system designer treats accuracy, latency, and cost as a single joint decision driven by requirements — and says so explicitly, naming which axis the requirements prioritize and choosing accordingly. That explicit, requirement-driven trade-off reasoning is the heart of ML system design.

★ Deep dive: availability, reliability, and the full menu of non-functional requirements▸

Latency, throughput, and cost are the headline non-functional requirements, but a complete design considers more. Availability (often expressed as "nines" — 99.9% uptime is "three nines," about 8.8 hours of downtime a year; 99.99% is "four nines," about 53 minutes) sets how much downtime is tolerable, which drives redundancy: high-availability systems run multiple replicas across data centers so no single failure takes them down. Reliability/correctness asks how the system behaves under failure — does it degrade gracefully (fall back to a simpler model or cached result) or fail hard? Scalability is whether the design holds up as load grows by 10× or 100×. Consistency matters when data is replicated — do all users see the same state immediately, or is some staleness acceptable? Security and privacy (especially for user data and models) and fairness (the model behaving acceptably across groups — Track 11) round out the list. Each is a requirement you elicit and design against, and each trades against the others — the famous example being the CAP-theorem tension between consistency and availability in distributed systems.

The deeper principle is that non-functional requirements are where most of the system's real complexity and cost live. The functional requirement ("recommend videos") is often a one-liner; the non-functional ones ("at 7,000 QPS, p99 < 150ms, 99.99% available, under $X/month, fair across user groups") are what force the architecture to be sophisticated — the replicas, load balancers, caches, shards, fallbacks, and monitoring all exist to satisfy non-functional requirements, not functional ones. This is why experienced designers spend so much time pinning them down: they are the design constraints. It also explains why the same functional system (say, a recommender) can range from a weekend project to a thousand-server platform depending entirely on its non-functional requirements. As a designer, the move is to elicit these explicitly, quantify them (a number and a percentile, not an adjective), prioritize them against each other (you can't maximize all — which matters most here?), and then justify every architectural choice by the requirement it serves. That discipline — requirements as the north star, every component traceable to a requirement it satisfies — is what makes a design coherent rather than a grab-bag of fashionable components, and it's the thread that runs through every system you'll design in the rest of this track, starting with a recommender next.
