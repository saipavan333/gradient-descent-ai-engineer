Track 8 · AI Agents & Tool Use

Memory (short & long-term)

⏱ 24 min Lesson 8.5 ⬇ Runnable notebook
Why this existsAn agent's context window is short-term memory that fills up and resets. External memory — often a vector database — gives long-term recall across steps and sessions, so it can remember facts, preferences, and past results.
By the end, you can
  • Contrast short-term (context) and long-term memory
  • Explain external memory via a vector store
  • Describe storing and recalling facts
  • Manage a filling context window

Concept Two kinds of memory: working and long-term

An agent's reasoning, plan, and tool results all live in its context window (Lesson 5.1) — which is its short-term memory, the "working memory" it reasons over right now. But the context window is limited (even large ones fill up on long tasks) and it's forgotten when the conversation ends. For agents that run long, or that should remember a user across sessions, that isn't enough. So agents add long-term memory: an external store that persists and can be searched.

The two work together. Short-term (context) holds the immediate task state — what's been said and done recently. Long-term (an external store, typically a vector database) holds durable knowledge — user preferences, past conversations, accumulated facts — and is retrieved when relevant, which is exactly RAG (Track 7) applied to the agent's own memory. As the context fills, older content is summarized (compressed by the LLM) and the summary kept, or important facts are written to long-term memory and recalled later. This lets an agent maintain continuity over arbitrarily long interactions without exceeding its context window.

Worked example A simple long-term memory store

Long-term memory, at its core, is just store-and-retrieve. Here's a minimal version: remember facts, then recall the relevant ones by keyword. (Real systems retrieve by embedding similarity — Track 7 — but the store-then-recall structure is identical.)

long_term = []
def remember(fact): long_term.append(fact)
def recall(keyword):  # retrieve facts matching the cue (embedding similarity in production)
    return [m for m in long_term if keyword.lower() in m.lower()]

remember("The user's name is Alex.")
remember("The user prefers Python over Java.")
remember("The user is allergic to peanuts.")

print("stored", len(long_term), "facts")
print("recall('python') ->", recall("python"))
print("recall('peanut') ->", recall("peanut"))
Outputstored 3 facts recall('python') -> ['The user prefers Python over Java.'] recall('peanut') -> ['The user is allergic to peanuts.']

The agent stored three facts and later retrieved the relevant one on demand — recalling the Python preference when the topic is programming, and the peanut allergy when relevant (e.g., suggesting recipes). Crucially, these facts persist beyond the current conversation: a week later, the agent can recall that the user prefers Python and avoid peanut recipes, without those facts cluttering its context window the whole time. That's the value of long-term memory — durable, retrievable knowledge that gives continuity and personalization. In production, recall would embed the cue and search a vector database for semantically similar memories (so "what language does the user like?" retrieves the Python fact even without the word "Python"), and the retrieved memories get injected into the prompt exactly like RAG context (Track 7). The agent's memory is a RAG system over its own accumulated knowledge — a satisfying reuse of everything you learned last track.

★ Key points to remember
  • Short-term memory = the context window: the agent's working memory, but limited and forgotten when the session ends.
  • Long-term memory = an external store (vector DB) of durable facts, retrieved when relevant — RAG (Track 7) over the agent's own knowledge.
  • As context fills, summarize older content or write key facts to long-term memory and recall later.
  • Together they give continuity across long tasks and personalization across sessions.
  • Long-term memory is mechanically just store-and-retrieve — usually by embedding similarity.

Practice Build the reasoning yourself

1A customer-support agent should remember a user's past issues and preferences across separate conversations weeks apart. Which memory type, and how would it work?
Show solution & reasoning

Long-term memory. After each conversation, the agent writes durable facts to an external store — "user had a billing issue resolved on March 3," "user prefers email over phone," "account is on the Pro plan." These are embedded and stored in a vector database keyed to the user. When the user returns weeks later, the agent retrieves the relevant memories (by similarity to the current conversation) and injects them into context, so it "remembers" the history — exactly RAG (Track 7) over the user's memory.

Short-term memory (context) can't do this — it's wiped between sessions. The long-term store provides cross-session continuity and personalization, which transforms the experience from "starting over every time" to "the agent knows me." You'd also manage the memory carefully: store only useful, durable facts (not every message), respect privacy/consent (Track 11), and let users view/edit what's remembered. Memory done well is a major differentiator for assistant products — and it's the same retrieval machinery you already built.

2An agent on a long research task starts giving worse answers as the task drags on, seeming to "forget" earlier findings. What's likely happening, and how does summarization help?
Show solution & reasoning

The context window is overflowing. As the ReAct trace grows (every Thought, Action, and Observation accumulates, Lesson 8.3), it eventually exceeds the window, so either the system truncates older content (the agent literally loses early findings) or the model's attention is diluted across a huge context and it loses the thread. Either way, the agent "forgets" what it learned early on.

Summarization helps by compressing the older trace into a concise summary — "Findings so far: X, Y, Z; still need: W" — replacing pages of detailed history with a short recap that preserves the essentials within the window. This keeps the important state accessible without blowing the context budget. You might also write key findings to long-term memory and retrieve them as needed. Managing the context — summarizing, pruning, offloading to memory — is essential for long-running agents, and "the agent forgot earlier context" is a classic symptom of an unmanaged, overflowing window. The same finite-context constraint that shapes RAG (Track 7) shapes agent memory.

3What are the risks of an agent storing everything in long-term memory indefinitely, and how would you manage them?
Show solution & reasoning

Several risks. Noise: storing every message bloats the memory with trivia, making retrieval return junk and diluting the useful facts — so store selectively (important, durable facts, not every utterance). Staleness: facts change (the user's preferences, their plan), so old memories can become wrong — implement updates and expiry, and prefer recent info on conflicts. Privacy: remembering personal data raises consent and security concerns (Track 11) — store only what's needed, with user awareness and the ability to view and delete their memories, and protect it. Cost: unbounded memory grows storage and retrieval cost.

Management strategies: be selective about what to remember (often via an LLM step deciding "is this worth saving?"), summarize and consolidate related memories, expire or down-weight old ones, deduplicate, and give users transparency and control. Memory is powerful but it's not "store everything forever" — it's a curated, governed store, much like the production concerns of a RAG knowledge base (Track 7) plus the privacy considerations of Track 11. Thoughtful memory management is what makes the difference between an agent that helpfully remembers and one that's cluttered, stale, or a privacy liability.

★ Deep dive: memory architectures and the long-horizon challenge▸

Agent memory has grown into a rich design space because it's central to capable long-running agents. Beyond the basic short/long-term split, systems distinguish several memory types borrowed loosely from cognitive science. Episodic memory stores specific past experiences ("on March 3, the user asked about X and we did Y") — useful for recalling what happened. Semantic memory stores distilled facts and knowledge ("the user prefers Python") — the generalizations extracted from episodes. Procedural memory stores learned skills or workflows ("to handle a refund, do these steps"). Real systems often combine these: raw episodes get summarized into semantic facts, and successful action sequences become reusable procedures. The retrieval, too, gets sophisticated — recency weighting (prefer recent memories), importance scoring (some facts matter more), and reflection that periodically consolidates memories into higher-level insights (an idea from the "Generative Agents" research, where simulated agents maintained believable behavior over long periods via exactly this kind of memory architecture).

The deepest challenge memory addresses is the long-horizon problem: enabling agents to work coherently over very long tasks and very long relationships. A single context window — even a million tokens — eventually fills, and attention over enormous contexts degrades. Memory is the architectural answer: keep the immediately-relevant in context, offload the rest to a searchable store, and retrieve as needed — effectively giving the agent unbounded, organized memory on top of its bounded context. This connects to the context-length frontier of Track 5 (positional encoding, efficient attention) but takes a different, complementary tack: rather than making the context window bigger, give the agent a retrieval-based memory it can query, which scales far beyond any context window. The two approaches combine — bigger context plus smart memory — in the most capable systems. There are hard open problems here: knowing what to remember and when to retrieve it (too much retrieval clutters context, too little loses important facts), keeping memory consistent as facts change, and doing it all efficiently. Memory is also where agents start to feel genuinely personalized and persistent — an assistant that remembers you across months, learns your preferences, and builds on past interactions — which is a major frontier for assistant products, with real benefits (continuity, personalization) and real responsibilities (privacy, accuracy, user control — Track 11). The throughline: memory turns a stateless model into a stateful agent that can maintain continuity across the limited context window and across time, built largely from the retrieval machinery of Track 7, and it's essential infrastructure for any agent meant to do more than a single short task.

One agent with memory and tools is powerful — but some jobs are better split across several specialized agents. Next, in 8.6 Multi-agent patterns, we see how multiple agents coordinate, and when that helps versus hurts.
