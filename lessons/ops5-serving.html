Packaging & serving

⏱ 27 min Lesson 9.5 ⬇ Runnable notebook
Why this existsA trained model is just a file; serving turns it into a callable service. Packaging (a container) freezes the environment so it runs anywhere; serving patterns — online, batch, streaming — are chosen by how predictions are needed.
By the end, you can
  • Explain packaging and containers
  • Contrast online, batch, and streaming serving
  • Distinguish latency and throughput
  • Know why serving cost dominates training

Concept From a saved model to a service users can call

A trained model is a file — a blob of learned weights sitting on disk. That file can't do anything by itself; to be useful it has to serve predictions: receive an input, run the model, and return an output, on demand. Getting from "file on disk" to "service users can call" involves two steps. First, packaging: bundling the model file together with everything it needs to run — the inference code, and the software dependencies (the exact libraries and versions). Second, serving: running that package somewhere it can answer requests. Packaging without serving is a sealed box nobody opens; serving without proper packaging is the "works on my laptop" disaster waiting to happen.

The standard packaging tool is a container (Docker being the common one): a self-contained unit that bundles the model, the code, and all its dependencies — even a pinned version of Python and the operating-system libraries — so it runs identically on your laptop, a teammate's machine, or a cloud server. This solves the notorious "it works on my machine but breaks in production" problem, which in ML is acute because a slightly different library version can silently change a model's numerical output. A container freezes the entire environment, so the model that worked in testing is byte-for-byte the model that runs in production.

Concept Three ways to serve: online, batch, streaming

Once packaged, a model can be served in three patterns, chosen by how predictions are needed. Online serving (also called real-time or synchronous) wraps the model behind an API endpoint — a URL the application calls with one input and gets one prediction back immediately; this is what you use when a user is waiting (a fraud check at checkout, a recommendation as the page loads), and it optimizes for low latency (the delay before a response). Batch serving runs the model on a large pile of inputs at once, on a schedule — e.g., overnight, score all 10 million customers for churn and write the results to a database; nobody's waiting in real time, so it optimizes for throughput (total volume processed) rather than per-request speed. Streaming serving sits in between, scoring events continuously as they flow through a real-time data stream (e.g., scoring transactions as they happen).

Worked example The same model, served two ways

The model logic doesn't change between serving patterns — only how inputs arrive. Here a single predict function (standing in for the packaged model) is served online (one request) and batch (many inputs at once):

def predict(x): return x * 2                          # the packaged 'model'

# online: one request at a time, answered immediately
print("online (1 request) :", predict(5))

# batch: score many inputs at once, on a schedule
batch = [1, 2, 3, 4, 5]
print("batch (many at once):", [predict(x) for x in batch])
Outputonline (1 request) : 10 batch (many at once): [2, 4, 6, 8, 10]

The exact same predict is used both ways. Online, it's called with a single input (predict(5) → 10) — this is the shape of a real-time endpoint: one request in, one prediction out, immediately, while a user waits. Batch, it's mapped over a whole list at once ([predict(x) for x in batch] → [2, 4, 6, 8, 10]) — the shape of a scheduled job scoring millions of records and writing them somewhere. The model is identical; what differs is the serving pattern wrapped around it. That's the key insight: packaging produces one model artifact, and you choose the serving pattern based on the use case's needs — low latency for online, high throughput for batch — without changing the model itself. In a real deployment the online version lives behind a web endpoint and the batch version runs as a scheduled pipeline, but both load the same packaged model file; the predict here is that shared core.

★ Key points to remember
  • Packaging bundles the model + code + dependencies; a container (Docker) freezes the whole environment so it runs identically anywhere.
  • Serving runs the package so it can answer prediction requests.
  • Online serving = real-time endpoint, one request at a time, optimized for low latency (user waiting).
  • Batch serving = score huge volumes on a schedule, optimized for throughput (nobody waiting live).
  • Streaming serving = score events continuously as they flow in; pick the pattern by how predictions are needed.

Practice Build the reasoning yourself

1A model works perfectly in the data scientist's notebook but produces different predictions when an engineer runs it on the production server. The model file is identical. What's likely wrong, and how does packaging fix it?
Show solution & reasoning

The likely culprit is an environment mismatch: the production server has different versions of the libraries (or Python, or underlying numerical/OS libraries) than the notebook. ML predictions can depend on these in subtle ways — a different version of scikit-learn or NumPy can change how a computation rounds, how a preprocessing step behaves, or even how the saved model is deserialized, producing different outputs from the same model file. This is the classic "works on my machine" failure, and it's especially pernicious in ML because it doesn't crash — it just gives subtly different (wrong) numbers, much like train/serve skew (Lesson 9.3).

Packaging in a container fixes it by freezing the entire environment alongside the model: the container specifies the exact Python version, the exact library versions, and the system libraries, so the production server runs the identical environment the notebook used. The container the data scientist builds and tests is the same container that runs in production — there's no "the server has different versions" gap because the versions travel with the model. This is why containers are the standard ML packaging tool: they convert "depends on whatever happens to be installed on this machine" into "carries its own complete, pinned environment." The general principle is reproducibility extended from training (Lesson 9.2) to deployment — the same instinct (pin everything, leave nothing to chance) applied to the runtime rather than the training run.

2An e-commerce site needs product recommendations. Compare doing this with batch serving (precompute nightly) vs online serving (compute per request). What are the trade-offs?
Show solution & reasoning

Batch (precompute nightly): overnight, score recommendations for every user and store them; when a user visits, just look up their precomputed list. Pros: extremely fast at request time (it's a database lookup, not a model run), simple, and cheap (you run the model in bulk once). Cons: the recommendations are stale — they can't reflect what the user did in the last few hours (just browsed running shoes), and you waste compute scoring users who never visit that day. Online (compute per request): when the user visits, run the model live with their current context. Pros: fully fresh and personalized to the moment, reflecting their latest behavior. Cons: must be fast enough to not delay the page (a latency constraint), and more expensive/complex per request.

The trade-off is essentially freshness vs cost/latency. The right choice depends on how much recommendations benefit from real-time context: for slow-changing preferences, nightly batch is fine and far simpler; for behavior that shifts within a session (the user is clearly shopping for a gift right now), online's freshness is worth its cost. Many real systems use a hybrid: precompute a batch of candidate recommendations (cheap, broad) and then re-rank or adjust them online using live context (fresh where it matters) — getting most of the freshness at a fraction of the full online cost. Reasoning about this trade-off — rather than reflexively reaching for "real-time everything" — is exactly the systems judgment serving design requires, and it connects to the cost/latency optimization of the next lesson.

3Why is serving (inference) often a bigger long-term cost and engineering concern than training, even though training gets more attention?
Show solution & reasoning

Because of volume and duration. Training a model is typically a one-time (or periodic) cost — you train it, perhaps over hours or days, and you're done until the next retraining. Serving runs forever: every single prediction for every user, every day, for the entire life of the deployed model. A model trained once might then serve billions of predictions over its lifetime, so even a small per-prediction cost or latency multiplies into the dominant expense and the primary engineering challenge. The famous training runs grab headlines, but for most deployed systems the cumulative inference cost dwarfs the training cost.

This reframes where engineering effort should go. Serving must be reliable (it's a live service users depend on — downtime is an outage, unlike a training job that can just be rerun), scalable (it must handle peak traffic, which is the next lesson), fast (latency directly affects user experience and is often constrained), and cost-efficient (since it runs constantly, optimizations compound). It's a classic production-software problem — uptime, scaling, latency, cost — layered on top of ML, which is why serving is where MLOps meets traditional systems engineering hardest. The practical implication: when you design an ML system, you should think about serving from the start, not treat it as an afterthought once training "succeeds," because serving is where the model spends its actual working life and where most of the operational money and risk live. This is also why the next lesson is devoted entirely to scaling inference — making serving fast and cheap at volume is one of the central problems of applied ML.

★ Deep dive: the serving stack, model formats, and the latency budget▸

A production serving setup is a small stack. At the core is the packaged model, often exported to a serving-optimized format rather than served from raw training-framework objects: formats like ONNX (a portable, framework-independent model representation) or framework servers like TensorFlow Serving and TorchServe, which load the model efficiently and expose it over a standard protocol. Around it sits an API layer (commonly a REST or gRPC endpoint) that receives requests, runs the model's preprocessing and inference, and returns predictions — plus the container that packages it all, an orchestration system (like Kubernetes) that runs and scales the containers, and a load balancer in front (next lesson). The reason for serving-specific formats and servers is that training frameworks are built for flexibility and gradient computation, whereas serving wants the opposite: minimal, fast, stable inference with low overhead — so you "compile" the trained model into a lean form for production.

The discipline that organizes serving design is the latency budget. A user-facing request has a total time budget (say, the page must respond in 200 milliseconds), and the model's inference is only one slice of it — there's also network time, feature lookup (hitting the feature store's online side, Lesson 9.3), preprocessing, and post-processing. Designing serving means accounting for the whole budget: if feature lookup eats 50ms and the network 30ms, the model has maybe 100ms to run, which constrains how big and slow a model you can afford — sometimes forcing a smaller or compressed model (Lesson 9.6) even at a small accuracy cost, because a more accurate model that blows the latency budget is unusable in that slot. This is a recurring tension in applied ML: the "best" model offline is often not the best model to serve, because production imposes constraints — latency, cost, memory — that offline evaluation ignores. The engineers who ship successful ML systems are the ones who treat these production constraints as first-class design inputs from the beginning, co-designing the model and its serving so the whole thing meets its budget — rather than training the most accurate model possible and only then discovering it's too slow or expensive to run. Serving, in other words, is where ML stops being a modeling exercise and becomes systems engineering, and the next lesson pushes directly into the hardest part of that: making inference scale.

Serving one request is easy; serving millions per second without falling over is not. Next, 9.6 Scaling inference covers replicas, load balancing, autoscaling, and the techniques (batching, caching, model compression) that make serving fast and affordable at volume.
