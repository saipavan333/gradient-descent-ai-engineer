Track 7 · RAG & Vector Search

Building a RAG pipeline

⏱ 28 min Lesson 7.6 ▶ Open in Colab
Why this existsA RAG pipeline assembles the pieces into a working system: index offline, query online, generate cited answers. Knowing the two phases and prompt assembly is what turns components into a product.
By the end, you can
  • Separate offline indexing from online querying
  • Assemble a grounded, cited prompt
  • Manage the token budget and top-k
  • Instruct the model to abstain when context is missing

Concept Two phases: index offline, query online

You now have every ingredient — chunking (7.2), embeddings (7.3), vector search (7.4), reranking (7.5). A RAG pipeline assembles them into a working system with two distinct phases that run at different times.

The indexing phase happens offline, ahead of time (and again whenever documents change): ingest documents → chunk → embed → store in the vector database with metadata. It's a batch job, so it can be slow without hurting users. The querying phase happens online, per request, and must be fast: embed the question → retrieve (and optionally rerank) the relevant chunks → assemble them with the question into a prompt → generate the answer with citations. Separating the two is the key architectural decision: you pay the heavy embedding cost once at index time, so each query only embeds one short question and searches a pre-built index.

Concept Assembling the prompt — and the budget

The prompt assembly step is where retrieval meets generation. You build a prompt containing: an instruction (answer from the context, cite sources, say "I don't know" if the context lacks the answer), the retrieved chunks (each tagged with a citation marker like [1]), and the user's question. Done well, this yields grounded, cited, abstaining-when-appropriate answers (Lesson 7.1's deep dive). You also manage a token budget (Lesson 5.1): the chunks plus question plus instructions must fit the context window with room for the answer, so you cap how many chunks you include (top-k) and watch the cost, since you pay per token on every query.

Worked example A minimal end-to-end RAG pipeline

Here's the whole querying phase in a few lines: index a tiny corpus, retrieve the top chunks for a question, and assemble a cited, grounded prompt — the exact structure a production system uses, just scaled down.

The pipeline retrieved the two refund-related chunks (correctly excluding the irrelevant store-hours chunk [3]), tagged each with its citation id, and assembled a prompt that instructs the model to answer only from this context and cite its source. The model would respond something like "A refund is processed within 5 business days [1]" — grounded in a real document and traceable to it. That's a complete RAG system: the indexing built a searchable store, and the querying retrieved, assembled, and prepared a grounded prompt. Everything beyond this — a neural embedder instead of TF-IDF, a real vector DB instead of an in-memory list, reranking before assembly, streaming the answer — makes each step better or faster, but the architecture is exactly these two phases. Notice how the citation ids flow from the document store, through retrieval, into the prompt: that single thread is what lets the final answer cite its sources, which is much of RAG's value (Lesson 7.1).

★ Key points to remember
  • A RAG pipeline has two phases: indexing (offline: chunk → embed → store) and querying (online: embed → retrieve → rerank → prompt → generate).
  • Separating them pays the heavy embedding cost once, so each query only embeds a short question and searches a pre-built index.
  • Prompt assembly combines an instruction, retrieved chunks (with citation tags), and the question.
  • Instruct the model to answer only from context, cite sources, and abstain if the answer isn't there.
  • Mind the token budget (top-k chunks must fit the context window) and the per-query cost.

Practice Build the reasoning yourself

1Your RAG system is slow — each query takes 4 seconds. Walk through the pipeline steps and identify where the time likely goes and what you'd optimize first.
Show solution & reasoning

Per-query steps: embed the question (fast, ~ms), vector search (fast with ANN, ~ms), optional rerank (cross-encoder over candidates — can be slow), prompt assembly (instant), and LLM generation (usually the dominant cost — generating tokens is by far the slowest step). So the 4 seconds is almost certainly the LLM, with reranking a possible secondary contributor.

Optimize first by addressing generation: use a faster/smaller model, stream the answer (so the user sees tokens immediately rather than waiting for the full response), shorten the prompt (fewer chunks = fewer input tokens to process), or cache answers to repeated questions. If reranking is also heavy, rerank fewer candidates (Lesson 7.5). The lesson: profile the pipeline — most RAG latency is in LLM generation, not retrieval, so that's where the big wins are, and streaming alone transforms perceived speed even if total time is unchanged.

2You add 500 new documents to your knowledge base every day. Which pipeline phase do you re-run, and what must you keep consistent?
Show solution & reasoning

You re-run the indexing phase for the new documents: chunk them, embed the chunks, and add the vectors (with metadata) to the existing vector DB — typically an incremental update, not a full rebuild. The querying phase is unaffected; it just searches the now-larger index. This is exactly why the two phases are separated: you can keep knowledge fresh by re-indexing new content without touching the serving path.

The critical consistency requirement: use the same embedding model (and same chunking approach) for the new documents as for the existing ones (Lesson 7.3). If you embed the new docs with a different model, their vectors live in an incompatible space and retrieval across old and new content breaks. This is a real operational gotcha — if you ever upgrade your embedding model, you must re-embed the entire corpus, not just new documents, so everything stays in one comparable space. Incremental updates are cheap; embedding-model migrations are expensive.

3Why instruct the model to say "I don't know" when the retrieved context doesn't contain the answer, rather than letting it answer anyway?
Show solution & reasoning

If the relevant information wasn't retrieved (a retrieval failure, or the knowledge base genuinely lacks it), an LLM left to "answer anyway" will fall back on its parametric memory or simply hallucinate a plausible answer (Lesson 6.1) — defeating the entire point of grounding. Explicitly permitting "I don't know" gives the model a principled exit, so it abstains instead of fabricating.

This abstention is a key safety and trust feature of good RAG (Lesson 7.1's deep dive): a system that says "I couldn't find information on that" is far more trustworthy than one that confidently invents an answer when retrieval came up empty. It also turns silent retrieval failures into visible ones — if users frequently get "I don't know," that's a signal to improve chunking, embeddings, or coverage. The instruction is a small addition to the prompt with an outsized effect on reliability, and it embodies RAG's core promise: ground answers in real sources, and admit when there aren't any.

★ Deep dive: orchestration, production concerns, and where RAG meets agents▸

Real pipelines are rarely hand-coded from scratch; orchestration frameworks like LangChain and LlamaIndex provide the plumbing — document loaders for dozens of formats, chunkers, embedding and vector-DB integrations, prompt templates, and ready-made retrieval chains — so you assemble a pipeline from components rather than wiring every step yourself. They're convenient for getting started and for standard patterns, though many teams eventually write their own thin orchestration for control and to avoid abstraction overhead. The valuable skill isn't any one framework but understanding the underlying steps (which this track teaches), so you can debug and customize whatever you use — frameworks come and go, but chunk → embed → store → retrieve → rerank → assemble → generate is the durable mental model.

Production RAG has concerns the happy-path tutorial omits, and they're where systems succeed or fail. Caching: cache embeddings, retrieved results, and even full answers for repeated queries to cut latency and cost. Streaming: stream tokens to the user so responses feel instant. Observability: log what was retrieved for each query so you can debug "why did it answer that?" — usually the answer is in the retrieved chunks. Freshness and consistency: keep the index current and handle document updates/deletes. Security: enforce access control via metadata filtering (Lesson 7.4), and beware that retrieved content could contain prompt-injection attacks (Track 11) — a malicious document could try to hijack the model's instructions. Cost management: RAG adds retrieval cost and a larger prompt to every query, so you budget tokens and choose model sizes deliberately (Track 6, Track 9). And evaluation (the next lesson) to know it actually works. These operational matters are most of the engineering effort in a serious RAG deployment, and overlooking them is why many prototypes stall before production.

Finally, RAG is the gateway to something bigger. The pipeline here is static — retrieve once, then generate. But you can make it iterative and adaptive: let the model decide whether it needs to retrieve, what to search for, and retrieve again based on what it found — reasoning and acting in a loop. That's the leap from RAG to agents (Track 8): retrieval becomes one tool the model can choose to use, alongside web search, code execution, and APIs, orchestrated by the model's own reasoning. A RAG pipeline is, in a sense, the simplest agent — one that always takes exactly one action (retrieve) before answering. Understanding RAG deeply, including how to assemble and operate the full pipeline, is therefore the foundation for the agentic systems that come next, where the model takes the wheel. But first, the indispensable question for any RAG system: how do you know it's actually good? That's Lesson 7.7.

You can build a RAG pipeline — but is it any good? Retrieval can silently fail, and answers can look fluent yet be ungrounded. Next, in 7.7 Evaluating RAG, we measure retrieval and generation separately, with the metrics that catch each failure.
