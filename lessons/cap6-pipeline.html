Capstone 6: full training→serving pipeline

⏱ 150 min Lesson 13.7 ▶ Open in Colab
Why this existsThe capstone of capstones: a full training-to-serving pipeline automating the whole lifecycle — data, train, evaluate, deploy, monitor, retrain — with a baseline gate that makes automated retraining safe. Systems-level ML engineering.
By the end, you can
  • Chain the lifecycle stages into a pipeline
  • Enforce a deployment gate
  • Close the loop with drift-triggered retraining
  • Integrate the whole curriculum

Concept The capstone of capstones: automate the whole loop

The final project brings everything together into one automated system: a full training→serving pipeline that takes data all the way through training, evaluation, deployment, monitoring, and retraining — with as little manual intervention as possible. Where the earlier capstones each built one piece (a model, a served API), this one assembles them into the complete ML lifecycle loop as a running system. It's the most ambitious and most impressive portfolio project, because it demonstrates you can build not just a model but the machinery around it — the pipeline that turns a working model into a maintainable production system, which is what senior ML engineering is actually about.

The pipeline chains the lifecycle stages (Track 9) into an automated flow. Data is ingested and versioned. Training produces a model, with the run tracked. Evaluation measures it — and critically, a gate compares the new model against the current production baseline, deploying it only if it's at least as good. Deployment serves the model (Capstone 5) through a registry with rollback. Monitoring watches operational and model health, and its drift detection can trigger retraining — closing the loop. The whole thing is orchestrated to run reliably, ideally automatically (on a schedule or triggered by drift), so the system largely maintains itself.

Worked example The pipeline with a deployment gate

The heart of a safe automated pipeline is the gate: a new model deploys only if it passes evaluation against the baseline, so automated retraining can't silently ship a worse model. Here the stages run in sequence, with the gate deciding deployment:

The pipeline runs its stages — load data, train, evaluate — and then the gate decides: the new model's 0.88 accuracy beats the 0.85 baseline, so it deploys. Had it scored below the baseline (say 0.82), the gate would have printed BLOCKED and kept the current production model. That gate is what makes an automated pipeline safe: retraining on fresh data doesn't guarantee a better model (the data could be corrupted, unrepresentative, or the run could land poorly), so the pipeline validates each new model against the baseline before deploying — catching a bad retrain automatically instead of shipping it (Track 9's CI/CD/CT). This tiny example is the skeleton of the whole system; the real project makes each stage real and connects them: versioned data feeding a tracked training run, evaluation on a proper test set with the gate, deployment through a model registry with rollback (Capstone 5's serving underneath), monitoring that watches quality and detects drift, and a trigger that reruns the pipeline when drift is detected — so monitoring closes the loop back to retraining. Orchestrated together, it's a system that continuously produces, validates, deploys, and maintains models with humans setting the policy (the gate's threshold, the guardrails) rather than hand-running every step. Building even a working version of this — the stages chained, the gate enforced, the loop closed — demonstrates that you understand ML as a lifecycle to be engineered and automated, not a one-time training exercise, which is exactly the systems- level capability that defines a strong ML engineer and makes this the most impressive project in the portfolio.

★ Key points to remember
  • The project: a full training→serving pipeline — the whole ML lifecycle loop as one automated system.
  • Stages chain: data → train → evaluate → deploy → monitor → retrain, with data and models versioned.
  • The deployment gate validates each new model against the baseline, so automated retraining can't ship a worse model.
  • Monitoring's drift detection triggers retraining, closing the loop so the system maintains itself.
  • It demonstrates you can build the machinery around a model — the systems-level skill that defines senior ML engineering.

Show a strong plan

"Stages, chained: (1) Data — ingest and version the data (a data-version tag), with validation that incoming data meets expected schema/ranges. The plan demonstrates you understand the full ML lifecycle as an engineered, automated system, not a sequence of manual steps — which is exactly the systems-level capability that defines senior ML engineering. The reasoning skill is chaining the stages correctly, knowing the tools that connect them (orchestration, tracking, registry, serving, monitoring), and — most importantly — building in the safety mechanisms that make automation trustworthy: the deployment gate (so retraining can't ship a worse model), versioning (for rollback), and human-set policy (so automation serves judgment rather than replacing it). 2What could go wrong with a fully automated retraining pipeline that lacks strong validation, and how does the gate prevent it?
Show reasoning

"Without strong validation, automation becomes a machine for shipping bad models fast and silently. The core danger: retraining on fresh data doesn't guarantee improvement, and several things can make the new model worse — the recent data might be corrupted (a broken upstream pipeline), unrepresentative (a holiday week, an outage that skewed the data), or even poisoned (manipulated inputs). If the pipeline blindly retrains and deploys, it pushes that degraded model straight to production automatically, with no human noticing — and worse, feedback loops can compound it (a biased model's outputs become its next training data). You've built a system for amplifying data problems into production failures at full speed.

The gate is the safeguard: by validating each new model against a trusted held-out set and the current production baseline before deploying, it catches the bad retrain and refuses to ship it, keeping the current model instead. This is what makes automated retraining (CT — continuous training, Track 9) safe rather than reckless. The deeper principle is that automation multiplies whatever discipline you put into it: automated deployment with good gates lets you ship improvements safely and often; automated deployment without gates lets you ship disasters safely and often. This is why serious automated pipelines invest heavily in validation and often keep a human approval step or canary rollout for high-stakes models — you want the speed of automation with the safety of validation, never speed alone. Recognizing that a fully automated pipeline is dangerous without strong validation, and that the gate is what converts it from reckless to safe, demonstrates exactly the mature understanding of MLOps that distinguishes someone who can responsibly automate ML systems from someone who would build a fast way to break production."
