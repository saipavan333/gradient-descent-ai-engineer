Track 7 · RAG & Vector Search

Retrieval, reranking & hybrid search

⏱ 25 min Lesson 7.5 ⬇ Runnable notebook
Why this existsThe closest vector is not always the best chunk. Hybrid search (dense + keyword) and cross-encoder reranking add a precision layer that surfaces the truly-relevant chunks — often the biggest RAG quality win.
By the end, you can
  • Contrast bi-encoders and cross-encoders
  • Explain hybrid (dense + sparse) search
  • Use two-stage retrieve-then-rerank
  • Know when query rewriting helps

Concept The closest vector isn't always the best chunk

Embedding retrieval (Lesson 7.3) is fast because it embeds the query and each chunk independently — a bi-encoder — and compares vectors. But that independence is also its weakness: the model never looks at the query and a specific chunk together, so it can only judge relevance through a single similarity number. A chunk can be topically close yet not actually answer the question, or the true best chunk can rank slightly below a more generic one. Two techniques fix this: hybrid search and reranking.

First, hybrid search. Semantic (dense) retrieval is great at meaning but can fumble exact matches — specific names, codes, IDs, error messages — where the precise word matters. Old-fashioned keyword search (BM25, a strong sparse-retrieval algorithm) nails those exact matches but misses synonyms. Combining them captures both.

Concept Reranking: retrieve wide, then judge precisely

The second technique is reranking with a two-stage design. Stage one: a cheap bi-encoder retrieves a wide set of candidates (say the top 50) — fast, but coarse. Stage two: a more expensive, accurate cross-encoder reranks just those 50 and returns the best few. A cross-encoder reads the query and a candidate chunk together (concatenated) through a transformer, so it can judge relevance far more precisely than a single vector distance — but it's too slow to run over millions of chunks, which is why you only apply it to the 50 the fast retriever surfaced.

This "wide net, then precise filter" pattern is standard in production RAG because it gives you the best of both: the bi-encoder's scalability and the cross-encoder's accuracy, with the expensive model only touching a tiny candidate set. A related trick is query rewriting — using an LLM to rephrase or expand the user's query (fixing typos, adding synonyms, splitting a complex question) before retrieval — which improves the candidates the retriever finds in the first place.

Worked example Reranking changes the order

Let's see reranking in action. Five candidate chunks come back from a fast bi-encoder with one ranking; a cross-encoder, which actually reads each chunk with the query, scores them differently and reorders.

import numpy as np
docs   = ["A", "B", "C", "D", "E"]
dense  = np.array([0.80, 0.78, 0.60, 0.55, 0.40])   # bi-encoder retrieval scores (fast, coarse)
rerank = np.array([0.30, 0.95, 0.20, 0.88, 0.10])   # cross-encoder scores (slow, precise)

print("retrieval order (bi-encoder):", [docs[i] for i in dense.argsort()[::-1]])
print("rerank order (cross-encoder):", [docs[i] for i in rerank.argsort()[::-1]])
Outputretrieval order (bi-encoder): ['A', 'B', 'C', 'D', 'E'] rerank order (cross-encoder): ['B', 'D', 'A', 'C', 'E']

The fast retriever ranked A and B highest (closest vectors) and D fourth. But the cross-encoder — reading each chunk together with the query — discovered that B and D are actually the most relevant, promoting D from 4th to 2nd and dropping A from 1st to 3rd. If you only took the bi-encoder's top 2 (A, B), you'd hand the LLM a mediocre chunk (A) and miss the genuinely-best one (D). After reranking, the top 2 (B, D) are the truly best — better context, better answer. This is exactly why two-stage retrieval matters: the bi-encoder is good enough to get the right chunks into the candidate set, and the cross-encoder is good enough to order them correctly. You pay the cross-encoder's cost on only 50 candidates, not millions, so it's affordable — and the quality jump is often large, because the final answer depends entirely on which chunks make it into the prompt.

★ Key points to remember
  • A bi-encoder embeds query and chunks separately (fast, scalable) but judges relevance only by vector distance.
  • Hybrid search fuses dense (semantic) and sparse (BM25 keyword) retrieval — synonyms and exact matches.
  • Reranking: retrieve wide (~50) with a bi-encoder, then a cross-encoder reranks by reading query+chunk together.
  • The cross-encoder is accurate but slow, so it only runs on the small candidate set — wide net, then precise filter.
  • Query rewriting (rephrase/expand the query with an LLM) improves the candidates retrieved.

Practice Build the reasoning yourself

1A user searches your docs for the error code "ERR_2048." Semantic (dense) retrieval returns conceptually-related troubleshooting pages but not the one page that actually documents ERR_2048. Why, and what fixes it?
Show solution & reasoning

Dense embeddings capture meaning, and an exact code like "ERR_2048" carries little semantic content — it's essentially a token that embeds to roughly the same place as other codes, so the embedding can't distinguish the exact one. Semantic search blurs precise identifiers, which is one of its known weaknesses.

The fix is hybrid search: add a sparse/BM25 keyword retriever that matches "ERR_2048" exactly and ranks the documenting page highly, then fuse it with the dense results (e.g., Reciprocal Rank Fusion). Now you get semantic relevance and exact-match precision. This exact-identifier case is the textbook reason hybrid search exists — whenever your domain has codes, SKUs, names, function names, or other strings where the precise token matters, pure semantic search will frustrate users, and keyword search must be in the mix.

2Adding a cross-encoder reranker improved your RAG answer quality noticeably, but latency went up. Explain the tradeoff and one way to manage it.
Show solution & reasoning

The cross-encoder reads each candidate chunk together with the query through a transformer, which is far more compute per candidate than a single vector comparison — so reranking 50 candidates adds real latency (one model forward pass per candidate, though they can be batched). You bought accuracy with time. That's the classic quality-vs-latency tradeoff.

Manage it by tuning how many candidates you rerank: reranking the top 20 instead of 100 cuts the cost substantially while usually keeping most of the quality gain (the best chunk is almost always in the top 20 from the retriever). Other levers: use a smaller/faster reranker model, batch the candidate scoring, cache results for repeated queries, or rerank only when the retrieval scores are ambiguous. You'd pick the candidate count that meets your latency budget while keeping answer quality high — measured on a real evaluation set (Lesson 7.7). The principle: reranking's cost scales with the candidate count, which is a direct, tunable knob.

3A user asks "Compare our refund and warranty policies." A single embedding of this query retrieves a muddled mix. How might query rewriting help?
Show solution & reasoning

The query asks about two distinct topics (refund and warranty), so one embedding of the whole question lands "in between" them and retrieves chunks that are vaguely related to both but strongly about neither. The single query vector can't be close to both topic clusters at once.

Query rewriting/decomposition fixes this: use an LLM to split the question into sub-queries — "refund policy" and "warranty policy" — retrieve for each separately, and combine the results, so you get strong chunks for both topics. More generally, query rewriting can fix typos, expand abbreviations, add synonyms, or rephrase a conversational question into a retrieval-friendly form. It improves the input to retrieval, complementing reranking (which improves the output). For multi-part and conversational questions, this query-transformation step is often the difference between a confused answer and a complete one — and it's a natural place where the LLM helps its own retrieval, a small step toward the agentic patterns of Track 8.

★ Deep dive: RRF, cross-encoder mechanics, and the retrieval-quality stack▸

How exactly do you fuse two rankings in hybrid search? The popular answer is Reciprocal Rank Fusion (RRF), and its appeal is that it needs no score calibration. Dense cosine similarities (0–1) and BM25 scores (unbounded) live on incomparable scales, so you can't just add them. RRF instead uses each item's rank in each list: it scores an item by $\sum_{\text{lists}} \frac{1}{k + \text{rank}}$ (with a constant like $k=60$), so being near the top of either list contributes a lot and the contributions simply add. A chunk ranked #1 by dense and #3 by sparse gets a high fused score; one that's mediocre in both stays low. It's simple, robust, and parameter-light, which is why it's a common default — though learned fusion (training a small model to weight the signals) can do better when you have evaluation data. The deeper point: hybrid search works because dense and sparse retrieval make uncorrelated errors (they fail on different queries), so combining them is like ensembling (Lesson 3.8) — the union of their strengths beats either, exactly the wisdom-of-crowds principle applied to retrieval.

It's worth seeing why a cross-encoder is so much more accurate than a bi-encoder, because it illustrates a fundamental tradeoff. A bi-encoder must compress all of a chunk's meaning into one fixed vector before ever seeing the query — so it can't emphasize the parts of the chunk relevant to this specific question. A cross-encoder feeds the query and chunk into the transformer together, so attention (Lesson 5.4) can directly relate every query word to every chunk word and produce a relevance judgment tailored to the pair. That joint processing is strictly more expressive — but it means you can't precompute anything (every query–chunk pair is a fresh forward pass), so it can't scale to a whole corpus. This is the same expressiveness-vs-scalability tension throughout ML: the more two things are processed together, the better the judgment and the worse the scaling. The two-stage design — bi-encoder to scale, cross-encoder to refine — is the elegant resolution, and the same pattern (cheap-wide then expensive-narrow) recurs in recommendation and search systems (Track 10). Put it all together and you have the modern retrieval-quality stack: query rewriting improves the input, hybrid search casts a complete net (semantic + lexical), ANN makes it fast (Lesson 7.4), and cross-encoder reranking precisely orders the candidates. Each layer addresses a specific failure mode, and a production RAG system worth its name uses several of them — because, as this track keeps emphasizing, retrieval quality is the ceiling on RAG quality, and these are the techniques that raise that ceiling.
