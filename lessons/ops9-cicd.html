CI/CD for ML

⏱ 23 min Lesson 10.9 ▶ Open in Colab
Why this existsHand-running the lifecycle doesn't scale. CI/CD automates test → build → deploy; ML adds CT (continuous training) and a baseline gate — the new model must beat production, not just pass tests — which makes automated retraining safe.
By the end, you can
  • Explain CI/CD and the ML twist (CT)
  • Describe the baseline deployment gate
  • List ML test types (code, data, model)
  • Know why gates make CT safe

Concept Automating the path from change to production

By now the lifecycle loop has many moving parts — train, test, validate, package, deploy, monitor. Doing these by hand is slow and error-prone: someone forgets to run the tests, deploys the wrong version, or skips validation under deadline pressure. CI/CD is the practice of automating this path. CI (Continuous Integration) means automatically testing and validating every change as soon as it's made; CD (Continuous Delivery/Deployment) means automatically packaging and releasing changes that pass. Together they form a pipeline: an automated sequence — change → test → build → deploy — that runs without manual steps, so shipping becomes fast, consistent, and safe. CI/CD comes from traditional software engineering; ML adds a twist.

The twist is that in software, the thing you ship is code, so CI/CD tests and deploys code. In ML, the system depends on code plus data plus the trained model — so the pipeline has to handle all three. This extended version is sometimes called CT (Continuous Training): the pipeline can automatically retrain the model on new data, validate that the new model is actually good, and deploy it — the automation of the retraining loop from Lesson 10.8. So ML CI/CD tests not just "does the code work?" but also "is the data valid?" and "is the new model good enough to deploy?"

Concept The deployment gate: tests and model quality

The heart of an ML CI/CD pipeline is the gate before deployment — the automated checks a change must pass to reach production. In plain software, the gate is "do the tests pass?" In ML, the gate has an extra, ML-specific condition: "is the new model actually better (or at least not worse) than the one in production?" A new model might be the result of retraining on fresh data, but fresh data doesn't guarantee a better model — the retraining could have produced something worse. So the pipeline automatically evaluates the new model on a validation set and compares it to the current production model (the baseline), and only promotes it if it clears the bar. This prevents the failure mode where automated retraining silently ships a degraded model.

This is why ML testing is broader than code testing. Beyond ordinary unit tests (does the code run correctly?), an ML pipeline runs data validation (is the incoming data the right schema, range, and quality?), model validation (does the new model meet the accuracy/quality bar and beat the baseline?), and often checks for things like fairness or performance on important data slices. The pipeline encodes your standards as automated gates, so every deployment — whether triggered by a code change or by retraining — has to earn its way to production by passing them.

Worked example A deployment gate with an ML-specific check

Here's the gate in miniature: a change only deploys if the tests pass and the new model beats the production baseline. Watch how each scenario is handled.

Three scenarios, three outcomes. The first change has passing tests and a new model (0.88) that beats the production baseline (0.86), so it clears both gates and deploys. The second has a new model that's better, but the tests failed — so it's blocked regardless of model quality; broken code doesn't ship even with a good model. The third is the distinctively ML case: the tests pass (the code is fine), but the new model (0.84) is worse than the baseline (0.86), so the gate blocks it. That third check is what separates ML CI/CD from ordinary CI/CD — in plain software, passing tests would be enough to deploy, but here a model that's technically correct yet performs worse must be stopped, because shipping it would degrade production. This gate is exactly what makes automated retraining (CT) safe: when the pipeline retrains on fresh data, it doesn't blindly deploy the result — it validates the new model against the baseline first, so a bad retrain is caught and rejected automatically instead of silently going live. The whole pipeline is this idea scaled up: encode every standard you care about (code correctness, data quality, model performance, fairness) as an automatic gate, and let only changes that pass all of them reach users.

★ Key points to remember
  • CI/CD automates the path from change to production: CI tests every change, CD releases the ones that pass — an automated pipeline.
  • The ML twist: the pipeline must handle code and data and models — including CT (continuous training), automated retrain-validate-deploy.
  • The deployment gate adds an ML-specific check: the new model must beat (or match) the production baseline, not just pass code tests.
  • ML testing spans unit tests (code), data validation (schema/quality), and model validation (quality vs baseline).
  • This is what makes automated retraining safe — a bad retrain is caught by the gate, not shipped.

Practice Build the reasoning yourself

1Design the gates (automated checks) a new model version should pass before your CI/CD pipeline deploys it to production. Go beyond "tests pass."
Show solution & reasoning

A robust ML deployment gate layers several checks. Code tests: unit and integration tests confirm the pipeline code runs correctly (the ordinary CI baseline). Data validation: the training/serving data has the expected schema, types, and ranges, with no surge in missing values — catching data problems before they corrupt the model. Model-quality validation: the new model meets a minimum accuracy/metric bar and beats (or at least matches) the current production baseline on a held-out validation set, so you never ship a regression. Slice checks: the model performs acceptably not just overall but on important sub-populations (so an aggregate gain doesn't hide a collapse on a key segment) — which connects to fairness. Behavioral/operational checks: the model meets latency and size budgets for serving (Lesson 10.5), and perhaps passes specific "the model should behave sensibly on these known cases" tests.

The reasoning is that "deployable" for a model means more than "the code works" — it means the model is correct, the data feeding it is sound, its quality is validated against the incumbent, it's fair across segments, and it fits the serving constraints. Each gate corresponds to a way the system could fail in production, and encoding it as an automatic check turns your team's standards into something the pipeline enforces on every deployment, consistently, without relying on a human to remember. This is the essence of mature MLOps: the pipeline is the codification of "what does it mean for this model to be safe to ship?", and the more of that judgment you can express as automated gates, the more confidently and frequently you can deploy. The art is choosing gates that catch real failures without being so strict that nothing ever passes.

2What could go wrong if a team sets up fully automated retraining and deployment (CT) without a strong validation gate?
Show solution & reasoning

Without a strong gate, automation becomes a way to ship bad models fast and silently. The core danger: retraining on fresh data doesn't guarantee improvement, and several things can make the new model worse — the recent data might be corrupted (a broken upstream pipeline, a data-quality incident), unrepresentative (a holiday week, an outage that skewed the data), or poisoned (deliberately manipulated inputs). If the pipeline blindly retrains and deploys, it will push that degraded model straight to production automatically, with no human in the loop to notice. You've built a machine for amplifying data problems into production model failures at full speed.

Concrete failure modes: a feedback loop (Lesson 10.8) where the model's own biased outputs become its training data, making each automated retrain more skewed; a data outage that fills features with nulls, so the model retrains on garbage and deploys a broken model overnight; or simple variance, where a retrain happens to land on a worse local optimum. The validation gate is the safeguard against all of these — by validating each new model against a trusted held-out set and the production baseline before deployment, the pipeline catches the bad retrain and refuses to ship it (as in the worked example's third case), falling back to keeping the current model. The broader principle is that automation multiplies whatever discipline you put into it: automated deployment with good gates lets you ship improvements safely and often; automated deployment without gates lets you ship disasters safely and often. This is why serious CT setups invest heavily in validation and usually keep a human approval step or canary rollout (Lesson 10.4) for high-stakes models — you want the speed of automation with the safety of validation, never speed alone.

3Step back: how does CI/CD/CT tie together everything in this track into a single automated lifecycle loop?
Show solution & reasoning

CI/CD/CT is the automation layer that operationalizes the entire lifecycle loop (Lesson 10.1), wiring every stage of the track into one self-running system. Trace it: the pipeline pulls fresh data through validated feature pipelines (9.3), retrains a model while tracking the experiment (9.2), validates the new model against the baseline (this lesson's gate), registers and stages it in the model registry (9.4), packages and deploys it through safe rollout (9.5), serves it at scale (9.6), and feeds monitoring (9.7), whose drift detection (9.8) can trigger the whole pipeline again. Each lesson was one stage; CI/CD/CT is the connective automation that makes them run as a continuous, hands-off loop instead of a series of manual handoffs.

This is the culmination of the MLOps mindset. At the start of the track, the lifecycle loop was a concept; here it becomes a literal pipeline that executes it — detect drift, retrain, validate, deploy, monitor, repeat — with humans setting the standards (the gates) and intervening on exceptions rather than hand-running every step. That's the difference between the low and high maturity levels mentioned in Lesson 10.1: a mature MLOps organization has automated the loop so that maintaining many models is feasible, deployments are frequent and safe, and the system largely keeps itself current. The profound shift this represents is from thinking of ML as "train a model and hand it off" to thinking of it as "build an automated system that continuously produces, validates, deploys, and maintains models" — engineering the loop, not just the model. Internalizing that shift, and being able to reason about each stage and how they compose into the automated whole, is exactly what it means to understand MLOps — and it's what the interview check next will consolidate.

★ Deep dive: pipeline orchestration and the levels of MLOps maturity▸

The automated pipelines in this lesson are built and run by orchestration tools — systems that define a workflow as a sequence (really a directed graph) of steps and execute them reliably, handling ordering, retries on failure, scheduling, and passing data between steps. Tools like Airflow, Kubeflow Pipelines, and others let you express "pull data → validate it → train → evaluate → if it passes, deploy" as a versioned, repeatable pipeline rather than a pile of scripts someone runs by hand. The pipeline itself becomes a first-class, version-controlled artifact: you can see exactly what sequence of steps produced a deployed model, rerun it, and trust it runs the same way every time. This connects back to reproducibility (Lesson 10.2) — an automated pipeline is reproducibility enforced by construction, because the steps and their order are codified rather than remembered. Orchestration is also what makes CT practical: "retrain weekly, validate, and deploy if it beats baseline" is a scheduled pipeline the orchestrator runs autonomously.

Stepping all the way back, this track has traced a path along the levels of MLOps maturity. At level 0 (manual), every stage is hand-run: a data scientist trains in a notebook and tosses a model over the wall to be deployed by hand, with little automation or monitoring — workable for one model, but it doesn't scale and models silently rot. At higher levels, you automate the training pipeline (so retraining is a reliable, repeatable process rather than a manual ritual), then add CI/CD so deployment is automated and gated, then close the loop with monitoring that feeds drift-triggered retraining — arriving at a largely self-running lifecycle where humans set policy and handle exceptions. Most real teams sit in the middle and climb deliberately, automating the most painful stage next rather than chasing full automation for its own sake. The wisdom the whole track has been building toward is this: MLOps maturity isn't about having the fanciest tools, it's about understanding the lifecycle loop and progressively automating it so that your models stay reliable and current as you scale from one model to many. The discipline is the same at every level — version everything, validate everything, monitor everything, make every change reversible — applied with increasing automation. That is what it means to run ML in production: not a one-time act of deployment, but the ongoing, increasingly automated practice of keeping a living system healthy. The interview check next consolidates this whole picture into the questions and mental models that demonstrate you can think like an ML engineer, not just train a model.
