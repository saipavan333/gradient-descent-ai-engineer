Text to numbers: tokenization

⏱ 24 min Lesson 6.1 ▶ Open in Colab
Why this existsModels do math on numbers, not text. Tokenization is the first step of every NLP system — splitting text into subword pieces and mapping them to IDs, a choice that quietly shapes cost, quality, and what the model can even perceive.
By the end, you can
  • Explain the text→tokens→IDs pipeline
  • Compare character, word, and subword tokenization
  • Explain why subwords (BPE/WordPiece) avoid out-of-vocabulary
  • Estimate token counts and their cost

Concept Turning text into numbers a model can read

A neural network does arithmetic on numbers (Track 4), but language is text. The first step of every NLP system is tokenization: splitting text into small pieces called tokens, then mapping each token to an integer ID via a lookup table called the vocabulary. The model never sees letters — only these IDs, which it will turn into vectors (embeddings) in the next lesson.

▶
Take it to the lab

Open the Tokenizer Lab — a real byte-pair-encoding tokenizer that learns its vocabulary live and splits any text you type, so you can see exactly how a model reads your words.

So "the cat sat" might become tokens ["the", "cat", "sat"] and then IDs [1014, 2543, 3079]. Decoding reverses the process — IDs back to tokens back to text — which is how a model's numeric output becomes readable words. The whole pipeline is reversible and deterministic.

Concept How big should a token be? Characters, words, subwords

The crucial design choice is token granularity, and it's a trade-off. Characters give a tiny vocabulary (~100 symbols) but make sequences very long and force the model to learn spelling from scratch. Words are intuitive but the vocabulary becomes enormous (hundreds of thousands), and any word not in it — a typo, a rare name, a new term — becomes a useless "unknown" token (the out-of-vocabulary problem).

The winning compromise is subword tokenization: keep common words whole, but split rare words into meaningful pieces. "tokenization" might become ["token", "ization"]; an unseen word like "unhappiness" becomes ["un", "happi", "ness"]. With a fixed vocabulary of ~30,000–100,000 subwords you can spell any word — no out-of-vocabulary problem — while keeping sequences reasonably short.

The dominant algorithms are Byte-Pair Encoding (BPE, used by GPT) and WordPiece (used by BERT). Both learn the vocabulary from a big corpus by repeatedly merging the most frequent adjacent pairs of symbols — so the final tokens reflect what's actually common in real text. Vocabularies also include special tokens like a padding token, an end-of-text marker, and (for chat models) role markers.

Worked example A tokenizer you can build in five lines

To make the pipeline concrete, here's a minimal word-level tokenizer — build a vocabulary, encode text to IDs, decode back. (Real models use subword tokenizers, but the encode/decode structure is identical.)

"the cat sat on the mat" became the ID sequence [4, 0, 3, 2, 4, 1] — note "the" maps to 4 both times (same token, same ID), which is exactly what lets the model recognize repeated words. Decoding inverts the map and recovers the original text. A real subword tokenizer does the same thing, but its vocabulary is learned and its tokens are sub-word pieces, so it would never choke on a word it hasn't seen. This humble ID sequence is the input to everything that follows — embeddings turn each ID into a vector next.

★ Key points to remember
  • Tokenization splits text into tokens and maps each to an integer ID via the vocabulary.
  • Granularity is a trade-off: characters (long sequences), words (huge vocab + out-of-vocabulary), subwords (the sweet spot).
  • Subword tokenizers (BPE, WordPiece) spell any word from a fixed ~30k–100k vocabulary — no unknown-word problem.
  • The vocabulary is learned from a corpus by merging frequent pairs; it includes special tokens.
  • The model only ever sees the ID sequence; decoding reverses IDs back to text.

Practice Build the reasoning yourself

1A word-level tokenizer is trained on news articles, then used on a tweet containing "tokenizationnnn" and an emoji. What happens, and how does a subword tokenizer fare better?
Show solution & reasoning

The word-level tokenizer has never seen "tokenizationnnn" or the emoji, so both become unknown tokens — the model loses all information about them, a real failure on noisy real-world text. This is the out-of-vocabulary problem.

A subword tokenizer fares far better: it would break "tokenizationnnn" into known pieces (e.g., token + ization + nnn or individual characters) and represent the emoji via its underlying bytes — so nothing is unknown. Every input maps to a valid sequence of in-vocabulary pieces. This robustness to messy, unseen text is exactly why subword tokenization is universal in modern NLP.

2You're told a model has a context limit of 8,000 tokens, and your document is 6,000 words of English prose. Will it fit? What rough rule helps you estimate?
Show solution & reasoning

Probably not comfortably. A common rule of thumb for English is roughly 1 token ≈ 0.75 words, or equivalently about 1.3 tokens per word. So 6,000 words ≈ 6,000 × 1.3 ≈ 7,800 tokens — just under 8,000, leaving almost no room for a prompt or the model's response. You'd want to trim or chunk the document.

This matters practically because API costs and context limits are measured in tokens, not words, and the ratio varies: code, rare words, and other languages tokenize into more tokens per word, so the same word count can blow past the limit. Estimating token counts is a routine engineering skill (and it returns in Track 8 on RAG, where you budget tokens carefully).

3The same English sentence often uses noticeably fewer tokens than its translation into a less-common language. Why, and what's one consequence?
Show solution & reasoning

Tokenizer vocabularies are learned from training corpora that are predominantly English (and a few other high-resource languages). So English words and common pieces became single tokens, while text in under-represented languages gets split into many small pieces — sometimes individual characters or bytes — because those chunks weren't frequent enough to earn their own token.

The consequence is a real fairness and cost issue: the same meaning costs more tokens (and thus more money and more of the context window) in those languages, and models often perform worse on them partly because each piece carries less meaning. This "tokenization tax" on low-resource languages is an active concern in making AI equitable — a concrete example of how a low-level engineering choice has societal consequences (a theme of Track 12).

★ Deep dive: how BPE actually learns, and why tokenization quietly shapes everything▸

Byte-Pair Encoding is beautifully simple. Start with the text as a sequence of individual characters (or raw bytes). Count all adjacent pairs; find the most frequent pair (say t followed by h); merge it into a new single token th. Repeat — next maybe th + e → the — for a fixed number of merges (which sets the vocabulary size). The result: frequent sequences become single tokens while rare ones stay split into pieces, all learned purely from frequency statistics. Modern variants operate on bytes so they can represent any Unicode text (emoji, every language) without ever failing — there's a token for every byte as a fallback, which is why byte-level BPE never produces a true "unknown."

Tokenization seems like a mundane preprocessing step, but it quietly determines a surprising amount. It sets the cost (you pay per token) and the effective context length (a tokenizer that splits your domain's text into many pieces wastes the window). It affects quality: models are famously bad at character-level tasks like counting letters or spelling words backward, precisely because they see tokens, not letters — "strawberry" might be three tokens, so "how many r's?" is genuinely hard for the model to see. Arithmetic suffers too, since numbers tokenize inconsistently (1234 might be one token but 1235 two). And the choice bakes in the biases of the training corpus, including the language inequities above. Researchers increasingly treat tokenization as a first-class design decision rather than an afterthought, and some are exploring tokenizer-free, byte-level models to sidestep these issues entirely. For now, understanding that a model perceives the world in subword chunks — not letters and not words — explains a whole category of its strengths and quirks, and it's the foundation the embeddings of the next lesson are built on.

You can turn text into token IDs — but an ID like 2543 is just an arbitrary label with no meaning. Next, in 5.2 Word embeddings, we turn each ID into a dense vector where distance and direction actually capture meaning.
