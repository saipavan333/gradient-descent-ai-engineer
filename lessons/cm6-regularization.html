Regularization (L1 / L2)

⏱ 24 min Lesson 3.6 ⬇ Runnable notebook
Why this existsOverfitting's most common cure is to penalize complexity directly. Regularization keeps weights small — and one flavor performs automatic feature selection, a tool reused everywhere from linear models to neural nets.
By the end, you can
  • Add an L1 or L2 penalty to a loss
  • Explain why L1 gives sparsity but L2 does not
  • Tune the strength lambda on validation data
  • Connect regularization to the bias-variance tradeoff

Concept Penalize complexity directly

Lesson 3.5 said overfitting comes from a model that's too flexible — and for linear models, "too flexible" shows up concretely as large weights. A wildly overfit polynomial has huge coefficients that nearly cancel, producing the violent wiggles we saw. Regularization is the direct cure: add a penalty for big weights to the loss, so the optimizer has to balance fitting the data against keeping weights small. The model can still fit, but only if the improvement is "worth" the added penalty.

The new objective is just the old loss plus a penalty term, controlled by a knob $\lambda$ (lambda):

$$\text{total loss} = \underbrace{\text{MSE}}_{\text{fit the data}} + \lambda \cdot \underbrace{\text{(size of the weights)}}_{\text{stay simple}}$$

There are two standard ways to measure "size of the weights," and the choice has a striking consequence. L2 regularization (called ridge) penalizes the sum of squared weights, $\lambda\sum w_j^2$. L1 regularization (called lasso) penalizes the sum of absolute weights, $\lambda\sum |w_j|$. Ridge shrinks all weights smoothly toward zero; lasso shrinks them and drives some to exactly zero.

Concept Why L1 zeros weights but L2 doesn't

That difference isn't a quirk — it's geometry. Picture the penalty as a budget: "keep the weights inside this region." For L2 the region is a circle (a ball); for L1 it's a diamond with sharp corners on the axes. The model settles where the data's loss contours first touch that region. A circle has no special points, so the touch happens at a generic spot with all weights nonzero. A diamond's corners stick out on the axes — and a corner is exactly where some coordinate is zero. So L1 solutions love to land on corners, switching features off entirely.

This makes lasso a built-in feature selector: it not only fights overfitting but produces a sparse model that uses only some of the features, which is easier to interpret and cheaper to run. Ridge keeps every feature but tames them all — better when you believe most features genuinely contribute, or when features are correlated.

Concept The strength dial λ

The knob $\lambda$ sets how hard you penalize. At $\lambda = 0$ there's no penalty and you're back to plain regression (maximum overfitting risk). As $\lambda$ grows, weights are squeezed toward zero, the model gets simpler, bias rises and variance falls — it's the bias–variance tradeoff of Lesson 3.5, now with a continuous dial. Crank $\lambda$ too high and every weight collapses to near zero: the model underfits, predicting almost a constant.

So $\lambda$ is a hyperparameter you tune on validation data, exactly the U-curve search from the last lesson. There's no formula for the best $\lambda$; you try several and keep the one that generalizes best.

Worked example Ridge shrinks, lasso selects

We build data where only the first three features matter (true weights 3, −2, 1.5) and the other five are pure noise that the model should learn to ignore. Then we fit plain regression, ridge, and lasso, and compare what each does to the coefficients.

import numpy as np
from sklearn.linear_model import LinearRegression, Ridge, Lasso
from sklearn.preprocessing import StandardScaler

rng = np.random.default_rng(0)
X = rng.normal(0, 1, (100, 8))
true = np.array([3.0, -2.0, 1.5, 0, 0, 0, 0, 0])   # only first 3 features are real
y = X @ true + rng.normal(0, 0.5, 100)
Xs = StandardScaler().fit_transform(X)

print("true ", true)
print("OLS  ", np.round(LinearRegression().fit(Xs, y).coef_, 2))
print("Ridge", np.round(Ridge(alpha=10).fit(Xs, y).coef_, 2))
lasso = Lasso(alpha=0.3).fit(Xs, y).coef_
print("Lasso", np.round(lasso, 2), " zeros:", int((np.abs(lasso) < 1e-8).sum()))
Outputtrue [ 3. -2. 1.5 0. 0. 0. 0. 0. ] OLS [ 2.56 -1.76 1.29 -0.07 -0.04 0.06 -0.02 0.04] Ridge [ 2.32 -1.6 1.15 -0.09 -0.01 0.09 -0.02 0.09] Lasso [ 2.27 -1.49 1.01 -0. 0. 0. 0. 0. ] zeros: 5

Look at the last five coefficients — the noise features. Plain OLS gives them small but nonzero weights (−0.07, −0.04, …): it can't resist fitting a little noise. Ridge shrinks everything a bit but still leaves all five nonzero. Lasso sets all five to exactly zero — it discovered, on its own, that only three features matter, and switched the rest off. That's the sparsity property in action: lasso returned a simpler, more interpretable model that says "ignore those five." (Note we standardized first, so the penalty treats every feature fairly — penalizing raw weights would unfairly punish small-scale features.)

★ Key points to remember
  • Regularization adds a penalty on weight size to the loss, trading data-fit against simplicity.
  • L2 (ridge) penalizes squared weights → shrinks all of them smoothly; keeps every feature.
  • L1 (lasso) penalizes absolute weights → drives some to exactly zero (sparsity / feature selection).
  • The geometry: L1's diamond has corners on the axes, so solutions land there with weights = 0.
  • The strength λ is a hyperparameter: bigger = simpler model (more bias, less variance); tune it on validation data.
  • Always standardize features first so the penalty is fair across scales.

Practice Build the reasoning yourself

1In the worked example, why did lasso set the five noise-feature weights to exactly 0 while ridge left them small-but-nonzero?
Show solution & reasoning

It's the geometry from the diagram. Lasso's L1 penalty has a constant "pull" toward zero regardless of how small a weight already is, and its diamond-shaped constraint has corners on the axes — so when a feature contributes little, the cheapest place for the solution to sit is a corner where that weight is exactly 0. The noise features contribute nothing real, so lasso parks them at zero.

Ridge's L2 penalty pulls proportionally to the weight's size, so the pull vanishes as a weight approaches zero — there's never quite enough force to push it the last step to exactly 0. Ridge therefore shrinks the noise weights to small values (−0.09, 0.09, …) but not to zero. Same goal, different geometry, qualitatively different result.

2You have 10,000 features but suspect only a few dozen actually matter (e.g., gene-expression data). Which regularizer fits this belief, and what extra benefit do you get?
Show solution & reasoning

L1 (lasso), because it produces a sparse model — it will zero out the thousands of irrelevant features and keep only the handful that matter, which is exactly your prior belief about the problem.

The extra benefit beyond accuracy is interpretability and cost: a model that uses 30 of 10,000 features tells a scientist which genes are predictive (a finding in itself), and is far cheaper to deploy and explain than one touching all 10,000. When you think "most features are noise," lasso encodes that belief directly. (If features are highly correlated, Elastic Net — a blend of L1 and L2 — often works better; see the deep dive.)

3How would you choose the value of λ in practice? Why can't you pick it by minimizing training error?
Show solution & reasoning

You choose λ by cross-validation (Lesson 3.12): try a range of values (often spaced logarithmically, like 0.001, 0.01, 0.1, 1, 10), measure validation error for each, and keep the λ with the best validation score. Tools like scikit-learn's RidgeCV / LassoCV automate exactly this.

You can't use training error because training error is minimized at λ = 0 — no penalty always fits the training data best. Picking λ by training error would always tell you "don't regularize," defeating the purpose. Regularization trades a little training fit for better generalization, so its benefit is only visible on held-out data.

★ Deep dive: Elastic Net, weight decay, and the Bayesian view▸

Elastic Net combines both penalties, $\lambda_1\sum|w_j| + \lambda_2\sum w_j^2$, to get the best of each. Pure lasso struggles when features are highly correlated — it tends to arbitrarily pick one of a correlated group and zero the rest, which is unstable. Adding a little L2 encourages correlated features to be kept or dropped together, giving more stable selection while preserving sparsity. It's the default choice for many high-dimensional problems.

Regularization is everywhere, often under other names. In logistic regression the same L1/L2 penalties apply (scikit-learn's LogisticRegression is L2-regularized by default — its C parameter is $1/\lambda$). In neural networks, L2 regularization is called weight decay and is standard in every optimizer; dropout and early stopping are regularizers too. The instinct "penalize complexity to generalize" recurs at every scale of the field, right up to training large language models.

There's also a beautiful Bayesian interpretation (connecting to Lesson 1.10's maximum likelihood). Adding an L2 penalty is mathematically identical to placing a Gaussian prior on the weights — a belief that weights should be small and near zero — and finding the most probable weights given the data. L1 corresponds to a Laplace prior, which is more sharply peaked at zero, explaining its taste for exact zeros. So regularization isn't an ad-hoc trick; it's what you get when you state a prior belief that the world is simple and then do principled inference. The strength λ is just how strongly you hold that belief.

Linear models, regularized, are strong baselines — but they assume a straight boundary. Next, in 3.7 Decision trees, we meet a model that asks a cascade of yes/no questions, carving the space into boxes and capturing nonlinear patterns no line can.
