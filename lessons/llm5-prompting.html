Prompting techniques

⏱ 25 min Lesson 7.5 ▶ Open in Colab
Why this existsPrompting is the zero-cost, highest-leverage way to steer an LLM. Few-shot, chain-of-thought, and good prompt structure reliably get better answers from the same frozen model.
By the end, you can
  • Use zero-shot, few-shot, and chain-of-thought
  • Explain in-context learning
  • Structure a prompt (system/instruction/examples/format)
  • Know prompting's limits vs retrieval and fine-tuning

Concept Same model, better answers — just by asking well

A trained LLM is fixed, but how you ask it dramatically changes what you get back. Prompting is the craft of writing the input so the model produces the response you want — and it's the highest-leverage, zero-cost way to use an LLM (no training, instant). The simplest is zero-shot: just state the task ("Translate this to French: ..."). For many tasks that's enough, especially with aligned models.

When zero-shot struggles, add examples. Few-shot prompting puts a handful of solved examples right in the prompt before the real query, and the model infers the pattern — this is the in-context learning from Lesson 7.1, learning a task from the prompt with no weight updates. Examples teach the model the exact format, style, and edge cases you want.

For tasks needing reasoning (math, logic, multi-step problems), the breakthrough technique is chain-of-thought (CoT): prompt the model to "think step by step" and show its work before answering. Forcing the model to generate intermediate reasoning steps — rather than jumping straight to an answer — massively improves accuracy, because each step conditions the next (Lesson 6.3) and the model effectively "computes" on the page. A simple "Let's think step by step" can turn a wrong answer into a right one.

Concept The anatomy of a good prompt

Reliable prompts have recognizable parts. Assembling them deliberately — rather than writing one vague sentence — is what makes outputs consistent and steerable:

The system message sets the persona and constraints ("You are a careful medical assistant; never diagnose"). The instruction states the task precisely. Optional examples demonstrate format. The input is the user's content. And an explicit output format ("respond only with valid JSON matching this schema") makes the result machine-usable. Clear role + task + examples + format is the recipe for prompts that behave predictably.

Worked example Building a few-shot prompt

Few-shot prompting is mechanically just string assembly: concatenate a few solved examples, then the new query, and let the model continue the pattern. Here we build a sentiment classifier with three in-context examples.

The prompt ends mid-pattern — Text: wonderful -> — so the model's most natural continuation is to fill in the label (positive), having inferred the task from the three examples. No fine-tuning, no labels in a training set: the "training data" lives in the prompt, and the model learns the mapping on the fly. This is in-context learning in its simplest form, and it scales: the examples could be translation pairs, formatted extractions, or question–answer demonstrations. The art is choosing examples that cover the format and the tricky cases. Notice too that this is exactly how you'd steer output format — show the model two or three examples of the exact structure you want, and it conforms — often more reliably than describing the format in words.

★ Key points to remember
  • Prompting is the zero-cost, highest-leverage way to steer an LLM — the same model gives better answers when asked well.
  • Zero-shot states the task; few-shot adds in-prompt examples (in-context learning, no weight updates).
  • Chain-of-thought ("think step by step") greatly improves reasoning by making the model show its work.
  • A strong prompt has parts: system / instruction / examples / input / output format.
  • Few-shot is just string assembly: concatenate examples + query and let the model complete the pattern.

Practice Build the reasoning yourself

1An LLM answers "If a shirt costs $20 after a 20% discount, what was the original price?" incorrectly. How might chain-of-thought prompting fix it?
Show solution & reasoning

Add "Let's think step by step" (or provide a worked example). Instead of blurting a number, the model now generates intermediate reasoning: "A 20% discount means the price is 80% of the original. So $20 = 0.8 × original. Therefore original = $20 / 0.8 = $25." Each step conditions the next, and the model effectively does the arithmetic on the page rather than guessing the answer in one leap.

Without CoT, the model must produce the final number immediately, which for multi-step problems often fails — it pattern-matches to a plausible-looking answer (a common wrong one here is $24, from naively adding 20%). CoT works because the transformer can only "compute" by generating tokens; giving it room to write the steps lets it use that computation. It's one of the most reliable, free accuracy boosts for any reasoning task — and it's why modern models often reason in steps by default.

2You need an LLM to extract a person's name, age, and city from messy text and return strict JSON. Sketch a prompt using the anatomy from this lesson.
Show solution & reasoning

System: "You are a precise information-extraction tool. Output only valid JSON, nothing else." Instruction: "Extract the person's name, age, and city from the text. If a field is missing, use null." Example (few-shot): one input → exact JSON output, e.g. {"name": "Ana", "age": 30, "city": "Lima"}. Input: the messy text. Output format: reiterate the schema {"name": ..., "age": ..., "city": ...}.

Each part does a job: the system message constrains it to JSON-only (preventing chatty preamble), the instruction defines the task and the null rule, the example pins the exact structure (more reliable than describing it), and the format reminder reinforces it. For production reliability you'd also use constrained decoding (Lesson 7.4) to guarantee valid JSON. This deliberate assembly is the difference between a prompt that works 70% of the time and one that works 99% — exactly the steerability the anatomy buys you.

3A teammate says "prompting is just trial and error — there's no skill to it." Give two concrete, principled reasons prompting is a real skill.
Show solution & reasoning

(1) Principled techniques have predictable effects. Few-shot examples teach format via in-context learning; chain-of-thought reliably boosts reasoning by enabling step-by-step computation; explicit output-format specs and system roles improve consistency. These aren't random — they follow from how the model works (next-token prediction conditioning on context, Lesson 6.3), so you can reason about what will help, not just guess.

(2) Structure and specificity dominate luck. A vague prompt leaves the model to guess your intent; a well-structured one (clear role, task, examples, format, constraints) removes ambiguity and steers reliably. Skilled prompters also know to decompose hard tasks, give the model an "out" for uncertainty ("say 'unknown' if unsure" to reduce hallucination), and place key instructions where the model attends to them. There's genuine iteration involved, but it's informed iteration guided by an understanding of the model — which is exactly what makes it a skill rather than luck. (That said, the field is also moving toward making models robust to prompt phrasing, so the brittleness your teammate dislikes is a real, improving limitation.)

★ Deep dive: prompt patterns that move metrics, and the limits of prompting▸

Beyond the basics, a toolkit of patterns reliably improves results. Self-consistency: sample several chain-of-thought answers (with temperature) and take the majority vote — multiple reasoning paths catch each other's errors, often beating a single greedy CoT. Decomposition: break a hard task into a sequence of simpler prompts ("first outline, then write each section"), since models handle focused subtasks better than one sprawling request — this is the seed of agentic workflows (Track 9). ReAct: interleave reasoning with actions (like searching or calling a tool), letting the model gather information mid-reasoning (Track 9 builds this out). Giving an out: explicitly permitting "I don't know" measurably reduces hallucination, because you stop forcing a confident answer. Role and format priming: "You are an expert Python reviewer" and "respond in a markdown table" genuinely shift the output distribution toward the desired style. These patterns are studied, benchmarked, and move real metrics — prompting has matured from folklore into something closer to an engineering discipline (sometimes called "prompt engineering," though good models increasingly need less of it).

It's equally important to know prompting's limits, so you reach for the right tool. Prompting can't add knowledge the model lacks — for fresh or private facts you need retrieval (Track 8), not a cleverer prompt. It can't reliably enforce a complex behavior across all inputs — for that, fine-tuning (Lesson 7.2) bakes it in. It's brittle — small wording changes can swing results, and a prompt tuned for one model may flop on another. And long, example-heavy prompts cost tokens (Lesson 6.1) and context window. The mature view is that prompting, retrieval, and fine-tuning are complementary layers: prompt to direct the model in the moment, retrieve to supply knowledge, fine-tune to instill behavior — and reserve each for what it does best. Master prompting first because it's free and immediate and solves a surprising fraction of problems; reach for the heavier tools only when prompting genuinely can't get there. That judgment — knowing which layer a problem belongs to — is the real expertise, and it's the throughline connecting this lesson to the rest of the track and to Tracks 8 and 9.

Prompting steers a frozen model; sometimes you truly need to change its weights — affordably. Next, in 6.6 Parameter-efficient fine-tuning (LoRA/QLoRA), we see how to fine-tune a billion-parameter model by training less than 1% of it, even on a single GPU.
