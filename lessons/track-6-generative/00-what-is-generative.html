What is a generative model?

Every model you have built so far answers a question about data: which class, what value. A generative model does something stranger and more powerful — it learns what the data is, well enough to dream up brand-new examples that look like they belong.

Track 6 · Lesson 0 Concept + code ~22 min Builds on: Tracks 1, 3

This track is where models stop merely judging data and start creating it: images, audio, molecules, and the text completions you met in Track 5. The four families we will build from scratch — autoregressive models, VAEs, GANs, and diffusion — all chase the same goal in different ways. This lesson defines that goal precisely and gives you a mental map of the whole track.

💡 Concept: modelling the data, not just the label

Suppose every data point is a vector \(x\) (an image flattened into pixels, a sentence as token ids, a row of measurements). A discriminative model learns a conditional distribution \(p(y\mid x)\): given an input, predict a label \(y\). That is exactly what the classifiers in Track 3 did — they carve the input space with a decision boundary. They are superb at answering “which class?”, but they know nothing about how plausible any particular \(x\) is, and they cannot invent a new one.

A generative model instead learns the distribution of the data itself: either \(p(x)\) (the spread of all plausible inputs) or, per class, \(p(x\mid y)\). Once you hold a handle on \(p(x)\) you can do something a discriminator never can — sample from it: draw a fresh \(x \sim p(x)\) that the model considers typical. Generation is sampling from a learned distribution.

Link to Track 3. A discriminative model needs only the boundary between classes; a generative model must capture the shape of each class. That is strictly harder — you are modelling more — but it buys you sampling, density estimation, missing-data filling, and anomaly detection, all from one object.

📐 Four ways to model \(p(x)\)

Real data lives in very high dimensions where \(p(x)\) is fantastically complicated, so we cannot just write it down. The four families in this track are four strategies for representing and sampling from \(p(x)\), trading off three things: can you compute the exact likelihood \(p(x)\) of a point, how fast can you sample, and how sharp are the results.

Two quantities recur throughout the track, so meet them now. The likelihood \(p_\theta(x)\) is the probability your model (with parameters \(\theta\)) assigns to a real data point; training a likelihood-based model means turning \(\theta\) so that real data gets high probability — i.e. maximum likelihood, which you saw as a principle in Track 1. The second is a way to measure how far the model distribution sits from the true one, the KL divergence

\[ \mathrm{KL}\big(p \,\|\, q\big) \;=\; \mathbb{E}_{x\sim p}\!\left[\log \frac{p(x)}{q(x)}\right] \;\ge\; 0, \] (0.1)

which is zero exactly when \(p=q\) and grows as they diverge. Read it as “the average extra surprise from using \(q\) when the truth is \(p\).” It is not symmetric — \(\mathrm{KL}(p\|q)\neq\mathrm{KL}(q\|p)\) — and that asymmetry quietly shapes how each model behaves. We will derive it where it bites (the VAE’s training objective in Lesson 2).

🛠️ From scratch: classify and generate

Here is the whole idea in a few lines. We fit a discriminative model (logistic regression) and a generative one (a Gaussian per class) to the same two-moons data, then we sample from the generative model — something the classifier simply cannot do. This is the complete logic of gen_basics.py: 

import numpy as np
rng = np.random.default_rng(0)

# DISCRIMINATIVE: learn p(y|x) — a boundary. Cannot generate.
F = feats(X)                       # quadratic features of the points
w = np.zeros(F.shape[1])
for _ in range(4000):              # gradient descent on logistic loss
    p = 1/(1+np.exp(-F @ w))
    w -= 0.05 * F.T @ (p - y) / len(y)

# GENERATIVE: learn p(x|y) — one Gaussian per class. CAN generate.
for c in (0, 1):
    Xc = X[y == c]
    mu, cov = Xc.mean(0), np.cov(Xc.T)
    new_points = rng.multivariate_normal(mu, cov, 200)   # <-- sampling!

The single line rng.multivariate_normal(mu, cov, 200) is the entire difference in capability: it produces data that never existed. Everything else in this track is about making that sampled data dramatically more realistic than a Gaussian blob.

🔑 Key results to remember

  • Discriminative learns \(p(y\mid x)\); generative learns \(p(x)\) or \(p(x\mid y)\). Only the generative model can sample new data.
  • Generation = sampling from a learned distribution. The hard part is representing a complicated \(p(x)\) well enough to sample from it.
  • Likelihood \(p_\theta(x)\) is the probability the model assigns real data; maximising it over \(\theta\) is the default training principle for three of the four families.
  • KL divergence \(\mathrm{KL}(p\|q)\ge 0\) measures distribution mismatch and is asymmetric; it returns in the VAE objective.
  • The four families trade off exact-likelihood vs. fast-sampling vs. sharpness. No single model wins all three — hence four lessons.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. What can a generative model do that a purely discriminative classifier cannot?

  • Predict a class label for a new input.
    Both can do this — a generative model can classify via Bayes’ rule. This is not the distinguishing ability.
  • Achieve zero training error.
    Training error depends on capacity and data, not on the generative/discriminative split; either can overfit.
  • Draw brand-new samples that resemble the training data.
    Correct. Because it models \(p(x)\), you can sample \(x\sim p(x)\). A discriminator only models the boundary \(p(y\mid x)\) and has no notion of how to produce an \(x\).
  • Run faster at test time.
    Speed is unrelated; generative models are often slower to sample. The defining feature is sampling capability, not speed.

Q2. “Generation is just sampling from a learned distribution.” What makes this hard in practice?

  • Real \(p(x)\) lives in very high dimensions and has intricate structure that is hard to represent.
    Correct. A 64×64 colour image is a point in ~12,000 dimensions; the set of realistic images is a thin, curved sliver of that space. Capturing it is the whole challenge.
  • Random number generators are not truly random.
    Pseudo-randomness is more than good enough; this is not the bottleneck.
  • Sampling requires labelled data.
    Many generative models are trained unlabelled. Labels are optional, not the difficulty.
  • Probabilities must sum to exactly one in floating point.
    Normalisation is a technical detail handled routinely; it is not what makes generation hard.

Q3. The KL divergence \(\mathrm{KL}(p\|q)\) is…

  • symmetric: \(\mathrm{KL}(p\|q)=\mathrm{KL}(q\|p)\).
    No — KL is famously asymmetric, and the two directions encourage different failure modes (mode-covering vs. mode-seeking).
  • sometimes negative.
    It is always \(\ge 0\) (Gibbs’ inequality), reaching zero only when the distributions match.
  • a true distance metric.
    It violates symmetry and the triangle inequality, so it is a divergence, not a metric.
  • zero if and only if \(p=q\), and positive otherwise.
    Correct. \(\mathrm{KL}\ge 0\) with equality exactly when the distributions are identical, which is why minimising it pulls the model toward the data distribution.

Q4. In the Figure 0.1 demo, why are the generative model’s samples (right) a poor match to the crescent moons?

  • Logistic regression was trained for too few steps.
    Logistic regression is the discriminative model (left panel); it has nothing to do with the sampled points on the right.
  • A single Gaussian per class is too simple to capture a curved, crescent shape.
    Correct. A Gaussian is an ellipse of probability; it cannot bend into a moon. This limitation is exactly what autoregressive models, VAEs, GANs, and diffusion overcome.
  • There was not enough training data.
    Even with infinite data a single Gaussian stays elliptical — the model family, not the data amount, is the limit.
  • The samples were drawn from \(p(y\mid x)\).
    They were drawn from \(p(x\mid y)\), the per-class Gaussian. The issue is that Gaussian’s shape, not which distribution was sampled.

✍️ Practice

1. Run gen_basics.py. Then use the fitted per-class Gaussians to classify a point \(x\) by Bayes’ rule: \(p(y\mid x)\propto p(x\mid y)\,p(y)\). Does this generative classifier’s accuracy match the logistic-regression boundary?

Show solution

For each class compute the Gaussian density \(p(x\mid y=c)=\mathcal{N}(x;\mu_c,\Sigma_c)\), multiply by the class prior \(p(y=c)\) (here \(0.5\) each), and pick the larger. Because both classes share a similar covariance, the resulting boundary is close to a smooth curve and reaches roughly the high-0.8s to low-0.9s in accuracy on two-moons — competitive with, but not identical to, the discriminative logistic boundary. The lesson: a generative model gives you classification for free via Bayes, but when you only care about the label, modelling all of \(p(x\mid y)\) is more work than learning the boundary directly. This is the classic generative-vs-discriminative trade-off.

2. Replace each class’s single Gaussian with a mixture of two Gaussians (fit with a few steps of k-means then per-cluster mean/covariance). Do the sampled points hug the crescents better?

Show solution

Yes — markedly. Splitting each moon into two clusters lets the model place two tilted ellipses along the crescent, and their union curves far better than one ellipse. Sampling now means first picking a component (by its weight) then drawing from that Gaussian — your first taste of a latent variable (the component id), which is exactly the idea the VAE generalises in Lesson 2. Push this to many components and you approach a kernel density estimate; the catch is that such models do not scale to high-dimensional data like images, which is why we turn to neural generative models.

3. Conceptual: name one task where you specifically need \(p(x)\) (a generative model) and a discriminative classifier would not suffice.

Show solution

Several good answers: anomaly detection (flag points with low \(p(x)\) — a classifier has no \(p(x)\) to threshold); missing-data imputation (sample the missing coordinates from \(p(x_\text{missing}\mid x_\text{observed})\)); data augmentation / simulation (create new training examples); super-resolution or inpainting (sample plausible completions). All require a handle on how the data is distributed, not merely where the class boundary lies.
