Object detection: finding and boxing objects

⏱ 45 min Lesson 14.3 ▶ Open in Colab
Why this existsReal applications rarely just label a whole image — they need to find and count objects. Detection adds localization (a box per object) on top of classification, and IoU/mAP are how you measure whether it works.

Concept Not just what, but where — and how many

Classification answers "what is in this image?" with a single label. But real scenes have multiple objects, and often you need to know where each one is. Object detection does both: for every object in an image, it outputs a bounding box (a rectangle giving the object's location) and a class label with a confidence score. So detection = localization + classification, done for an unknown number of objects at once. It powers self-driving cars (find pedestrians, cars, signs), retail (count products), security, medical imaging (locate tumors), and much more — anywhere you need to find and identify things in an image, not just label the whole picture.

This is harder than classification for two reasons: the model must produce a variable number of outputs (a scene might have zero objects or fifty), and it must pin down locations, not just a label. Detectors handle this by proposing many candidate boxes across the image and, for each, predicting whether an object is there, which class, and how to adjust the box to fit tightly. To train and evaluate them, we need a way to measure how well a predicted box matches the true one — which is where IoU comes in.

Concept IoU: measuring box overlap

IoU (Intersection over Union) measures how well a predicted box overlaps the ground-truth box: it's the area of their overlap divided by the area of their union. IoU = 1.0 means a perfect match; IoU = 0 means no overlap. A predicted detection is usually counted as correct if its IoU with a true box exceeds a threshold (commonly 0.5) and the class is right. IoU is the fundamental metric of detection — it's how you decide whether a box is good, how the model is trained to tighten boxes, and how detectors decide which overlapping predictions to keep.

Worked example Computing IoU

IoU is simple arithmetic on box coordinates. Here we compute it for a predicted box against a ground-truth box (each as x1, y1, x2, y2 — top-left and bottom-right corners):

The ground-truth box is a 10×10 square at the origin; the prediction is the same size but shifted by 1 pixel in each direction. Their overlap is the 9×9 region where they coincide (area 81), and their union is the total area they jointly cover (100 + 100 − 81 = 119). So IoU = 81/119 = 0.681 — a good match, above the 0.5 threshold, so this counts as a correct detection. Trace the logic: the intersection is found by taking the inner-most edges (the max of the left/top corners, the min of the right/bottom corners), and the union is the two areas minus the double-counted overlap. This little function is doing exactly what every detector does millions of times — to decide if a prediction matches a true object, to train the model to tighten its boxes toward IoU = 1, and to remove duplicate boxes (if two predictions for the same object overlap heavily, keep the more confident one — a step called non-max suppression). The value 0.681 tells you this box is well-placed but not perfect (a perfect box would give 1.0); a prediction shifted far away would give a low IoU and count as a miss. Understanding IoU — that it's overlap-over-union, ranges 0 to 1, and gates whether a detection is "correct" — is the single most important metric concept in detection, because it's the ruler by which every box is judged.

★ Key points to remember
  • Object detection = localization + classification: a bounding box and a class (with confidence) for every object.
  • It's harder than classification: a variable number of outputs, and locations to pin down — not just one label.
  • IoU (intersection over union) measures box overlap, 0 to 1; a detection is "correct" if IoU ≥ a threshold (often 0.5) and the class matches.
  • IoU is used to judge boxes, train the model to tighten them, and suppress duplicate overlapping predictions (non-max suppression).
  • Detectors come as one-stage (fast, e.g., YOLO) or two-stage (accurate, e.g., Faster R-CNN); evaluated with mAP.

Practice Build the reasoning yourself

1You're building detection for a self-driving car (find pedestrians, cars, signs in real time) vs a medical tool (find tumors in scans, offline). Which detector style for each, and why?
Show solution & reasoning

For the self-driving car, prioritize a one-stage detector (like YOLO). The decisive constraint is real-time speed — the car must detect obstacles many times per second to react safely, so low latency is non-negotiable, and one-stage detectors (which predict boxes directly in a single pass) are built for exactly this. You accept slightly lower accuracy for the speed you must have. For the medical tool, prioritize a two-stage detector (like Faster R-CNN) or the most accurate model available. Here the analysis is offline (no real-time constraint — a radiologist reviews scans), and accuracy is paramount because missing a tumor (a false negative) is catastrophic, so you'd spend the extra compute of a two-stage, propose-then-refine approach for the best possible detection, and likely favor high recall (catch every possible tumor for a human to confirm).

The reasoning is the same requirements-driven trade-off that runs through system design (Track 10): the right choice depends on the constraints and costs, not on which detector is "best" in the abstract. Real-time + safety → optimize latency (one-stage); offline + high-stakes accuracy → optimize correctness (two-stage/most-accurate). You'd also tune the IoU and confidence thresholds to the stakes: the car needs reliable detection but can tolerate some imperfect boxes; the medical tool wants high recall (don't miss anything) even at the cost of more false alarms a human can dismiss. Recognizing that detector choice is a speed-vs-accuracy decision governed by the application's requirements — and being able to reason about which side each use case favors and why — is exactly the practical judgment that distinguishes someone who understands detection as an engineering choice from someone who just knows the model names. It's the accuracy/latency trade-off of the whole curriculum, applied to vision.

2A detector outputs three heavily-overlapping boxes for what is clearly one object. What's happening, and how is it fixed?
Show solution & reasoning

This is normal and expected: detectors propose many candidate boxes across the image, so a single object often triggers several nearby, overlapping predictions (the model is confident there's an object around here from multiple starting boxes). Left as-is, you'd report three detections for one object — clearly wrong. The fix is non-max suppression (NMS): among a cluster of overlapping boxes of the same class, keep the one with the highest confidence and suppress (remove) the others that overlap it heavily (IoU above a threshold with the kept box). Concretely: sort boxes by confidence, take the top one, discard any remaining box that has high IoU with it, then repeat with the next surviving box. The result is one clean box per object.

The reasoning ties back to IoU: IoU is the tool that decides which boxes are "the same detection." Two boxes with high mutual IoU are almost certainly the same object, so NMS uses IoU to collapse them to one. This reveals something important about how detectors work — they're not magically producing exactly one box per object; they produce many candidates and then clean up with a post-processing step (NMS) that relies on IoU. Understanding this demystifies detector output and explains a common practical concern: the NMS IoU threshold is a knob — too aggressive and you merge distinct nearby objects (two people standing close become one), too lenient and you keep duplicates. So NMS, and its IoU threshold, is a real part of tuning a detector. Being able to explain that overlapping boxes are the raw output and NMS (via IoU) resolves them into final detections shows you understand the detection pipeline, not just the model — the kind of practical, end-to-end understanding that matters on the job.

3How would you evaluate a detector overall (across all its predictions on a test set), and why isn't simple accuracy enough?
Show solution & reasoning

Simple accuracy doesn't fit detection because there's no single label per image — there are many objects, each needing a correct box and class, plus the model can produce false positives (boxes where there's no object) and false negatives (missed objects). The standard metric is mAP (mean Average Precision). Here's the logic: for each class, a prediction counts as a true positive if its IoU with a ground-truth box exceeds a threshold (e.g., 0.5) and the class matches; otherwise it's a false positive, and unmatched true objects are false negatives. From these you compute precision and recall (Track 11) across all the model's predictions ranked by confidence, trace the precision–recall curve, and take the area under it — the Average Precision (AP) for that class. Averaging AP over all classes gives mAP, the single headline number for a detector (often reported at IoU 0.5, written mAP@0.5, or averaged over several IoU thresholds).

The reasoning is that detection evaluation must jointly account for localization quality (via IoU), classification correctness, and the precision/recall balance across many predictions — which mAP does by combining them. Simple accuracy can't, because it assumes one label per example; detection needs a metric that handles multiple objects, variable output counts, and the trade-off between finding everything (recall) and not crying wolf (precision). This is a direct application of the evaluation discipline from Track 11 — precision, recall, the PR curve, choosing metrics that match the task — to the detection setting, where IoU adds the localization dimension. Understanding that a detector is evaluated with mAP (built from IoU-gated precision/recall), and why that's the right metric rather than accuracy, demonstrates you grasp both detection specifically and the general principle that the metric must fit the task's structure — exactly the evaluation judgment that separates rigorous practitioners from those who report a single misleading number.

★ Deep dive: how detectors actually find objects, and the field's evolution▸

The core challenge of detection — producing a variable number of located outputs from a fixed-size image — has been solved in evolving ways worth understanding. Early deep detectors were two-stage (the R-CNN family, culminating in Faster R-CNN): stage one proposes candidate regions likely to contain objects, stage two classifies each region and refines its box. This is accurate but slower (two passes). One-stage detectors (YOLO — "You Only Look Once" — and SSD) predict boxes and classes directly in a single pass over the image, dividing it into a grid and predicting objects per cell, which is much faster and enabled real-time detection at some accuracy cost. Both classic families relied on anchors — predefined box shapes at each location that the model adjusts — and on non-max suppression to clean up overlapping predictions. More recently, transformer-based detectors (DETR and successors) reframed detection as directly predicting a set of objects with attention, removing the need for hand-designed anchors and NMS — part of the same transformer wave reshaping all of vision. Across all of them, IoU remains the fundamental currency: it defines what a correct box is, drives the box-refinement loss, and powers NMS.

The broader lesson is that detection is classification's harder sibling, built on the same backbone but adding the localization problem — and its solutions illustrate a recurring pattern in applied ML: a hard structured-output problem gets tackled by clever framing (propose-and-refine, grid prediction, set prediction), and the field steadily simplifies the pipeline as models get more capable (from many hand-designed pieces — anchors, region proposals, NMS — toward end-to-end learned approaches). For a practitioner, the key takeaways are practical: detection gives you boxes + classes for many objects; the speed/accuracy trade-off (one-stage vs two-stage) is chosen by the application's requirements; IoU is the metric that gates correctness and drives training and NMS; and mAP is how you evaluate overall. You'd rarely implement a detector from scratch — you'd use a proven model (a YOLO variant, or a framework's detector) pretrained and fine-tuned on your data (transfer learning again), and spend your effort on the data, the thresholds, and the evaluation. Understanding detection at this level — what it outputs, how IoU governs it, the architectural families and their trade-offs, and how it's evaluated — equips you to build and reason about one of the most widely-deployed vision capabilities, and it sets up the next step: when a box isn't precise enough and you need the exact shape of an object, you turn to segmentation.
