Interview check: Python & Data

⏱ 15 min Lesson 2.8
Why this existsData and Python rounds test fluency under pressure: vectorize a loop on the spot, debug a shape error, and spot a leak in a described pipeline. This checkpoint rehearses those moves with worked answers.
By the end, you can
  • Vectorize a loop and explain why it's faster
  • Debug a broadcasting / shape-mismatch error
  • Reason about train/test splits and catch leakage
  • Explain a groupby and a merge clearly

Concept What the Python & data round tests

Data and Python interviews test fluency under mild pressure: can you vectorize a loop on the spot, read a shape error and fix it, write a groupby without hesitating, and spot a data leak in a pipeline someone describes? These are the daily motions of the job, so interviewers want to see them as reflexes, not puzzles. This checkpoint rehearses exactly those moves.

Practice Rapid-fire conceptual questions

QWhat is "vectorization," and why is it faster than a Python loop?
Model answer

Vectorization means expressing a computation as operations on whole arrays (a + b, np.sqrt(a)) instead of looping over elements. It's faster because the loop runs in NumPy's compiled C over a single contiguous block of one-dtype memory — no per-element Python interpreter overhead, cache-friendly access, and CPU SIMD parallelism. Typical speedups are 10–100×, and the same idea scales to GPUs for deep learning.

QExplain broadcasting in one sentence, then give the rule.
Model answer

Broadcasting lets NumPy combine arrays of different shapes by virtually stretching the smaller one, with no copying. The rule: align shapes from the right; each dimension must be equal, or one of them must be 1 (or missing, treated as 1). So (64,10) + (10,) works (adds the same 10 values to every row), but (4,3) + (4,) fails — you'd need (4,1) via keepdims=True.

QDifference between df.loc and df.iloc?
Model answer

loc selects by label (the index/column names); iloc selects by integer position. They coincide when the index is a default 0,1,2,…, but diverge after filtering or sorting, when the first row's label may not be 0. Use loc when you mean names, iloc when you mean positions.

QWhat's the difference between a NumPy view and a copy, and why does it matter?
Model answer

A slice usually returns a view — a window onto the same memory — so modifying the slice also modifies the original. Boolean and fancy indexing return copies. It matters because an accidental view means a write "spreads" to data you didn't intend to change (and pandas' SettingWithCopyWarning flags the same hazard). When you need independence, call .copy(); check with np.shares_memory.

QWhen would you use groupby versus merge?
Model answer

groupby aggregates within one table — "what's the X per Y?" (split–apply–combine: total sales per region). merge combines two tables by matching rows on a shared key — "attach each order's customer name." One summarizes; the other joins. They're often used together: merge to bring fields together, then groupby to summarize.

Practice Coding challenges (say your reasoning aloud)

C1Vectorize this loop: out = []; for i in range(len(a)): out.append(a[i]*b[i] if a[i] > 0 else 0).
Model answer

"It's an element-wise product, gated by a condition on a — that's a mask plus a multiply, no loop needed."

import numpy as np
a, b = np.asarray(a), np.asarray(b)
out = np.where(a > 0, a * b, 0)

np.where(condition, if_true, if_false) chooses element-wise. Equivalent: compute a * b then zero the non-positive spots with (a > 0) * (a * b). Either is a single vectorized expression that runs in C — the kind of "remove the loop" move interviewers look for.

C2Debug it: X is shape (100, 5), col_means is (5,) from X.mean(0), and X - row_means (with row_means = X.mean(1)) throws a broadcasting error. Fix it.
Model answer

"X.mean(1) collapses the columns, giving row means of shape (100,). Aligned from the right against (100, 5), that's 100 vs 5 — mismatch. I keep the axis so it broadcasts down the columns:"

row_means = X.mean(axis=1, keepdims=True)   # shape (100, 1)
centered = X - row_means                    # now broadcasts fine

"keepdims=True turns (100,) into (100,1), which lines up with (100,5) and subtracts each row's mean across its columns. The column version worked because (5,) already aligns with the right dimension 5." Reading the two shapes from the right is the whole debugging trick.

C3Spot the leak: a teammate scales all features with StandardScaler().fit_transform(X), then does train_test_split, trains, and reports 0.97 test accuracy. What's wrong and how do you fix it?
Model answer

"Scaling before splitting leaks: the scaler computed its mean and std over the entire dataset, including the test rows, so test information bled into preprocessing. The 0.97 is optimistic and won't hold in production."

# Fix: split first, fit on train only — ideally via a Pipeline
X_tr, X_te, y_tr, y_te = train_test_split(X, y, random_state=0)
pipe = make_pipeline(StandardScaler(), model)   # scaler fits on train inside fit()
pipe.fit(X_tr, y_tr)
print(pipe.score(X_te, y_te))                   # honest estimate

"The Pipeline guarantees the scaler — and every preprocessing step — fits only on the training fold, even under cross-validation. The rule: anything that learns from data (means, encoders, imputers, feature selection) must be fit on train, applied to test." Naming the rule, not just the fix, shows depth.

✓ Three traps interviewers love Score: 0 / 3

Q1. A pandas operation is slow because the code loops over df.iterrows() doing arithmetic. Best fix?

  • ABuy a faster computer
  • BUse a bigger for loop
  • CReplace the row loop with vectorized column operations (or .groupby/.agg)
  • DConvert the DataFrame to a Python list first
Hardware won't fix an algorithmic mismatch; iterrows is slow by design.
More looping is the opposite of the fix.
Correct. iterrows is notoriously slow; vectorized column math (or groupby aggregations) runs in C and is dramatically faster — the pandas equivalent of "remove the loop."
A Python list loses vectorization entirely and would be slower still.

Q2. You filter a DataFrame and write df[df.age > 30]["score"] = 0. It runs but nothing changes (and you get a SettingWithCopyWarning). Why?

  • AThe condition is wrong
  • BChained indexing may assign to a temporary copy; use df.loc[df.age > 30, "score"] = 0
  • Cpandas can't assign to columns
  • DYou must convert to NumPy first
The filter is fine; the problem is the two-step (chained) assignment.
Correct. df[...][...]= may modify a temporary copy, so the write is lost. A single .loc[rows, col] = value assigns to the original reliably.
pandas assigns to columns fine — when you use one indexer, not chained ones.
No NumPy conversion needed; just use .loc.

Q3. A model predicting 30-day churn includes the feature "days_since_last_login," computed as of today (after the 30-day window). The interviewer asks if that's okay.

  • ANo — it uses information from after the prediction point (temporal/target leakage); compute features only from data available at prediction time
  • BYes, more features always help
  • CYes, as long as there are no missing values
  • DIt only matters for deep learning
Correct. A feature measured after the prediction window leaks future information; features must be built only from data known at the moment of prediction. This is temporal leakage.
More features hurt when they leak — this one encodes the future.
Missingness is irrelevant; the timing of the measurement is the problem.
Leakage ruins any model regardless of type.
★ How to perform in the room
  • Reach for vectorization first. If you start writing a loop over array/DataFrame rows, say "actually, this vectorizes as…" — it's the signal of fluency.
  • Debug shapes out loud. Read the two shapes from the right and name the clashing axis; it shows method, not guessing.
  • Name the rule, not just the fix. "Fit preprocessing on train only" beats silently wrapping things in a Pipeline.
  • Be leak-paranoid. Volunteer "would I know this at prediction time?" — interviewers love candidates who distrust suspiciously good results.
  • Check before you trust. Mention .shape, .head(), row counts before/after a merge — the habits of someone who's been burned (in a good way).
