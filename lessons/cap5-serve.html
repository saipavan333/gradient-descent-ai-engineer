Capstone 5: ship a monitored model API

⏱ 120 min Lesson 13.6 ▶ Open in Colab
Why this existsShip a model as a monitored API — the production skills many candidates lack. Wrap the model in an endpoint and log every request for observability (latency, errors, drift). Turns a model into an operable service.
By the end, you can
  • Wrap a model in a serving API
  • Add two-layer monitoring
  • Detect silent drift
  • Explain why serving skills stand out

Concept The project: turn a model into an operable service

A trained model is a file; a product is a service users can call, reliably, that you can watch and maintain. This capstone builds exactly that: wrap a model in a serving API — an endpoint that receives requests and returns predictions — and add the monitoring that makes it operable. It's a high-value portfolio project because it demonstrates the production skills that many candidates lack: not just training a model, but deploying it as a real service and building the observability to run it. Serving and monitoring are where ML meets software/systems engineering, and showing you can do them signals you can ship, not just experiment.

The project has two parts. Serving: wrap the model behind an API endpoint (a web service — e.g., a small Flask or FastAPI app) that loads the model once, accepts requests, runs inference, and returns predictions as JSON. Package it (ideally in a container, Track 9) so it runs anywhere. Monitoring: log every request — its inputs, the prediction, and the latency — and track operational health (latency, error rate) and model behavior (prediction distribution, and drift signals). This observability is what turns a fragile script into an operable service: you can see when it's slow, when it errors, and when its predictions start drifting — the difference between catching a problem and being blindsided by it.

Worked example Serving with monitoring built in

The core pattern — serve predictions while logging every request for monitoring — is simple and essential. Here a serve function runs the model and records each request, so you can observe the system's behavior:

log = []
def predict(x): return x * 2                    # the "model"
def serve(x):
    y = predict(x)
    log.append({"input": x, "output": y})        # monitoring: log every request
    return y

for x in [3, 7, 10]:
    print(f"serve({x}) -> {serve(x)}")
print(f"logged {len(log)} requests for monitoring")
print(f"avg output: {sum(r['output'] for r in log)/len(log):.1f}")
Outputserve(3) -> 6 serve(7) -> 14 serve(10) -> 20 logged 3 requests for monitoring avg output: 13.3

Each call to serve runs the model and logs the request — so after serving three requests, you have a record you can analyze: 3 requests logged, average output 13.3. That logging is what makes the service observable: from the log you can compute the metrics that matter — request volume, latency (time each prediction), error rate (failed requests), and the distribution of predictions (which is how you'd detect drift, e.g., if the average output suddenly shifts, signaling the model is seeing different inputs than before, Track 9). In the real project, serve lives inside a web endpoint (Flask/FastAPI) so clients call it over HTTP, you'd time each request for latency, wrap inference in error handling, and send the logged metrics to a monitoring dashboard with alerts. But the principle is exactly this: every request flows through a path that both serves the prediction and records what happened, so the model isn't a black box you deploy and forget but a service you can watch. This is the operational discipline of Track 9 made concrete — the recognition that deploying the model is the middle of the work, and that observability (you can't operate what you can't see) is what lets you keep it healthy. Building this — a real endpoint serving a model with real monitoring — proves you understand production ML, not just model training, which is exactly the rarer, more valuable skill that makes this a strong portfolio piece.

★ Key points to remember
  • The project: wrap a model in a serving API (an endpoint) and add the monitoring that makes it operable.
  • Serving: a web service (Flask/FastAPI) that loads the model once, accepts requests, runs inference, returns predictions — ideally containerized.
  • Monitoring: log every request; track operational health (latency, errors) and model behavior (prediction distribution, drift).
  • Observability is essential — you can't operate what you can't see; it turns a script into a service.
  • Demonstrates production skills (serving + monitoring) that many candidates lack — a high-value portfolio signal.

Practice Build and reason about it

1Plan this project: what would you build, and what would you monitor?
Show a strong plan

"Serving: take a model from an earlier capstone (the image classifier or a simple model), and wrap it in a web API — a FastAPI/Flask app with a /predict endpoint that loads the model once at startup, accepts an input (image or features), runs inference, and returns the prediction as JSON. I'd add input validation and error handling (return a clear error, not a crash, on bad input), and containerize it (Docker) so it runs identically anywhere and is deployable (Capstone 0's reproducibility, extended to serving). Monitoring — two layers (Track 9): operational — latency per request (and percentiles, since the tail matters), error rate, request volume, uptime; and model — the distribution of predictions and inputs (to detect drift), and, where I can get labels later, actual accuracy over time. I'd log every request (input, prediction, latency, timestamp) and surface the metrics on a simple dashboard, ideally with alerts when latency or error rate crosses a threshold or the prediction distribution shifts. Bonus: a health-check endpoint, and load-testing to see how it behaves under concurrent requests."

The plan demonstrates you understand production serving as more than 'put the model behind a URL' — it includes the software-engineering realities (loading once, validation, error handling, containerization) and, crucially, the two-layer monitoring that Track 9 emphasized: operational health and model quality, because a service can be up and fast while its predictions silently decay. The reasoning skill is recognizing that a deployed model is a living service that needs observability on both axes, and knowing specifically what to monitor (latency percentiles, error rate, prediction/input distributions for drift, delayed accuracy). Being able to explain this — the serving setup and the deliberate two-layer monitoring — shows production maturity that's genuinely rarer than model-training skill, which is exactly why this project is such a strong differentiator: many candidates can train a model, far fewer can articulate how they'd serve and operate it reliably. That production judgment is a powerful thing to demonstrate."

2Your served model's operational metrics (latency, errors) are all green, but users report worse results over time. What's happening?
Show reasoning

"This is the classic silent model failure: the system is healthy (green operational metrics — it's up, fast, not erroring) but the model's quality has decayed, almost certainly due to drift (Track 9) — the live data has shifted away from what the model was trained on, so its predictions are increasingly wrong even though nothing crashed and the code is unchanged. Operational monitoring is blind to this by design, because it measures infrastructure, not prediction quality — which is exactly why you need the second monitoring layer. Model monitoring would catch it: tracking the prediction distribution might show it shifting; tracking the input distribution would reveal the drift directly (the incoming data no longer looks like training data); and any available outcome/label signal (or a business metric users care about) would show the measurable quality drop. To fix it, once confirmed, I'd retrain on recent data reflecting the new reality."

The reasoning demonstrates the single most important lesson of production ML monitoring: 'the dashboards are green' is dangerously incomplete for ML, because it answers 'is the service up?' but not 'is the model still good?' — and the gap between those questions is exactly where ML systems fail silently. Recognizing that green operational metrics say nothing about model quality, that the cause is likely drift, and that the fix is the model-monitoring layer plus retraining, shows you understand the two-layer monitoring principle and the reality that deployed models decay on their own. This is precisely the production-minded insight that this capstone (and Track 9) is built to instill, and being able to diagnose 'green metrics but worse results' as silent quality decay from drift — not a mysterious bug — is a strong, mature answer that signals you've thought about operating models, not just deploying them. It also motivates why you built the model-monitoring layer in the first place: to catch exactly this."

Demonstrating it signals you can bridge the gap from 'model that works in a notebook' to 'service that works in production,' which is precisely the gap where many projects (and candidates) fall short.

It also shows a mature understanding of the ML lifecycle: that deploying is the middle of the work, not the end, and that a deployed model is a living service needing observability and maintenance (Track 9). This production mindset — 'you're not shipping a model, you're committing to operating one' — is a strong differentiator, because it's the mindset of an engineer who ships things that keep working, not just prototypes that demo once. ★ Deep dive: the production gap, and why serving skills are undervalued and overneeded▸

There's a well-known gap in ML education and in many candidates' skills: enormous attention on model training (the exciting, tutorial-friendly part) and comparatively little on serving and operations (the part where value is actually delivered). Yet in industry the ratio is reversed — a model is trained once but served and operated indefinitely, and a model that isn't reliably served does nothing, so the engineering effort and business value concentrate in production, not training. This capstone deliberately targets that gap. Wrapping a model in an API, containerizing it, handling the software realities (loading, validation, errors, concurrency, latency), and building two-layer monitoring (operational health and model quality) are the concrete skills that turn a model into a product. They're also where ML engineering overlaps most with traditional software and systems engineering — uptime, scaling, observability, reliability — which is why the strongest ML engineers are those who can operate in both worlds. Demonstrating these skills, which many candidates lack precisely because they're under-taught, is a powerful differentiator.

The deeper reason this matters connects to everything in the MLOps track (Track 9): a deployed model is a living system in a changing world, so operating it — serving it reliably, monitoring both its infrastructure and its quality, detecting drift, and maintaining it — is not an afterthought but the larger part of the real work. The famous industry observation that the trained model is often less than 10% of a production ML system, with the rest being data pipelines, serving, and monitoring, captures exactly why this capstone is valuable: it exercises that other 90%. And it embodies the production mindset that this whole curriculum has emphasized — that shipping a model is a commitment to operating it, that you can't manage what you don't measure, and that observability is what separates a system that quietly rots from one that catches its own problems. Building a served, monitored model, and being able to reason about serving architecture, monitoring layers, drift detection, and operational trade-offs, is compelling evidence that you understand ML as a production discipline, not just a modeling exercise. For many employers, that's the most important thing a portfolio can show, because it predicts that you'll ship AI that actually works and keeps working — which is precisely what they're hiring for. The final capstone brings it all together: the complete pipeline from data to a monitored, retraining production system.
