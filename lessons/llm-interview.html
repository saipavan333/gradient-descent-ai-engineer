Concept What an LLM / GenAI interview tests

LLM interviews probe whether you understand how these systems are built and used end to end: the training pipeline, the practical levers (decoding, prompting, fine-tuning), and the trade-offs. The questions below are the ones that come up constantly; each model answer shows the reasoning interviewers want to hear.

Model answer "Explain RLHF."

"RLHF — reinforcement learning from human feedback — is how you align a model to human preferences after supervised fine-tuning. Three steps. First, collect preference data: show humans two model responses to the same prompt and have them pick the better one. Second, train a reward model — a network that scores a response — to agree with those human choices; it's essentially logistic regression on preference pairs, using the Bradley-Terry model where the probability a human prefers A is the sigmoid of the reward difference. Third, optimize the policy (the LLM) with reinforcement learning, usually PPO, to produce responses the reward model scores highly, with a KL penalty that keeps it from drifting too far from the original model. The result is a model that's helpful, honest, and harmless rather than just capable. A simpler modern alternative is DPO, which optimizes directly on the preference pairs and skips the separate reward model and RL loop."

Strong addition: mention reward hacking — the policy can game an imperfect reward model (sycophancy, padding).

Model answer "What's the difference between temperature and top-p?"

"Both control sampling, but differently. Temperature rescales the logits before the softmax: below 1 it sharpens the distribution (more deterministic, more repetitive), above 1 it flattens it (more diverse, riskier), and near 0 it approaches greedy. It changes how peaked the whole distribution is. Top-p (nucleus sampling) instead truncates: it keeps the smallest set of tokens whose cumulative probability reaches p (say 0.9) and samples only among them, cutting the unreliable tail. Its cutoff is adaptive — few tokens when the model is confident, many when it's uncertain — unlike top-k's fixed count. In practice you combine them: top-p removes the garbage tail, temperature controls diversity among the sensible options. A common default is temperature around 0.7–1.0 with top-p around 0.9; lower temperature for factual or code tasks, higher for creative writing."

Model answer "How does LoRA save memory?"

"LoRA freezes the pretrained weights and trains a small low-rank update instead. For a weight matrix W of size d×d, rather than updating all d² parameters, it learns two skinny matrices B (d×r) and A (r×d) with rank r much smaller than d, so the update ΔW = BA has only 2dr trainable parameters — often under 1% of the full count. The memory savings are twofold: you train far fewer parameters, and crucially you only need gradients and optimizer state (Adam keeps two extra values per trainable weight) for those few parameters, not the billions in the frozen base. QLoRA adds 4-bit quantization of the frozen base to shrink the memory just to hold it, which together let you fine-tune a 65B model on a single GPU. It works because the weight change a task needs is empirically low-rank, and a side benefit is no catastrophic forgetting plus swappable adapters, since the base stays frozen."

Model answer "Fine-tune, RAG, or prompt — how do you decide?"

"I match the tool to the need. Prompting changes what you ask — it's free and instant, so I start there; it handles a huge range of tasks via instructions and few-shot examples. RAG (retrieval) changes what the model knows — when the task needs fresh, private, or factual knowledge the model lacks, I retrieve relevant documents and put them in the prompt rather than trying to bake facts into weights, which is unreliable. Fine-tuning changes how the model behaves — I reach for it when I need a consistent skill, format, or style that prompting can't reliably enforce, or to make a smaller, cheaper model do one thing well. The mantra: prompt to direct, retrieve for knowledge, fine-tune for behavior. They're complementary layers, and most real systems use a combination — often a fine-tuned model with RAG and a carefully engineered prompt."

Model answer "How does a diffusion model generate images?"

"It learns to reverse a noising process. The forward process is fixed: gradually add Gaussian noise to a training image over many steps until it's pure static. The model is trained to undo one step — specifically, given a noisy image, predict the noise that was added, with a simple MSE loss. That's self-supervised, since you generated the noise and know it exactly. To generate, you start from pure random noise and run the learned denoising steps repeatedly, and a coherent image condenses out. To make it useful you condition on a text prompt via cross-attention so it denoises toward a matching image, and classifier-free guidance lets you dial up prompt adherence. Latent diffusion (Stable Diffusion) runs the whole thing in a compressed latent space from a VAE, which is what makes it efficient enough for a single GPU. Diffusion overtook GANs because its training is stable and it produces higher-quality, more diverse images, despite being slower to sample."

★ Rapid-fire answers to have ready
  • Base vs chat model? Base = pretrained only (autocompletes); chat = + SFT + alignment (follows instructions).
  • Why hallucination? Trained for plausible text, not verified truth; fix with retrieval (RAG).
  • In-context learning? Learn a task from prompt examples, no weight updates.
  • Chain-of-thought? "Think step by step" — intermediate steps let the model compute, boosting reasoning.
  • Chinchilla? Scale params and data together (~20 tokens/param); many early models were undertrained.
  • DPO vs RLHF? DPO optimizes directly on preferences — no reward model or RL, simpler and more stable.

Practice Mock interview prompts

1"Walk me through how ChatGPT was built, from raw text to deployed assistant." (2-minute answer.)
Show a strong answer

"Start with pretraining: a decoder-only transformer trained with next-token prediction on a massive web/book/code corpus — self-supervised, hugely expensive, producing a knowledgeable but not-yet-helpful base model. Then supervised fine-tuning on high-quality (instruction → response) demonstrations teaches it to follow instructions and act like an assistant. Then alignment with RLHF (or DPO): collect human preference comparisons, train a reward model, and optimize the policy to produce preferred responses with a KL leash — making it helpful, honest, and harmless. At deployment, behavior is shaped by the system prompt and user prompt (prompting, possibly few-shot or chain-of-thought), decoding settings (temperature, top-p) control the output style, and the system is often augmented with retrieval for fresh knowledge and tools for actions. Smaller customized variants can be made cheaply with LoRA. So it's: pretrain for knowledge, fine-tune for instruction-following, align for values, then prompt/retrieve/decode at serve time."

This hits every stage of the pipeline in order and connects them — exactly the end-to-end fluency interviewers want.

2"Your fine-tuned model started giving sycophantic, padded answers after alignment. Diagnose it."
Show a strong answer

"That's reward hacking. The RLHF reward model is an imperfect proxy for 'what humans actually want,' and RL ruthlessly maximizes whatever you actually measure. If raters slightly preferred polite, thorough-looking responses, the reward model learned 'flattery and length = good,' and the policy exploited that to the extreme — Goodhart's law in action. I'd address it by improving the preference data (more diverse, penalize padding and sycophancy explicitly), strengthening the KL penalty so the policy doesn't drift as far, evaluating the reward model directly for these failure modes, and possibly switching to DPO or adding rule-based checks. More broadly it's a reminder that alignment optimizes a proxy, and any gap between the proxy and true intent gets exploited — which is a core AI-safety concern (Track 11)."

Naming reward hacking, Goodhart's law, and concrete mitigations shows real depth about the alignment process, not just its happy path.

3"Explain to a product manager why a smaller, well-trained model might be better than the biggest available one."
Show a strong answer

"Bigger isn't automatically better for a product. First, inference cost and latency: you pay for the model on every single request, forever, and a model half the size is roughly half the cost and faster — which matters enormously at scale and for user experience. Second, the Chinchilla insight: a smaller model trained on more data can match a larger, undertrained one in quality, so size doesn't equal capability. Third, diminishing returns: scaling laws mean each quality increment costs exponentially more, so the biggest model may be only marginally better while costing far more. And a smaller model is easier to fine-tune (LoRA), deploy on modest hardware, and even run on-device for privacy. So I'd recommend choosing the smallest model that meets our quality bar on our evaluation, not the biggest one available — and revisit if quality falls short."

This translates technical concepts (inference economics, Chinchilla, diminishing returns) into business language — exactly the communication skill senior roles require.

★ Deep dive: reasoning about LLM systems end to end▸

The strongest LLM candidates think in terms of the full system, not just the model. When asked to build something with an LLM, they reason through the layers: which base model (size/cost/quality trade-off), do we need fine-tuning (is there a consistent behavior to instill?) or just prompting, do we need retrieval (does it require knowledge the model lacks?), what decoding settings (deterministic for structured output, sampled for creative), how do we evaluate it (Track 11), and how do we serve it within latency and cost budgets (Track 9). That decomposition — model + adaptation + knowledge + decoding + evaluation + serving — is the mental checklist of someone who's actually shipped LLM products, and it turns a vague "use GPT" into a concrete, defensible architecture.

Three habits sharpen LLM reasoning. First, always reason about cost — tokens, inference, and training are real money, and the right answer is usually the cheapest thing that meets the quality bar, not the most powerful. The prompt → retrieve → fine-tune ordering is partly a cost ordering (free → cheap → expensive). Second, connect to fundamentals: a reward model is logistic regression (Track 3); LoRA exploits low-rank structure (Track 3); attention conditions diffusion on text (Track 5); the next-token objective is cross-entropy (Track 3); fine-tuning is transfer learning (Track 4). Showing these threads proves you understand why things work, not just that they do. Third, hold the dual truth about scale: LLMs are astonishingly capable and fundamentally limited (hallucination, frozen knowledge, brittleness), so good systems pair a strong model with retrieval, tools, evaluation, and guardrails. The candidate who can say both "here's why this is incredibly powerful" and "here's exactly how it fails and what I'd add to handle that" demonstrates the mature, system-level understanding that the best teams hire for — and it's precisely what the rest of this curriculum (Tracks 7–11) equips you to do."
