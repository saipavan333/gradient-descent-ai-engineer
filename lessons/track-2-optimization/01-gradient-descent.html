Gradient descent from scratch

One line — “step a little way downhill, then repeat” — is how essentially every model in modern AI is trained. Here we derive it, watch it work, learn its one critical knob, and use it to fit a real model end to end.

Track 2 · Lesson 1 Concept + code ~28 min Builds on: Lesson 2.0

💡 Concept: just keep stepping downhill

Last lesson gave us the arrow \(-\nabla f\), pointing in the direction of steepest descent. Gradient descent is the obvious, beautiful idea that follows: stand at your current guess, take a small step in that downhill direction, and repeat until you stop moving. Writing the parameter vector as \(\boldsymbol{\theta}\), one step is:

$$ \boldsymbol{\theta}_{\text{new}} = \boldsymbol{\theta}_{\text{old}} - \alpha\,\nabla f(\boldsymbol{\theta}_{\text{old}}). $$
(★)

Every symbol earns its place. \(\boldsymbol{\theta}\) holds whatever we’re optimizing (one number, or a billion). \(\nabla f(\boldsymbol{\theta})\) is the gradient there — the uphill arrow. The minus sign flips it downhill. And \(\alpha\) (alpha), the learning rate, is a small positive scalar setting how big a step to take. That’s the entire algorithm: evaluate the gradient, step against it, repeat. Notice it is nothing but the two operations from Track 1 — scale a vector (\(\alpha\,\nabla f\)) and subtract it.

🎯 The one knob that matters: the learning rate

Almost everything that can go wrong with gradient descent traces back to \(\alpha\). Too small and training crawls, wasting compute and possibly never arriving. Too large and each step overshoots the bottom, bouncing up the far wall — the iterates oscillate, or diverge to infinity. In between is a “just right” range where it descends quickly and stably:

⚠️ Divergence looks like NaN

In practice, an \(\alpha\) that’s too large makes the loss climb and then become inf or NaN within a few steps. If your loss explodes to NaN early in training, the first thing to try is a smaller learning rate. It’s the single most common training bug.

Two honest caveats we’ll resolve later. First, gradient descent finds a local minimum — wherever downhill leads from the start. On a convex bowl (Lesson 2.2) that’s also the global minimum; on a wrinkled surface like a neural network’s loss, it may not be — yet it works astonishingly well anyway. Second, real datasets are huge, so we usually compute the gradient on a small random mini-batch of data each step rather than all of it; that noisy version is stochastic gradient descent (SGD), the workhorse of deep learning, and the same rule (★) with a noisy \(\nabla f\).

🛠️ From scratch: gradient descent, then a real model

First the bare algorithm on a function we can check, then the payoff: using it to fit a linear regression — which, from Track 1 Lesson 12, means maximum likelihood under Gaussian noise.

code/track-2/gradient_descent.py
import numpy as np

def gradient_descent(grad, theta0, lr=0.1, steps=100):
    """Follow -gradient downhill. `grad` returns the gradient at a point."""
    theta = np.array(theta0, float)
    history = [theta.copy()]
    for _ in range(steps):
        theta = theta - lr * grad(theta)      # the entire algorithm: eq (★)
        history.append(theta.copy())
    return theta, np.array(history)

# Minimize f(x,y) = x^2 + 2y^2, whose gradient is (2x, 4y); minimum at (0,0)
grad_f = lambda t: np.array([2*t[0], 4*t[1]])
theta, hist = gradient_descent(grad_f, [3.0, 1.0], lr=0.1, steps=60)
print("converged to:", np.round(theta, 4))   # ~[0. 0.]

# ---- The payoff: fit y = w*x + b by gradient descent on Mean Squared Error ----
rng = np.random.default_rng(0)
X = rng.uniform(-3, 3, size=200)
y = 2.0 * X - 1.0 + rng.normal(0, 0.5, size=200)   # true line: w=2, b=-1, + noise

def mse_grad(params):
    w, b = params
    pred = w * X + b
    err  = pred - y                                # residuals
    # d/dw and d/db of  mean(err^2)
    return np.array([2*np.mean(err * X), 2*np.mean(err)])

params, _ = gradient_descent(mse_grad, [0.0, 0.0], lr=0.05, steps=400)
print("learned w, b:", np.round(params, 3))        # ~[2. -1.]  recovered the true line!
🤖 You just trained a model

That second block is machine learning in miniature: a model (\(y=wx+b\)), a loss (mean squared error = Gaussian negative log-likelihood, Track 1), and gradient descent driving the parameters to the values that best explain the data. Swap the linear model for a neural network and MSE for cross-entropy, and you have how GPT is trained — same loop (★).

🔑 Key results to remember

Lock these in

  1. The update: \(\boldsymbol{\theta} \leftarrow \boldsymbol{\theta} - \alpha\,\nabla f(\boldsymbol{\theta})\). Evaluate gradient, step downhill, repeat.
  2. Steps shrink automatically near a minimum because \(\nabla f \to 0\) there.
  3. Learning rate \(\alpha\): too small = slow; too large = overshoot/diverge (→ NaN); tune it first.
  4. GD finds a local min (global on convex problems). On huge data we use noisy mini-batch gradients: SGD.
  5. Fitting a model = running GD on its loss. Linear regression by GD on MSE recovers the true line — that’s training.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. In \(\theta \leftarrow \theta - \alpha\nabla f(\theta)\), why is there a minus sign?

  • To make the step size positive.
    Step size is set by \(\alpha\) (already positive); the sign is about direction, not magnitude.
  • The gradient points uphill, so we step against it to go downhill.
    Correct. \(\nabla f\) is steepest ascent; subtracting it moves toward steepest descent, decreasing \(f\).
  • To cancel the learning rate.
    The minus doesn’t cancel \(\alpha\); both appear in the step \(-\alpha\nabla f\). It flips direction.
  • Convention only; a plus sign works too.
    A plus sign would do steepest ascent, climbing the loss — the opposite of training. The sign is essential.

Q2. Your loss becomes NaN after three steps of training. The most likely fix?

  • Train for more steps.
    More steps of a diverging process make it worse, not better. The dynamics are unstable, not unfinished.
  • Add more parameters.
    Model size isn’t the issue; the optimization is blowing up. More parameters won’t stabilize an overshooting step.
  • Use a larger learning rate.
    Larger \(\alpha\) makes overshoot worse — exactly what produces the explosion. You want the opposite.
  • Use a smaller learning rate.
    Correct. A NaN explosion early in training is the signature of too-large \(\alpha\): steps overshoot and grow. Shrinking \(\alpha\) restores stable descent.

Q3. Near a minimum, what happens to the size of gradient-descent steps, and why?

  • They grow, to escape the minimum.
    GD seeks minima, it doesn’t flee them. Nothing makes steps grow as you approach one (with a sane \(\alpha\)).
  • They stay constant, set only by \(\alpha\).
    The step is \(\alpha\nabla f\) — it depends on the gradient too, which changes as you move. So step size isn’t constant.
  • They shrink, because the gradient approaches zero.
    Correct. At a minimum the slope is flat (\(\nabla f\to 0\)), so \(\alpha\nabla f\) shrinks and the algorithm gently settles.
  • They become negative.
    “Step size” is a magnitude; it doesn’t go negative. The direction can vary, but near a min the steps simply get small.

Q4. Fitting \(y=wx+b\) by gradient descent on mean squared error is equivalent to…

  • maximum likelihood under Gaussian noise on the targets.
    Correct (Track 1, Lesson 12). Minimizing MSE is the negative log-likelihood of a Gaussian noise model — GD just performs that minimization numerically.
  • maximizing the number of parameters.
    Linear regression has a fixed two parameters here; GD adjusts their values, it doesn’t add parameters.
  • a method unrelated to probability.
    It’s deeply tied to probability: squared error is the Gaussian likelihood in disguise, the central bridge from Track 1.
  • finding the median of the data.
    Squared-error fitting targets means, not medians (absolute-error fitting relates to medians). The connection here is the Gaussian/mean.

✍️ Practice

Problem 1 · by hand

Minimize \(f(x) = (x-3)^2\) by gradient descent from \(x_0 = 0\) with \(\alpha = 0.25\). Compute the first three iterates.

Show solution

\(f'(x) = 2(x-3)\). Update: \(x \leftarrow x - 0.25\cdot 2(x-3) = x - 0.5(x-3)\).

\(x_1 = 0 - 0.5(0-3) = 1.5\). \(x_2 = 1.5 - 0.5(1.5-3) = 1.5 + 0.75 = 2.25\). \(x_3 = 2.25 - 0.5(2.25-3) = 2.25 + 0.375 = 2.625\). The iterates \(0, 1.5, 2.25, 2.625, \dots\) close in on the minimum \(x=3\), halving the remaining distance each step.

Problem 2 · reasoning

For \(f(x) = (x-3)^2\), what is the largest learning rate \(\alpha\) for which gradient descent still converges? (Hint: write the update as \(x \leftarrow x - 2\alpha(x-3)\) and look at the multiplier on the distance to the minimum.)

Show solution

Let \(d = x - 3\) be the distance to the minimum. The update gives \(d_{\text{new}} = d - 2\alpha d = (1 - 2\alpha)d\). Each step multiplies the distance by \((1-2\alpha)\). This shrinks (converges) exactly when \(|1 - 2\alpha| < 1\), i.e. \(0 < \alpha < 1\). At \(\alpha = 1\) it oscillates forever between two points; above \(\alpha=1\) the distance grows and it diverges. The stability limit is set by the curvature (here \(f''=2\)): in general \(\alpha < 2/f''\).

Problem 3 · code

Modify the linear-regression fit to also report the final MSE, and try learning rates \(0.01\), \(0.05\), and \(0.5\). What happens at \(0.5\)?

Show solution 
import numpy as np
rng = np.random.default_rng(0)
X = rng.uniform(-3, 3, size=200); y = 2*X - 1 + rng.normal(0, 0.5, size=200)
def fit(lr, steps=400):
    w, b = 0.0, 0.0
    for _ in range(steps):
        err = (w*X + b) - y
        w -= lr * 2*np.mean(err*X); b -= lr * 2*np.mean(err)
    return w, b, np.mean(((w*X+b)-y)**2)
for lr in [0.01, 0.05, 0.5]:
    print(lr, np.round(fit(lr), 3))

At \(0.01\) it converges slowly (may not fully reach \(w{=}2,b{=}{-}1\) in 400 steps); at \(0.05\) it converges nicely to MSE \(\approx 0.25\) (the noise variance \(0.5^2\) — you can’t beat the noise floor!); at \(0.5\) the step is too large and the parameters explode to inf/NaN. A perfect demonstration of Figure 1.3.

Problem 4 · reasoning

Why does stochastic gradient descent (using a small random batch each step) sometimes help training, beyond just being faster per step?

Show solution

The batch gradient is a noisy estimate of the true gradient, so SGD takes slightly random steps. That noise lets it jump out of shallow local minima and saddle points that a deterministic step might get stuck in or near — a free form of exploration on the wrinkled, non-convex loss surfaces of deep networks. It also acts as a mild regularizer. The trade-off is a noisier path that never perfectly settles (it jitters around the minimum), which is why learning-rate schedules shrink \(\alpha\) over time. We’ll see these dynamics again in Track 4.
