The learning problem & generalization

Track 2 taught us to drive the training loss to zero. Track 3 asks the deeper question: so what? A model that aces its training data but fails on new data has learned nothing. Generalization — performing well on data you’ve never seen — is the whole point.

Track 3 · Lesson 0 Concept + code ~26 min Builds on: Tracks 1–2

It is trivial to get zero error on data you’ve already seen: just memorize it. The hard, real goal of machine learning is to do well on data you haven’t seen. This lesson makes that goal precise, names the two ways models fail at it, and shows the U-shaped curve that governs every model you will ever train.

💡 Concept: training error is not the goal

We train by minimizing error on a finite training set — call this the training error (or empirical risk). But what we actually care about is the test error (the true risk): the error our model would make, on average, over all data drawn from the same source, including examples it never saw. The difference between them is the generalization gap:

$$ \underbrace{\text{generalization gap}}_{\text{what we fear}} \;=\; \underbrace{\text{test error}}_{\text{what we want small}} \;-\; \underbrace{\text{training error}}_{\text{what we measure}}. $$
(★)

A model can drive training error to zero yet have huge test error — it memorized the training set, gap and all. The only honest way to estimate test error is to hold out data the model never trains on: split your data into a training set (to fit) and a separate test set (to evaluate), and never let the model see the test set during training. This rests on one assumption — that training and test data are drawn independently from the same distribution (the “i.i.d.” assumption). When that breaks (the world shifts after you train), all bets are off, which is why monitoring for distribution shift matters in deployment (Track 9).

📐 Two ways to fail: underfitting and overfitting

Imagine fitting curves of increasing flexibility to noisy data drawn from some true underlying function. There are two opposite failure modes, with a sweet spot between them:

Underfitting means the model lacks the capacity to represent the real pattern — like fitting a straight line to a curve. Both training and test error stay high. Overfitting is the seductive opposite: the model is so flexible it fits the training data too well, memorizing the random noise specific to those exact points. Training error looks fantastic, but the model has learned quirks that won’t recur, so test error is terrible. The art of machine learning is finding the capacity that captures signal without fitting noise.

🎯 The U-shaped curve that governs everything

Plot error against model complexity and a universal picture emerges. Training error falls monotonically — more flexibility always fits the seen data better. But test error is U-shaped: it falls as the model gains enough capacity to capture the signal, bottoms out at the sweet spot, then rises as the extra capacity starts fitting noise. The gap between the two curves is exactly the generalization gap (★), and it widens as complexity grows.

⚠️ Never tune on your test set

If you pick the model that minimizes test error, you’ve secretly used the test set to make a decision — and your test error is now an optimistic lie. The fix is a third split, the validation set: tune complexity on validation data, and touch the test set exactly once, at the very end, to report an honest number. We make this rigorous with cross-validation later in the track.

🤖 A modern twist: double descent

The classic U-curve is the foundation — but very large models (like deep nets and LLMs) show a surprising sequel: push complexity far past the point of fitting the training data perfectly, and test error can fall again, a phenomenon called double descent. It doesn’t overturn the U-curve; it reveals a second regime the classical theory didn’t anticipate. We’ll meet it again when the classic story and modern deep learning collide in Lesson 3.3.

🛠️ From scratch: watch overfitting happen

We’ll fit polynomials of increasing degree to noisy data and measure training and test error directly. The U-curve will appear in the numbers.

code/track-3/generalization.py
import numpy as np
rng = np.random.default_rng(0)

true_f = lambda x: np.sin(1.3*x) + 0.3*x
def make_data(n):
    x = rng.uniform(0, 5, n)
    return x, true_f(x) + rng.normal(0, 0.3, n)

x_train, y_train = make_data(20)
x_test,  y_test  = make_data(500)        # large, unseen test set

def error(deg):
    coef = np.polyfit(x_train, y_train, deg)         # fit on TRAIN only
    train_err = np.mean((np.polyval(coef, x_train) - y_train)**2)
    test_err  = np.mean((np.polyval(coef, x_test)  - y_test )**2)
    return train_err, test_err

print(f"{'degree':>6} {'train':>8} {'test':>8}")
for deg in [1, 2, 3, 5, 9, 15]:
    tr, te = error(deg)
    print(f"{deg:>6} {tr:>8.3f} {te:>8.3f}")
# train error keeps dropping; test error dips then EXPLODES -> overfitting

best = min(range(1, 16), key=lambda d: error(d)[1])
print("best degree by test error:", best)    # a small/medium degree, not 15

🔑 Key results to remember

Lock these in

  1. We minimize training error but care about test error on unseen data. Gap = test − train.
  2. Estimate test error with a held-out test set; assume train/test are i.i.d. from the same source.
  3. Underfitting: too simple → high train & test error. Overfitting: too flexible → low train, high test (fits noise).
  4. Training error falls monotonically; test error is U-shaped in complexity. Aim for the bottom of the U.
  5. Tune on a validation set, never the test set; report test error only once. (Modern caveat: double descent.)

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. A model has 0% training error and 35% test error. This is a classic case of…

  • underfitting.
    Underfitting shows high error on both sets. Here training error is zero — the model fit the training data perfectly, the opposite problem.
  • the i.i.d. assumption failing.
    Possible in general, but the textbook signature here — perfect train, poor test — is overfitting. A distribution shift would be diagnosed differently.
  • overfitting.
    Correct. Zero training error with high test error means the model memorized the training set (including its noise) and fails to generalize — the definition of overfitting.
  • a good model.
    A 35% test error is what matters, and it’s poor. The perfect training score is a warning sign, not a success.

Q2. As model complexity increases, what happens to training error and test error?

  • Both decrease forever.
    Training error decreases, but test error eventually rises as the model starts fitting noise — that’s the whole U-shape.
  • Training error keeps falling; test error is U-shaped (falls, then rises).
    Correct. More capacity always fits seen data better (train ↓), but past the sweet spot it fits noise, so test error turns back up.
  • Both stay constant.
    Complexity clearly changes fit quality; neither curve is flat. Training error in particular drops steadily.
  • Training error rises; test error falls.
    Backwards. More flexibility lowers training error; it’s test error that can rise (from overfitting).

Q3. Why do we keep a separate test set untouched until the very end?

  • To have more data to train on.
    The test set is deliberately withheld from training; its purpose is evaluation, not extra training data.
  • Because test data is higher quality.
    It’s the same kind of data; its value is being unseen, not being better.
  • To make training faster.
    Holding out data doesn’t speed up training (it slightly reduces training data). The reason is unbiased evaluation.
  • So our error estimate is honest — using it to choose the model would bias the estimate.
    Correct. Any decision made using the test set leaks information; the reported error then overstates real performance. One untouched final check keeps the number trustworthy.

Q4. What does the i.i.d. assumption underpinning generalization actually claim?

  • Training and test data are drawn independently from the same distribution.
    Correct. “Independent and identically distributed” — same source, independent draws. It’s what lets training performance predict test performance.
  • The model is independent of the data.
    The model very much depends on the training data (that’s learning). i.i.d. is about how the data is sampled, not the model.
  • The data has no noise.
    i.i.d. says nothing about noise levels; noisy data can still be i.i.d. The assumption concerns the sampling process.
  • Test data comes later in time.
    Timing isn’t the point (and time-ordered data often violates i.i.d.). The claim is same-distribution, independent sampling.

✍️ Practice

Problem 1 · reasoning

You fit a degree-15 polynomial to 16 data points and get exactly zero training error. Should you be pleased? What do you predict for its test error?

Show solution

No — be worried. A degree-15 polynomial has 16 free coefficients, enough to pass exactly through 16 points, so zero training error is guaranteed and meaningless (it would happen for any 16 points, signal or pure noise). Between the data points the curve will swing wildly to thread every point, so it has almost certainly fit the noise. Predicted test error: high. This is overfitting in its purest form — the model has as many knobs as data points and used them all to memorize.

Problem 2 · reasoning

A colleague reports “92% accuracy” but admits they tried 50 model variants and reported the best one’s score on the same test set. Why is 92% likely optimistic?

Show solution

By selecting the best of 50 variants on the test set, they turned the test set into a tuning set. With 50 tries, one will look good partly by luck (it happened to match the test set’s particular noise), so the maximum score overstates true performance — a form of overfitting to the test set itself. The honest protocol: tune all 50 variants on a validation set, then evaluate only the single chosen model on the test set, once. The more choices you make against a dataset, the less that dataset can be trusted for evaluation.

Problem 3 · code

Run generalization.py and read off the degree with the lowest test error. Then increase the training set size from 20 to 200 and rerun — does the best degree change?

Show solution

With 20 training points the best degree is small (often 3–5); the test error explodes for high degrees because there’s little data to constrain the wiggly fit. With 200 training points, higher degrees become safer — more data restrains the extra flexibility, so the best degree typically rises and the test error at high degrees grows far less. This previews a central theme: more data lets you safely use more complex models. Capacity and data size must be balanced — exactly the subject of Lesson 3.3.

Problem 4 · synthesis

Connect this lesson back to Track 2: gradient descent can drive training loss arbitrarily low on a flexible model. Why is “the optimizer worked perfectly” not the same as “the model is good”?

Show solution

Track 2’s optimizers minimize training loss — that’s their job, and they can do it perfectly on a high-capacity model by memorizing the data. But training loss is the wrong target on its own: a perfectly optimized overfit model has near-zero training loss and poor test performance. So optimization success (low training loss) and learning success (low test loss) are different things. This is why Track 3 exists alongside Track 2: optimization gets you a solution that fits the data; theory and regularization (next lessons) steer you toward one that also generalizes. A great optimizer applied to the wrong objective just overfits faster.
