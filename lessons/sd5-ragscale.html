Track 10 · AI System Design

Designing a RAG assistant at scale

⏱ 27 min Lesson 10.5 ▶ Open in Colab
Why this existsScaled RAG separates a heavy offline indexing pipeline from a light online serving path. The retrieved chunks become LLM input tokens, so k (chunks retrieved) is a direct cost and latency knob — token economics drive the design.
By the end, you can
  • Separate offline indexing from online serving
  • Reason about RAG token cost
  • Tune k for cost vs recall
  • Apply the ML lifecycle to RAG

Concept RAG as a production system, not a demo

Track 7 taught how RAG works: retrieve relevant documents, then have an LLM generate a grounded answer. Building a RAG demo is a weekend project. Designing a RAG assistant at scale — serving millions of users over millions of documents, fast and affordably — is a system-design problem that combines everything so far. The good news is that RAG fits the pattern you already know (Lessons 10.3–10.4): a heavy offline stage builds the index, and a light online stage answers each query. The new wrinkles are that the "expensive scorer" is now an LLM (costly per call, in both money and latency), and that the system has more moving parts — embedding model, vector database, reranker, LLM — each of which must scale.

The offline indexing pipeline does the heavy lifting once: chunk the documents (Lesson 7.2), embed each chunk, and load the vectors into a vector database (Lesson 7.4). This is expensive and slow, but it's done ahead of time and only re-run when documents change. The online serving path runs per query and must be fast: embed the query, retrieve the top chunks from the vector DB, optionally rerank, assemble a prompt with the retrieved context, and call the LLM to generate the answer. Separating them lets you scale each independently — the vector DB scales with corpus size, the LLM serving with query volume — and cache at multiple points.

Worked example The token cost of retrieved context

A RAG-specific cost driver: the retrieved chunks become input tokens to the LLM, and you pay per token. The number of chunks you retrieve (k) directly sets the cost and latency of every query:

Each query sends the system prompt (100 tokens) + the question (50) + the 5 retrieved chunks × 200 tokens = 1,000, for 1,150 input tokens. At $0.003 per thousand tokens, that's about a third of a cent per query — trivial for one query, but at a million queries a day it's ~$3,450 daily, over a million dollars a year just for input tokens (before counting the generated output tokens, which cost more). Now the design lever is visible: the retrieved context dominates the token count (1,000 of 1,150), so k — how many chunks you retrieve — is a direct cost and latency knob. Retrieve 10 chunks instead of 5 and you nearly double the input cost and slow every call; retrieve 3 and you save, but risk missing the answer-bearing chunk. This is a concrete instance of the recurring accuracy-vs-cost trade-off: more context can improve answer quality but linearly raises cost and latency, so you tune k (and chunk size, and whether to rerank so fewer chunks suffice) against your budget. Reasoning about token economics like this — recognizing that in RAG the retrieved context is the cost, and k is the dial — is exactly the quantitative thinking a scaled-RAG design demands, and it's why reranking (fewer, better chunks) and caching become so valuable at scale.

★ Key points to remember
  • Scaled RAG follows the pattern: heavy offline indexing (chunk → embed → vector DB) + light online serving (embed query → retrieve → LLM).
  • Separating them lets you scale each independently (vector DB with corpus size, LLM serving with query volume) and cache each.
  • The retrieved chunks become LLM input tokens — so k (chunks retrieved) is a direct cost and latency knob.
  • At scale, per-query token cost multiplies into real money — token economics drive the design.
  • Reranking (fewer, better chunks) and caching are high-value levers for cost and quality.

Practice Build the reasoning yourself

1A company's RAG assistant works but costs too much and is too slow. Give three levers to reduce cost/latency and the trade-off each carries.
Show solution & reasoning

Several levers, each with a trade-off. (1) Reduce k / rerank: retrieve fewer chunks, or retrieve more then rerank down to the few best (Lesson 7.5), so the LLM gets less context — directly cutting input tokens (cost) and latency. Trade-off: too few chunks risks missing the answer, so reranking (fewer better chunks) is the sweet spot, at the cost of a rerank step. (2) Cache: cache answers to repeated or semantically-similar queries (Lesson 10.7), so common questions skip the LLM entirely. Trade-off: staleness if answers change, managed with expiry. (3) Use a smaller/cheaper LLM (or a smaller model for easy queries, escalating to a big one only when needed — a "router"). Trade-off: potential quality drop, so you'd route by difficulty. Other levers: a cheaper embedding model, a smaller/quantized reranker, shorter chunks, or trimming the system prompt.

The reasoning framework is to attack the dominant costs first. In RAG, the LLM call (input context + generation) usually dominates, so the highest-leverage moves target it: send it less (smaller k via reranking, shorter context), call it less often (caching), or call a cheaper one (smaller model / routing). Each trades some quality or adds complexity for cost/latency savings, and the right mix depends on the requirements — which is exactly the accuracy-vs-cost balancing that defines system design. A strong answer names the levers, quantifies roughly where the cost goes (the token math from the worked example), and picks the trade-offs that fit the requirement (e.g., "for a customer- support bot where most questions repeat, caching is the biggest win; for a research assistant with unique queries, reranking to cut k matters more"). Tailoring the levers to the workload is the mark of real design thinking rather than a generic checklist.

2Your document corpus updates constantly (news articles added every minute). How does this change the RAG design compared to a static corpus?
Show solution & reasoning

A constantly-updating corpus turns the "offline, occasional" indexing pipeline into a continuous, streaming one, which is a real design shift. With a static corpus you index once and serve; with a live corpus you need incremental indexing: as each new article arrives, chunk and embed it and insert its vectors into the vector DB in near-real-time, so it becomes retrievable within minutes, not at the next nightly rebuild. This requires a vector DB that supports efficient incremental inserts (and deletes/updates for changed or removed articles), and a streaming pipeline (Lesson 9.3's streaming features) feeding it. You also need to handle freshness in retrieval — possibly boosting recent documents or filtering by recency — since for news, a slightly-less-semantically-similar but newer article may be more useful than an older exact match.

Additional consequences: caching becomes riskier (a cached answer can go stale as new articles arrive, so cache TTLs must be short for time-sensitive queries), and you must decide the index-freshness SLA (how quickly must a new article be searchable — seconds, minutes?), which drives how aggressively you stream. You'd also manage index growth — a constantly-growing corpus needs a strategy for old documents (archive, down-weight, or drop), or the index grows without bound. The reasoning skill is recognizing that corpus dynamics are a first-class requirement that reshapes the indexing side of the architecture: "how often does the data change, and how fresh must retrieval be?" is exactly the kind of requirement (Lesson 10.2) you must elicit, because a static-corpus design and a live-corpus design differ substantially on the offline/indexing side even though the online serving path looks similar. Spotting that a streaming corpus demands incremental indexing, freshness handling, and shorter cache lifetimes shows you understand RAG as a living system, not a one-time index-and-serve.

3How do the ML-lifecycle concerns from Track 9 (monitoring, drift, evaluation) apply to a production RAG assistant?
Show solution & reasoning

They apply throughout, because a RAG assistant is a deployed ML system subject to all of Track 9's realities. Monitoring has the two layers (Lesson 9.7): operational (latency of retrieval and generation, error rates, LLM API availability, token spend) and quality (are answers grounded and correct?). Quality is hard to measure without labels, so you use proxies — retrieval metrics (are relevant chunks being fetched?), faithfulness checks (does the answer cite/match the context, or hallucinate?), user feedback (thumbs up/down), and abstention rate (how often it says "I don't know"). Drift appears in a RAG-specific form: the corpus drifts (new topics, changing facts) and the queries drift (users ask new kinds of questions the retrieval wasn't tuned for), both degrading quality even though the code is unchanged — caught by monitoring retrieval quality and query patterns over time. Evaluation (Lesson 7.7) is eval-driven development: a maintained test set of question–answer pairs with retrieval and faithfulness metrics, so any change (new embedding model, different k, new LLM) is measured, not guessed.

The deeper point is that designing the system is only half the job; operating it is the other half, and a complete design accounts for both. A strong RAG-at-scale answer doesn't stop at the architecture — it includes how you'll monitor answer quality (not just latency), detect corpus/query drift, evaluate changes against a held-out set, and update the index and models over time. This is where Track 10 (design) and Track 9 (MLOps) explicitly meet: the architecture determines what the system is, and the lifecycle discipline determines whether it stays good. Interviewers notice when a candidate closes this loop — designing not just a system that works at launch but one that can be observed, evaluated, and maintained — because that's the difference between a design that survives contact with production and a demo. RAG at scale is a perfect capstone for this because it touches retrieval, LLMs, cost, latency, and the full operational lifecycle, all at once.

★ Deep dive: the full scaled-RAG architecture and its failure modes▸

A production RAG assistant has more components than "retrieve then generate." The offline side: a data ingestion pipeline (pull documents, handle formats), chunking, an embedding service (itself a model to serve at scale for indexing), and a vector database that may be sharded (Lesson 10.7) across machines for a large corpus, plus metadata storage for filtering and citations. The online side, per query: optional query understanding/rewriting, query embedding, retrieval (often hybrid — dense + keyword, Lesson 10.4), reranking with a cross-encoder to get the best few chunks, prompt assembly with the context and citation instructions, the LLM call (itself a serving platform to scale — Lesson 10.6), and post-processing (extract citations, safety-filter the output). Wrapped around it all: caching at multiple layers (embedding cache, retrieval cache, answer cache — Lesson 10.7), monitoring, and cost controls. Each component is a sub-system that scales on its own axis, which is why the offline/online decoupling matters so much — it lets you scale the embedding service, the vector DB, and the LLM serving independently according to their different bottlenecks.

The characteristic failure modes are worth internalizing because they shape the design. Retrieval failures dominate (as in all RAG — Track 7): the relevant chunk wasn't fetched, so the answer is wrong no matter how good the LLM — which is why retrieval quality gets disproportionate attention and monitoring. Context overflow/distraction: stuffing too many chunks (high k) not only costs more but can degrade answers, as the LLM is distracted by irrelevant context — an argument for reranking to fewer, better chunks. Hallucination despite retrieval: the model ignores or misreads the context, addressed with prompt discipline ("answer only from context") and faithfulness monitoring. Stale index: the corpus changed but the index didn't, so answers are outdated — addressed with incremental indexing. Cost blowouts: unbounded k, no caching, or an expensive LLM on every query — addressed with the levers in this lesson and the next. Prompt injection via retrieved content (Track 8): a malicious document can carry instructions into the LLM's context, a genuine security concern for RAG over untrusted corpora. A mature scaled-RAG design anticipates these — heavy investment in retrieval quality and its monitoring, reranking to control k, multi-layer caching, incremental indexing, and guardrails on the LLM — because they are the ways this specific system predictably fails. Seeing RAG at scale as this full architecture-plus-failure-modes, rather than the tidy two-box demo, is the synthesis this track is built to produce: you're combining retrieval (Track 7), LLM serving (Track 6 and the next lesson), cost engineering, and the operational lifecycle (Track 9) into one coherent system reasoned against real requirements. That is exactly the capstone-level thinking that the design track exists to build, and the LLM serving platform underneath it is important enough to be its own lesson, next.
