Track 9 · AI Agents & Tool Use

Function / tool calling

⏱ 25 min Lesson 9.2 ▶ Open in Colab
Why this existsTools are how an agent acts — search, code, APIs, databases. Function calling lets the model emit a structured request the system runs, turning a text generator into something that can actually do things.
By the end, you can
  • Explain tool use / function calling
  • Describe a tool schema (name, parameters)
  • Trace the call → execute → observe cycle
  • Know what makes a tool well-designed

Concept How a text model reaches into the world

An LLM only outputs text — so how does it do things: search the web, run code, query a database, send an email? Through tool calling (also called function calling). The mechanism is elegant: you tell the model what tools are available, and when it wants to use one, it doesn't run it — it emits a structured request (typically JSON) naming the tool and its arguments. Your code reads that request, executes the real function, and feeds the result back to the model, which then continues. The model decides; your code acts.

This separation is fundamental and deliberate. The model never executes anything itself — it only proposes an action as structured text, and your code is the gatekeeper that decides whether and how to run it. That's what makes tool use controllable and safe: you validate the arguments, enforce permissions (Lesson 9.7), and handle errors, all in your code, before any real action happens. Modern chat models are specifically fine-tuned to produce these structured tool calls reliably, so the model "knows" to emit clean JSON when a tool would help.

Concept The tool schema: telling the model what's available

For the model to use a tool correctly, it needs to know the tool exists and how to call it. You provide a schema for each tool: its name, a natural-language description of what it does, and its parameters with their types.

The description is the most important part: it's how the model decides when to use the tool. A vague description ("weather stuff") leaves the model guessing; a clear one ("Get the current temperature and conditions for a given city") helps it call the right tool at the right time with the right arguments. Writing good tool schemas — clear descriptions, well-typed parameters, sensible defaults — is a real skill, because the model's tool use is only as good as its understanding of what each tool does. The parameter types also let your code validate the call before executing.

Worked example A tool call, end to end

Let's run the full round trip: the model decides to call a tool, your code executes it, and the result becomes the answer. (We hard-code the model's decision here; a real LLM would emit this JSON, having read the tool's schema.)

Trace the three steps. The model decided to call get_weather with {"city": "Tokyo"} — a structured request, not a guess at the weather. Your code looked up the tool and executed it with those arguments, getting "25C sunny" from the real function. That result was fed back, and the model produced a grounded answer. Crucially, the model never knew the weather and never ran any code — it knew to ask for it via the tool, and your code did the rest. This is the whole pattern, and it generalizes: swap get_weather for search_web, run_python, query_database, or send_email, and the model can take real actions in the world — always by proposing a structured call that your code validates and executes. The loop from Lesson 9.1 plus this tool-calling mechanism is the core of every agent: decide which tool to call, execute it, observe, repeat.

★ Key points to remember
  • Tool calling: the model emits a structured request (JSON) naming a tool + arguments; your code executes it and returns the result.
  • The model decides, never executes — your code is the gatekeeper that validates and runs the action (control & safety).
  • Each tool has a schema: name, description, typed parameters. The description tells the model when to use it.
  • Modern chat models are fine-tuned to emit clean tool calls; clear schemas make their tool use reliable.
  • Swap the tool and the model can search, run code, call APIs — the same mechanism powers all agent actions.

Practice Build the reasoning yourself

1You give an agent two tools: search and calculator. For "What is 17% of last year's revenue of $4.2M?", what sequence of tool calls would you expect, and why two tools?
Show solution & reasoning

The model would likely first decide it has the numbers it needs (revenue $4.2M is in the question, 17% is given), so it might just call calculator with "0.17 * 4200000" → $714,000, then answer. If the revenue weren't given, it would first call search to find it, observe the result, then call calculator with that value — two tools in sequence, each call informed by the previous observation.

This shows why agents chain tools: a real task often needs information (search) and computation (calculator), and the model decides the order based on what it has and what it observes. It also shows why LLMs are given calculators at all — they're unreliable at arithmetic (Lesson 6.1's tokenization quirks), so offloading math to a real calculator tool is far more accurate than asking the model to compute. Giving the model precise tools for things it's bad at (arithmetic, current facts, code execution) is a core design principle: let the model orchestrate reliable tools rather than do everything itself.

2Your agent keeps calling the wrong tool or passing malformed arguments. Before blaming the model, what would you inspect first?
Show solution & reasoning

Inspect the tool schemas — especially the descriptions and parameter definitions. The model chooses and fills tools based entirely on their schemas, so vague or ambiguous descriptions ("get data") lead to wrong tool choices, and unclear parameter specs (missing types, no examples, ambiguous names) lead to malformed arguments. Two tools with overlapping descriptions are especially confusing. Often the "model is dumb" is really "the schema is unclear."

Fixes: write precise, distinct descriptions that say exactly when to use each tool; specify parameter types, constraints, and ideally an example; name parameters unambiguously; and reduce the number of tools (fewer, clearer tools beat many overlapping ones). Then validate arguments in your code and return a helpful error message to the model if a call is malformed, so it can correct itself. Tool/schema design is one of the highest-leverage things you control in an agent — the analog of prompt engineering (Track 7), applied to tools.

3A tool call fails (the API is down, or returns an error). How should the agent system handle it, and why feed the error back to the model?
Show solution & reasoning

Your code should catch the error and return it to the model as an observation — e.g., "Error: weather API timed out" — rather than crashing the whole agent. Feeding the error back lets the model react: it might retry, try a different tool or argument, or tell the user it couldn't get the information. The model can only adapt to what it observes, so a silent failure (or a crash) removes its ability to recover.

This is exactly the ReAct pattern of the next lesson — each action's result (success or error) becomes an observation that informs the next decision. Robust agents treat tool errors as normal, recoverable events: catch them, surface them as informative observations, and let the model's reasoning route around them, with retry/budget limits (Lesson 9.7) so a persistently-failing tool doesn't cause an infinite retry loop. Graceful error handling is much of what separates a brittle agent demo from a dependable one — the real world is full of flaky tools, and the agent must cope.

★ Deep dive: how tool calling is trained, and the standardization of tools▸

How does a model "know" to emit a clean tool call? Through fine-tuning (Lesson 7.2). Chat models are trained on examples where, given a question and a set of tool schemas, the correct response is a structured tool call with valid arguments — so the model learns to recognize when a tool would help and to format the call correctly. The API then enforces or parses the structure (modern systems use constrained decoding, Lesson 7.4, to guarantee the tool call is valid JSON matching the schema, so it can't emit malformed calls). This is why tool calling "just works" with capable models but can be unreliable with weaker ones that weren't well-trained for it. It also means the model's tool-use ability is a learned skill that improves with model quality — better models choose tools more aptly, chain them more sensibly, and recover from errors more gracefully, which is part of why agent reliability has improved alongside base-model capability.

The ecosystem is rapidly standardizing how tools are exposed. Early on, every framework had its own way to define tools; now there's strong momentum behind the Model Context Protocol (MCP) and similar standards that define a common interface for tools and data sources, so a tool written once can be used by many different agents and models — like USB for AI tools. This matters because the value of an agent scales with the tools available to it, and standardization means a growing, shared library of tools (file systems, databases, web search, SaaS APIs, code execution) that any agent can plug into. The frontier is agents with access to hundreds of tools, which raises new challenges — the model can't reason well over hundreds of schemas at once, so techniques like tool retrieval (retrieve the relevant tools for a query, like RAG over tools — Track 8) and hierarchical tool organization are emerging. The throughline: tool calling is the bridge between a language model's reasoning and real-world action, it's a trained and standardizing capability, and designing good tools (and good schemas) is the key lever you control. With this mechanism in hand, the next lesson shows the dominant pattern for orchestrating tools — ReAct — where the model interleaves reasoning with these tool calls to solve multi-step problems.

The model can call tools — but how should it orchestrate them across a multi-step task? Next, in 8.3 ReAct: reasoning + acting, we meet the pattern that interleaves the model's reasoning with its tool calls, grounding each step in real results.
