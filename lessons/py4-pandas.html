pandas: Series & DataFrames

⏱ 27 min Lesson 2.4 ⬇ Runnable notebook
Why this existsReal data arrives as messy labeled tables, not tidy arrays. pandas loads, filters, groups, and joins that data — where an ML engineer spends a huge fraction of every project.
By the end, you can
  • Explain the Series and DataFrame models
  • Select, filter, and sort rows and columns
  • Aggregate with groupby and join with merge
  • Go from a raw file to an analysis-ready DataFrame

Concept Real data is a labeled table

NumPy arrays are perfect for pure numbers, but real data almost never arrives as a clean grid. It comes as labeled tables: columns with names like "customer" and "amount," rows identified by a label, and a mix of numbers, text, and dates — the kind of thing you'd open in a spreadsheet or a CSV file (a CSV, "comma-separated values," is the most common plain-text table format). The tool for this is pandas, and its two structures are the everyday vocabulary of data work.

A Series is a one-dimensional labeled array — think of a single spreadsheet column, where each value has a row label. A DataFrame is a two-dimensional table: a collection of Series (the columns) that all share one common set of row labels. That shared set of row labels is called the index. If a NumPy array is a bare grid of numbers, a DataFrame is that grid with names on the columns and labels on the rows.

Concept Selecting and filtering

You pull data out of a DataFrame by column, by label, by position, or by condition:

  • A column: df["amount"] returns that column (a Series).
  • By label: df.loc[row_label, "amount"] selects using the index/column names (loc = "by location-label").
  • By position: df.iloc[0, 2] selects by integer position, counting from 0 (iloc = "integer location," like NumPy).
  • By condition (the workhorse): df[df["amount"] > 100] keeps the rows where the test is true — the same true/false masking you learned in NumPy, now applied to whole rows.

Concept groupby: split, apply, combine

The most powerful everyday pattern in pandas answers questions of the form "what's the X per Y?" — total sales per region, average score per student. It's called split–apply–combine: split the rows into groups by some key column, apply an aggregation (a summarizing calculation like sum, mean, or count) to each group, and combine the per-group results into a new little table.

Concept merge: joining tables on a key

Data is usually spread across several tables — orders in one, customer details in another. merge (also called a "join") combines two tables by matching their rows on a shared key column (a column both tables have in common, like a customer ID), so you can bring related information together into one table.

Worked example From raw orders to insight

A small orders table. We'll inspect it, filter it, summarize per region, and join in customer info — the four moves you'll make on every dataset.

import pandas as pd

orders = pd.DataFrame({
    "customer": ["Ann", "Bo", "Ann", "Cy", "Bo"],
    "region":   ["West", "East", "West", "East", "East"],
    "amount":   [120, 80, 200, 50, 300],
})

print(orders.shape)                     # (5, 3)  -> always check the shape first
print(orders.describe())                # a quick numeric summary of the 'amount' column

# Filter: only orders over $100 (a true/false mask on whole rows)
big = orders[orders["amount"] > 100]
print("\nbig orders:\n", big)

# Group: total revenue per region (split-apply-combine)
by_region = orders.groupby("region")["amount"].sum()
print("\nrevenue per region:\n", by_region)
Output(5, 3) big orders: customer region amount 0 Ann West 120 2 Ann West 200 4 Bo East 300 revenue per region: region East 430 West 320 Name: amount, dtype: int64

Three lines turned raw rows into a per-region revenue summary (East 430, West 320 — split the rows by region, sum each group, combine). Now join in a separate customers table to attach each order's customer tier:

customers = pd.DataFrame({
    "customer": ["Ann", "Bo", "Cy"],
    "tier":     ["gold", "silver", "gold"],
})

merged = orders.merge(customers, on="customer")    # match rows by the shared 'customer' key
print(merged.groupby("tier")["amount"].sum())
Outputtier gold 370 silver 380 dtype: int64

The merge attached each customer's tier to their orders by matching the customer key, and a second groupby revealed revenue by tier. This load → inspect → filter → group → join loop is the daily rhythm of data work, and it's exactly what the EDA lesson (2.7) systematizes.

★ Key points to remember
  • A Series is a labeled 1-D column; a DataFrame is a table of Series sharing one index (row labels).
  • Select by column (df["x"]), by label (df.loc), by position (df.iloc), or by condition (df[df["x"] > 0]).
  • groupby = split–apply–combine: df.groupby("key")["val"].agg() answers "what's the X per Y?"
  • merge joins two tables by matching a shared key column.
  • Always check .shape, .head(), .info(), and .describe() first.

Practice Build the reasoning yourself

1Given a DataFrame df with columns ["product", "category", "price"], write one line to get the average price per category.
Show solution & reasoning
df.groupby("category")["price"].mean()

The reasoning: "average price per category" is split–apply–combine. Split by category, pick the price column, apply mean(). The result is a Series indexed by category. (For several stats at once: .agg(["mean", "max", "count"]).)

2What's the difference between df.loc[0] and df.iloc[0]? When could they return different rows?
Show solution & reasoning

df.iloc[0] selects by integer position — always the first row. df.loc[0] selects by the index label 0 — the row whose label is 0.

They differ whenever the index isn't a clean 0,1,2,…. For example, after filtering or sorting, the first row might carry label 7; then df.iloc[0] returns it, but df.loc[0] returns the (possibly different) row labeled 0 — or errors if no such label exists. The reasoning: loc is by name, iloc is by position; mixing them up is a classic bug, so always know which you mean.

3You merge orders (1000 rows) with customers, but the result has only 950 rows. What likely happened, and how would you investigate?
Show solution & reasoning

A default (inner) merge keeps only rows whose key exists in both tables. Losing 50 rows means 50 orders had a customer_id with no match in customers (missing or mistyped IDs), so they were dropped.

Investigate: check for unmatched keys, e.g. orders[~orders["customer_id"].isin(customers["customer_id"])], and consider how="left" to keep all orders (filling unmatched customer columns with "missing") so you can see and handle them rather than silently lose them. The reasoning: silent row loss in joins is a sneaky data bug — always compare row counts before and after a merge, exactly as you'd check .shape.

★ Deep dive: tidy data, vectorized text, and the SettingWithCopyWarning▸

Tidy data. pandas works best when data is "tidy": each row is one observation, each column is one variable, each cell one value. Messy data — values hidden in column names (a separate column per year), or several variables crammed in one cell — fights every operation. Reshaping tools (melt to go from wide to long, pivot for long to wide) exist to tidy data first; doing so up front makes groupby, merge, and plotting fall into place. Recognizing tidy vs. messy structure is a core data-wrangling instinct.

Vectorized text and dates. Just as NumPy vectorizes math, pandas vectorizes text and time through "accessors": df["name"].str.lower(), df["email"].str.contains("@"), df["date"].dt.year. Prefer these over looping or .apply() with a custom function — they're faster and cleaner. .apply() is the flexible fallback for logic with no vectorized form, but reach for it last, not first.

The SettingWithCopyWarning. This famous warning appears when you assign to a slice that might be a view or a copy, so pandas can't promise your write lands where you intend — e.g. df[df.age > 18]["score"] = 0 (called "chained indexing") may modify a temporary copy and silently do nothing. The fix is to do it in one step with .loc: df.loc[df.age > 18, "score"] = 0. The deeper lesson echoes NumPy's views-vs-copies: be explicit about which object you're modifying. When in doubt, build a new column or use .copy(), and prefer single-step .loc assignments. These three habits — tidy first, vectorize text/dates, assign with .loc — separate fluent pandas from the frustrating kind.
