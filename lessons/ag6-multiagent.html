Track 8 · AI Agents & Tool Use

Multi-agent patterns

⏱ 24 min Lesson 8.6 ▶ Open in Colab
Why this existsSome tasks decompose into specialties — a researcher, a writer, a checker. Multi-agent systems coordinate focused agents (supervisor, pipeline, debate), trading coordination cost for specialization — powerful, but easy to over-use.
By the end, you can
  • Describe supervisor, pipeline, and debate patterns
  • Explain the benefit of specialized agents
  • Weigh multi-agent costs and failure points
  • Decide when a single agent is better

Concept Many agents, each with a specialty

So far we've had one agent with tools, memory, and a loop. For complex jobs, an alternative is a multi-agent system: several agents, each specialized for a role, coordinating to solve the task together. The intuition is a team — instead of one generalist doing everything, you have a researcher, a writer, a reviewer, each focused on what it does best, with a coordinator orchestrating them. Specialization can improve quality (each agent has a focused role and prompt) and modularity (you can improve or swap one agent independently).

Three common patterns. Supervisor/worker: a supervisor agent breaks the goal into subtasks and delegates each to a specialized worker, then assembles the results — like a manager and a team. Pipeline: agents in sequence, each transforming the previous one's output (draft → edit → fact-check). Debate: multiple agents argue or critique each other's answers, converging on a more robust conclusion than any one would reach alone (an ensemble-like effect, Lesson 3.8). The agents coordinate by passing messages and sometimes sharing state, with one usually acting as orchestrator.

Worked example A supervisor delegating to workers

Here's the supervisor/worker pattern in miniature: a supervisor coordinates two specialized worker agents (a researcher and a writer), passing one's output to the next. (Each worker would be its own LLM with a focused role; here they're simple functions.)

The supervisor decomposed the goal and routed work: the researcher gathered information, its output fed the writer, which composed the result. Each worker had a focused role — and in a real system, each would be an LLM with a specialized prompt ("You are a meticulous researcher...", "You are a concise writer...") and its own tools (the researcher gets web search; the writer doesn't need it). The benefit is separation of concerns: a focused agent often outperforms a generalist trying to do everything in one tangled prompt, and you can improve the researcher without touching the writer. But notice the structure adds overhead — two LLM calls instead of one, plus the supervisor's coordination — so multi-agent isn't free. This toy shows the skeleton; the judgment is when the specialization is worth the added complexity and cost, which is the crux of multi-agent design.

★ Key points to remember
  • Multi-agent systems use several specialized agents coordinating — like a team of focused experts.
  • Patterns: supervisor/worker (delegate subtasks), pipeline (sequential transforms), debate (critique to converge).
  • Benefits: focused roles (better quality per agent) and modularity (improve/swap one agent independently).
  • Costs: more LLM calls, coordination overhead, and more failure points — multi-agent isn't free.
  • Use multi-agent when specialization genuinely helps; a single well-prompted agent often suffices.

Practice Build the reasoning yourself

1Design a multi-agent system to produce a fact-checked news summary. What agents would you use, in what pattern?
Show solution & reasoning

A pipeline fits well: (1) a researcher agent gathers articles on the topic (with web search); (2) a summarizer agent condenses them into a draft; (3) a fact-checker agent verifies each claim against sources (and flags unsupported ones); (4) an editor agent polishes the final text. Each transforms the previous output, with a focused role and appropriate tools. You might add a supervisor to orchestrate and handle failures (e.g., send it back to the fact-checker if claims fail).

The separation helps because each step needs different skills and prompts — researching, condensing, verifying, and editing are genuinely distinct, and a single agent prompt trying to do all four would be unwieldy and error-prone. Critically, the dedicated fact-checker (and giving it the sources to check against — RAG, Track 7) directly targets hallucination, a key risk in news. That said, weigh it against a simpler design: a single well-prompted agent with search and a "verify your claims" instruction might suffice for low-stakes summaries. Use the multi-agent pipeline when the quality/verification demands justify the extra calls and complexity — which, for fact-checked news, they plausibly do.

2A team replaces a single working agent with a 5-agent system and finds it became slower, more expensive, and less reliable. What likely went wrong?
Show solution & reasoning

They likely added multi-agent complexity where it wasn't warranted. Slower and more expensive is expected (five LLM calls and coordination instead of one), but less reliable reveals the deeper issue: more agents mean more handoffs where information gets lost or distorted, more failure points (any one agent erroring can derail the chain), and compounding errors across agents (Lesson 8.1) — five agents at 90% reliability each give ~59% end-to-end. If the task didn't genuinely need specialization, the coordination overhead added fragility without benefit.

The fix is often to simplify: collapse back to a single well-designed agent (or fewer agents), and add multi-agent structure only where a specific specialization clearly helps. The lesson — echoing the autonomy principle of Lesson 8.1 — is that multi-agent is not automatically better; it's a tool with real costs, and "more agents" is a common form of over-engineering. The most reliable systems use the fewest agents that get the job done, with clear interfaces between them. Reach for multi-agent when a task genuinely decomposes into distinct specialties, not because multiple agents sound sophisticated.

3Why might the "debate" pattern (agents critiquing each other) produce better answers than a single agent, and how does this connect to Track 3?
Show solution & reasoning

Debate works because independent critique catches errors: when one agent proposes an answer and another scrutinizes it ("is this claim supported? is the reasoning sound?"), flaws that the first agent missed get surfaced and corrected. Multiple agents with different framings also bring different perspectives, and converging across them tends to discard idiosyncratic mistakes while keeping shared, well-supported conclusions.

This is exactly the ensemble principle from Lesson 3.8: combining multiple models (here, multiple agent "votes" or critiques) reduces variance and errors, because independent mistakes cancel while correct signal reinforces — the wisdom of crowds. Debate is ensembling applied to reasoning: instead of averaging predictions, agents average arguments, with critique as the mechanism that filters out errors. It's most useful for hard reasoning problems where a single pass is unreliable, and it trades extra compute (multiple agents, multiple rounds) for higher answer quality — the same compute-for-reliability trade as self-consistency (Lesson 6.5) and reflection (Lesson 8.4), just distributed across agents. The recurring theme: multiple grounded perspectives beat one, whether they're trees in a forest, samples in self-consistency, or agents in a debate.

★ Deep dive: orchestration, emergent dynamics, and the single-vs-multi debate▸

Multi-agent orchestration has spawned frameworks (AutoGen, CrewAI, LangGraph, and others) that provide the plumbing — defining agents with roles and tools, routing messages between them, managing shared state, and handling the control flow (who acts next, when to stop). The hard parts are communication (how agents pass information without losing or distorting it — natural-language handoffs are flexible but lossy; structured messages are precise but rigid), control flow (who decides what happens next — a central supervisor, a fixed graph, or emergent negotiation), and shared state (a common scratchpad or memory all agents can read/write, versus isolated contexts). These mirror the challenges of human teams and distributed systems, and there's no one-size-fits-all answer — the right structure depends on the task. A practical pattern that works well is a supervisor with a fixed set of specialist tools-as-agents: the supervisor's reasoning drives the process (so control is clear), and each specialist is essentially a powerful tool it can call — which keeps the benefits of specialization while avoiding the chaos of fully peer-to-peer agent negotiation.

The most active debate in the field is single powerful agent vs multi-agent system. The multi-agent case: specialization, modularity, parallelism, and debate-style error-catching. The single-agent case: as base models get more capable, a single strong agent with good tools and a well-structured prompt can often match or beat a complex multi-agent system without the coordination overhead, handoff losses, and compounding failure points — and it's far simpler to build and debug. Empirically, many elaborate multi-agent systems have been found to underperform a simpler single-agent approach, because the coordination costs and error-multiplication outweigh the specialization gains. The emerging consensus leans toward simplicity: start with a single capable agent; add multi-agent structure only where a task genuinely benefits from distinct specialties or parallelism, and keep the number of agents and handoffs minimal. This echoes the whole track's reliability theme — every added agent is an added failure point, and the compounding-error math (Lesson 8.1) punishes complexity. There's also a fascinating frontier of emergent multi-agent dynamics: research on many interacting agents (simulated societies, negotiation, collaborative problem-solving) shows surprising collective behaviors, both useful and concerning (agents can develop unexpected strategies, or collectively fail in correlated ways). For now, the practical wisdom is grounded and a little deflationary: multi-agent is a real and sometimes valuable tool, but it's not a magic upgrade — it's a design choice with clear costs, best used sparingly and deliberately, and a single well-built agent is a strong default. With agents able to reason, act, plan, remember, and coordinate, the final and most important question is how to make them safe and reliable — which is the last lesson.

Agents that act in the world can also cause harm — wrong actions, runaway loops, security holes. Next, in 8.7 Guardrails & reliability for agents, we make agents safe and dependable enough to trust with real actions.
