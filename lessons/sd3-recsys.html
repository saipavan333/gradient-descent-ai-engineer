Track 10 · AI System Design

Designing a recommendation system

⏱ 28 min Lesson 10.3 ⬇ Runnable notebook
Why this existsRecommenders pick the best few items from millions — you can't score everything with a heavy model. The two-stage funnel (cheap candidate generation → expensive ranking) is forced by cost and reused across ML systems.
By the end, you can
  • Explain the two-stage funnel
  • Contrast candidate generation and ranking
  • Justify the funnel with cost math
  • Reason about objectives and feedback loops

Concept The most common ML system design question

"Design the recommendation feed" — for videos, products, posts, music — is the single most common ML system design prompt, because recommenders are everywhere and they exercise every part of the framework. A recommendation system selects, from a huge catalog of items, the handful most likely to interest a specific user right now. The central challenge is scale meets personalization: you might have 10 million items and 100 million users, you must respond in tens of milliseconds, and the "best 10 items" is different for every user and changes by the hour. You cannot score all 10 million items with a heavy model on every request — that's far too slow and expensive. The design that resolves this tension is the key insight of the whole lesson.

The architecture is a two-stage funnel. Candidate generation (also called retrieval) is stage one: a fast, cheap, approximate method narrows the full catalog down to a few hundred plausible candidates — using techniques like embedding similarity (find items near the user's taste vector via ANN, Track 7) or simple heuristics (popular items, items similar to recently-liked ones). It favors recall (don't miss good items) over precision. Ranking is stage two: a heavier, more accurate model scores just those few hundred candidates with rich features and orders them precisely, producing the final ranked list. It favors precision. The funnel works because the expensive model only ever runs on hundreds of items, not millions.

Worked example Why the funnel is essential — the cost math

The two-stage funnel isn't a stylistic choice; it's forced by cost. Compare ranking the whole catalog with the heavy model versus ranking only the candidates the cheap stage produced:

catalog, candidates, rank_cost = 10_000_000, 500, 100
naive     = catalog    * rank_cost     # rank the whole catalog with the heavy model
two_stage = candidates * rank_cost     # rank only the 500 candidates
print(f"rank all {catalog:,}: {naive:,} units")
print(f"rank {candidates} candidates: {two_stage:,} units")
print(f"cheaper by: {naive // two_stage:,}x")
Outputrank all 10,000,000: 1,000,000,000 units rank 500 candidates: 50,000 units cheaper by: 20,000x

If the heavy ranker costs 100 units to score one item, ranking all 10 million items costs a billion units per request — utterly infeasible at scale. But the cheap candidate-generation stage first narrows to ~500 items, so the ranker runs on only those: 50,000 units. That's 20,000× cheaper, which is the difference between impossible and routine. This single ratio explains why essentially every large-scale recommender and search system on earth uses a funnel: the expensive, accurate model is too costly to run on everything, so a cheap approximate stage first shrinks the problem to a size the expensive stage can afford. Candidate generation's job is to be fast and never miss the good items (high recall) even if it lets some junk through; the ranker's job is to be precise on the small set it's handed. The two stages have complementary jobs and complementary cost profiles — cheap-and-wide then expensive-and-narrow — and composing them is what makes personalized recommendation possible at 10-million-item scale. In an interview, deriving this cost gap out loud is exactly how you justify the funnel architecture rather than just asserting it.

★ Key points to remember
  • A recommender picks the few best items for a user from a huge catalog — scale meets personalization.
  • You can't score millions of items with a heavy model per request, so recommendation uses a two-stage funnel.
  • Candidate generation (retrieval): fast, cheap, approximate; narrows millions → hundreds; favors recall.
  • Ranking: heavier, precise model scores only the candidates; favors precision; produces the final list.
  • The funnel is forced by cost — running the ranker on hundreds instead of millions is thousands of times cheaper.

Practice Build the reasoning yourself

1Design candidate generation for a video recommender. Give two complementary sources of candidates and why you'd combine them.
Show solution & reasoning

A good candidate-generation stage combines multiple sources, because each captures a different notion of "plausibly relevant." Two complementary ones: (1) Embedding-based (personalized): represent the user and each video as vectors in a shared space (learned from watch history), and retrieve the videos whose vectors are nearest the user's vector via ANN search (Track 7). This captures personalized taste — "videos like the ones this user watches." (2) Popularity/trending (non-personalized): retrieve currently popular or trending videos, and perhaps recent uploads from channels the user subscribes to. This captures freshness and broad appeal that a purely history-based embedding might miss.

You combine them because relying on one source creates blind spots. Pure embedding retrieval can trap the user in a filter bubble — only ever surfacing more of the same — and struggles with the cold-start problem (a new user or new video has no history, so no meaningful embedding). Adding a popularity/trending source injects fresh, broadly-appealing, and new content, covering exactly those gaps. In practice recommenders blend several candidate sources (embeddings, trending, subscriptions, co-watch "people who watched X also watched Y") into one candidate pool for the ranker. The reasoning skill is recognizing that candidate generation is about coverage — you want the good items to be in the pool from multiple angles, since the ranker can only order what it's given — and that different sources cover different failure modes (personalization vs freshness vs cold-start). Naming the sources and the specific weakness each one patches is what makes this a designed stage rather than a vague "use embeddings."

2The ranking model needs training labels. Where do the labels come from, and what's a subtle danger in using them?
Show solution & reasoning

The labels come from user interactions logged in production: clicks, watches, watch-time, likes, purchases — implicit signals that a recommendation was good. You train the ranker to predict these (e.g., "probability this user watches this video > 30 seconds"), using historical logs of what was shown and what the user did. This implicit feedback is abundant and cheap, which is why recommenders can train on enormous datasets.

The subtle danger is a feedback loop (Lesson 9.8): the training labels are collected from items the current system chose to show, so the model learns from a biased sample of its own past behavior. If the recommender only ever showed popular items, it only has interaction data on popular items, so it learns to keep recommending popular items — a self-reinforcing loop that narrows recommendations and buries good-but-unshown content. This is presentation/position bias (users click what's shown near the top, regardless of true relevance) and exposure bias (you never learn about items you never showed) combined. It's dangerous because the model looks like it's improving on its logged metrics while actually collapsing diversity and entrenching itself. Mitigations include exploration (deliberately showing some uncertain items to gather unbiased feedback — the explore/exploit trade-off), de-biasing the training data by weighting for position, and monitoring diversity metrics, not just click-through. The deep reasoning is that in a recommender the model's outputs become its future training data, so naively optimizing logged clicks can quietly degrade the system — recognizing this reflexive trap, and proposing exploration to break it, is exactly the sophistication that distinguishes a strong recommender design.

3An interviewer asks: "Beyond predicting clicks, what else should a recommender optimize, and why does optimizing clicks alone go wrong?"
Show solution & reasoning

Optimizing clicks (or any single engagement metric) alone tends to go wrong because the metric is a proxy, and proxies get gamed by the optimizer. Pure click-optimization pushes toward clickbait (sensational thumbnails/titles that get clicks but disappoint), toward addictive but low-value content (maximizing short-term watch-time at the cost of user well-being and long-term retention), toward a filter bubble (narrow, homogeneous recommendations that reduce discovery), and can amplify misinformation or outrage because those drive engagement. The system optimizes exactly what you told it to — clicks — and in doing so degrades things you care about but didn't measure.

So a mature recommender optimizes a balanced objective reflecting multiple goals: engagement, yes, but also diversity (varied recommendations for discovery and against bubbles), freshness/novelty (not just the same items), long-term satisfaction and retention (are users still happy weeks later, not just clicking now), creator/ecosystem health (surfacing new creators, not only established ones), and sometimes explicit quality or trust signals (downranking clickbait or misinformation). These are often combined into a weighted objective or enforced as constraints/re-ranking rules on top of the engagement score. The reasoning skill — and the reason interviewers ask this — is recognizing that optimization is powerful and literal: it will maximize precisely the metric you specify, including in harmful ways you didn't intend, so choosing the objective is a design decision with real ethical and business stakes. This connects to the broader theme of AI safety and responsible design (Track 11): specifying the right objective, not just optimizing efficiently, is often the hardest and most consequential part of building a system that affects millions of people. A candidate who raises this shows they think about what a system should do, not just what it can do.

★ Deep dive: the full recommender stack, and the offline/online split▸

A production recommender is a stack of stages, more elaborate than "generate then rank." A common pipeline is: candidate generation from multiple sources (embeddings, trending, co-visitation, subscriptions), filtering (remove already-seen items, apply business rules and eligibility — age-appropriateness, regional availability, freshness), ranking with a heavy model scoring the survivors on rich features, and finally re-ranking that adjusts the ranked list for objectives beyond the per-item score — injecting diversity so the top 10 aren't all near-duplicates, ensuring freshness, applying fairness or exploration policies, and respecting constraints (don't show three ads in a row). Each stage narrows or reshapes the set, and each has its own model or logic. This staged structure is the recommender-specific instance of the general principle that ML systems compose cheap-broad and expensive-narrow steps to make personalization tractable at scale — and it maps directly onto the funnel diagram, just with more layers.

Underlying it all is the offline/online split that recurs across ML system design. The heavy work happens offline, on a schedule: training the embedding and ranking models, computing item embeddings for the whole catalog, and building the ANN index — all far too expensive to do per request. The online path, per request, does only the cheap work: look up the user's embedding, run ANN retrieval against the pre-built index, fetch pre-computed features, run the ranker on a few hundred items, and re-rank — all within the latency budget (Lesson 10.2). This split is why recommenders can feel instant despite operating over millions of items: the expensive computation was done ahead of time, and serving just assembles pre-built pieces. You'll see the exact same offline-build/online-serve pattern in search (next lesson) and RAG at scale (Lesson 10.5), because it's the fundamental way ML systems reconcile heavy models with tight latency — do the expensive part offline, keep the online part light. Recognizing this pattern, and being able to place each component on the offline or online side with a reason (does it need to be per-request-fresh, or can it be precomputed?), is one of the most transferable skills in ML system design. It turns "design a recommender" from a daunting blank page into a familiar shape: two-stage funnel, offline-built indexes and models feeding a light online path, re-ranked for objectives beyond raw engagement — a template you can adapt to almost any large-scale ML product.

Search is recommendation's close cousin — a funnel driven by a query instead of a user profile. Next, 10.4 Designing a search / ranking system covers retrieval, ranking, and how the two-stage pattern reappears.
