Track 11 · AI System Design

Interview check: System Design

⏱ 20 min Lesson 11.8
Why this existsSystem design interviews test whether you can drive a vague prompt to a scaled, affordable design while reasoning about trade-offs out loud. This checkpoint rehearses those questions with model answers.
By the end, you can
  • Approach any prompt with the framework
  • Design a recommender/search/RAG system
  • Estimate scale and control cost
  • Articulate a design philosophy

Concept What a system design interview tests

System design is the interview that most separates senior from junior engineers, because it can't be crammed — it tests whether you can take a vague, open-ended problem and drive it to a coherent design while reasoning about scale, cost, and trade-offs out loud. Interviewers aren't looking for a memorized "correct" architecture; they're watching how you think: do you clarify before designing, structure your approach, estimate scale, justify choices by requirements, and acknowledge trade-offs? The model answers below show the reasoning and the habits that make a design discussion land.

Model answer "How do you approach a system design question you've never seen?"

"I use the same framework every time so an unfamiliar prompt doesn't throw me. First, clarify requirements — I ask questions to turn the vague prompt into concrete functional requirements (what it must do) and non-functional ones (scale, latency, cost, availability), and I estimate the scale with back-of-the-envelope math (users, QPS, data size) because the scale drives the architecture. Second, high-level design — I sketch the major components and data flow, getting the whole system on the board before diving in. Third, deep-dive the ML — data sources, features, model choice, training, evaluation. Fourth, scale it — serving, replicas, caching, sharding to meet the load. Fifth, trade-offs — I discuss the choices I made, their alternatives, what I'd monitor, and how it could fail. The key is that I'm driving the conversation through this structure while staying responsive to the interviewer's steers, and reasoning out loud the whole time — because a design I can't seen before is just this process applied to new specifics. I'd rather show a sound method than a rehearsed answer."

Strong close: emphasize clarifying first — jumping to a solution before understanding the problem is the classic failure.

Model answer "Design a recommendation feed. Walk me through it."

"After clarifying (how many users/items, latency budget, what we optimize for), I'd use the two-stage funnel, because we can't score a 10-million-item catalog with a heavy model per request. Candidate generation: fast, cheap, high-recall retrieval narrows millions to a few hundred — using embedding similarity via ANN for personalization, plus trending and subscriptions for freshness and cold-start. Ranking: a heavier model scores just those candidates with rich features from a feature store, ordering them precisely. Then a re-ranking pass adds diversity and freshness so the top items aren't near-duplicates. The heavy work — training, item embeddings, the ANN index — is done offline; the per-request path just looks up the user embedding, retrieves, ranks, and re-ranks within the latency budget. I'd justify the funnel with the cost math: ranking hundreds instead of millions is thousands of times cheaper. And I'd flag the trap of optimizing clicks alone — it breeds clickbait and filter bubbles — so I'd optimize a balanced objective (engagement, diversity, long-term satisfaction) and watch for feedback loops. Finally, I'd note the operational side: monitor quality, detect drift, and retrain."

Model answer "How would you estimate the scale of a system?"

"Back-of-the-envelope, out loud, with round numbers — because I only need orders of magnitude to choose an architecture. Say it's a feed for 10 million daily users making ~20 requests each: that's 200 million requests a day, divided by ~86,400 seconds, so ~2,300 QPS average — and I'd design for peak, maybe 3× that, so ~7,000 QPS. That single number tells me I need horizontal scaling with a load balancer and autoscaling, not one server. Then I'd size the fleet: if each request takes 100ms, one worker does ~10 QPS, so ~700 replicas plus headroom. For storage, if I'm holding embeddings — 50 million docs × 768 dims × 4 bytes ≈ 150 GB — I know the index may need sharding or compression. The point isn't precision; it's that the scale determines the design, so I estimate early and let it drive my choices. And I state my assumptions — '10 million users, 20 requests each' — so the interviewer can correct my inputs. Estimation is how I ground the architecture in the actual numbers instead of hand-waving."

Model answer "This system is too expensive. How do you cut cost?"

"I'd first find where the cost goes — usually the expensive model inference (GPU-seconds for an LLM) — then attack it from three directions. Do less work: a two-stage funnel so the heavy model runs on few items, smaller k in RAG, shorter outputs. Do it less often: caching — exact and semantic — so repeated and similar requests skip the model entirely; savings scale directly with hit rate, so a 60% hit rate cuts model cost 60%. Do it cheaper: quantization or a distilled/smaller model, batching to keep GPUs full, routing easy queries to a small model and only hard ones to the big one, and autoscaling down off-peak. Each trades something — accuracy, freshness, or complexity — so I'd pick the levers that fit the requirements and quantify the savings. The unifying principle is to spend expensive computation only where it adds value, and use cheap resources — filters, caches, precomputation, approximations — everywhere else. I'd also set a cost budget per request and monitor spend as a first-class metric, because at scale inefficiency is real recurring money."

★ Rapid-fire answers to have ready
  • First move on any prompt? Clarify requirements (functional + non-functional) and estimate scale — never jump to a solution.
  • Recommender / search design? Two-stage funnel: cheap candidate generation (recall) → expensive ranking (precision).
  • Why the funnel? You can't run the heavy model on millions of items; narrow first, then rank hundreds.
  • Recurring pattern? Offline (build indexes/models) + online (light per-request path) — reused in recsys, search, RAG.
  • Latency SLA? Written on percentiles (p95/p99), not averages — the tail is what users feel.
  • Scale a service? Replicas + load balancer + autoscaling; batch, cache, quantize to cut per-request cost.
  • LLM serving? Queue → batch → GPU workers with KV cache; capacity = QPS × tokens ÷ per-GPU throughput.
  • Cost lever #1? Caching — savings equal the hit rate; then funnel, batching, compression, sharding.

Practice Mock interview prompts

1"Design an AI customer-support assistant for a large SaaS company." (Full 30-minute answer — outline your structure.)
Show a strong answer

"Clarify: scope (which products, languages), volume (tickets/day → QPS), latency (chat, so interactive), what 'good' means (resolution rate, escalation rate), and whether answers must cite sources. High-level design: this is a RAG assistant — retrieve from the company's docs/tickets, generate a grounded, cited answer. Offline: ingest and chunk docs, embed, build a vector index (incrementally, since docs update). Online: query understanding → hybrid retrieval → rerank → assemble prompt with citations and 'say I don't know' → LLM → post-process. Scale: LLM serving platform (queue, batch, GPU pool) or a hosted API depending on volume/build-vs-buy; autoscale for peak; shard the index if large. Cost: cache common questions (exact + semantic — support questions repeat heavily, so a high hit rate), rerank to keep k small, route easy questions to a small model, set a cost budget. Quality & ops: monitor faithfulness and resolution rate, detect drift as products change, evaluate on a held-out Q&A set, and human-escalation for low-confidence answers. Trade-offs: build vs buy for the LLM, k vs cost, cache TTL vs freshness, and guardrails against prompt injection from retrieved content."

This shows you can assemble RAG (Track 8), LLM serving (10.6), cost control (10.7), and the operational lifecycle (Track 10) into one coherent, requirement-driven system — and that you lead with clarification and close with trade-offs and ops. That end-to-end synthesis is exactly the senior signal.

2"Midway through your design, the interviewer says: 'Now assume traffic grows 100×.' How do you respond?
Show a strong answer

"I'd treat it as a scaling stress-test and reason through where the system breaks and how each part scales. Stateless serving scales horizontally — 100× the replicas behind the load balancer, with autoscaling — so that's mostly a cost question, not an architecture change. The data stores are the harder part: a vector index or database that fit on one machine at 1× may need sharding across many at 100×, which introduces fan-out latency and coordination I'd design for (replication for fault tolerance, hedging for tail latency). Caching becomes even more valuable at 100× — more traffic means more repetition to absorb, so I'd invest in a strong caching hierarchy to keep the hit rate high and shield the expensive model. Bottlenecks shift: at 100× I'd look for any single point that doesn't scale — a shared database, a rate-limited external API, a coordinator — and address it. And cost becomes dominant, so the efficiency levers (batching, compression, routing) that were optional at 1× are now essential. I'd also revisit whether a hosted API still makes sense versus self-hosting at this volume."

Handling a mid-interview curveball by calmly reasoning about what scales trivially (stateless replicas), what needs rework (sharding stateful stores), and what becomes critical (caching, cost, bottleneck-hunting) — rather than freezing — demonstrates exactly the adaptive, scale-aware thinking these interviews probe.

3"What separates a good AI system design from a bad one, in your view?" (Tests design philosophy.)
Show a strong answer

"A good design is driven by requirements and honest about trade-offs; a bad one is a pile of fashionable components with no justification. Concretely, a good AI system design (1) starts from clear functional and non-functional requirements and sizes itself to the actual scale, so it's neither over- nor under-engineered; (2) makes every component traceable to a requirement it satisfies — nothing is there because it's trendy; (3) reasons explicitly about the trade-offs (accuracy vs latency vs cost, freshness vs cost, simplicity vs scale) and picks the side the requirements favor; (4) plans for the full lifecycle — not just launch, but monitoring, drift, evaluation, and cost over time; and (5) fails gracefully, with the failure modes anticipated and guarded. A bad design skips requirements, over-engineers for scale it doesn't have (or ignores scale it does), can't justify its choices, and treats deployment as the finish line. The deepest mark of a good design is simplicity proportional to the requirements: the simplest architecture that meets the needs, adding complexity (sharding, multi-agent, custom serving) only where a requirement forces it — because every added component is added cost, latency, and failure surface. Great design is knowing what not to build as much as what to build."

Articulating a coherent design philosophy — requirements-driven, trade-off-honest, lifecycle-aware, and biased toward the simplest sufficient system — shows the interviewer you have judgment that generalizes beyond any single problem, which is the ultimate thing a system design interview is trying to find.

★ Deep dive: thinking like a system designer, end to end▸

The strongest system-design candidates share one habit: they treat every prompt as the same process applied to new specifics, not a new puzzle to solve from scratch. That process — clarify requirements, sketch architecture, deep-dive the ML, scale, control cost, weigh trade-offs — is a scaffold that turns an intimidating open-ended question into a structured walk, and it's why a good designer stays calm on an unfamiliar prompt: they trust the method. Layered on top is pattern recognition: most large-scale AI systems are variations on a few templates — the two-stage funnel, the offline/online split, the retrieve-then-generate shape — so recognizing which template a prompt maps to gives you an instant skeleton to adapt. "Design search," "design a feed," "design a RAG assistant" are, underneath, the same funnel with different inputs and a different expensive stage. Seeing that shared skeleton is what lets experienced designers produce a sensible architecture in minutes, then spend their time on the interesting specifics and trade-offs rather than reinventing structure.

The deepest signal, though, is judgment expressed as trade-off reasoning. There is no single right design, so what's evaluated is whether you can name the trade-offs (accuracy vs latency vs cost, freshness vs cost, simplicity vs scale, build vs buy), reason about which way the requirements push, and justify your choice — while acknowledging what you'd monitor and how it could fail. This is where system design connects to everything else in the curriculum: it's the capstone where classical ML, deep learning, NLP, LLMs, RAG, agents, and MLOps stop being separate topics and become building blocks you compose under real constraints. A recommender uses embeddings (Track 3–6) and a feature store (Track 10); a RAG assistant uses retrieval (Track 8) and LLM serving (Track 7) and drift monitoring (Track 10); an agent platform adds guardrails (Track 9). System design is the discipline of assembling these into a coherent whole that meets requirements at scale and stays affordable and maintainable — and reasoning about the whole out loud. That synthesis, more than any single technique, is what "senior ML engineer" means, and it's what this track has built. The final piece — how to evaluate whether these powerful systems are actually good, safe, and fair — is the concern of the next track, and it's the responsibility that comes with the ability to build systems that affect millions of people."

That completes Track 11 — you can design AI systems end to end, from vague prompt to scaled, affordable architecture.
