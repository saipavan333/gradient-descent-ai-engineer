Designing a search / ranking system

⏱ 27 min Lesson 10.4 ⬇ Runnable notebook
Why this existsSearch is recommendation with a query. It reuses the retrieve-then-rank funnel: hybrid retrieval (keyword + semantic) gathers candidates, a ranking model orders them. Cheap-wide then expensive-narrow, the same pattern.
By the end, you can
  • Relate search to recommendation
  • Contrast keyword and semantic retrieval
  • Explain hybrid retrieval
  • Apply the two-stage cost split

Concept Search is recommendation with a query

Designing a search / ranking system — "design search for an e-commerce site," "design a document search engine" — is recommendation's close cousin, and it reuses the same two-stage funnel. The difference is the input: a recommender starts from a user profile ("what would this person like?"), while search starts from an explicit query ("what matches these words?"). Both must find the best few items from a huge collection and return them fast. So the architecture rhymes: a cheap, wide retrieval stage gathers candidates that match the query, then an expensive, precise ranking stage orders them. If you understand the recommender funnel (Lesson 10.3), you already understand the shape of search.

What's distinctive about search is how retrieval works, because a query is words. Two retrieval methods complement each other. Keyword retrieval (classically an inverted index with BM25 scoring) matches the query's exact words — an inverted index is a map from each word to the list of documents containing it, so it finds documents sharing terms with the query, fast, and nails exact matches (names, codes, SKUs). Semantic retrieval (vector/ANN search, Track 7) matches by meaning via embeddings, so it finds relevant documents even with no shared words (query "laptop", document "notebook computer"). Each has a blind spot the other covers, so production search uses hybrid retrieval: run both and combine.

Worked example The two-stage cost split, for search

Search faces the same economics as recommendation: a cheap scorer can touch everything, an expensive one cannot. Here retrieval scores all documents cheaply, and a heavy reranker (a cross-encoder, Track 7) refines only the top few hundred:

docs, rerank_k = 10_000_000, 500
retrieve_cost, rerank_cost = 1, 50
stage1 = docs     * retrieve_cost      # cheap: score every document
stage2 = rerank_k * rerank_cost        # expensive: rerank the top-k
print(f"stage 1 retrieve all : {stage1:,}")
print(f"stage 2 rerank top {rerank_k}: {stage2:,}")
print(f"two-stage total: {stage1 + stage2:,}")
print(f"rerank everything: {docs * rerank_cost:,}")
Outputstage 1 retrieve all : 10,000,000 stage 2 rerank top 500: 25,000 two-stage total: 10,025,000 rerank everything: 500,000,000

The cheap retrieval scorer (cost 1 per document) can afford to touch all 10 million documents: 10 million units. The expensive reranker (cost 50 — think a cross-encoder that reads query and document together) runs only on the top 500 that retrieval surfaced: 25,000 units. The two-stage total is ~10 million units, dominated by the cheap retrieval pass. Reranking everything with the heavy model would cost 500 million units — about 50× more, and that's the affordable version of the same idea from recommendation. The lesson generalizes: you use a cheap scorer where you need to cover everything (retrieval, favoring recall) and an expensive scorer only where you need precision on a small set (reranking). The costs are complementary: retrieval is cheap-per-item but runs on many items; reranking is expensive-per-item but runs on few. Composing them keeps total cost near the cheap stage's while getting the precision of the expensive stage on the items that matter — exactly the funnel logic, now applied to search. The reranker is what turns "documents that mention the words" into "documents that actually answer the query," and it can afford to be smart precisely because retrieval already did the cheap work of narrowing the field.

★ Key points to remember
  • Search reuses the two-stage funnel: cheap retrieval (wide, recall) → expensive ranking/reranking (narrow, precision).
  • Difference from recommendation: the input is an explicit query, not a user profile.
  • Keyword retrieval (inverted index + BM25) nails exact terms; semantic retrieval (vector/ANN) matches meaning.
  • Hybrid retrieval runs both and combines — each covers the other's blind spot.
  • A cheap scorer touches everything (recall); an expensive scorer refines only the top-k (precision) — the same cost logic as recsys.

Practice Build the reasoning yourself

1A user searches "error code E-4501" on a support site and gets irrelevant results, even though a document with that exact code exists. The search uses only semantic (embedding) retrieval. What went wrong, and how do you fix it?
Show solution & reasoning

The failure is a classic weakness of pure semantic retrieval: it blurs exact identifiers. Embeddings capture meaning, but a specific code like "E-4501" carries its signal in the exact token, not in a fuzzy meaning — and the embedding of "E-4501" is nearly indistinguishable from "E-4502" or other codes, so semantic search returns documents that are topically about errors rather than the one with that precise code. Semantic search is great for "how do I fix a login problem" but poor for "E-4501," because there's no meaningful "meaning" to a serial number; there's just the exact string.

The fix is hybrid retrieval: add keyword retrieval (an inverted index with BM25) alongside the semantic retriever. Keyword search will match the exact token "E-4501" and surface the document containing it immediately, while the semantic retriever continues to handle meaning-based queries. Combining the two (via score fusion, e.g., reciprocal rank fusion) gives you both exact-match precision and semantic recall. This is exactly why production search — especially over technical, enterprise, or product content full of codes, SKUs, and names — almost always uses hybrid rather than semantic alone. The reasoning skill is diagnosing the symptom ("exact identifier not found despite existing" → semantic blind spot) and prescribing the specific complementary method (keyword) that covers it, rather than vaguely "improving the model." It's the same hybrid lesson from RAG (Track 7), applied in a search-design context, and recognizing that different query types (exact-token vs conceptual) need different retrieval methods is the core insight.

2How is a "learning to rank" model for search trained — what are the inputs and labels, and why is ranking framed differently from ordinary classification?
Show solution & reasoning

A learning-to-rank model scores each (query, document) pair for relevance, and the system sorts documents by that score. The inputs (features) combine query-document signals (keyword match strength/BM25, semantic similarity, title match), document signals (quality, freshness, popularity, authority), and sometimes user/context signals — the "rich features" the ranker affords because it runs on only a few hundred candidates. The labels come from relevance judgments: either human-rated relevance (graded: perfect/good/fair/bad) or, more commonly at scale, implicit feedback from logs — clicks, dwell time, whether the user stopped searching (a "successful" search) — as proxies for relevance.

Ranking is framed differently from ordinary classification because what matters is the relative order, not each item's absolute label. You don't ultimately care whether document A scores 0.8 or 0.6 in isolation; you care that the truly-relevant document is ranked above the less-relevant ones. This motivates pairwise and listwise training objectives: pairwise learns "document A should rank above document B for this query" from pairs, directly optimizing relative order; listwise optimizes a whole ranked list against ranking metrics like NDCG (which reward putting relevant items near the top). A plain pointwise classifier ("is this relevant: yes/no") ignores that the goal is ordering and that errors near the top matter far more than errors at the bottom. The reasoning skill is recognizing that the objective should match the goal: since search success is about the order of the top results, you train on relative comparisons and evaluate with rank-aware metrics — not accuracy on independent items. This mirrors the broader theme that choosing the right objective (Lesson 10.3's balanced recommender objective) is central to designing a system that does what you actually want.

3Both recommendation (10.3) and search (10.4) use the same two-stage funnel and the same offline/online split. Why does this pattern recur, and where else have you seen it?
Show solution & reasoning

The pattern recurs because it's the general solution to a general problem: find the best few items from a huge collection, fast, using an expensive-but-accurate scorer you can't afford to run on everything. Whenever you face that shape — large corpus, tight latency, a heavy relevance model — the answer is the same two moves. First, a cheap wide stage then an expensive narrow stage (the funnel): a light scorer covers the whole collection for recall, an accurate model refines the top few for precision, keeping total cost near the cheap stage. Second, push heavy work offline: precompute embeddings, build indexes, and train models ahead of time, so the per-request online path only assembles pre-built pieces within the latency budget. Together these reconcile "huge scale" with "accurate and fast," which no single-stage, all-online design can do.

You've now seen it in recommendation and search, and you'll see it again immediately in RAG at scale (Lesson 10.5) — retrieve cheaply from a pre-built vector index, then let an expensive LLM generate on just the retrieved context, with indexing done offline. It also underlies RAG generally (Track 7, retrieve-then-generate) and even the candidate-then-rerank structure inside retrieval itself. Recognizing this as one reusable pattern rather than three separate designs is a major leap in system-design maturity: it means that when you're handed "design X over a huge corpus with low latency," you don't start from a blank page — you reach for the funnel plus the offline/online split and adapt them to X's specifics. This kind of pattern-recognition, seeing the shared skeleton beneath superficially different problems (recommend / search / RAG), is exactly what lets experienced engineers design complex systems quickly and what interviewers are hoping to see. The specific systems are just instances; the transferable asset is the pattern.

★ Deep dive: query understanding, evaluation, and what makes search hard▸

Real search has an important stage before retrieval: query understanding. A raw query is messy — misspelled, ambiguous, underspecified — so production search first processes it: spelling correction ("recieve" → "receive"), query expansion (adding synonyms so "laptop" also matches "notebook"), intent classification (is "apple" the fruit, the company, or a product?), entity recognition (spotting that "iphone 15 case" contains a product and an accessory type), and tokenization/normalization. Better query understanding improves everything downstream, because retrieval and ranking can only work with what the query representation gives them. This stage is where a lot of search quality is won or lost, and it's easy to forget in a design because it's "before the interesting ML" — but a strong candidate mentions it, because garbage query-in means garbage results-out regardless of how good the ranker is.

Search is also distinguished by how it's evaluated, which shapes the design. Because what matters is the order of top results, search uses rank-aware metrics: NDCG (normalized discounted cumulative gain, which rewards putting highly-relevant results near the top and discounts relevance lower down), MRR (how high the first relevant result ranks), and precision/recall@k. These come from graded relevance judgments (human-labeled or from logs), and offline metric gains are validated with online A/B tests measuring real user behavior (click- through, query reformulation rate, task success), because — as with recommenders — offline relevance and real user satisfaction can diverge. What makes search genuinely hard is the combination: queries are ambiguous and vary enormously, relevance is subjective and query-dependent, the corpus is huge and constantly changing, latency budgets are tight, and the feedback signals (clicks) are biased by position and presentation (Lesson 10.3's feedback-loop problem reappears). A complete search design therefore spans query understanding, hybrid retrieval, learning-to-rank, re-ranking for diversity and freshness, the offline/online split, and an evaluation regime pairing rank-aware offline metrics with online A/B tests — all reasoned against the requirements. Seeing search as this full system, and knowing which stage each concern lives in, is what turns "design a search engine" from intimidating into a structured walk through a familiar architecture — the same confidence the whole track is building toward as the problems get larger, starting with RAG at scale next.
