Critiquing claims & evidence

A result is only as good as its evidence. The final reading skill is the most important: separating what a paper showed from what it claimed — and recognizing the handful of ways results routinely mislead.

Track 8 · Lesson 2 Method + code ~24 min Builds on: Lessons 0–1, Track 3

You can read a paper and reproduce it; now learn to judge it. Most published results are honest but imperfect, and a few are mirages produced by avoidable mistakes. Critical reading isn’t cynicism — it’s a checklist of questions, each of which the authors (and you) should be able to answer. We’ll demonstrate the most common trap in runnable code.

💡 The questions a strong result can answer

When a paper reports “our method beats the baseline,” a critical reader asks a fixed set of questions before believing it. None of these is an accusation; each is a check the evidence should pass.

🛠️ From scratch: how cherry-picking manufactures a result

The most common trap is comparing single runs of stochastic methods. Our cherry_picking.py makes two methods, A and B, with the identical true mean (0.80) — B’s “improvement” is a useless tweak that changes nothing. Run-to-run randomness alone produces a tempting story: 

# A and B have the SAME true mean; only the random seed differs per run
A = evaluate("A", seeds);  B = evaluate("B", other_seeds)

# DISHONEST: report B's luckiest run vs A's unluckiest
print(best(B), "vs", worst(A))          # "B beats A by +0.14" (!)

# HONEST: mean +/- 95% CI over 20 seeds, then a t-test
print(mean(A), "±", ci(A));  print(mean(B), "±", ci(B))
t = (mean(B) - mean(A)) / pooled_se     # |t| < 2  ->  not significant

The output is stark. Cherry-picked, “B beats A by \(+0.14\)” — a huge, entirely fictional gain. Evaluated honestly, \(A=0.799\pm0.013\) and \(B=0.806\pm0.014\): the confidence intervals overlap and the t-statistic is well under 2, so the difference is not statistically significant. The “improvement” was an artifact of selection, not a real effect.

The fix is the same every time. Run multiple seeds, report mean ± a confidence interval (or a significance test), and compare against a baseline you tuned as hard as your own method. If the gain is smaller than the noise, it isn’t a result yet. This connects directly to Track 9, where we make honest evaluation a habit.

🔑 Key results to remember

  • Critique by checklist: baseline strength, error bars, typical (not cherry-picked) examples, test-set hygiene, compute parity, claim scope.
  • Single-run comparisons of stochastic methods are unreliable; seed noise alone can fabricate large “improvements.”
  • Honest evaluation = multiple seeds, mean ± confidence interval, significance test, equally tuned baselines.
  • If the gain is smaller than the noise, it is not a result.
  • Critical reading is constructive: each red flag is a question the evidence should be able to answer.

🧩 Quiz

🧩 Check your understanding
Answer from memory, then read why.

Q1. In the demo, methods A and B have identical true means, yet a single-run comparison showed “B beats A by +0.14.” What produced that gap?

  • A real algorithmic improvement in B.
    By construction B’s tweak changes nothing; there is no real improvement to find.
  • A bug in the evaluation code.
    The code is correct; the gap comes from how the runs were selected, not from a bug.
  • Run-to-run (seed) randomness, exploited by picking B’s best run against A’s worst.
    Correct. With identical means but noisy runs, selecting favorable runs for one side manufactures a difference that has no underlying cause — the definition of cherry-picking.
  • B was trained on more data.
    Both use the same setup; only the random seed differs. No data advantage exists.

Q2. What is the standard remedy for unreliable single-run comparisons?

  • Run multiple seeds and report mean ± a confidence interval (and a significance test).
    Correct. Averaging over seeds estimates the true mean, and the interval/test tells you whether an observed gap exceeds the noise. This is the honest comparison the demo’s right panel shows.
  • Pick the seed that gives the cleanest result.
    That is the cherry-picking the lesson warns against — it inflates apparent performance.
  • Use a larger learning rate.
    Hyperparameters don’t address evaluation reliability; the issue is statistical, not optimization.
  • Only report training accuracy.
    Training accuracy says nothing about generalization or about whether a difference is real.

Q3. A paper shows its new method beating a baseline, but the baseline was used with default settings while the new method was carefully tuned. Which red flag is this?

  • Missing error bars.
    A different issue — this is about unequal effort, not about reporting variance.
  • Cherry-picked qualitative examples.
    That concerns showing best-case samples, not how the baseline was configured.
  • Claims exceeding evidence.
    Related but distinct; here the specific problem is the comparison itself being unfair.
  • A weak/untuned baseline — the new method may only be beating a strawman.
    Correct. If the baseline wasn’t tuned as hard as the new method, the “win” may reflect tuning effort, not a better method. Fair comparisons tune both sides equally.

Q4. Hyperparameters are selected by evaluating on the test set, then the same test set is reported as the final result. Why is this a problem?

  • It uses too much compute.
    Compute isn’t the issue; the issue is information leakage from the test set.
  • The reported number is optimistically biased — the test set was used to make choices, so it no longer measures generalization to unseen data.
    Correct. Tuning on the test set leaks it into model selection; the final number reflects fit to that specific set, not true generalization. A separate validation set must be used for tuning.
  • It makes training slower.
    Speed is irrelevant; the harm is statistical bias in the reported result.
  • It only matters for classification, not regression.
    Test-set leakage biases any task — classification, regression, or otherwise.

✍️ Practice

1. Run cherry_picking.py a few times with different seed ranges. Does the cherry-picked “winner” stay the same? What does that instability tell you?

Show solution

The cherry-picked “winner” flips around — sometimes B looks far ahead, sometimes A — precisely because there is no real difference, only noise. That instability is the tell: a genuine effect is robust to reseeding, while an artifact of selection is not. The honest mean ± CI, by contrast, stays close to 0.80 ± a small interval for both methods every time. The practical rule: if your headline result changes when you change the random seed, you don’t yet have a result — you have noise.

2. Suppose B genuinely improves the mean by 0.005 with per-run std 0.03. Roughly how many seeds would you need for the improvement to clear the noise (be detectable)?

Show solution

The standard error of the mean is \(\sigma/\sqrt{N}\). To detect a true difference of \(0.005\) you want the SE comfortably below it — say SE \(\approx 0.0017\) so the effect is ~3 SEs. Then \(\sigma/\sqrt{N}=0.0017\Rightarrow \sqrt{N}=0.03/0.0017\approx 18\Rightarrow N\approx 320\) seeds per method. The point: tiny true improvements require many runs to verify, which is why small reported gains without many seeds deserve skepticism — the experiment may simply lack the statistical power to support the claim.

3. Conceptual: you reproduce a paper and your honest, multi-seed result is clearly worse than the paper’s single reported number. List three innocent explanations before concluding the paper is wrong.

Show solution

Innocent explanations are common: (1) the paper reported a best-of-many or tuned run while you averaged seeds — a reporting-convention mismatch; (2) an undocumented detail (data preprocessing, a specific initialization, a longer training schedule) that you didn’t replicate; (3) a different data split or evaluation protocol. Only after ruling these out — ideally by contacting the authors or checking released code — is “the result doesn’t hold” a fair conclusion. Critical reading cuts both ways: be as rigorous about your own reproduction as about the paper’s claims.
