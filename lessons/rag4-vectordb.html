Track 7 · RAG & Vector Search

Vector databases & ANN search

⏱ 26 min Lesson 7.4 ▶ Open in Colab
Why this existsComparing a query to millions of vectors naively is too slow. Approximate nearest-neighbor indexes (HNSW, IVF) make retrieval near-instant by trading a little accuracy for a huge speedup — packaged in vector databases.
By the end, you can
  • Contrast exact and approximate NN search
  • Explain the recall-latency tradeoff
  • Describe HNSW and IVF intuitively
  • Know what a vector database provides

Concept Finding nearest neighbors fast, at scale

Retrieval means finding the chunk vectors closest to a query vector (Lesson 7.3). The obvious way — brute-force search — compares the query to every stored vector and keeps the closest. That's exact and simple, but it's $O(N)$: with 10 million chunks, every single query computes 10 million similarities, which is far too slow for a real application answering many queries per second. We need a smarter way to search.

The answer is approximate nearest neighbor (ANN) search: instead of checking all vectors, use a pre-built index that lets you check only a small, promising subset and still find (almost always) the true nearest neighbors. You trade a tiny bit of accuracy for an enormous speedup.

This accuracy-for-speed trade is the central knob of vector search. The metric is recall — the fraction of the true nearest neighbors the approximate search actually finds (Lesson 3.4's recall, applied here). You tune the index to hit, say, 98% recall at a latency you can afford. For the vast majority of RAG systems, missing the occasional borderline neighbor is harmless, and the 100× speedup is what makes interactive retrieval possible.

Concept How ANN indexes work: HNSW and IVF

Two index families dominate. HNSW (Hierarchical Navigable Small World) builds a multi-layer graph of vectors: the top layer is sparse with long-range links, lower layers progressively denser. A search starts at the top, takes big jumps toward the query's neighborhood, then descends layer by layer to zero in — like finding a house by taking the highway, then main roads, then the final street. It achieves roughly logarithmic search time over millions of vectors.

The other family, IVF (Inverted File index), first clusters the vectors (k-means, Lesson 3.10) and, at query time, searches only the few clusters nearest the query — skipping the rest. A vector database (like FAISS, Pinecone, Weaviate, Milvus, pgvector) packages one of these indexes with the practical machinery: storing vectors and metadata, updating the index as documents change, filtering by metadata (Lesson 7.2), and scaling across machines. You rarely implement ANN yourself — you choose and configure a vector DB.

Worked example Exact nearest-neighbor search (what ANN approximates)

Let's do exact brute-force search — the ground truth that ANN approximates — over 1,000 vectors. We build a query near a known vector (index 42) and confirm search finds it.

Exact search correctly found vector 42 as the top match (similarity 0.975 — the query was built right next to it), followed by the next-closest vectors. The whole search is one matrix-vector product (db @ query), computing all 1,000 similarities at once (vectorization, Lesson 2.2). That's fine for 1,000 vectors, but for 100 million it would be far too slow per query. An ANN index would return almost this same top-5 — usually identical, occasionally swapping a borderline neighbor — while examining only a tiny fraction of the vectors, making it orders of magnitude faster. This exact computation is the "correct answer" that ANN methods race to approximate; the art of vector search is getting 99% of this accuracy at 1% of the cost. Notice too that normalizing the vectors let us use a plain dot product for cosine similarity — exactly the optimization from Lesson 7.3, and what vector-DB hardware is tuned to do billions of times per second.

★ Key points to remember
  • Brute-force search is exact but $O(N)$ — too slow for millions of vectors per query.
  • ANN (approximate nearest neighbor) checks a smart subset: ~100× faster for ~99% accuracy.
  • The trade is measured by recall (fraction of true neighbors found); you tune recall vs latency.
  • HNSW (layered graph, "highway then streets") and IVF (cluster, search nearest clusters) are the main indexes.
  • A vector database packages the index with storage, metadata filtering, updates, and scaling — you configure it, not build it.

Practice Build the reasoning yourself

1Your vector search returns results in 5 ms but occasionally misses a clearly relevant chunk a human would expect. You suspect the ANN index. What single knob would you adjust, and what's the cost?
Show solution & reasoning

Increase the index's search effort — for HNSW this is typically a parameter like ef_search (how many candidates to explore); for IVF, the number of clusters to probe (nprobe). A higher value examines more vectors, raising recall (finding more of the true neighbors, fewer misses) — at the cost of higher latency (your 5 ms might become 15 ms).

This is the fundamental recall-vs-latency dial of ANN. You'd tune it to the minimum search effort that achieves your target recall (say 98%) on a test set of queries with known correct neighbors, accepting the latency that buys. If 5 ms is non-negotiable, you might instead improve recall by upgrading the index build parameters or the embedding quality. The key insight: most "the retriever missed an obvious chunk" issues are an under-tuned ANN recall setting, and there's a direct, measurable knob to trade latency for accuracy.

2You need to retrieve only chunks from documents the current user is authorized to see. How do metadata filtering and vector search combine to do this?
Show solution & reasoning

Store each chunk's access metadata (e.g., allowed roles or owner) alongside its vector (Lesson 7.2). At query time, the vector database does filtered search: it restricts the nearest-neighbor search to vectors whose metadata matches the user's permissions, so the results contain only authorized chunks. This is a core production feature — semantic relevance and access control together.

It matters enormously for security: without filtering, a RAG system could leak confidential documents to unauthorized users by retrieving them as "relevant" context. (How the DB implements efficient filtered ANN — pre-filter, post-filter, or filter-during-search — is a real engineering concern, since naive post-filtering can return too few results.) The broader point: a production vector DB isn't just fast search, it's governed search — respecting permissions, freshness (filter by date), and scope (filter by source) — which is exactly why you use a real vector database rather than a bare ANN library for anything serious.

3For a prototype with only 5,000 chunks, a colleague insists on setting up a distributed ANN index. Is that the right call? Why or why not?
Show solution & reasoning

No — it's premature over-engineering. At 5,000 vectors, even exact brute-force search is instant (a single small matrix-vector product, as in the worked example), so an ANN index — let alone a distributed one — adds complexity, infrastructure, and approximation error for zero benefit. ANN's value appears at scale (hundreds of thousands to millions of vectors); below that, exact search is simpler and better (no recall loss).

The right call: use exact search (or a simple in-memory index like FAISS flat) for the prototype, and switch to an ANN index only when the corpus grows large enough that latency becomes a real problem — and measure to confirm. This mirrors a recurring engineering lesson: match the complexity of your solution to the actual scale of your problem, and don't pay the costs of scale (approximation, distributed systems) until you have the scale that justifies them. Start simple, measure, and add machinery only when the numbers demand it.

★ Deep dive: the recall–latency frontier, and the vector-search landscape▸

The defining artifact of vector search is the recall–latency curve: for a given index, plotting achievable recall against query latency traces a frontier, and tuning parameters (HNSW's ef, IVF's nprobe) moves you along it. Better index designs shift the whole curve up-and-left (more recall per millisecond), which is what the field competes on. There's also a build-time vs query-time tradeoff: HNSW gives excellent query speed and recall but is slower to build and memory-heavy (it stores the graph); IVF is faster to build and more memory-efficient but typically needs more probing for the same recall. Newer methods like product quantization (PQ) compress vectors to a few bytes each — trading a little accuracy for massive memory savings, enabling billion-scale indexes on modest hardware — and are often combined with IVF (IVF-PQ) for huge corpora. Understanding that every vector index is a point on a multi-way tradeoff (recall, latency, memory, build time, update cost) is what lets you choose and tune one intelligently rather than cargo-culting a default.

The vector-search landscape has matured rapidly because RAG made it essential. FAISS (from Meta) is the foundational open-source library with many index types. Dedicated vector databases — Pinecone (managed), Weaviate, Milvus, Qdrant — add the production layer (persistence, filtering, distribution, hybrid search). And increasingly, general databases have grown vector support: pgvector brings ANN to PostgreSQL, Elasticsearch and others added vector fields — so you can often add vector search to infrastructure you already run, avoiding a separate system. The right choice depends on scale, existing stack, and whether you want managed convenience or self-hosted control; for many applications, pgvector or a single-node FAISS index is plenty, and a dedicated distributed vector DB is warranted only at large scale. A subtle but important operational concern is updates: documents change, and re-indexing or incrementally updating an ANN structure (especially HNSW graphs) has its own costs and consistency challenges — a real factor for systems with frequently-changing knowledge (which is much of RAG's value proposition). The throughline: vector search is a deep, well-engineered field, but for the RAG practitioner the essential mental model is simple — it's nearest-neighbor search made fast by approximation, configured through a recall–latency dial, and packaged in a database that also handles storage, filtering, and scale. Get the retrieval fast and accurate enough, and you've built the engine that makes RAG responsive; the remaining lessons make its results better (reranking) and prove they're good (evaluation).

Fast retrieval gives you candidate chunks — but the closest vector isn't always the most useful chunk. Next, in 7.5 Retrieval, reranking & hybrid search, we add a precision layer that reorders candidates and blends semantic with keyword search.
