ReAct: reasoning + acting

⏱ 24 min Lesson 9.3 ▶ Open in Colab
Why this existsReAct interleaves Reasoning and Acting: the agent writes a Thought, takes an Action, reads an Observation, and loops. Thinking before acting makes tool use deliberate and the trace debuggable — chain-of-thought extended into a loop.
By the end, you can
  • Explain the Thought → Action → Observation loop
  • Say why reasoning improves tool choice
  • Connect ReAct to chain-of-thought
  • Use the reason-trace to debug an agent

Concept Interleaving thinking and doing

An agent has a loop (Lesson 9.1) and tools (Lesson 9.2) — but how should it use them across a multi-step problem? The dominant pattern is ReAct (Reasoning + Acting): the model alternates between Thought (reasoning about what to do next), Action (calling a tool), and Observation (reading the tool's result), looping until it has enough to answer. It's chain-of-thought reasoning (Lesson 7.5) interleaved with real tool calls — so the model both thinks and acts, with each grounding the other.

Why interleave, rather than reason all at once or act blindly? Two failure modes it avoids. Pure reasoning without acting (chain-of-thought alone) can't get external facts and may confidently reason from wrong assumptions. Pure acting without reasoning (just calling tools) has no plan and can't synthesize results. ReAct combines them: the Thought decides a sensible next Action, and the Observation grounds the next Thought in reality — so the model corrects course based on what it actually finds, rather than charging ahead on assumptions. This grounding-in-feedback is what makes ReAct reliable enough to be the default agent pattern.

Worked example A ReAct trace, step by step

Let's solve "What's the combined population of France and Germany?" — a question needing two lookups and a calculation — with an explicit ReAct trace. The model generates the Thoughts and Actions; your code runs the Actions and supplies the Observations.

Watch the interleaving. Each Thought reasons about what's needed and decides the next Action; each Observation (the tool's real result) feeds the next Thought. The model didn't try to recall the populations from memory (which it might get wrong, Lesson 7.1) — it looked them up, grounding its answer in retrieved facts. And it didn't blindly search everything at once — it reasoned about what to do at each step. After two lookups, the final Thought recognized it had enough information and produced the answer. This trace is the agent loop from Lesson 9.1, made concrete: Thought = think, Action = act, Observation = observe, repeat until done. In a real system, the entire growing trace (all previous Thoughts, Actions, and Observations) is fed back into the model's context each step, so it always reasons with the full history of what it's tried and learned — which is how it avoids repeating itself and builds toward an answer.

★ Key points to remember
  • ReAct interleaves Thought (reason) → Action (call a tool) → Observation (read result), looping until it can answer.
  • It's chain-of-thought + tools: reasoning guides each action; observations ground each thought.
  • It avoids reasoning-from-wrong-assumptions (no feedback) and blind tool-calling (no plan).
  • The full growing trace is fed back each step, so the model reasons with everything it's learned.
  • ReAct is the default agent pattern because grounding each step in real results makes it robust.

Practice Build the reasoning yourself

1Write a plausible ReAct trace for: "Is it warmer in Paris or Cairo right now?" Assume a get_weather(city) tool.
Show solution & reasoning

Thought: I need both cities' temperatures to compare. Let me get Paris first. → Action: get_weather("Paris") → Observation: "12°C". → Thought: Now Cairo. → Action: get_weather("Cairo") → Observation: "28°C". → Thought: 28 > 12, so Cairo is warmer. → Answer: Cairo is currently warmer (28°C vs Paris's 12°C).

The structure is the same as the worked example: a Thought decides each lookup, the Observations supply the real data, and a final Thought compares them and triggers the answer. Note the model couldn't answer this from memory at all — current weather is live data outside its training (Lesson 7.1) — so the tools aren't optional here; they're the only way to answer correctly. This is the essence of ReAct: reason about what information you need, act to get it, observe, and reason again until you can answer — turning a question the model can't answer alone into one it can.

2In a ReAct loop, an Action returns "Observation: Error — no results found." How does the ReAct structure let the agent recover, and what might its next Thought be?
Show solution & reasoning

Because the error comes back as an Observation, the model's next Thought can react to it — exactly like any other result. A sensible next Thought: "The search for 'X' found nothing; let me try different keywords / a broader query / a different tool." Then it issues a new Action. The error didn't crash the loop; it became information the model reasons about.

This is ReAct's built-in robustness: every action's outcome — success or failure — is an observation that informs the next decision, so the model can route around failures (reformulate the query, switch tools, or, if it keeps failing, tell the user it couldn't find the answer). This is why feeding errors back as observations (Lesson 9.2) is so important — it's what gives the agent the chance to recover. Pair it with retry/step limits (Lesson 9.7) so a persistently-failing action doesn't loop forever, and you have an agent that handles the messy, error-prone real world gracefully rather than falling over at the first hiccup.

3How is ReAct related to chain-of-thought prompting (Lesson 7.5), and what does it add?
Show solution & reasoning

ReAct is essentially chain-of-thought extended with actions. Chain-of-thought (Lesson 7.5) has the model write out its reasoning steps before answering, which improves multi-step reasoning by letting it "compute on the page." ReAct keeps that reasoning (the Thoughts) but interleaves real tool calls (Actions) and their results (Observations) between the reasoning steps.

What it adds is grounding in external reality. Pure chain-of-thought reasons entirely within the model's own knowledge — so if it needs a fact it doesn't have, or computes something it gets wrong, the reasoning proceeds on a false foundation. ReAct lets the reasoning reach out: fetch the fact, run the calculation, check the result — so each reasoning step can be grounded in something real rather than the model's possibly-faulty internal state. In short: chain-of-thought makes the model reason; ReAct makes it reason and verify against the world. That combination — think, then check, then think again — is why ReAct is the foundation of practical agents, and it builds directly on the chain-of-thought idea you already know.

★ Deep dive: ReAct variants, the reasoning-action synergy, and limits▸

ReAct (from a 2022 paper) crystallized why combining reasoning and acting beats either alone, and it spawned a family of refinements. The core insight is a synergy: reasoning helps acting (the model plans which tool to use and how to interpret results), and acting helps reasoning (real observations correct the model's assumptions and reduce hallucination, because it's reading facts rather than recalling them). Empirically, ReAct outperformed both chain-of-thought alone (which hallucinated facts) and act-only agents (which lacked a plan) on knowledge-intensive and decision-making tasks. Variants build on it: Reflexion adds a reflection step where the agent reviews its own trace, diagnoses failures, and tries again with that self-critique (Lesson 9.4); Plan-and-Solve and related methods front-load more explicit planning before acting (Lesson 9.4); and tree-search methods (like Tree of Thoughts) explore multiple reasoning/action paths and pick the best. But ReAct's simple interleaved loop remains the workhorse because it's robust and easy to implement, and most production agents are some flavor of it.

It's worth understanding ReAct's limits, because they shape when it struggles. The whole trace lives in the context window, so for long tasks the trace grows until it strains or exceeds the context (motivating memory and summarization, Lesson 9.5). The model is greedy — it commits to each action without exploring alternatives, so a wrong early action can derail the trace (motivating planning and reflection to catch and correct mistakes, Lesson 9.4). And errors still compound over many steps (Lesson 9.1's deep dive): each Thought→Action→Observation cycle has some failure probability, and long traces multiply them, so ReAct agents become less reliable as tasks lengthen — which is why bounding the number of steps and the task scope (Lesson 9.7) matters. The frontier is making the loop more reliable over longer horizons: better planning so the agent doesn't wander, reflection so it catches its own errors, memory so it doesn't lose context, and verification so it checks its work. Each of those is a lesson ahead. But the fundamental pattern you've now seen — reason, act, observe, repeat, grounding every step in real feedback — is the beating heart of how an LLM accomplishes things in the world, and everything else in this track makes that heart beat more reliably. The next lesson tackles ReAct's biggest weakness directly: a purely reactive agent can wander, so we add planning to give it direction and reflection to let it catch its own mistakes.
