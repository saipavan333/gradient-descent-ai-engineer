Honest evaluation & pitfalls

The fastest way to fool yourself in ML is to let the test set leak into your decisions. We’ll watch a “state-of-the-art” result appear out of pure noise — and learn the simple discipline that prevents it.

Track 9 · Lesson 2 Method + code ~24 min Builds on: Lessons 0–1, Track 3

You can design a clean experiment and run honest ablations, and still be wrong — if your evaluation is leaky. The most common and dangerous mistake is letting the test set influence your choices. We make the danger concrete by manufacturing a fake result from noise, then fix it with the train/validation/test discipline.

💡 The one rule: the test set is touched once

Split data into three roles. The training set fits model parameters. The validation set chooses everything you tune by hand — hyperparameters, architecture, when to stop. The test set is used exactly once, at the very end, to estimate generalization. The moment you use the test set to make a choice, it stops measuring generalization and starts measuring how well you fit that particular set.

🛠️ From scratch: a “result” conjured from noise

Here is the danger in code. In evaluation_pitfalls.py we generate \(K\) random models — each just guesses labels — and keep the one with the best test accuracy. Because we selected on the test set, its reported accuracy climbs far above the true value of 0.50, purely from selection noise: 

best = -1
for i in range(K):                       # try K random 'models'
    preds = random_guesses(n_test)
    acc = (preds == test_labels).mean()
    if acc > best:                       # keep the best ON THE TEST SET
        best, best_model = acc, i
# 'best' is optimistically biased: we used the test set to choose
This is not a contrived edge case. Every hyperparameter you pick by checking the test set, every “let me just try one more architecture and see if test improves,” every benchmark leaderboard with thousands of submissions — all are versions of this same selection. The defenses: tune only on validation; keep a final test set you touch once; use cross-validation when data is scarce; and treat any gain that vanishes on truly fresh data as the noise it is.

🔑 Key results to remember

  • Three roles: train fits parameters, validation selects choices, test estimates generalization once.
  • Adaptive overfitting: selecting among many models by test accuracy inflates that accuracy — even for random models.
  • The inflation grows with the number of things you try; a fresh test set reveals the true (lower) performance.
  • Defenses: tune on validation only, hold out a final test set, use cross-validation, and re-check surprising gains on new data.
  • Data leakage (any test info entering training or selection) makes results optimistic and irreproducible.

Q1. In the demo, all models are random (true accuracy 0.50), yet the best-on-test accuracy reached 0.62. What caused the inflation?

  • The random models secretly learned.
    They make pure random guesses; there is nothing to learn. The inflation isn’t real skill.
  • Selecting the maximum over many noisy test scores picks up favorable noise — the test set was used to choose.
    Correct. The max of many noisy 0.50-centered scores is above 0.50 by chance; using the test set to select that maximum bakes the noise into the reported number.
  • The test set was too large.
    A larger test set reduces per-model noise; the inflation comes from selection across many models, not test size.
  • The learning rate was wrong.
    There’s no training here — just random guessing — so learning rate is irrelevant.

Q2. What is the validation set for?

  • Fitting the model’s weights.
    That’s the training set’s job; validation isn’t used for parameter fitting.
  • Reporting the final headline number.
    The final estimate comes from the test set, used once; validation is for choices made along the way.
  • Choosing hyperparameters, architecture, and when to stop — the decisions you make by hand.
    Correct. Validation absorbs all the selection so the test set stays untouched and can give an honest generalization estimate at the end.
  • Adding more training data.
    Validation is held out from training on purpose; folding it back in defeats its role (except as a final retrain after selection).

Q3. The inflated, test-selected model is re-evaluated on a fresh test set and scores 0.50. What does this confirm?

  • The earlier 0.62 was an artifact of selecting on the test set, not real performance.
    Correct. On data that played no role in selection, the true performance (0.50) reappears — proving the inflation was selection bias, not skill. A fresh hold-out is the antidote.
  • The fresh test set is broken.
    It’s working correctly — it’s revealing the truth the contaminated set hid.
  • The model degraded over time.
    Nothing changed about the model; only the evaluation set changed from contaminated to clean.
  • Random models are actually 0.62 accurate.
    They are 0.50 by construction; the 0.62 was selection noise, as the fresh test confirms.

Q4. Which practice is a form of test-set leakage?

  • Choosing the learning rate using the validation set.
    That’s the correct use of validation — no leakage.
  • Normalizing features using training-set statistics.
    Using only training statistics is the right way to normalize; no test information leaks.
  • Reporting test accuracy once, at the end.
    That’s exactly the honest protocol — the opposite of leakage.
  • Repeatedly tweaking the model and re-checking test accuracy until it improves.
    Correct. Each re-check uses the test set to guide a decision, so the test set is being optimized against — adaptive overfitting. Selection must happen on validation, not test.

✍️ Practice

1. Run evaluation_pitfalls.py. How does the inflated accuracy change as \(K\) (number of models tried) grows, and why?

Show solution

The inflation grows with \(K\): more models tried means a higher maximum among their noisy test scores, so the “best” climbs further above 0.50 (here from ~0.50 at \(K=1\) to ~0.62 at \(K=2000\)). It’s the statistics of maxima — the expected maximum of \(K\) samples increases with \(K\). The practical implication is sobering: the more hyperparameter settings, architectures, or random seeds you try while watching the test set, the more optimistic your final number becomes. This is the mechanism behind leaderboard overfitting, and why a held-out final test set (touched once) is essential.

2. You have only 200 labeled examples — too few to carve out a big validation and test set. What evaluation strategy gives an honest estimate?

Show solution

Use k-fold cross-validation: split the data into \(k\) folds, train on \(k-1\) and evaluate on the held-out fold, rotating so every example is tested exactly once, then average. This uses all data for both training and evaluation without ever testing on data the model trained on. For honest model selection with small data, nest it: an inner cross-validation chooses hyperparameters, an outer one estimates performance — so selection never sees the data used for the final estimate. It’s more compute, but it prevents the optimistic bias that a single tiny test set would suffer from selection noise.

3. Conceptual: a famous benchmark has had thousands of papers evaluate on its fixed test set over years. Why might reported progress on it be partly illusory?

Show solution

The community as a whole is doing the demo’s selection across thousands of “models” — each paper tunes against, and reports, the same fixed test set, and the field keeps the methods that scored best on it. That collective selection adaptively overfits the benchmark, so some of the measured “progress” reflects fitting the specific test set rather than genuine generalization. The check is exactly the demo’s green line: when researchers build a fresh test set for the same task and re-evaluate top models, scores typically drop somewhat — confirming part of the gain was benchmark-specific. This is why new, held-out test sets and diverse evaluations matter for tracking real progress.
