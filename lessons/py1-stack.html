Why Python, and the scientific stack

A note before Track 2

This track is about the everyday tools. You don't need prior coding experience — we'll define every term, including the plain programming words. If you've never written Python, that's fine: read the code as "recipes," and the companion notebooks let you run each one and watch what happens. Going slowly here pays off for the rest of the course.

Concept Why Python, and what "the stack" means

First, two words. Python is a programming language — a way of writing instructions a computer can follow, designed to read almost like English. It's the language nearly all of machine learning is written in. But Python has a reputation for being slow at heavy number-crunching, so how does it run billion-number models? The answer is the scientific stack: a set of add-on toolkits that do the heavy lifting in fast, pre-compiled code (written in lower-level languages like C and C++ that run close to the hardware) while you write simple Python on top. You get Python's readability with C's speed.

Each toolkit is called a library — a bundle of ready-made code someone else wrote that you can pull into your own program with an import statement (a line like import numpy that says "make this library's tools available here"). Knowing which library does what, and how they stack on top of each other, is the first practical skill of an ML engineer.

Reading the stack from the bottom up:

  • NumPy — fast lists and grids of numbers (it calls them "arrays") and the math on them. The foundation everything else stands on (you'll spend the next two lessons here).
  • pandas — labeled tables (rows and named columns) for loading and tidying messy, real-world data. Built on top of NumPy.
  • Matplotlib — drawing plots and charts, so you can actually see your data.
  • scikit-learn — classical machine-learning models (the regressions, trees, and clusterings of Track 3) with one clean, consistent way to use them.
  • PyTorch / TensorFlow — deep learning (neural networks), with automatic gradients and the ability to run on a GPU. Tracks 4–6 live here.

Concept The workflow is a loop, not a straight line

Beginners imagine machine learning as "load data → train model → done." In reality it's a loop: you explore the data, clean it, build a model, check how it did, and what you learn sends you right back to explore and clean some more. Most of the real work — and most of this track — is the unglamorous left side of that loop.

Concept Notebooks vs. scripts, and keeping work reproducible

You'll write code in two forms. A notebook (a Jupyter notebook, with the file extension .ipynb) runs in small chunks called cells, one at a time, mixing code, its output, and your notes on one page — perfect for exploring, where you want to poke at data and see results instantly (it's why this course ships runnable notebooks). A script (a plain .py file) runs straight through from top to bottom, and is what you use for anything repeatable — training pipelines, production jobs, anything a person or machine must run reliably. A good rule: explore in a notebook, then move the keepers into scripts.

!
Reproducibility starts on day one

"It works on my machine" is the original sin of data work. Two habits prevent it. First, pin your dependencies (the libraries your code needs) by recording their exact version numbers in a file, so anyone can install the same ones. Second, set a random seed — a fixed starting number for the random-number generator — so that code using randomness produces the same result every run. ("Reproducible" means someone else, or future-you, can re-run your work and get identical numbers.) These habits cost minutes now and save days later.

Worked example The whole stack in fifteen lines

A miniature touch of every layer: NumPy makes numbers, pandas organizes them into a table, and Matplotlib (shown but not run here) would plot them. It's deliberately tiny — the point is to see the pieces click together. Read the comments (the text after #, which Python ignores) — they explain each line.

import numpy as np       # load NumPy, and nickname it "np" so we can type less
import pandas as pd      # load pandas, nicknamed "pd"

# NumPy: make some data, with reproducible randomness via a fixed seed (42)
rng = np.random.default_rng(42)            # a random-number generator seeded at 42
heights = rng.normal(170, 8, size=5).round(1)   # 5 random heights, avg 170, spread 8
weights = rng.normal(70, 10, size=5).round(1)   # 5 random weights, avg 70,  spread 10

# pandas: organize the two lists into a labeled table (a "DataFrame")
people = pd.DataFrame({"height_cm": heights, "weight_kg": weights})
print(people)
print("\nsummary:\n", people.describe().round(1))   # .describe() = quick stats per column
Output (identical every run, thanks to the seed) height_cm weight_kg 0 172.4 57.0 1 161.7 71.3 2 176.0 66.8 3 177.5 69.8 4 154.4 61.5 summary: height_cm weight_kg count 5.0 5.0 mean 168.4 65.3 std 10.0 6.0 ...

Those few lines touched NumPy (random arrays), pandas (a table and its .describe() summary), and reproducibility (the seeded generator, which is why your output will match exactly). The next lessons unpack each layer; scikit-learn and PyTorch arrive in later tracks. You don't need to memorize any of this yet — you need the mental map of which tool to reach for.

★ Key points to remember
  • Python is the language of ML because the scientific stack does the heavy number-crunching in fast compiled code underneath.
  • The layers: NumPy (arrays) → pandas (tables) → Matplotlib (plots), scikit-learn (classic ML), PyTorch/TensorFlow (deep learning).
  • The workflow is a loop (collect → explore → clean → train → evaluate → repeat); data prep dominates.
  • Notebooks for exploring, scripts for anything repeatable.
  • Pin dependency versions and set random seeds for reproducibility from the very start.

1For each task, name the library you'd reach for: (a) load a 2 GB spreadsheet and filter rows; (b) train a random forest; (c) make a histogram; (d) fine-tune a neural network; (e) multiply two large grids of numbers fast.
Show solution & reasoning

(a) pandas — labeled tables, file loading, filtering. (b) scikit-learn — classical models like random forests. (c) Matplotlib — plotting. (d) PyTorch (or TensorFlow) — neural networks with autograd/GPU. (e) NumPy — fast array/matrix math.

The reasoning: each library owns a layer of the stack. Knowing the map means you never waste time forcing one tool to do another's job (e.g. looping over a pandas table row by row when a vectorized NumPy operation would be 100× faster — a Lesson 2.2 lesson).

2A colleague sends you a notebook that "worked yesterday" but now crashes on your machine with a version error. What two habits would have prevented this, and how?
Show solution & reasoning

(1) Pin dependencies — a file (commonly requirements.txt) recording the exact version of every library. Then you install the same versions they used, so the tools behave the same. (2) Use a fresh, isolated environment per project (a "virtual environment"), so projects don't share — and accidentally break — each other's libraries.

The reasoning: most "works on my machine" failures are version mismatches. Capturing the exact environment turns an invisible, fragile setup into an explicit, shareable one. This is the seed of MLOps reproducibility (Track 9), where the same idea scales up to Docker images and locked pipelines.

3True or false, and explain: "Once I have a model, I'm done — the workflow ends at training."
Show solution & reasoning

False. The workflow is a loop. After training you evaluate, and the results almost always send you back: maybe the data had a leak, a feature was missing, the metric was wrong, or performance drops on new data. You re-explore, re-clean, re-train. And once deployed, models drift as the world changes (Track 9), forcing another lap.

The reasoning: treating ML as a one-shot pipeline is the most common beginner misconception. Internalizing the loop — and that data work dominates it — sets realistic expectations, which is exactly why this Python/data track comes before any modeling.

★ Deep dive: why NumPy is fast, and what "vectorization" really buys you▸

A plain Python list (Python's built-in "bag of items") is a collection of pointers to separate objects scattered around memory, each carrying type information and overhead. Looping over it, Python handles one item at a time, interpreting each step — flexible, but slow. A NumPy array is different: a single, unbroken block of raw numbers, all the same type (the dtype), laid out end to end. That layout unlocks three speedups at once.

(1) No per-element Python overhead. One array operation (a + b) makes a single call into compiled C that loops internally — the slow Python interpreter is out of the hot path. (2) Cache friendliness. Numbers sitting next to each other in memory let the processor stream them efficiently instead of chasing scattered pointers. (3) SIMD. Modern processors can apply one instruction to several numbers at the same time, and compiled array code exploits this; on a GPU the same idea scales to thousands of parallel lanes — which is why deep learning needs GPUs. This is what people mean by "vectorization": expressing computation as operations on whole arrays so the fast, parallel machinery underneath can do its job. You'll feel the difference firsthand next lesson, where a vectorized operation beats a hand-written Python loop by 10–100×. It also explains a recurring theme of the whole field: the art is often in removing loops by reshaping work into array (and matrix) operations — the same matrix multiplications from Track 1, now running on hardware built to fly through them.
