Capstone C: a live reinforcement-learning agent

Make learning visible. Deploy the Q-learning agent from Track 7 as a web app where visitors click “Train” and watch the value heatmap and policy arrows sharpen in real time — the most dynamic capstone, and a perfect teaching tool.

Track 10 · Capstone C Project + deployment ~35 min Builds on: Track 7

The first two capstones show a model’s output. This one shows a model learning. RL GridWorld — Live (in capstones/rl-gridworld-live/) keeps a Q-learning agent on the server and lets the browser drive its training, visualizing the value function and policy after each batch of episodes. It turns the abstract idea of “learning from reward” into something you can watch happen.

💡 What you’re building

A web app with a persistent agent: the Q-table lives in server memory, so each “Train 200 episodes” click continues training the same agent and returns the updated grid. The browser renders cells colored by learned value and labeled with the current best action — so convergence is literally visible.

🛠️ The agent: Q-learning that trains on demand

The core is the Track 7 agent, refactored so training is incremental and the state is JSON-serializable. The temporal-difference update is unchanged — only the packaging is new: 

def train(self, episodes=200, eps=0.15):
    for _ in range(episodes):
        s = random_nonterminal_state()              # exploring start
        while not terminal(s):
            a = epsilon_greedy(self.Q, s, eps)       # explore or exploit
            ns, r = env.step(s, a)                    # slippery dynamics
            best = max(self.Q[ns, b] for b in actions)
            self.Q[s, a] += alpha * (r + gamma*best - self.Q[s, a])   # TD update
            s = ns
        self.episodes += 1

State changes everything. The name generator and the double-descent lab are stateless — each request is independent. This agent is stateful: the Q-table accumulates across requests, so the server must keep it in memory and serve a single worker. That’s a real architectural distinction worth understanding — and the README notes the fix for scaling (move the Q-table to a shared store like Redis so multiple workers share one agent).

📐 Tests that prove it learns, and deployment

The test suite asserts the agent actually learns — not just that the code runs. test_learning_improves_return trains for 1500 episodes and checks the greedy return rose substantially; test_learned_policy_reaches_goal checks it reaches near-optimal from the start. If a refactor ever broke the learning rule, CI would catch it. Deployment uses the same pipeline as the other capstones, with one caveat baked into the start command: run a single Gunicorn worker, because the agent is in-memory. Push to GitHub, then deploy on Render (start: gunicorn app:app --bind 0.0.0.0:$PORT) or as a Docker Space.

🔑 Key takeaways

  • Visualizing learning (not just output) is uniquely instructive — and makes a memorable demo.
  • Incremental training over HTTP: a persistent server-side agent trains a batch per request and returns its state.
  • Stateful vs stateless services: this app holds the Q-table in memory, so it needs a single worker (or a shared store to scale).
  • Tests assert learning (return improves, policy reaches the goal), so CI guards the algorithm, not just the code.
  • Same deploy pipeline — the only change is the single-worker constraint from statefulness.

Q1. Why must this app run as a single server worker, unlike the name generator?

  • Because Q-learning can’t be parallelized at all.
    Q-learning can be parallelized in general; the constraint here is about where this app stores its state.
  • Because Flask only supports one worker.
    Flask/Gunicorn support many workers; the limitation comes from the app’s design, not the framework.
  • The agent’s Q-table is held in memory, so multiple workers would each have a different, partially-trained agent.
    Correct. With several workers, requests would hit different in-memory agents, and the displayed learning would be inconsistent. One worker keeps a single coherent agent; scaling needs a shared store (e.g. Redis).
  • Because the grid is too large.
    The grid is tiny; size isn’t the issue. State location is.

Q2. What makes this capstone “stateful” where the others are “stateless”?

  • The Q-table persists across requests and accumulates learning; each request depends on previous ones.
    Correct. Training is incremental — the result of one request (an updated Q-table) carries into the next. The name generator and lab treat every request independently.
  • It uses more CSS.
    Styling has nothing to do with statefulness.
  • It returns JSON instead of images.
    Response format is unrelated to state; the lab also returns JSON (with an image inside).
  • It has more endpoints.
    Endpoint count isn’t what makes it stateful — the persistent in-memory agent is.

Q3. test_learning_improves_return trains 1500 episodes and asserts the return rose. Why test this rather than just that the code runs?

  • Because tests must always train models.
    Not all tests train; this one does because the claim being guarded is about learning.
  • To measure server latency.
    It checks learning quality, not speed.
  • To confirm the grid renders.
    Rendering is a front-end concern; this test checks the algorithm’s behavior.
  • A refactor could keep the code running while silently breaking the learning rule; asserting that return improves catches that.
    Correct. “It runs” is a weak guarantee for an RL agent — a sign error in the TD update would still run but never learn. Asserting measurable improvement guards the algorithm itself, the property users actually care about.

Q4. In the live grid, what do the cell colors and arrows represent?

  • Colors are random; arrows show the mouse.
    Both encode the agent’s learned knowledge, not decoration or input.
  • Color = a cell’s learned state value; arrow = the current greedy (best) action there.
    Correct. The color encodes \(V(s)=\max_a Q(s,a)\) (greener = higher value), and the arrow is \(\arg\max_a Q(s,a)\) — so you can literally see the value function and policy the agent has learned, and watch them sharpen with training.
  • Color = training speed; arrow = wind direction.
    Neither; the visualization shows values and the greedy policy.
  • They show the test-set accuracy.
    This is RL, not supervised learning — there’s no test-set accuracy; the metric is expected return.

✍️ Practice & extensions

1. Run the app and use “Auto-train.” Watch which cells learn good values first, and explain the order.

Show solution

Values propagate backward from the goal. Cells next to the goal learn high values first, because they directly receive the +1 reward; cells further away only gain value once their neighbors have learned theirs, so reward information has to “flow” outward step by step over many episodes. This is the temporal-difference bootstrap in action — each state’s value is learned from its neighbor’s current estimate (Track 7). Watching the green region grow outward from the goal makes the credit-assignment idea — how a delayed reward reaches the actions that earned it — concrete and visible.

2. Add a “slippiness” control that changes how often moves slip sideways, and retrain. How does the optimal policy change?

Show solution

With more slip, the environment is riskier, and the optimal policy becomes more cautious: it tends to route further from the pit, even if that path is slightly longer, because a slip near the pit is costly. With near-zero slip the agent takes the most direct route, including squeezing past the pit. This is a vivid demonstration that the optimal policy depends on the environment’s stochasticity, not just the reward layout — and that an agent trained in one slip regime can be suboptimal in another. Implementing it means exposing the env’s slip parameter through a slider and resetting the agent on change.

3. Conceptual: to scale this to many simultaneous users sharing one agent, the README suggests moving the Q-table to Redis. Why does that solve the multi-worker problem?

Show solution

The multi-worker problem is that each Gunicorn worker has its own copy of the in-memory Q-table, so training requests routed to different workers update different agents. Moving the Q-table to an external store (Redis) makes it shared: every worker reads and writes the same table, so they collectively train one agent regardless of which worker handles a request. This is the standard pattern for stateful services — keep the process stateless and put the shared state in a database or cache — which then lets you run many workers (and even many machines) behind a load balancer. It’s the same train-offline/serve-online and shared-artifact thinking that recurs across all three capstones, applied to live state.
