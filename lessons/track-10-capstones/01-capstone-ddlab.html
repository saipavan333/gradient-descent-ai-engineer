Capstone B: reproduce a classic, live

Turn a published result into an interactive web app. The double-descent reproduction from Track 8 becomes a live lab where anyone can drag sliders and watch the interpolation spike appear and vanish — a deployable scientific artifact.

Track 10 · Capstone B Project + deployment ~35 min Builds on: Tracks 3, 8, 9

Reproducing a result (Track 8) proves you understand it. Packaging that reproduction so others can interact with it — and so a test suite re-verifies the science on every commit — is a different, equally valuable skill. This capstone, in capstones/double-descent-lab/, makes the double-descent phenomenon explorable in a browser and deployable to the web.

💡 What you’re building

A web app with two sliders — label noise and ridge regularization — that recompute the double-descent curve on demand. The server runs the exact NumPy experiment from Track 8, renders the curve as an image, and streams it back; the browser shows it updating live. It’s a research result you can poke at.

📐 The result it reproduces

Classical theory predicts test error is U-shaped in model size. Double descent shows that past the interpolation threshold (parameters \(=\) data points), test error spikes and then descends again. The app makes the mechanism tangible: raise the ridge slider and the spike shrinks, because regularization prevents the wild exact-interpolation fit that causes it.

🛠️ The science as a tested module

The experiment is one function, parameterized so the UI can call it with any settings, from ddlab/experiment.py: 

def double_descent_curve(n=40, noise=0.5, ridge=1e-6, d=8, repeats=20):
    Xtr, ytr, w = make_data(n, d, noise)             # synthetic nonlinear target
    Xte, yte = fresh_test_set(w)
    ratios, errs = [], []
    for P in feature_counts_crossing(n):             # sweep model size across P = n
        e = mean(test_error(..., P, ridge) for _ in range(repeats))
        ratios.append(P / n); errs.append(e)
    return ratios, errs

What makes this a professional artifact rather than a script is the test suite. It doesn’t just check the code runs — it encodes the scientific claims and fails CI if they ever break: 

def test_curve_has_peak_near_interpolation_threshold():
    ratios, errs = double_descent_curve()
    peak = argmax(errs)
    assert 0.8 <= ratios[peak] <= 1.3        # the spike is at P/n ~ 1

def test_large_ridge_suppresses_peak():
    _, sharp  = double_descent_curve(ridge=1e-6)
    _, smooth = double_descent_curve(ridge=1.0)
    assert max(smooth) < max(sharp)          # regularization removes the spike
Tests that encode science. This is the deepest idea in the capstone. Ordinary tests check “does the function return the right type?” These tests assert “does the phenomenon still appear, in the right place, for the right reason?” The second test is a controlled ablation (Track 9) frozen into CI — it verifies that removing the mechanism (via regularization) removes the effect. A reproduction you can’t re-verify automatically is one bug away from quietly becoming wrong.

📐 Deploying it

The same five-step pipeline from Capstone A applies: test locally, push to GitHub (CI runs the science tests), then deploy. Because this app renders plots server-side with matplotlib, the only difference is that matplotlib is in requirements.txt. On Render set build pip install -r requirements.txt and start gunicorn app:app --bind 0.0.0.0:$PORT; on Hugging Face Spaces push the included Docker image. Full steps are in the project README.md.

🔑 Key takeaways

  • A reproduction can be a product: wrap the experiment in an API and a UI so others can explore it interactively.
  • Parameterize the science (noise, ridge, size) so the same function powers both the tests and the live app.
  • Tests can encode claims: assert the peak is at \(P/n\approx1\) and that regularization removes it — a controlled ablation in CI.
  • Server-side rendering keeps the client trivial: the browser just displays the returned plot image.
  • Reproducible + interactive + tested is a far stronger artifact than a one-off figure in a notebook.

Q1. The test test_large_ridge_suppresses_peak asserts the spike shrinks with strong regularization. Why is this test especially valuable?

  • It checks the code returns a list.
    That’s a trivial type check; this test does something deeper — it verifies the mechanism.
  • It freezes a controlled ablation into CI: removing the mechanism (allowing regularization to prevent exact interpolation) removes the effect.
    Correct. It encodes a causal claim — the spike is caused by unregularized interpolation — and fails if that stops holding. It’s Track 9’s ablation logic, automated.
  • It makes the app run faster.
    Tests don’t affect runtime performance of the app; they guard correctness.
  • It trains a larger model.
    No training of a large model occurs; it compares two curve computations.

Q2. Why does the same double_descent_curve function serve both the tests and the web app?

  • To make the code longer.
    It actually keeps code shorter and consistent — one source of truth.
  • Because Flask requires it.
    Flask imposes no such requirement; this is a design choice for correctness.
  • So the exact computation users see is the one the tests verify — no drift between “what’s tested” and “what’s served.”
    Correct. Sharing one parameterized function means CI validates the very code path the app runs, so a passing build guarantees the live result is the verified one.
  • To avoid using NumPy.
    The function uses NumPy throughout; that’s unrelated.

Q3. The app renders the curve into a PNG on the server and sends the image. What’s the main benefit versus sending raw numbers?

  • The browser stays trivial — it just displays an image, with no plotting code to maintain.
    Correct. Server-side rendering moves the plotting (matplotlib) to where the science already runs, so the front end is a few lines that set an <img> source — simple and consistent with the notebooks’ figures.
  • It hides the result from the user.
    It shows the result as a plot; nothing is hidden.
  • It makes the science more accurate.
    Accuracy comes from the computation, not from where the plot is drawn.
  • It removes the need for a server.
    The server is doing the rendering — it’s very much needed.

✍️ Practice & extensions

1. Run the app and sweep the ridge slider from tiny to large. Describe what happens to the interpolation spike and connect it to the mechanism.

Show solution

As ridge increases, the sharp peak at \(P/n=1\) flattens and eventually disappears. The spike is caused by the model fitting the training data exactly at the threshold, where the single interpolating solution is wildly contorted (huge variance). Regularization penalizes large weights, so the model no longer interpolates exactly — it trades a little training fit for a much smoother solution, and the variance explosion never happens. Watching this live is more convincing than reading it: you’re manipulating the cause and seeing the effect respond, which is exactly what the test_large_ridge_suppresses_peak test asserts numerically.

2. Add a third slider for the number of training points \(n\). Predict what happens to the location of the spike, then implement and check.

Show solution

The spike marks where parameters equal data points, \(P=n\). The x-axis is already \(P/n\), so in those units the spike should stay at \(P/n=1\) regardless of \(n\) — what changes is the absolute number of features at which it occurs, and the noise level of the curve (more data → smoother, often a less violent peak). Implementing it means threading an n parameter from the slider through double_descent_curve (it’s already an argument) and into the API handler. Add a test asserting the spike stays near \(P/n=1\) as \(n\) varies — extending the science-encoding test suite, which is the spirit of the capstone.

3. Conceptual: your app recomputes the curve on every slider change, which takes a moment. How would you make it feel instant, without faking the science?

Show solution

Several honest options: cache results by (noise, ridge) so repeated settings are instant; precompute a grid of curves offline and interpolate/display the nearest; reduce the repeats for the interactive view while keeping the high-fidelity version for the tests; or compute asynchronously and show a quick low-repeat preview that sharpens. Each preserves the real computation — none fabricate the curve. This is the same train-offline/serve-online and caching thinking from Capstone A’s scaling discussion: make the expensive part happen rarely, and serve cached or lighter results for interactivity.
