Capstone A: ship a from-scratch GPT

Everything you built, deployed. Take the character-level Transformer from Track 5, train it to invent names, wrap it in a real web service, and put it online — a complete, professional, real-time app you can push to GitHub today.

Track 10 · Capstone A Project + deployment ~40 min Builds on: Tracks 4, 5

A capstone is where knowledge becomes a portfolio. This one turns the transformer you implemented by hand into MiniGPT Name Generator — a live web app that invents new names character by character. The full project is in capstones/minigpt-namegen/: model, server, tests, Docker, CI, and one-click deploy. Nothing here uses PyTorch; the model is the ~120 lines of NumPy you already understand.

💡 What you’re building

The app has three layers, each something you can read end to end: a model (the Track 5 Transformer, packaged with train/save/load/sample), a server (a small Flask API), and a browser UI (vanilla JavaScript that calls the API and renders names). A user types a prefix and a “creativity” setting; the model samples names in real time.

🛠️ The model: a Transformer that invents names

The model is the single-head causal Transformer from Track 5, trained on a list of names separated by newlines. Generation starts at the newline boundary and samples characters — feeding each back as context — until the next newline. That accumulated string is a brand-new name. The sampling core, from minigpt/model.py: 

def sample_name(m, stoi, itos, temperature=0.9, prefix=""):
    ids = [stoi["\n"]] + [stoi[c] for c in prefix]      # start at boundary
    out = list(prefix)
    for _ in range(max_len):
        logits, _, _ = m.forward(np.array(ids[-m.T:]))   # look at recent context
        p = softmax(logits[-1] / temperature)            # temperature = creativity
        nxt = rng.choice(len(p), p=p)                    # sample next character
        if nxt == stoi["\n"]: break                      # word finished
        ids.append(nxt); out.append(itos[nxt])
    return "".join(out).capitalize()

The temperature exposes the sampling trick from Track 6 as a user control: dividing the logits by a small number sharpens the distribution (safe, common names); a larger number flattens it (wild, novel names). Trained on 161 names for a few thousand steps, it produces plausible inventions like Alexan, Mica, and Casana — and conditioning on a prefix like “al” yields names that start that way.

🛠️ The server: a real API

A Flask app loads the trained weights once at startup and exposes a clean endpoint. The model is trained ahead of time (python -m minigpt.train writes model.npz), or the server trains it on first boot if missing: 

@app.route("/api/generate")
def generate():
    count = clamp(request.args.get("count", 6), 1, 20)
    temp  = clamp(request.args.get("temperature", 0.9), 0.1, 1.5)
    prefix = only_letters(request.args.get("prefix", ""))
    names = [sample_name(MODEL, STOI, ITOS, temperature=temp, prefix=prefix)
             for _ in range(count)]
    return jsonify(names=names)

That is a production-shaped API: input validation, sensible limits, JSON out. A /health endpoint reports the model dimensions for liveness checks. The whole repo ships with a pytest suite (forward-pass shapes, training reduces loss, save/load round-trips, generation is clean) that runs in under a second — and a GitHub Actions workflow that runs those tests on every push.

📐 Deploying it: laptop to public URL

Concretely, after git push to GitHub you have two easy hosting paths. On Render: create a Web Service from the repo, set the build command to pip install -r requirements.txt && python -m minigpt.train and the start command to gunicorn app:app --bind 0.0.0.0:$PORT. On Hugging Face Spaces: create a Docker Space and push — the included Dockerfile trains the model at build time and serves it. Either way you get a shareable URL in minutes. The full step-by-step is in the project’s README.md.

🧩 Check your understanding
Answer from memory, then read why.

Q1. In sample_name, what does raising the temperature do?

  • Speeds up generation.
    Temperature reshapes the probability distribution, not the compute time.
  • Forces the model to output real dictionary names.
    There’s no dictionary lookup; the model only samples characters from its learned distribution.
  • Flattens the next-character distribution, yielding more varied/novel (and riskier) names.
    Correct. Dividing logits by a larger temperature makes the softmax more uniform, so lower-probability characters get chosen more often — more creativity, more nonsense. Lower temperature sharpens toward the most likely, safer names.
  • Changes the model’s weights.
    Weights are fixed after training; temperature only affects sampling at generation time.

Q2. Why does the app train the model ahead of time (or at container build), rather than on every request?

  • So each request is fast — training once amortizes the cost; requests only run quick forward passes.
    Correct. Training is the expensive step; doing it once and loading model.npz means each /api/generate call is just a few forward passes — milliseconds, suitable for real-time use.
  • Because training can’t be done in Python.
    Training is pure Python/NumPy here; that’s not the reason.
  • Because Flask forbids loops.
    Flask has no such restriction; the reason is latency and efficiency.
  • To avoid using the test set.
    Unrelated — this is about serving latency, not evaluation hygiene.

Q3. What is the purpose of the GitHub Actions CI workflow in this repo?

  • To train a bigger model in the cloud.
    CI here runs the test suite; it isn’t a training cluster.
  • To deploy automatically to production.
    That would be CD; this workflow runs tests. Deployment is handled by the host.
  • To format the code.
    Formatting could be a CI step, but this workflow’s job is running pytest.
  • To run the test suite on every push, so a broken change is caught before it ships.
    Correct. CI re-runs the unit tests automatically on each push/PR, giving you a green check that the model, server, and save/load all still work — the safety net that makes frequent deployment safe.

Q4. A recruiter asks how your name generator works. Which answer best reflects what this capstone gives you?

  • “I called an API from a library and it worked.”
    That’s the opposite of this capstone — here nothing is a black box; you wrote the model.
  • “A character-level Transformer I implemented in NumPy samples names one character at a time; I serve it with Flask and deploy it via Docker.”
    Correct. Because every layer is yours, you can explain the model (attention, sampling), the serving (Flask API), and the deployment (Docker/CI) — exactly the end-to-end command a capstone is meant to demonstrate.
  • “It uses magic.”
    The whole point of building from scratch is that there’s no magic left — you understand each step.
  • “I’m not sure, it’s a big neural network.”
    This capstone’s model is small and fully transparent; uncertainty here would mean the build didn’t achieve its goal.

✍️ Practice & extensions

1. Run the project locally (python -m minigpt.train then python app.py) and deploy it to a free host. Swap data/names.txt for a different corpus and redeploy.

Show solution

After training, the app serves at localhost:8000. To deploy, push to GitHub and create a Render Web Service (build: pip install -r requirements.txt && python -m minigpt.train; start: gunicorn app:app --bind 0.0.0.0:$PORT) — you’ll have a public URL in a couple of minutes. Swapping the corpus is the fun part: replace the names with Pokémon, startup names, or chemical compounds (anything line-delimited), retrain, and the model invents in that new style. This demonstrates that the system is general; only the data changes. Mention the live link and the repo in your portfolio.

2. Add a second attention head or a feed-forward block (Track 5, Lesson 2) to the model. Does sample quality improve? What’s the cost?

Show solution

More capacity (a second head, or an MLP block after attention) generally lets the model capture richer letter patterns, improving sample quality on a larger corpus — at the cost of more parameters, slower training, and a bigger model.npz. The honest way to claim improvement is the Track 9 method: hold the corpus and training budget fixed, train both versions across a few seeds, and compare a quantitative metric (e.g. validation perplexity, or the fraction of generated names that are pronounceable) with error bars — not a cherry-picked sample. This connects the capstone back to rigorous evaluation: even a fun project deserves an honest comparison.

3. Conceptual: the production model is trained at container build time. What changes if the corpus were gigabytes instead of kilobytes?

Show solution

Training at build time only works because this model is tiny and trains in seconds. With a large corpus and model, you’d separate the stages: train offline (a one-time or scheduled job, perhaps on a GPU), save the weights to durable storage (an artifact registry, S3, or the HF Hub), and have the server load those weights at startup rather than retraining. The container build would just install dependencies and fetch the artifact. This train-offline / serve-online split is standard MLOps, and the boundary is already in the code — model.npz is the artifact passed from training to serving. Scaling the capstone is mostly about where that artifact lives and how it’s produced.
