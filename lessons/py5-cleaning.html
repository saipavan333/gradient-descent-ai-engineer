Data cleaning & preparation

⏱ 26 min Lesson 2.5 ⬇ Runnable notebook
Why this existsModels are only as good as their data, and raw data is full of missing values, text categories, and unscaled numbers. One mistake here — data leakage — silently inflates scores and ruins models in production.
By the end, you can
  • Handle missing values by dropping or imputing
  • Encode categoricals and scale numerics
  • Split train/test and fit preprocessing on train only
  • Recognize and prevent data leakage

Concept Garbage in, garbage out

A model is only as good as the data you feed it, and raw data is almost never ready to use: it has missing entries, values stored as the wrong type, text categories a model can't read, and numbers on wildly different scales. Data cleaning and preparation is the work of turning that mess into a clean table of numbers a model can learn from. It's unglamorous, it's where you'll spend much of your time, and one specific mistake — data leakage, which we'll meet at the end — silently wrecks more models than any modeling error. We'll build the right habits from the start.

Concept Missing values

Real datasets have holes — a sensor failed, a form field was left blank — shown in pandas as NaN (short for "Not a Number," the marker for a missing value). You have two basic choices: drop the affected rows or columns, or impute them (fill the gaps with a sensible value such as the column's average, its median, or a model's guess). ("Impute" simply means "fill in a missing value with an estimate.")

Dropping is fine when missingness is rare and random; imputing preserves data when it isn't. The key question is always why a value is missing — random noise, or a meaningful pattern (for example, "income left blank" might itself correlate with the thing you're predicting).

Concept Encoding categories into numbers

Models do arithmetic, so text categories ("red," "Tuesday," "gold tier") have to become numbers. The default tool is one-hot encoding: turn one text column into several 0/1 columns, one per category, with a single 1 marking the value present in that row.

i
Why not just number the categories 1, 2, 3?

Labeling red=1, green=2, blue=3 implies an order and a magnitude — that "blue is three times red," and that green sits "between" them. All false, and it misleads the model. One-hot avoids this by giving each category its own independent column. The exception is genuinely ordinal data — categories with a real order, like small < medium < large — where a single ordered number is appropriate.

Concept Scaling numeric features

Features on different scales (age in years vs. income in dollars) can dominate distance- and gradient-based models — remember from Lesson 1.1 how raw distance was swamped by the largest feature. Scaling fixes this. Standardization (subtract the mean, divide by the standard deviation — the broadcasting trick from Lesson 2.3) makes every feature have average 0 and spread 1. Min–max scaling squeezes every value into the range [0, 1]. Many models (linear, SVM, neural nets, k-means) train better and faster on scaled inputs.

Concept The split, and the leakage that ruins everything

To estimate how a model will do in the real world, you hold out a test set — a chunk of data the model never trains on, used only at the very end to check performance. But here's the trap: any preparation step that learns something from the data — a scaler learning the mean, an encoder learning the categories, an imputer learning the fill value — must learn it from the training set only, then merely apply it to the test set. If you instead learn from all the data first, information from the test set "leaks" into training, and your scores become over-optimistic fantasies that collapse in production. This mistake is called data leakage.

Worked example The right way to scale: fit on train only

We split the data first, then "fit" the scaler (let it learn the mean and spread) on the training data, and apply those same learned numbers to the test data. This is the pattern for every preparation step.

import numpy as np
from sklearn.model_selection import train_test_split
from sklearn.preprocessing import StandardScaler

X = np.array([[1800.], [950.], [2400.], [1500.], [1200.], [3000.]])  # one feature: house size
y = np.array([350, 210, 560, 300, 250, 700])                          # price ($k)

# split into a training set and a held-out test set
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.33, random_state=0)

scaler = StandardScaler()
scaler.fit(X_train)                      # LEARN mean & spread from the TRAIN set only
X_train_s = scaler.transform(X_train)    # apply them to train
X_test_s  = scaler.transform(X_test)     # apply the SAME numbers to test

print("train mean used:", scaler.mean_.round(1), " std:", scaler.scale_.round(1))
print("test (scaled):", X_test_s.ravel().round(2))
Output (the split depends on random_state)train mean used: [1362.5] std: [318.9] test (scaled): [5.13 3.25]

Notice the test data was scaled using the training mean and spread — the test set's own statistics were never peeked at. (Here the two test houses happen to be large, so their scaled values come out big and positive — a reminder that on tiny data a random split can be lopsided.) Contrast the leaky version, which learns from everything:

# WRONG -- leakage: the scaler sees test data when computing the mean/spread
bad = StandardScaler().fit(X)            # fit on ALL the data
# now the test rows influenced the scaling used in training -> optimistic, invalid scores

The fix in practice is a Pipeline, which bundles the preparation steps and the model together so the "fit on train only" rule is enforced for you, automatically:

from sklearn.pipeline import make_pipeline
from sklearn.linear_model import LinearRegression
pipe = make_pipeline(StandardScaler(), LinearRegression())
pipe.fit(X_train, y_train)               # the scaler fits on train, inside the pipeline
print("R^2 on test:", round(pipe.score(X_test, y_test), 3))

Pipelines make the correct thing the easy thing — which is why they're the professional default, and why they matter even more once cross-validation (Track 3) repeats the split many times.

★ Key points to remember
  • Clean data first: handle missing values (drop or impute), encode categories, scale numbers.
  • One-hot encode plain categories (no false ordering); use a single ordered number only for truly ordinal data.
  • Standardize (mean 0, spread 1) or min–max scale so no feature dominates by its units.
  • Data leakage: fit any step that learns from data (scaler, encoder, imputer) on the training set only, then apply to test.
  • Use a Pipeline to enforce fit-on-train-only automatically.

Practice Build the reasoning yourself

1A categorical column "city" has values {NYC, LA, SF}. After one-hot encoding, how many new columns are created, and what does the row for an "LA" record look like?
Show solution & reasoning

Three new 0/1 columns: city_NYC, city_LA, city_SF. An "LA" record is [0, 1, 0] — a single 1 in the LA column, 0 elsewhere.

The reasoning: one column per category, exactly one "hot" (1) per row. (As an optimization, some workflows drop one column — the "dummy variable trap" — since the last category is implied when the others are all 0; this avoids the perfect-redundancy problem from Lesson 1.4 in linear models.)

2Spot the leak: to predict whether a customer will churn (leave) next month, your features include "account_closed_date." Why is this a problem?
Show solution & reasoning

It's target leakage: account_closed_date exists only because the customer churned — it's information from the future, basically the answer itself. The model will look brilliant in testing (it's essentially reading the label) but be useless in production, where that field isn't known at the time you need to predict.

The reasoning: leakage isn't only a preparation-order issue; it's also features that wouldn't be available at the moment of prediction. The fix: drop any feature that's a consequence of, or recorded after, the event you're predicting. Always ask "would I actually know this at prediction time?"

3You impute missing ages with the mean. Where should that mean be computed — over the whole dataset, or just the training set — and why?
Show solution & reasoning

Just the training set. The mean is a number learned from data, so computing it over the whole dataset lets test-set ages influence the fill value used during training — leakage, exactly like the scaler example.

The reasoning: the rule generalizes — anything a preparation step learns (a mean, a median, the list of categories, scaling factors) must be learned from train and merely applied to test. Bundle the imputer into a Pipeline and this happens correctly even across cross-validation folds, with no chance to forget.

★ Deep dive: the many faces of leakage (temporal, group, and target)▸

Leakage is subtle because it wears many disguises beyond the fit-on-all-data version. The best engineers develop a reflexive suspicion of "too-good" results, because near-perfect scores usually mean a leak, not a breakthrough.

  • Target leakage (practice 2): a feature that encodes the answer or is created after the event — account_closed_date, or "number of late-payment reminders" when predicting default. The model "cheats."
  • Temporal leakage: training on data from after the prediction moment. If you predict tomorrow's price using a feature that summarizes the whole month (including tomorrow), you've used the future. Time-series problems require splitting by time, not randomly.
  • Group leakage: the same entity appearing in both train and test. If one patient has 10 visits and some land in train and some in test, the model can memorize that patient rather than learn to generalize — inflating scores. The fix is to split by group (all of a patient's rows go to one side).
  • Preparation leakage (this lesson): fitting scalers/encoders/imputers — or doing feature selection — using the test set.

The unifying principle: the model, at training time, must only see information it would genuinely have at prediction time. Anything else is leakage, and it turns evaluation into self-deception. When a result looks suspiciously perfect, your first hypothesis should be a leak. This vigilance carries straight into Track 3's cross-validation (where leaking across folds is a classic mistake) and Track 11's responsible-AI evaluation, where trustworthy measurement is the entire game.
