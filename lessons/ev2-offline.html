Offline evaluation & datasets

⏱ 25 min Lesson 11.2 ⬇ Runnable notebook
Why this existsOffline evaluation measures quality on a fixed dataset before deployment. Split data into train/validation/test (never peek at test), and choose metrics by error cost — precision, recall, and F1 from the confusion matrix, not misleading accuracy.
By the end, you can
  • Split data and keep test unseen
  • Read the confusion matrix
  • Compute precision, recall, F1
  • Choose metrics by error cost

Concept Evaluate before you touch real users

Offline evaluation is measuring a model's quality on a fixed dataset, in the lab, before deploying it to real users. It's the workhorse of ML evaluation because it's fast, cheap, and repeatable — you can test a hundred model variants overnight without risking a single real user. The foundation is honest data splitting: you divide your labeled data into a training set (to fit the model), a validation set (to tune choices like hyperparameters and pick the best variant), and a test set (to judge final performance once). The iron rule is that the test set stays unseen until the very end — the moment you make decisions based on the test set, it's no longer an honest estimate of performance on new data, a subtle form of the data leakage from Track 3.

The second half of offline evaluation is choosing the right metric for the task, because (Lesson 11.1) the wrong metric certifies bad models. For classification, the key metrics come from the confusion matrix — a table of correct and incorrect predictions broken down by class — which lets you compute precision and recall instead of misleading raw accuracy.

Worked example Precision, recall, and F1

From a confusion matrix — the counts of each prediction outcome — the meaningful metrics follow directly. Here a classifier flagged some items, with 80 true positives, 30 false positives, 20 false negatives, and 870 true negatives:

TP, FP, FN, TN = 80, 30, 20, 870
precision = TP / (TP + FP)        # of what we flagged, how much was right?
recall    = TP / (TP + FN)        # of the real positives, how many did we catch?
f1 = 2 * precision * recall / (precision + recall)   # their harmonic mean
print(f"precision: {precision:.3f}")
print(f"recall   : {recall:.3f}")
print(f"F1       : {f1:.3f}")
Outputprecision: 0.727 recall : 0.800 F1 : 0.762

Each metric answers a different question. Precision (0.727): of the 110 items the model flagged (80 + 30), 80 were truly positive — so ~73% of its alarms were real. Recall (0.800): of the 100 actual positives (80 + 20), it caught 80 — so it found 80% of what it should have. These trade off: make the model stricter (flag only when very confident) and precision rises but recall falls (you miss more); make it looser and recall rises but precision falls (more false alarms). F1 (0.762) is their harmonic mean, a single number balancing both — useful when you care about precision and recall together. Notice that accuracy here would be (80+870)/1000 = 95%, which sounds great but hides that the model misses 20% of the positives — exactly why precision and recall are the honest metrics on imbalanced data. The design point: which metric you optimize depends on the costs. For fraud or disease detection, missing a positive is catastrophic, so you prioritize recall (catch them all, tolerate false alarms). For spam filtering, wrongly flagging a real email is annoying, so you prioritize precision (don't cry wolf). Choosing the metric that matches your error costs — rather than defaulting to accuracy — is the core skill of offline evaluation, and the confusion matrix is what makes those trade-offs visible.

★ Key points to remember
  • Offline evaluation measures quality on a fixed dataset before deployment — fast, cheap, repeatable.
  • Split data into train (fit), validation (tune/select), test (judge once); the test set must stay unseen.
  • Peeking at the test set (tuning to it, or picking the best of many peeks) leaks information and inflates scores.
  • From the confusion matrix: precision = TP/(TP+FP), recall = TP/(TP+FN); they trade off, F1 balances them.
  • Choose the metric by error costs: recall when misses are catastrophic (fraud), precision when false alarms are costly (spam).

Practice Build the reasoning yourself

1A colleague reports their model improved from 0.90 to 0.94 F1. Then you learn they tried 200 model variants and reported the best one's score on the test set. What's wrong, and how should they have done it?
Show solution & reasoning

The problem is test-set overfitting through repeated evaluation. By trying 200 variants and reporting the best test-set score, they've effectively tuned to the test set — with 200 tries, some variant will score high on that particular test set partly by luck (it happened to fit its quirks), not because it's genuinely better. The reported 0.94 is optimistically biased and won't hold on truly new data. This is a subtle form of data leakage: the test set was supposed to be touched once, but selecting the best of 200 peeks quietly leaked test information into the model selection. It's the multiple-comparisons problem — evaluate enough things and something looks good by chance.

The correct procedure keeps the test set pristine: use the validation set for all the variant selection — try the 200 variants, compare them on validation, pick the best — and only then evaluate that single chosen model on the test set, once, to get an honest estimate. The validation set can be "used up" by repeated selection (that's its job), which is exactly why it's separate from the test set. For even more robustness, use cross-validation (rotate which fold is validation) for selection, and consider a fresh held-out set for the final number. The reasoning skill is recognizing that the test set's honesty is a one-time resource — every decision informed by it spends some of that honesty — so all the searching and tuning must happen on validation data, with the test set reserved for a single final verdict. A 0.94 that came from picking the best of 200 test-set peeks is not a real 0.94, and understanding why (selection leakage inflates it) is fundamental to trustworthy offline evaluation.

2You're evaluating a model on a rare-disease dataset that's 99% healthy, 1% sick. Design an offline evaluation that won't be fooled.
Show solution & reasoning

The design must be built around the class imbalance, since accuracy is worthless here (99% by predicting "healthy" always). Metrics: report precision and recall on the "sick" class specifically — recall is likely the priority (missing a sick patient is the catastrophic error), with precision tracked so false alarms stay manageable, and F1 or a precision-recall curve to see the trade-off across thresholds. I'd use the PR curve and average precision rather than ROC/AUC, because ROC can look deceptively good on severe imbalance while PR curves better reflect performance on the rare positive class. Confusion matrix: examine the actual TP/FP/FN/TN counts, not just summary numbers, to see exactly how many sick patients are missed. Threshold: choose the decision threshold deliberately based on the cost trade-off (how many false alarms are acceptable per catch), not the default 0.5.

Data considerations: ensure the test set has enough sick cases to estimate recall reliably (with 1% prevalence, a small test set might have only a handful of positives, making recall estimates noisy — so I'd want many positive examples, possibly stratified sampling to guarantee coverage). I'd also evaluate on subgroups (does recall hold across ages, sexes, sites?) to catch hidden failures, and check for leakage (no patient appears in both train and test). The reasoning is that imbalanced, high-stakes evaluation demands class-specific metrics, careful thresholding by error cost, enough positive examples for stable estimates, and subgroup checks — everything that raw accuracy on a random split would paper over. Designing the evaluation around the rare class that matters, rather than around convenient aggregate metrics, is what makes it trustworthy for a life-or-death task.

3Offline evaluation says your new model is clearly better, so you ship it — and real-world performance is worse. List the reasons offline results can fail to transfer to production.
Show solution & reasoning

Several gaps can make an offline win fail in production. Distribution shift: the test set doesn't match live data — it's stale, or sampled differently, so real inputs differ from what you evaluated on (and will drift further over time, Track 9). Train/serve skew: features are computed differently in production than in your offline pipeline (Lesson 9.3), so the deployed model sees different inputs than the evaluated one. The metric doesn't capture the real goal: you optimized offline accuracy, but real success is user engagement or task completion, which can move the opposite way (Lesson 11.1's proxy gap — and the offline/online gap of the next lesson). Test-set leakage/overfitting: the offline number was inflated (as in practice item 1), so "better offline" was partly illusory. Feedback effects: the model changes user behavior in ways static offline data can't show (a recommender creating filter bubbles). Latency/cost constraints: the "better" model is too slow or expensive to serve well, degrading the real experience even if its predictions are more accurate.

The unifying lesson is that offline evaluation is a proxy for real-world performance, and the proxy can be wrong — it measures accuracy on a fixed past dataset, while production is a living, shifting environment where the metric, the data, and user behavior all differ from the lab. This is precisely why offline evaluation, however rigorous, is necessary but not sufficient, and why it must be paired with online evaluation on real users (the next lesson) to confirm that offline gains actually translate. A mature practitioner treats a strong offline result as a hypothesis — "this should be better" — to be validated online, not as a final verdict. Recognizing the specific ways offline and online diverge (shift, skew, wrong metric, leakage, feedback, constraints) is what lets you anticipate and close those gaps, and it motivates the entire practice of online evaluation that comes next.

★ Deep dive: the full metric toolbox and building good evaluation datasets▸

Beyond precision/recall/F1, offline evaluation has a metric for every task shape, and choosing well is half the battle. For classification, the ROC curve and AUC summarize performance across all thresholds (though on severe imbalance, precision-recall curves are more informative). For ranking and retrieval (recommenders, search — Track 10), rank-aware metrics like NDCG, MRR, and recall@k reward putting the right items near the top. For regression, MAE and RMSE measure error magnitude (RMSE punishing large errors more). For generation, reference-overlap metrics like BLEU/ROUGE exist but correlate weakly with quality, which is why the field increasingly uses human evaluation and LLM-as-judge (Lesson 11.4). The meta-skill is matching the metric to what "good" means for your task and error costs — a mismatch (accuracy on imbalanced data, overlap metrics on open-ended text) is one of the most common evaluation mistakes, and it's a direct instance of Lesson 11.1's metric-goal gap.

Equally important, and often neglected, is the quality of the evaluation dataset itself — because your metrics are only as trustworthy as the data you compute them on. A good eval set is representative of the real distribution (including the hard and rare cases, not just easy majority examples), correctly labeled (label errors put a ceiling on measured accuracy and can make a better model look worse), large enough for statistically stable estimates (especially for rare classes), free of leakage from the training data, and sliced so you can evaluate subgroups separately (overall metrics hide failures on specific segments — a model can be 95% accurate overall yet fail badly for one demographic, which matters enormously for fairness, Lesson 11.5). Building and maintaining such datasets is real, ongoing work — curating examples, getting reliable labels (often multiple annotators with agreement checks), refreshing as the world changes, and deliberately including adversarial and edge cases. The professional reality is that a team's evaluation set is one of its most valuable assets: it's what lets every change be measured, it encodes the team's definition of "good," and its quality bounds the quality of everything built against it. Teams that invest in rigorous, representative, well-sliced eval data can improve steadily and catch regressions; teams with sloppy eval data optimize confidently in the wrong direction. "You ship your evaluation, not just your model" is the mantra — and it's why this track treats evaluation as a first-class engineering discipline, with online testing next to close the gap offline data can't.
