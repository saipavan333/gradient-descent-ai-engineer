Capstone 0: project setup & reproducibility

Concept Build a portfolio, prove you can ship

You've learned the concepts; the capstones are where you prove it by building. This track is a set of end-to-end projects — an image classifier, a fine-tuned LLM, a RAG assistant, an agent, a served API, a full pipeline — that together form a portfolio demonstrating you can build real, working AI systems, not just discuss them. In interviews (Track 12), your projects are often the strongest signal, because a project you built and can deep-dive proves applied judgment that no memorized answer can. This capstone-zero lays the foundation every other project rests on: project setup and reproducibility — the disciplined starting point that separates a professional project from a throwaway script.

Reproducibility is the ability to run your project again — or have someone else run it — and get the same result. It sounds basic, but it's where amateur projects fail: the "model that worked last week" that no one can rebuild, the notebook that runs only on your laptop, the result you can't explain because you don't know what produced it. Getting reproducibility right from the start is both good engineering and a strong signal to anyone reviewing your work (including interviewers) that you build like a professional.

The recipe is to pin the five things a result depends on. Code: version-control everything in git, so the exact code is recoverable. Data: use a specific, versioned dataset (a fixed file, a data-version tag) so you know exactly what the model trained on. Environment: pin library versions in a requirements.txt lockfile (or a container), since a different NumPy or library version can change results. Seed: fix the random seed so "random" initialization and shuffling repeat identically. Config: keep hyperparameters and settings in a config file (not scattered magic numbers), logged with each run. Pin all five and your project is reproducible; leave any floating and it may not be.

Worked example The seed: making "random" repeatable

The most commonly-forgotten reproducibility determinant is the random seed. Setting it makes the random-number generator deterministic, so anything "random" — weight initialization, data shuffling — comes out identically every run:

import numpy as np
def run():
    np.random.seed(42)              # fix the seed before any randomness
    return np.random.rand(3).round(4).tolist()

print("run 1:", run())
print("run 2:", run())
print("identical:", run() == run())
Outputrun 1: [0.3745, 0.9507, 0.732] run 2: [0.3745, 0.9507, 0.732] identical: True

Both runs produce the exact same "random" numbers, because seed(42) makes the generator deterministic. Without the seed, each run would differ — and so would your model's initialization, its data shuffling, and therefore its final weights and results, making the run impossible to reproduce. In a real project you'd seed every source of randomness (NumPy, the deep-learning framework, Python's random) at the start, and log the seed with the run. This tiny discipline is the difference between "I got 92% once and can't get it again" and "here's exactly how to reproduce 92%." It generalizes to the other four determinants: the same instinct — fix everything the result depends on and record it — applies to versioning your code (git commit), your data (a version tag), your environment (a lockfile), and your config (a settings file logged per run). A project that does all five can be rebuilt, debugged, shared, and trusted; one that does none is a fragile artifact that works until it mysteriously doesn't. Starting every capstone with this foundation is what makes the rest of your work solid — and it's exactly the professionalism that impresses anyone who reviews your projects.

★ Key points to remember
  • The capstones build a portfolio that proves you can ship real AI systems — often your strongest interview signal.
  • Reproducibility = getting the same result again; it's where amateur projects fail and where professionalism shows.
  • Pin all five: code (git), data (version), environment (lockfile/container), seed, config.
  • Fix the random seed so initialization and shuffling repeat identically — the most-forgotten determinant.
  • Keep config (hyperparameters, settings) in a file logged per run, not as scattered magic numbers.

Practice Set up your capstone

1Outline the folder structure and setup you'd create at the start of any capstone project, and why each piece matters.
Show a strong setup

A clean project scaffold makes everything downstream easier. Version control: a git repo from commit one, with a .gitignore (exclude data, model files, secrets). Structure: separate directories — a data/ (with a note on how to obtain/version the data, not the raw data in git), src/ or notebooks/ for code, models/ for saved artifacts, and a configs/ for settings. Environment: a requirements.txt (or environment file) pinning exact library versions, so anyone can recreate the environment; ideally a container for full reproducibility. Config: a config file (YAML/JSON) holding hyperparameters and paths, so runs are parameterized and logged rather than hardcoded. Seed: a single place that sets all random seeds at startup. README: explaining what the project does, how to set it up, and how to run it — the first thing a reviewer reads.

Each piece pays off: git makes work recoverable and collaborative; the structure keeps a growing project navigable; pinned dependencies prevent "works on my machine"; the config makes experiments systematic and reproducible; the seed makes results repeatable; and the README makes the project understandable to others (including your future self and interviewers). The reasoning is that a project's foundation determines how far it can go — a well-scaffolded project is easy to extend, debug, share, and present, while a pile of ad-hoc scripts becomes unmaintainable and unreproducible fast. Setting this up takes twenty minutes and saves hours later, and it's exactly the professional discipline that signals, to anyone reviewing your portfolio, that you build real systems, not one-off hacks. Every capstone in this track assumes this foundation, and building it as a habit is one of the most transferable things you'll take into a job.

2Why is experiment tracking (Track 9) especially valuable across a portfolio of capstone projects?
Show reasoning

Across multiple projects, each involving many experiments, experiment tracking (logging each run's parameters, metrics, and artifacts — Lesson 9.2) becomes the memory that makes your work cumulative rather than chaotic. Within a project, it lets you compare runs and find the best configuration systematically instead of guessing which of thirty variants was good — turning "I think lr=0.01 worked?" into a queryable record. Across projects, it builds a habit and a body of documented, reproducible results you can point to. And critically for a portfolio, tracked experiments let you tell a rigorous story: "I tried these approaches, here's how they compared, here's why I chose the winner" — which is exactly the decision-and-trade-off narrative that makes a project deep-dive compelling in an interview (Lesson 12.7).

The deeper value is that tracking embodies the reproducibility discipline in action: it captures, at the moment each run happens, the parameters and results you'd otherwise forget, so any result is recoverable and comparable. It also enforces good habits — once you're logging runs, you naturally version data, fix seeds, and parameterize configs, because tracking only means something if the runs are reproducible. For capstones specifically, using a tracking tool (even a simple one) demonstrates production-grade practice to reviewers and gives you the material to discuss your projects rigorously. The meta-point is that the reproducibility foundation of this capstone and the experiment-tracking discipline of Track 9 are two sides of the same professionalism: capturing everything a result depends on, at the time it happens, so your work is trustworthy, comparable, and presentable. Building that habit across your portfolio is what turns a collection of projects into evidence of how you actually work.

3How should you think about choosing which capstone projects to build and how to present them for a portfolio?
Show reasoning

Choose projects that demonstrate breadth and depth in the areas you're targeting, and that you can discuss deeply. The capstones in this track are deliberately chosen to span the field — a classic supervised-learning project (image classifier), modern LLM work (fine-tuning, RAG, agents), and production skills (a served API, a full pipeline) — so together they show you can work across the stack, from a model to a deployed system. For your own portfolio, I'd build a subset that matches your target roles (an LLM-focused role → emphasize the RAG, fine-tuning, and agent projects; an ML-engineering role → emphasize the serving and pipeline projects) while keeping at least one that shows end-to-end production capability, since that's rarer and more impressive than another model notebook. Depth beats quantity: two projects you built thoughtfully and can deep-dive are worth more than five shallow clones of a tutorial.

For presentation, treat each project as a story with the professional foundation visible: a clear README (problem, approach, results, how to run), reproducible code (the five determinants), honest results (including limitations and what you'd improve — which reads as mature, not weak), and ideally a short write-up of the key decisions and trade-offs you made and why. The trade-off narrative is what elevates a project from "I followed a tutorial" to "I made engineering judgments" — exactly what interviewers probe in a deep-dive (Lesson 12.7). Where possible, make it runnable or show it working (a demo, results, a deployed endpoint), since evidence it actually works is powerful. The reasoning is that a portfolio isn't judged by how many models you trained but by what it reveals about how you think and build — so choose projects that let you demonstrate real judgment, build them on a reproducible foundation, and present them as the decision-rich stories they are. That's how the capstones become not just learning exercises but the concrete proof of capability that gets you hired.

★ Deep dive: from tutorials to real projects — the mindset shift▸

The capstones mark a transition that's central to becoming an engineer: moving from following tutorials to building real projects. A tutorial hands you a cleaned dataset, a defined problem, and a known answer, so you learn the mechanics without the hard parts. A real project starts from ambiguity — you define the problem, find and clean the data, make design decisions under uncertainty, hit obstacles the tutorial never mentioned, and have no answer key telling you if you're right. That gap is exactly where real learning and real skill live, and it's why building projects (even imperfect ones) teaches far more than passively completing tutorials: you're forced to make judgments, debug the undocumented, and take responsibility for the whole system. The reproducibility foundation of this capstone is the first piece of that professionalism — the recognition that a real project is something to be built solidly, recoverably, and trustworthily, not a script that happens to run once.

This mindset shift is also what the entire curriculum has been building toward. The earlier tracks gave you the knowledge — math, ML, deep learning, LLMs, RAG, agents, MLOps, system design, evaluation, and safety — and the interview track showed how to communicate it. The capstones are where it all becomes doing: assembling that knowledge into working systems, making the trade-offs the design track taught, applying the operational discipline of MLOps, evaluating rigorously as the safety track insisted, and building it all reproducibly. Each capstone is a small integration of many tracks — the RAG assistant uses embeddings, retrieval, LLMs, evaluation, and serving; the pipeline uses training, tracking, registries, deployment, and monitoring — so completing them is what turns separate knowledge into integrated capability. And that capability, demonstrated in a portfolio of real, reproducible, thoughtfully-built projects, is ultimately what makes you employable: not that you know the concepts (many do) but that you can build with them, make sound judgments, and ship systems that work. The habit to carry from this capstone into every project — and into your career — is to build like a professional from the first commit: reproducible, versioned, configured, tracked, and documented, so your work is solid, shareable, and trustworthy. That foundation is what makes everything you build on top of it real. Now let's build the first one.
