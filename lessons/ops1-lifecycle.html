Track 10 · MLOps & Serving

The ML lifecycle

⏱ 20 min Lesson 10.1 ▶ Open in Colab
Why this existsMLOps is the engineering of running ML in production — the 90% of work around the model. The lifecycle is a loop (data → train → deploy → monitor → retrain) because models decay as the world changes, unlike ordinary software.
By the end, you can
  • Define MLOps and the production gap
  • Describe the lifecycle loop
  • Explain why ML systems decay silently
  • See deployment as the middle, not the end

Concept A model is the easy part; running it is the job

Every track so far has been about building models — training them, evaluating them, prompting them. This track is about everything that happens around the model to make it actually useful in the real world. That discipline is called MLOps (Machine Learning Operations): the engineering practices for deploying, running, monitoring, and maintaining ML systems in production — meaning the live environment where real users depend on the system, as opposed to a notebook on your laptop. A famous truth of the field is that the trained model is often less than 10% of the work; the other 90% is data pipelines, serving infrastructure, monitoring, and the machinery that keeps it all running.

The reason is that a model in a notebook and a model in production are completely different animals. In a notebook you run it by hand, once, on data you already have, and if it crashes you just rerun the cell. In production it must serve predictions automatically, reliably, at scale, on data you've never seen, while you monitor it for failures and quality decay — and when something breaks at 3am, someone gets paged. MLOps is the set of tools and practices that closes that gap.

Concept The lifecycle is a loop, not a line

It's tempting to think of building an ML system as a straight line: get data, train a model, ship it, done. In reality it's a lifecycle — a repeating loop. You collect and clean data, train a model, evaluate it, deploy it to production, and monitor it. But monitoring is not the end: in production the model's quality slowly decays (the world changes, Lesson 10.8), problems surface, and new data arrives — all of which feed back into collecting fresh data and training a new version. The loop turns continuously for the entire life of the system.

This loop is the single most important mental model in MLOps, and it's what makes ML systems harder to operate than ordinary software. Traditional software does the same thing forever until you change the code. An ML system can silently get worse while the code stays identical, because its behavior depends on data — and data drifts. So the lifecycle never closes; "deploying the model" is the middle of the story, not the end. Every lesson in this track is one stage of this loop: tracking experiments (the train stage), pipelines and feature stores (the data stage), registries and serving (the deploy stage), monitoring and drift (the monitor stage), and CI/CD to automate the whole cycle.

Here's the lifecycle as five simple stages, run once. Each stage is a function; the output of one feeds the next, and the final monitor stage decides whether the deployed model is healthy or needs retraining — the feedback that closes the loop.

The stages run in order — data, train, evaluate, deploy, monitor — exactly the lifecycle loop. The monitor stage is the crucial one: it checks the live model's accuracy against a threshold and returns "healthy" or "RETRAIN". Here accuracy is 0.86, comfortably above 0.80, so the model is healthy. But imagine months pass and drift drags accuracy to 0.74 — monitor would return "RETRAIN", kicking off another turn of the loop (collect new data, train fraud-v2, and so on). That single feedback check is what separates a real ML system from a one-off script: it never assumes the model stays good, it keeps watching. Every tool in this track exists to make one of these five stages reliable and automatic — so the loop runs continuously without a human hand-running cells.

★ Key points to remember
  • MLOps is the engineering discipline of deploying, running, monitoring, and maintaining ML in production (the live system real users depend on).
  • The trained model is a small fraction of the work; pipelines, serving, and monitoring are the bulk of it.
  • The ML lifecycle is a loop: data → train → evaluate → deploy → monitor → back to data.
  • Unlike normal software, an ML system can decay silently while the code is unchanged — because its behavior depends on drifting data.
  • "Deploying the model" is the middle of the story; maintaining it through the loop is the real job.

Practice Build the reasoning yourself

1A data scientist says "the model is done — it hits 92% accuracy in my notebook, so we can ship it and move on." What's wrong with this view?
Show solution & reasoning

The "done" framing is the mistake. A 92% notebook accuracy is necessary but nowhere near sufficient for a production system, for two distinct reasons. First, the gap to production: the model still needs to be packaged, served reliably at scale, given a data pipeline that computes its features the same way in production as in the notebook, versioned, and wrapped in monitoring — that's the bulk of the engineering, and none of it is done. Second, and deeper, a model is never "done": once it's live, its accuracy will decay as the world changes (drift, Lesson 9.8), so it needs ongoing monitoring and periodic retraining for its entire life.

The right mental model is the lifecycle loop: shipping the model is the middle, not the end. "We can move on" is exactly the attitude that produces systems that work great at launch and silently rot three months later. The mature view is that you're not shipping a model, you're committing to operating one — which means building the pipelines, serving, versioning, and monitoring that keep it healthy as the loop turns. This reframing, from "build a model" to "run a system," is the core of what MLOps adds to everything in the earlier tracks.

2Map each of these tools to a stage of the lifecycle loop: a feature store, a model registry, a monitoring dashboard, an experiment tracker.
Show solution & reasoning

Each tool serves one stage of the loop. An experiment tracker (Lesson 10.2) supports the train stage: it logs every training run's parameters and metrics so you can compare experiments and reproduce the best one. A feature store (Lesson 10.3) supports the data stage: it computes and serves features consistently for both training and serving. A model registry (Lesson 10.4) supports the deploy stage: it versions trained models and manages their promotion to production. A monitoring dashboard (Lesson 10.7) supports the monitor stage: it watches the live system's health and the model's quality.

The point of the exercise is to see that MLOps isn't a random pile of tools — each one exists to make a specific stage of the lifecycle reliable and automatic. When you understand the loop, the entire MLOps toolchain organizes itself in your head: every tool is answering "how do I make this stage of the loop trustworthy at scale?" That's also how you'd reason about adopting tools on a real team — you don't add a feature store because it's trendy, you add it because your data stage has a consistency problem it solves.

3Why does the "monitor → retrain" feedback edge matter so much that it defines the whole discipline? Connect it to a concept from an earlier track.
Show solution & reasoning

The feedback edge matters because it encodes the defining fact about ML systems: they degrade on their own. Without that edge, you'd have ordinary software thinking — build it, ship it, and assume it keeps working. The edge forces the opposite assumption: the deployed model will get worse, so you must continuously watch it and refresh it. Everything distinctive about MLOps — drift detection, retraining pipelines, model versioning, continuous training — exists to serve this one edge.

It connects directly to generalization from Track 3: a model is trained on one data distribution and only performs well as long as the live data resembles it. In Track 3 we worried about generalizing from training data to a fixed test set; in production the "test set" is the never-ending stream of real-world data, and that stream shifts over time, so yesterday's well-generalizing model becomes today's poorly-generalizing one. The monitor → retrain loop is generalization extended across time: it continually checks whether the model still generalizes to current data and rebuilds it when it doesn't. Seeing MLOps as "keeping a model's generalization valid as the world moves" ties the operational discipline back to the statistical foundations — which is exactly the kind of connection that signals deep understanding.

★ Deep dive: why MLOps emerged, and how it differs from DevOps▸

MLOps grew out of DevOps — the older discipline of automating and streamlining the build-test-deploy cycle for traditional software — but it has to handle things DevOps never did. In classic software, the artifact you ship is code, and code is deterministic: the same code produces the same behavior forever. In ML, the behavior of the system is determined by three things that all change independently: code, data, and the trained model. This is the root of every extra complication. You have to version not just code but datasets and models. Your tests can't just check logic; they have to check data quality and model performance, which are statistical, not binary. And your deployed system can fail not by crashing but by quietly becoming less accurate — a failure mode that doesn't exist in ordinary software and that no amount of unit testing catches. The whole field is essentially DevOps stretched to cover data and models as first-class, versioned, testable, monitored artifacts.

The maturity of an organization's MLOps is often described in levels. At level 0, everything is manual: a data scientist trains a model in a notebook and hands it to engineers who deploy it by hand, with no automation and little monitoring — fine for a first model, but it doesn't scale and models go stale unnoticed. At higher levels, the training pipeline is automated (retraining can be triggered on a schedule or by drift), deployment is automated through CI/CD (Lesson 10.9), and monitoring continuously feeds back into retraining — approaching the fully automated lifecycle loop. Most real teams live somewhere in the middle and climb gradually, automating the stages that hurt most first. The practical wisdom is that you don't need maximum automation on day one; you need to understand the loop and automate it incrementally as your number of models and their importance grows. The cost of not doing so is the industry's most common ML failure: not a model that was never good, but a model that was good at launch and silently decayed because nobody built the loop to catch it. The rest of this track builds that loop, stage by stage — starting with how you keep the training stage organized and reproducible.
